## 1. Python concurrency для відео-стримінгу (GIL)

### L1 — Junior
**Аналогія — кухня з одним кухарем.** Python має **GIL (Global Interpreter Lock)** — «правило, що лише один кухар (потік) готує JS-код одночасно». Поки він **чекає** (вода кипить — це I/O), може перемкнутись на іншу справу. Але **рахувати** (кодувати відео — це CPU) одночасно двома потоками не може.

**Просте правило:**
- **Чекання (мережа, диск)** → `threading` / `asyncio` ✅
- **Обчислення (енкодинг, обробка кадрів)** → `multiprocessing` / код на C ✅

### L2 — Middle
```python
# I/O-bound (роздача стриму, мережа) → asyncio
import asyncio
async def serve_stream(client):
    async for chunk in source:
        await client.send(chunk)   # поки шлемо — обслуговуємо інших

# CPU-bound (енкодинг кадрів) → процеси (обхід GIL)
from concurrent.futures import ProcessPoolExecutor
with ProcessPoolExecutor() as ex:
    encoded = list(ex.map(encode_frame, frames))  # реальний паралелізм по ядрах
```

**Три моделі й коли що:**
| Модель | Для чого у відео | GIL |
|---|---|---|
| `asyncio` | роздача стриму, мережеві з'єднання, оркестрація | не обходить (I/O) |
| `threading` | I/O-bound + виклики C-бібліотек | не обходить |
| `multiprocessing` | паралельний енкодинг/транскодинг | обходить (свій інтерпретатор) |

**Gotcha (ключове для цієї ролі):** **C-бібліотеки відпускають GIL.** GStreamer, PyAV, OpenCV, numpy під час важкого обчислення в C **звільняють GIL** → потоки **можуть** паралелити цю роботу. Тому «Python повільний для відео» — міф: важке роблять C-бібліотеки, Python лише диригує.

### L3 — Senior
**Чому GIL існує:** reference counting CPython не атомарний → GIL захищає лічильники без дорогих замків на кожен об'єкт. Наслідок — потоки дають **конкурентність I/O**, але **не паралелізм CPU** для чистого Python.

**Архітектура відео-сервісу на Python (правильний підхід):**
```
asyncio event loop (диригент) — приймає з'єднання, керує потоками
        │
        ├─→ GStreamer/PyAV pipeline (C, відпускає GIL) — енкодинг
        ├─→ ProcessPool — паралельний транскодинг по ядрах
        └─→ async I/O — роздача сегментів клієнтам
```

**Трейдофи:**
- `asyncio` — десятки тисяч з'єднань на потік (роздача HLS-сегментів), але один блокуючий виклик вішає loop.
- `multiprocessing` — справжній паралелізм енкодингу, але дорога передача даних (відеокадри великі → pickle/shared memory; краще `SharedMemory` або zero-copy).
- **Делегування в C** — найважливіше: важку роботу (енкодинг) роблять GStreamer/FFmpeg/PyAV у нативному коді, Python лише керує pipeline. Тоді GIL не вузьке місце.

**Сучасне:** Python 3.13 free-threaded (no-GIL) може змінити баланс, але для відео головне лишається — **обчислення в C, Python як orchestration layer**.

> ⚠️ Поведінка вивільнення GIL залежить від бібліотеки (numpy/OpenCV/PyAV відпускають у важких операціях). Звіряй у docs конкретної бібліотеки.

## 2. Codec vs Container (H.264/H.265, MP4/TS/fMP4)

### L1 — Junior
**Аналогія — вино й пляшка.**
- **Кодек (codec)** — це саме **вино**: алгоритм **стиснення** відео/аудіо (як саме дані запаковані). Приклади: H.264, H.265.
- **Контейнер (container)** — це **пляшка з етикеткою**: обгортка, що тримає закодоване відео + аудіо + субтитри + метадані + синхронізацію. Приклади: MP4, MKV, TS.

**Один контейнер може містити різні кодеки.** MP4 може тримати H.264 або H.265.

### L2 — Middle
**Кодеки (стиснення):**
| Кодек | Особливість |
|---|---|
| **H.264 / AVC** | стандарт де-факто, всюди підтримується |
| **H.265 / HEVC** | ~50% менший розмір за тієї ж якості, але дорожче кодувати + ліцензії |
| **AV1** | royalty-free, ще краще стиснення, дуже важкий для енкодингу |
| **VP8 / VP9** | Google, для WebM/WebRTC |
| Аудіо: **AAC, Opus** | Opus — для WebRTC/low-latency |

**Контейнери (обгортка):**
| Контейнер | Для чого |
|---|---|
| **MP4** | VOD (записане відео), широка підтримка |
| **MPEG-TS** | **live/broadcast** — стійкий до втрат, можна під'єднатись посеред потоку |
| **fMP4** (fragmented MP4) | **low-latency live** (HLS/DASH, CMAF) — шматками |
| **WebM** | VP8/VP9 + Opus, для веб |
| **FLV** | legacy RTMP |

**Gotcha (важливо для live):** звичайний **MP4 погано підходить для live** — він тримає індекс (`moov atom`), який зазвичай **в кінці файлу**, тож потрібен повний файл. Для live беруть **MPEG-TS** (потоковий від народження) або **fMP4** (шматки, кожен самодостатній).

### L3 — Senior
**Чому розділення кодек/контейнер важливе архітектурно:** дозволяє **міняти одне, не чіпаючи інше**. Той самий H.264-потік можна перепакувати (**remux**, без перекодування) з TS у fMP4 для різних плеєрів — дешево (копіювання, не енкодинг). А **transcode** (зміна кодека/бітрейту) — дорого (повне перекодування).

**Вибір для live-стримінгу (трейдофи):**
- **MPEG-TS** — стійкий до втрат пакетів (кожен TS-пакет 188 байт самодостатній), декодер може під'єднатись на наступному ключовому кадрі. Класика broadcast/contribution. Overhead ~?% на TS-заголовки.
- **fMP4 / CMAF** — сучасний для HLS+DASH (один формат на обидва → менше дублювання сегментів на CDN). Підтримує low-latency. Витісняє TS.
- **H.264 vs H.265 трейдоф:** H.265 економить смугу (важливо для high-load/CDN-витрат), але дорожче кодувати (CPU/латентність), складніші ліцензії, гірша сумісність зі старими плеєрами. Для live часто H.264 (швидкість, сумісність), H.265/AV1 — де економія смуги критична.

**Ключове senior-розуміння:** **remux ≠ transcode**. Remux (зміна контейнера) — `O(копіювання)`, transcode (зміна кодека/якості) — `O(енкодинг)`, у сотні разів дорожче. Грамотний pipeline уникає зайвого transcode (кодуй раз → remux/repackage під різні протоколи).

> ⚠️ Конкретні цифри стиснення (H.265 «−50%») — орієнтовні, залежать від контенту/налаштувань. Підтримка кодеків браузерами/пристроями змінюється — звіряй матриці сумісності.

## 3. Протоколи стримінгу (RTMP / HLS / DASH / WebRTC)

### L1 — Junior
**Аналогія — способи доставки.**
- **RTMP** — «кур'єр від студії на склад» (ingest: енкодер → сервер).
- **HLS / DASH** — «розкласти по магазинах мережі» (масштабована роздача через CDN, але повільніша).
- **WebRTC** — «прямий дзвінок» (миттєво, але важко на мільйони).

Головна вісь — **затримка (latency) vs масштабованість**.

### L2 — Middle
| Протокол | Транспорт | Латентність | Призначення |
|---|---|---|---|
| **RTMP** | TCP | ~2–5с | **ingest** (енкодер → сервер) |
| **HLS** (Apple) | HTTP/TCP | 6–30с (LL-HLS: 2–5с) | масштабована роздача через CDN |
| **DASH** | HTTP/TCP | подібно HLS | роздача, codec-agnostic, open |
| **WebRTC** | UDP/RTP | **<500мс** | real-time (дзвінки, інтерактив) |
| **SRT / RIST** | UDP | низька | надійний contribution по поганих мережах |

**Як працює HLS (типовий потік):**
```
Енкодер → RTMP/SRT → Сервер ріже на сегменти (по 2-6с) + .m3u8 манифест
       → CDN кешує сегменти → Плеєр тягне манифест + сегменти по HTTP
```

**Gotcha:** HLS/DASH — це **HTTP-файли** (сегменти + манифест), тому масштабуються звичайним CDN-кешуванням (мільйони глядачів). Ціна — **висока латентність** (плеєр буферизує кілька сегментів). WebRTC навпаки — низька латентність, але CDN-кешування не працює (peer-з'єднання).

### L2.5 — типова повна архітектура
```
[Студія/камера] → RTMP/SRT ingest → [Media Server: транскод + ABR ladder]
   → HLS/DASH сегменти → [CDN/edge] → [HTML5 плеєр] (масовий, latency 6-30с)
   → WebRTC → [SFU] → [WebRTC плеєр] (інтерактив, latency <1с)
```

### L3 — Senior
**Фундаментальний трейдоф — latency vs scale:**
- **HLS/DASH:** масштаб через **HTTP CDN-кешування** (stateless, мільйони глядачів дешево) ↔ латентність 6-30с (буферизація сегментів). LL-HLS/LL-DASH (chunked transfer, часткові сегменти) знижують до 2-5с ціною складності.
- **WebRTC:** sub-second латентність (UDP, без сегментної буферизації) ↔ погано масштабується (кожен peer — стан; потрібні SFU-каскади; немає простого CDN-кешу). Для мільйонів — дорого/складно.
- **RTMP:** простий, низька латентність для **ingest**, але TCP (head-of-line blocking) і legacy (Flash мертвий, але ingest живий). Витісняється **SRT** (UDP + надійність + шифрування) для contribution.

**Коли що (senior-вибір):**
- Ingest зі студії по ненадійній мережі → **SRT** (надійність по UDP) або RTMP.
- Масова роздача (спорт, концерт, мільйони) → **HLS/DASH + CDN** (latency прийнятна).
- Інтерактив (аукціон, ставки, дзвінок, sub-second) → **WebRTC** (SFU).
- Гібрид: WebRTC для активних учасників + HLS для пасивної масової аудиторії.

**Чому це питатимуть саме на цій ролі:** «scalable streaming server» + «WebRTC» прямо у вакансії. Знати **trade-off latency↔scale** і де яка ланка — основа дизайну такого сервера.

> ⚠️ Цифри латентності орієнтовні (залежать від конфігу, розміру сегментів, мережі). LL-HLS/LL-DASH/WebRTC-екосистема швидко розвивається — звіряй з актуальними джерелами.

## 4. GStreamer pipeline (elements, pads, caps, bins)

### L1 — Junior
**Аналогія — конвеєр на заводі.** GStreamer будує **pipeline (конвеєр)** з блоків:
- **Element (елемент)** — станція конвеєра: джерело (камера), фільтр (енкодер), стік (вихід у мережу).
- **Pad** — «роз'єм» елемента, яким він з'єднується з наступним.
- **Caps** — «специфікація» даних на роз'ємі (формат, роздільність, fps).
- **Bin** — група елементів (pipeline — це великий bin).

Дані течуть зліва направо: **source → filter → sink**.

### L2 — Middle
**Найпростіша pipeline (CLI):**
```bash
# Камера → конвертація → H.264 енкодинг → RTP-пакування → відправка по UDP
gst-launch-1.0 v4l2src ! videoconvert ! x264enc ! rtph264pay ! udpsink host=10.0.0.5 port=5000
#              [source]   [filter]      [encoder]  [payloader]  [sink]
#                    ! = з'єднання pad'ів (link)
```

**Складові:**
- **Element** — `v4l2src` (камера), `x264enc` (H.264 енкодер), `udpsink` (вихід).
- **Pad** — `src pad` (вихід елемента) → `sink pad` (вхід наступного). Дані течуть src→sink.
- **Caps** — формат на pad: `video/x-raw, width=1920, height=1080, framerate=30/1`. Сусідні елементи **домовляються** (caps negotiation), чи сумісні.
- **Bin / Pipeline** — контейнер елементів; має **bus** (повідомлення: помилки, EOS) і **стани** (NULL → READY → PAUSED → PLAYING).

**На Python (PyGObject):**
```python
import gi
gi.require_version("Gst", "1.0")
from gi.repository import Gst

Gst.init(None)
pipeline = Gst.parse_launch(
    "v4l2src ! videoconvert ! x264enc tune=zerolatency ! rtph264pay ! udpsink host=10.0.0.5 port=5000"
)
pipeline.set_state(Gst.State.PLAYING)   # запустити конвеєр
# ... слухати bus на помилки/EOS ...
```

**Gotcha:** `tune=zerolatency` на `x264enc` — критично для live (вимикає B-frames і буферизацію, питання 5). Без нього енкодер буферизує → секунди затримки.

### L3 — Senior
**Caps negotiation — серце GStreamer:** елементи **домовляються** про формат на льоту. Якщо формати несумісні (`v4l2src` дає YUYV, а `x264enc` хоче I420) — потрібен конвертер (`videoconvert`). Помилка «not negotiated» — найчастіша проблема pipeline (несумісні caps між елементами).

**Динамічні pads (складніший рівень):** деякі елементи (`decodebin`, `rtpbin`) створюють pad'и **в рантаймі** (не знають наперед, скільки потоків у вхідних даних) → треба підписуватись на сигнал `pad-added` і лінкувати динамічно. Це часта складність у реальних pipeline.

**Чому GStreamer, а не FFmpeg-subprocess (трейдоф):**
- **GStreamer** — модульний граф, тонкий контроль, динамічна реконфігурація, обробка подій/метаданих у потоці, відпускає GIL (питання 1, 7). Складніший API.
- **FFmpeg (як процес)** — простіше для лінійного транскоду, але монолітний (важко втручатись у середину потоку), керування через CLI/pipe.
Для «event-based handler + кастомний encoding service» (вакансія) GStreamer гнучкіший — можна вставляти `appsink`/`appsrc` для своєї логіки в Python.

**Архітектура encoding-сервісу на GStreamer (вакансія):**
```
appsrc (кадри з твого коду) → encoder → tee → [RTP/UDP sink]
                                            └→ [appsink (метрики/запис)]
bus → обробник подій (стан, помилки, QoS) → твій event handler
```
**Senior-розуміння:** GStreamer pipeline — це **граф потоку даних**; ти керуєш ним через стани, bus-повідомлення (QoS, помилки, EOS) і pad-проби (perf-моніторинг, вставка логіки). `appsrc`/`appsink` — міст між pipeline і Python-кодом (твоя бізнес-логіка, event handlers).

> ⚠️ GStreamer API глибокий і має багато нюансів (threading model, pad probes, dynamic pads). Це тема, яку **треба практикувати**, а не лише читати — звіряй з офіційною документацією GStreamer і прикладами. Якщо GStreamer для тебе новий — це пріоритет №1 у підготовці.

## 5. I/P/B-кадри та GOP

### L1 — Junior
**Аналогія — як економно описати відео.**
- **I-frame (ключовий кадр)** — **повна картинка** (як фото). Самодостатній, але важкий.
- **P-frame** — «що **змінилось** від попереднього кадру» (рух м'яча). Легкий.
- **B-frame** — «що змінилось, дивлячись і **назад, і вперед**». Найлегший, але потребує майбутній кадр.

**GOP (Group of Pictures)** — група від одного I-frame до наступного: `I B B P B B P ...`

### L2 — Middle
```
GOP:  I  B  B  P  B  B  P  B  B  P  ...  (наступний I)
      ↑                                    ↑
   ключовий кадр                     новий ключовий
```
- **I-frame** — повний, ~10× більший за P. Точка входу (під'єднатись/перемотати можна лише з I).
- **P-frame** — посилається на попередні. Середній розмір.
- **B-frame** — посилається на **попередні + майбутні** → найменший, **але** декодеру треба дочекатись майбутнього кадру → **додає затримку** й переупорядкування.

**Keyframe interval (GOP length):**
- **Короткий GOP** (часті I-frames) → менша латентність, швидша перемотка, краще відновлення після втрат, **АЛЕ** більший бітрейт (I-frames важкі).
- **Довгий GOP** → краще стиснення, **АЛЕ** повільніший вхід/перемотка, гірше відновлення.

**Gotcha (критично для live):** **B-frames додають латентність** (декодер чекає майбутній кадр + переупорядкування). Тому для **low-latency live** B-frames часто **вимикають** (`tune=zerolatency` у x264, питання 4). Класичний trade-off: стиснення ↔ затримка.

### L3 — Senior
**Чому keyframe interval критичний для стримінгу:**
- **HLS/DASH сегментація:** сегмент має починатись з I-frame (щоб плеєр міг почати з будь-якого сегмента). Тому **GOP вирівнюють із довжиною сегмента** (напр. GOP = 2с = сегмент 2с). Незбіг → плеєр не може чисто нарізати → артефакти/затримка.
- **Latency:** глядач може під'єднатись лише на I-frame → довгий GOP = довге очікування першого кадру (time-to-first-frame).
- **Error recovery:** втрата I-frame псує весь GOP (P/B залежать від нього); короткий GOP → швидше відновлення.

**Closed vs Open GOP:**
- **Closed GOP** — кадри не посилаються за межі GOP → кожен GOP самодостатній → потрібно для чистої сегментації (HLS/DASH). 
- **Open GOP** — краще стиснення, але кадри можуть посилатись через межу → проблеми з нарізкою.

**Трейдоф (senior-баланс):** налаштування GOP/frames — це **компроміс «стиснення ↔ латентність ↔ стійкість»** під конкретний сценарій:
- Low-latency інтерактив (WebRTC) → короткий GOP, **без B-frames**, closed GOP.
- Масова VOD-якість → довгий GOP, з B-frames, максимальне стиснення.
- Live broadcast → GOP = довжина сегмента, closed, помірні B-frames або без них.

**Чому це питатимуть:** розуміння I/P/B + GOP пояснює **причини** багатьох проблем стримінгу (висока латентність → довгий GOP/B-frames; артефакти після втрат → пошкоджений I-frame; погана перемотка → рідкі ключові кадри). Це фундамент і для troubleshooting (питання 6), і для encoding-сервісу (вакансія).

> ⚠️ Деталі (макс. кількість reference frames, рівні B-pyramid, специфіка кодеків) залежать від кодека/енкодера. Звіряй із докою x264/x265/конкретного енкодера.

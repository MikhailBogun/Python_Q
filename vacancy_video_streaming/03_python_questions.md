## ⚖️ Як вони важать Python для цієї ролі
Вакансія: «Python knowledge» + «performance optimization» + «scalable server» + 24/7 production. Тому Python-питання будуть **не про синтаксис**, а про:
1. **Конкурентність** (як обробляти потоки/кадри паралельно — GIL).
2. **Пам'ять** (довгоживучий сервіс не має текти за тижні аптайму).
3. **Бінарні дані** (відеокадр = масив байтів; копіювання вбиває перформанс).
4. **Надійність** (помилки в проді обробляються, не валять сервіс).
5. **Продуктивність** (профілювання, делегування в C).

## 🅱️ Блок 2 — Пам'ять і бінарні дані (критично для 24/7 + відео)

### 5. 🔗 Керування пам'яттю: GC + reference counting
> «Як Python керує пам'яттю? Reference counting vs garbage collector? Як знайти й полагодити memory leak у сервісі, що працює тижнями?»
- **Чому дуже ймовірне:** **24/7 production** — витоки за дні аптайму вбивають сервіс. Класична проблема live-систем.
- **Де вчити:** [answers/06_python_advanced.md](../answers/06_python_advanced.md) п.75, 77.

### 6. 🆕 Бінарні дані: `bytes`, `bytearray`, `memoryview`, buffer protocol ⭐
> «Як працювати з бінарними даними (відеокадр) у Python без зайвого копіювання? Що таке `memoryview` і buffer protocol? `bytes` vs `bytearray`?»
- **Чому дуже ймовірне:** **відеокадр = масив байтів**; zero-copy через `memoryview` — головна оптимізація відео (180MB/с при 30fps копіювання — питання Q7). **Role-specific, ще не розкрито детально.**
- **Складність:** Middle/Senior, **важливе саме для цієї ролі**.

### 7. 🆕 `struct` та робота з бінарними протоколами
> «Як розпарсити бінарний заголовок (напр. RTP/TS-пакет) у Python? Модуль `struct`, big/little-endian?»
- **Чому ймовірне:** парсинг мережевих/медіа-пакетів (RTP, MPEG-TS заголовки) — типова задача відео-інженера. **Role-specific.**
- **Складність:** Middle.

### 8. 🔗 Mutable vs immutable + пастка mutable default args
> «Які типи змінювані/незмінювані? Чому `def f(x, buf=[])` — баг?»
- **Чому ймовірне:** базова перевірка + класична пастка на буферах/акумуляторах.
- **Де вчити:** [answers/02_python.md](../answers/02_python.md) п.25.

## 🅲 Блок 3 — Структури даних і алгоритми (вимога «basic»)

### 9. 🆕 Ring buffer / `deque` для буферизації кадрів ⭐
> «Як реалізувати буфер останніх N кадрів (ring buffer)? Чому `collections.deque`, а не `list`? Складність операцій?»
- **Чому дуже ймовірне:** **буферизація кадрів** (jitter buffer, останні N кадрів) — пряма задача стримінгу; `deque` з `maxlen` — ідіоматичне рішення. **Role-specific практична задача.**
- **Складність:** Middle, **дуже релевантне**.

### 10. 🔗 Складність структур: list/deque/dict/set — коли що
> «`list` vs `deque` vs `dict`/`set` — складність операцій, коли яку брати? Чому `list.pop(0)` погано?»
- **Чому ймовірне:** «basic algorithms + complexity» (вимога) + вибір структури під буферизацію/чергу.
- **Де вчити:** [answers/02_python.md](../answers/02_python.md) п.47, 48, 50 + [answers/algorithms/README.md](../answers/algorithms/README.md).

### 11. 🆕 Producer-Consumer: `queue.Queue` / `asyncio.Queue` ⭐
> «Реалізуй producer-consumer: один потік захоплює кадри, інший кодує. Як синхронізувати? `queue.Queue`, backpressure?»
- **Чому дуже ймовірне:** **capture → encode** — класичний producer-consumer; thread-safe черга + backpressure (буфер не росте безмежно). **Role-specific патерн.**
- **Складність:** Middle/Senior, **дуже релевантне**.

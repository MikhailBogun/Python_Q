# Кодеки та контейнери — повний deep-dive для відео-інженера

> Усе про відео/аудіо кодеки й контейнери для ролі. Як працює стиснення, ключові кодеки, контейнери, **remux vs transcode**, мапінг під протоколи стримінгу.
> Прямо з вакансії: «Knowledge about video codecs and containers». Формат — від простого до глибокого.

---

## 0. Codec vs Container — фундамент (не плутати!)

**Аналогія — вино й пляшка.**
- **Кодек** = саме **вино**: алгоритм **стиснення** медіа (як дані запаковані). H.264, H.265, AV1, AAC, Opus.
- **Контейнер** = **пляшка з етикеткою**: обгортка, що тримає закодовані потоки + метадані + синхронізацію. MP4, MKV, MPEG-TS, WebM.

```
КОНТЕЙНЕР (MP4)
├── відео-потік: закодований H.264 (кодек)
├── аудіо-потік: закодований AAC (кодек)
├── субтитри
├── метадані (тривалість, мова, автор)
└── індекс/тайминг (для синхронізації й перемотки)
```
**Один контейнер тримає різні кодеки.** MP4 може містити H.264 або H.265 + AAC або Opus. Кодек і контейнер — **незалежні виміри**.

# ЧАСТИНА A — КОДЕКИ

## A1. Що таке кодек і навіщо стиснення

**Codec = COder + DECoder.** Сире відео величезне:
```
1080p, 30fps, 8-bit, RGB:  1920×1080×3 байти × 30 = ~186 MB/с = ~1.5 Гбіт/с
```
Без стиснення передати/зберегти неможливо. Кодек стискає у **100–1000 разів** (H.264 ~5-10 Мбіт/с для 1080p).

**Lossy vs Lossless:**
- **Lossy (з втратами)** — викидає «непомітне» для ока → величезне стиснення. H.264/265/AV1 — усі lossy. Для стримінгу — завжди lossy.
- **Lossless (без втрат)** — точне відновлення, мале стиснення. FFV1, FLAC (аудіо). Для архіву/майстерів.

## A2. Як працює стиснення відео (під капотом) ⭐

Чотири кроки (спрощено):

### Крок 1 — Prediction (передбачення)
- **Intra-prediction (просторове)** — у межах одного кадру (як JPEG): передбачити блок із сусідніх пікселів. → **I-frame** (Q5).
- **Inter-prediction (часове)** — між кадрами: знайти, куди зрушив блок (**motion estimation**), зберегти лише **різницю (residual)** + вектор руху. → **P/B-frames** (Q5).

**Це й є джерело I/P/B-кадрів і GOP** (зв'язок з Q5): I = повний, P = різниця від минулого, B = різниця від минулого+майбутнього.

### Крок 2 — Transform (перетворення)
Залишок (residual) переводять у частотну область — **DCT (Discrete Cosine Transform)**. Низькі частоти (загальна структура) важливіші за високі (дрібні деталі).

### Крок 3 — Quantization (квантування) — ТУТ втрати
Округлення коефіцієнтів. **QP (quantization parameter)** керує силою: вище QP → грубіше → менший розмір, гірша якість. **Це головний важіль якість↔розмір** (CRF/bitrate керують ним).

### Крок 4 — Entropy coding (ентропійне кодування)
Безвтратне ущільнення квантованих даних: **CABAC** (краще, повільніше) або **CAVLC** (простіше) у H.264.

```
Кадр → Prediction (intra/inter) → DCT → Quantization (lossy!) → Entropy → бітстрім
```

## A3. Колірний простір, chroma subsampling, бітність ⭐

### YUV/YCbCr замість RGB
Відео-кодеки працюють у **YCbCr**: **Y** (яскравість/luma) + **Cb, Cr** (кольороріність/chroma). Око чутливіше до яскравості, ніж до кольору → можна стиснути chroma сильніше.

### Chroma subsampling (часте питання!)
| Формат | Що це | Де |
|---|---|---|
| **4:4:4** | повна chroma (без субдискретизації) | майстеринг, скриншоти, текст |
| **4:2:2** | chroma вдвічі по горизонталі | професійне відео, broadcast |
| **4:2:0** | chroma вдвічі по обох осях (¼ даних) | **стандарт для стримінгу/відео** |

**4:2:0** — дефолт стримінгу: економить ~50% chroma-даних, око майже не помічає. **Gotcha:** для тексту/UI (screen sharing) 4:2:0 розмиває кольорові краї → там 4:4:4.

### Бітність (bit depth)
- **8-bit** — стандарт (256 рівнів на канал).
- **10-bit** — HDR, менше бендингу (смуг на градієнтах), 4K. Дорожче.

### HDR / колірні стандарти
- **BT.601** (SD), **BT.709** (HD — стандарт), **BT.2020** (UHD/HDR).
- HDR (HDR10, Dolby Vision) — ширший динамічний діапазон, потребує 10-bit.

## A4. Відео-кодеки (порівняння) ⭐

| Кодек | Рік | Стиснення | CPU енкодингу | Ліцензії | Підтримка |
|---|---|---|---|---|---|
| **H.264 / AVC** | 2003 | базове | низький | патенти (зрілі) | **всюди** (HW скрізь) |
| **H.265 / HEVC** | 2013 | ~50% краще H.264 | високий | складні патенти | широка (4K/HDR) |
| **AV1** | 2018 | ~30% краще HEVC | **дуже високий** | **royalty-free** | зростає (YT, Netflix) |
| **VP9** | 2013 | ~HEVC-рівень | високий | royalty-free | веб (YouTube/WebM) |
| **VP8** | 2008 | ~H.264 | середній | royalty-free | WebRTC (legacy) |
| **VVC / H.266** | 2020 | ~50% краще HEVC | дуже високий | патенти | рання |
| **MPEG-2** | 1995 | слабке | низький | застарілі | legacy (DVD/broadcast) |

**Як обирати (трейдоф):**
- **H.264** — дефолт: сумісність + швидкий HW-енкодинг + низька латентність. Для **live** найчастіше H.264.
- **H.265/AV1** — де економія смуги критична (4K, high-load CDN-витрати), але дорожче кодувати й гірша сумісність.
- **VP8/VP9/AV1** — royalty-free, для веб/WebRTC (немає патентних виплат).

**Аудіо-кодеки:**
| Кодек | Особливість |
|---|---|
| **AAC** | стандарт для стримінгу/MP4/HLS |
| **Opus** | **low-latency**, чудовий на низьких бітрейтах → **WebRTC/WebM** |
| **MP3** | legacy |
| **AC-3 / E-AC-3** | Dolby, broadcast/surround |
| **PCM** | без стиснення (raw) |
| **FLAC** | lossless |

## A5. Керування бітрейтом (CBR/VBR/CRF) ⭐

| Режим | Що | Коли |
|---|---|---|
| **CBR** (Constant Bitrate) | фіксований бітрейт | **live/broadcast** (передбачувана смуга) |
| **VBR** (Variable Bitrate) | бітрейт плаває за складністю сцени | VOD (краща якість/розмір) |
| **CRF** (Constant Rate Factor) | цільова **якість** (x264/x265), бітрейт плаває | **VOD** (найкраще якість/розмір) |
| **Capped CRF** | CRF з лімітом бітрейту | гібрид (якість + контроль піків) |

**Gotcha для live:** для стримінгу зазвичай **CBR** (або capped) — бо канал/CDN потребує **передбачуваної** смуги; чистий VBR/CRF дає сплески, що ламають буфер плеєра. Для VOD — **CRF** (оптимальна якість на байт).

## A6. Profiles і Levels

- **Profile** — набір можливостей кодека: H.264 **Baseline** (просто, для слабких пристроїв, без B-frames), **Main**, **High** (макс. стиснення). Вище → краще стиснення, але потрібен потужніший декодер.
- **Level** — обмеження ресурсів (макс. роздільність/бітрейт/fps): Level 4.0 ≈ 1080p30, 5.1 ≈ 4K тощо.

**Навіщо:** сумісність із пристроями. Старий/слабкий декодер може не тягнути High Profile / Level 5.1 → обираєш профіль/рівень під цільові пристрої.

## A7. NAL units — структура H.264-потоку (важливо для пакування) ⭐

H.264/H.265 **елементарний потік** = послідовність **NAL units** (Network Abstraction Layer):
- **SPS** (Sequence Parameter Set) — глобальні параметри (роздільність, профіль).
- **PPS** (Picture Parameter Set) — параметри кадрів.
- **IDR slice** — ключовий кадр (I-frame, точка входу).
- **non-IDR slice** — P/B-кадри.

**Два формати запису:**
- **Annex B** — NAL'и розділені start-кодами `0x000001` → для **потокової передачі** (RTP, MPEG-TS, live).
- **AVCC / length-prefixed** — кожен NAL з префіксом довжини → для **MP4** (random access).

**Чому важливо:** при пакуванні в RTP/контейнер треба **правильний формат**; `h264parse` у GStreamer конвертує між ними. SPS/PPS мають передаватись (in-band або out-of-band), інакше декодер не стартує — часта причина «чорного екрана».

# ЧАСТИНА B — КОНТЕЙНЕРИ

## B1. Що всередині контейнера

Контейнер **мультиплексує** (склеює) кілька потоків + керує синхронізацією:
```
КОНТЕЙНЕР
├── Track 1: відео (H.264) + timestamps (PTS/DTS)
├── Track 2: аудіо (AAC) + timestamps
├── Track 3: субтитри
├── Метадані (мова, тривалість, теги, обкладинка)
└── Індекс (де що лежить — для перемотки/random access)
```
**Головні задачі контейнера:** (1) тримати кілька потоків разом, (2) **синхронізація** A/V через timestamps, (3) **індекс** для перемотки, (4) метадані.

## B2. Контейнери (огляд) ⭐

| Контейнер | Основа | Призначення | Для live? |
|---|---|---|---|
| **MP4** | ISOBMFF | VOD, універсальний | ❌ (потрібен fMP4) |
| **fMP4** (fragmented MP4) | ISOBMFF | **low-latency live** (HLS/DASH, CMAF) | ✅ |
| **MPEG-TS** | TS (188-байт пакети) | **broadcast/live**, стійкий | ✅ |
| **MKV** (Matroska) | EBML | гнучкий, багато кодеків, open | частково |
| **WebM** | підмножина MKV | веб (VP8/VP9/AV1 + Opus) | частково |
| **MOV** | QuickTime (≈MP4) | Apple/майстеринг | ❌ |
| **FLV** | Flash | legacy RTMP | (ingest) |
| **AVI** | RIFF | старий, обмежений | ❌ |

## B3. Live vs VOD — ключова відмінність ⭐

**Чому звичайний MP4 поганий для live:**
```
MP4 має "moov atom" — ІНДЕКС усього файлу (де кожен кадр).
За замовчуванням moov В КІНЦІ файлу → потрібен ПОВНИЙ файл, щоб грати.
Для live (файл ще пишеться) — неможливо! 
```
**Рішення для live:**
- **MPEG-TS** — потоковий від народження: 188-байт пакети, кожен самодостатній, можна **під'єднатись посеред потоку** (на наступному ключовому кадрі). Стійкий до втрат. Класика broadcast.
- **fMP4 (fragmented MP4)** — MP4, розбитий на **самодостатні фрагменти** (кожен зі своїм мінііндексом) → можна стрімити шматками, не чекаючи кінця. Основа сучасного low-latency HLS/DASH.
- **`-movflags +faststart`** (ffmpeg) — для VOD-MP4 переносить moov **на початок** → можна почати грати до повного завантаження (progressive download).

**Gotcha-питання:** «чому не можна просто стрімити MP4?» → moov atom в кінці; для live → TS або fMP4.

## B4. Muxing / Demuxing

- **Muxing (мультиплексування)** — склеїти закодовані відео+аудіо потоки в контейнер (`mp4mux`, `mpegtsmux` у GStreamer).
- **Demuxing** — розібрати контейнер назад на потоки (`qtdemux`, `tsdemux`).

```
Encode video (H.264) ─┐
Encode audio (AAC)  ──┼─→ MUX → контейнер (MP4)
                      │
контейнер → DEMUX → відео-потік + аудіо-потік → Decode
```

## B5. Remux vs Transcode — критична концепція ⭐⭐

| | **Remux (repackage)** | **Transcode** |
|---|---|---|
| Що міняється | лише **контейнер** | **кодек / бітрейт / роздільність** |
| Кодек | **той самий** (копіюється) | перекодовується |
| Швидкість | **дуже швидко** (копіювання байтів) | **повільно** (повний енкодинг) |
| CPU | майже нуль | високий |
| Втрата якості | **немає** (копія) | є (повторне lossy-стиснення) |
| Приклад | TS → MP4 (той самий H.264) | H.264 → H.265, 1080p → 720p |

```bash
# REMUX — швидко, без втрат (той самий кодек, інша обгортка)
ffmpeg -i input.ts -c copy output.mp4          # -c copy = НЕ перекодовувати

# TRANSCODE — повільно, з втратами (інший кодек/якість)
ffmpeg -i input.mp4 -c:v libx265 -crf 28 output.mp4
```

**Senior-інсайт (дуже важливо для архітектури):** **remux у сотні разів дешевший за transcode.** Грамотний pipeline **кодує раз** (один майстер-енкод) → **remux/repackage** під різні протоколи/контейнери (TS для одного, fMP4 для іншого) замість багаторазового transcode. Зайвий transcode = марний CPU + втрата якості (кожне перекодування деградує). Зв'язок з Q2 і архітектурою стримінг-сервера.

## B6. CMAF — уніфікація HLS і DASH

Раніше HLS вимагав MPEG-TS, DASH — fMP4 → доводилось **дублювати** сегменти на CDN (×2 зберігання). **CMAF (Common Media Application Format)** — fMP4-формат, який розуміють **і HLS, і DASH** → **один набір сегментів** на обидва протоколи. Економія зберігання + основа low-latency (LL-HLS/LL-DASH). Сучасний стандарт.

# ЧАСТИНА C — ПРАКТИКА (під роль)

## C1. Мапінг: протокол → кодек + контейнер ⭐

| Протокол | Відео-кодек | Аудіо | Контейнер/пакування |
|---|---|---|---|
| **HLS** | H.264/H.265 | AAC | MPEG-TS (legacy) або **fMP4/CMAF** |
| **DASH** | будь-який | AAC/Opus | **fMP4/CMAF** |
| **WebRTC** | VP8/VP9/H.264/AV1 | **Opus** | **RTP** (немає контейнера — RTP-пакетизація!) |
| **RTMP** (ingest) | H.264 | AAC | FLV |
| **SRT** (contribution) | H.264/H.265 | AAC | зазвичай MPEG-TS |

**Gotcha:** **WebRTC не використовує контейнер** — медіа пакується прямо в **RTP-пакети** (RTP payloader, зв'язок із UDP/06 і GStreamer/07 `rtph264pay`). Контейнер (MP4/TS) — для файлів/HLS/DASH, не для WebRTC.

## C2. Bitrate ladder / ABR (adaptive bitrate)

Для адаптивного стримінгу (HLS/DASH) кодують **кілька рендицій** (один контент, різна якість):
```
1080p @ 5 Мбіт/с  ┐
720p  @ 3 Мбіт/с  ├─ один контент, bitrate ladder
480p  @ 1.5 Мбіт/с├─ плеєр перемикає за смугою (ABR, зв'язок з Q9 web player)
360p  @ 0.8 Мбіт/с┘
```
Кожна рендиція — окремий transcode (різний бітрейт/роздільність) + сегментація. Це основне навантаження транскодинг-ферми (high-load, nice to have у вакансії).

## C3. Інспекція файлів — інструменти ⭐

```bash
# ffprobe — що всередині (кодеки, контейнер, потоки, бітрейт)
ffprobe input.mp4
ffprobe -v error -show_streams -show_format input.mp4
ffprobe -show_entries stream=codec_name,width,height,bit_rate input.mp4

# Приклад виводу:
#   Stream #0:0: Video: h264 (High), yuv420p, 1920x1080, 5000 kb/s, 30 fps
#   Stream #0:1: Audio: aac, 48000 Hz, stereo, 128 kb/s
#   ↑ кодек    ↑ profile ↑ chroma  ↑ розмір  ↑ бітрейт ↑ fps

# mediainfo — детальніше, людиночитабельно
mediainfo input.mp4

# GStreamer аналог
gst-discoverer-1.0 input.mp4
```

**Python (PyAV) — програмна інспекція:**
```python
import av
container = av.open("input.mp4")
for stream in container.streams:
    if stream.type == "video":
        print(stream.codec_context.name)       # 'h264'
        print(stream.width, stream.height)      # 1920 1080
        print(stream.average_rate)              # 30 fps
        print(stream.pix_fmt)                    # 'yuv420p' (= 4:2:0, 8-bit)
```

**Strong Junior must:** уміти `ffprobe` файл і прочитати — який кодек, контейнер, роздільність, бітрейт, chroma. Це перший крок діагностики «чому не грає» (несумісний кодек/профіль).

## 🔑 Топ gotcha-питання на інтерв'ю
1. **«Codec vs container?»** → вино vs пляшка; незалежні.
2. **«Чому не можна стрімити MP4 для live?»** → moov atom у кінці; → TS/fMP4.
3. **«Remux vs transcode?»** → контейнер (швидко, копія) vs кодек (повільно, втрати).
4. **«H.264 vs H.265?»** → ~50% стиснення ціною CPU/ліцензій/сумісності.
5. **«Що таке 4:2:0?»** → chroma субдискретизація, стандарт стримінгу.
6. **«Який кодек/контейнер для WebRTC?»** → VP8/H.264 + Opus, RTP (без контейнера!).
7. **«CBR чи VBR для live?»** → CBR (передбачувана смуга).

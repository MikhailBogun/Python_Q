# Професія Live Video Streaming Engineer — процес, терміни, практики

> Цілісна картина: **що ти робитимеш**, **головні терміни** й **практики** — простою мовою. Це «карта», що зв'язує всі deep-dive у розуміння професії.
> ⚠️ Конкретика залежить від компанії; тут — типова картина для такої ролі на основі вакансії.

---

## 1. Велика картина — у чому суть роботи

**Одним реченням:** ти допомагаєш, щоб **«прямий ефір» доходив до глядачів безперебійно 24/7** — від камери в студії до плеєра в браузері.

**Аналогія — як телеканал/трансляція.** Є студія (джерело сигналу), є складна «труба», якою сигнал стискається, пакується й розноситься тисячам глядачів, і є плеєр, у якому вони дивляться. Твоя робота — **будувати й підтримувати цю трубу** (Python-сервіси + GStreamer), **стежити**, щоб вона працювала, і **лагодити**, коли щось ламається (навіть о 3 ночі).

**Дві половини роботи:**
1. **Розробка (build)** — писати/покращувати Python-сервіси, що обробляють відео (encoding, streaming server, web player, обробники подій).
2. **Експлуатація (operate)** — моніторити живі стріми, реагувати на інциденти, діагностувати проблеми (24/7 production).

Ти не сам — поруч **сеньйори, студійні інженери, DevOps**. Ти вчишся домену в процесі.

## 2. Signal flow — головна ментальна модель ⭐

Усе, що ти робитимеш, крутиться навколо **шляху сигналу** (signal flow) — від джерела до глядача. Запам'ятай цей ланцюг — це **карта всієї роботи й troubleshooting**:

```
┌─────────┐   ┌─────────┐   ┌─────────┐   ┌──────────┐   ┌─────────┐   ┌────────┐   ┌────────┐
│ СТУДІЯ  │──>│ CAPTURE │──>│ ENCODE  │──>│ PACKAGE  │──>│TRANSPORT│──>│  CDN/  │──>│ PLAYER │
│ камера/ │   │захоплен-│   │стиснення│   │пакування │   │доставка │   │  edge  │   │ глядач │
│ сигнал  │   │  ня     │   │(GStream)│   │(контейн.)│   │(UDP/TCP)│   │розповс.│   │(браузер)│
└─────────┘   └─────────┘   └─────────┘   └──────────┘   └─────────┘   └────────┘   └────────┘
     │             │             │             │             │             │            │
     └─────────────┴─────────────┴── МОНІТОРИНГ + TROUBLESHOOTING (накладається на все) ──┴──┘
                                  + EVENT HANDLER (реагує на події обладнання)
```

**Кожна ланка — зона твоєї роботи:**
| Ланка | Що відбувається | Твоя робота |
|---|---|---|
| **Студія** | камера/мікрофон дають сигнал (SDI/NDI) | взаємодія зі студійними інженерами |
| **Capture** | сигнал заходить у систему | джерела в pipeline (v4l2src, appsrc) |
| **Encode** | сире відео стискається (H.264) | **GStreamer encoding service** (твій обов'язок) |
| **Package** | складається в контейнер (TS/fMP4) | мукс, сегментація |
| **Transport** | передається по мережі | UDP/RTP/RTMP/SRT |
| **CDN/edge** | розноситься тисячам глядачів | scalable streaming server |
| **Player** | глядач дивиться в браузері | **web player** (твій обов'язок) |

Прив'язано до обов'язків вакансії:

| **scalable video streaming server** | сервіс, що приймає й роздає стріми багатьом; масштабування | [09 asyncio](09_deep_dive_asyncio.md), [03 протоколи](01_answers_q1_q5.md) |

| **event-based handler for equipment** | код, що реагує на події студійного обладнання (камера перемкнулась, сигнал зник) | [09 asyncio](09_deep_dive_asyncio.md), event loop |

### 🚨 Експлуатація (operate) — 24/7
- **Моніторити** дашборди живих стрімів (чи все ОК, латентність, дропи).
- **Реагувати на інциденти** — стрім завис/сипле → діагностувати й полагодити ([Q6 troubleshooting](02_answers_q6_q10.md)).
- **On-call** — чергування (можливо вночі/вихідні, бо 24/7).
- **Post-mortem** — після інциденту розібрати корінну причину.

### 👥 Командна робота
- **Сеньйори** — навчають, рев'ю коду, дизайн.
- **Студійні інженери** — сигнал, обладнання.
- **DevOps** — деплой, інфраструктура, моніторинг.
- **Розробники** — суміжні сервіси.

**Реалістичний тиждень:** ~60% розробка (Python/GStreamer), ~30% моніторинг/інциденти, ~10% мітинги/навчання/рев'ю. На чергуванні — більше операційки.

## 4. Головні терміни (глосарій) ⭐

Згруповано за зонами. Одне-два речення кожне — деталі в deep-dive.

### 📹 Відео-основи
| Термін | Простими словами |
|---|---|
| **Frame (кадр)** | одна «фотографія» з відео |
| **Resolution** | розмір кадру (1920×1080 = 1080p) |
| **FPS / framerate** | кадрів за секунду (30fps) |
| **Bitrate** | скільки даних за секунду (5 Мбіт/с) — якість vs смуга |
| **Codec (кодек)** | алгоритм стиснення (H.264, H.265) |
| **Container (контейнер)** | обгортка для потоків (MP4, TS) |
| **I/P/B-frames** | повний кадр / різниця від минулого / двостороння різниця |
| **GOP** | група кадрів від одного ключового до наступного |
| **Keyframe** | ключовий (повний) кадр = точка входу/перемотки |

### 🎬 Сигнал і студія
| Термін | Простими словами |
|---|---|
| **Signal flow** | шлях сигналу від джерела до глядача |
| **Source / Ingest** | джерело / прийом сигналу в систему |
| **Capture** | захоплення сигналу (з камери/карти) |
| **SDI / NDI** | професійні способи передачі відеосигналу (кабель / по мережі) |
| **Studio engineer** | людина, що відповідає за обладнання студії |

### 🌐 Стримінг і протоколи
| Термін | Простими словами |
|---|---|
| **RTMP** | протокол для **прийому** сигналу (енкодер→сервер) |
| **HLS / DASH** | роздача глядачам через HTTP/CDN (масштабовано, але затримка) |
| **WebRTC** | майже миттєва доставка (sub-second), для інтерактиву |
| **SRT** | надійна доставка по поганій мережі |
| **RTP** | пакування медіа для real-time (поверх UDP) |
| **Segment** | шматок відео (2-6с) у HLS/DASH |
| **Manifest** | список сегментів і якостей (.m3u8 / .mpd) |
| **Latency** | затримка від «події» до «глядач побачив» |
| **ABR** | адаптивний бітрейт — плеєр сам обирає якість за смугою |
| **CDN** | мережа серверів, що розносить контент ближче до глядачів |

### 🔌 Мережа
| Термін | Простими словами |
|---|---|
| **UDP / TCP** | швидко-ненадійно / надійно-повільніше |
| **Packet loss** | втрата пакетів → артефакти |
| **Jitter** | нерівномірність прибуття пакетів → ривки |
| **Multicast** | один потік → багато глядачів (студія/IPTV) |
| **MTU** | макс. розмір пакета (~1500) |

### 🎛️ GStreamer
| Термін | Простими словами |
|---|---|
| **Pipeline** | конвеєр обробки медіа |
| **Element** | блок конвеєра (джерело/фільтр/стік) |
| **Pad** | роз'єм елемента для з'єднання |
| **Caps** | опис формату даних на роз'ємі |
| **Bus** | канал повідомлень pipeline (помилки, події) |
| **appsrc / appsink** | міст між pipeline і твоїм Python-кодом |

### 📊 Експлуатація (24/7)
| Термін | Простими словами |
|---|---|
| **Monitoring** | стеження за «здоров'ям» системи в реальному часі |
| **Metrics** | числові показники (RPS, латентність, дропи) |
| **Logging** | запис подій для розслідування |
| **Alert** | сповіщення при проблемі |
| **On-call** | чергування (готовність реагувати) |
| **Incident** | збій у проді, що треба полагодити |
| **Dropped frames** | кадри, що не встигли оброблятись/передатись |
| **Post-mortem** | розбір інциденту після (корінна причина) |
| **SLA / SLO** | обіцяний/цільовий рівень надійності |

### 🐍 Python (інструменти ремесла)
| Термін | Простими словами |
|---|---|
| **GIL** | один потік виконує Python одночасно → CPU-важке в процеси/C |
| **asyncio** | конкурентність для мережі (роздача стрімів) |
| **Zero-copy** | обробка кадрів без копіювання (memoryview) — швидкість |
| **Profiling** | вимірювання, де код повільний |
| **Backpressure** | стримування швидкого продюсера, щоб не залити пам'ять |

## 5. Головні практики (як робиться робота) ⭐

### 🧩 Pipeline thinking (мислення конвеєром)
Усе у відео — це **потік даних крізь стадії**. Ти постійно думаєш: «дані заходять сюди → перетворюються тут → виходять туди». Це і GStreamer, і архітектура сервісу. Навчишся бачити систему як граф потоку.

### 🔍 Troubleshooting-методологія (найважливіша практика!)
Коли щось ламається, **не міняй наосліп**:
1. **Масштаб + час** — один стрім чи всі? коли почалось? що змінилось?
2. **Дані** — дашборди, метрики, логи (не здогадки).
3. **Шарова ізоляція** — перевір кожну ланку signal flow по черзі (мережа → encode → package → CDN → player), знайди, де ламається.
4. **Гіпотеза → перевірка → мінімальна зміна**.
5. **Post-mortem** — щоб не повторилось.
Це **головна практика** ролі (двічі «Excellent troubleshooting» у вакансії).

### 📈 Observability (спостережуваність)
Перш ніж лагодити — треба **бачити**. Три стовпи: **логи** (що сталося), **метрики** (скільки/як швидко), **трейси** (де затик). Дивишся дашборди (Grafana), реагуєш на алерти.

### ⚡ Performance profiling
Не вгадуєш, де повільно — **вимірюєш** (cProfile, py-spy). Знаходиш bottleneck, оптимізуєш точково (часто — прибрати копіювання кадрів, винести важке в C).

### 🔧 Інженерна гігієна (загальна)
- **Version control (Git)** — гілки, коміти, pull requests.
- **Code review** — сеньйори рев'юять твій код (і ти вчишся).
- **Testing** — тести, щоб зміни не ламали стрім ([10 тестування](10_deep_dive_testing.md)).
- **CI/CD** — автоматична перевірка/деплой.
- **Документація** — фіксувати, як що працює.

### 🚨 Incident response (реагування на інциденти)
- Спокій під тиском (прод лежить — не панікувати).
- Комунікація (інформувати команду, ескалювати).
- Безпечні зміни (rollback-план).
- Документувати, що зробив.

### 📡 Event-driven мислення
Студійне обладнання генерує **події** (камера перемкнулась, сигнал зник, формат змінився). Ти пишеш **обробники**, що реагують (часто — конечні автомати / стани, event loop, черги повідомлень).

## 8. 🎯 Що з цього найважливіше засвоїти (пріоритет)

**Для розуміння роботи (must):**
1. **Signal flow** (розділ 2) — карта всього.
2. **Troubleshooting-методологія** (розділ 5) — головна практика.
3. **Базовий словник** (розділ 4) — щоб розуміти колег.

**Для впевненості (corе Python — справжня планка вакансії):**
4. Python-фундамент: asyncio, GIL, профілювання, структури даних.
5. Pipeline thinking + observability.

**Як фора (бонус, не обов'язково):**
6. Глибина відео-домену (GStreamer, кодеки, протоколи) — робить тебе сильнішим кандидатом, але вони цього навчать.

> **Головна думка:** ця професія = **Python-інженерія + домен живого відео + операційна дисципліна 24/7**. Реальна планка — сильний Python + troubleshooting + готовність вчити домен. Решта (відео-глибина) додається з досвідом і з цих deep-dive.

## 6. Troubleshooting-сценарій: стрім завис о 3 ночі ⭐

> Це **найважливіше** питання ролі (24/7 production, «excellent troubleshooting»). Перевіряють **підхід**, не зазубрені факти.

### L1 — Junior
**Аналогія — лікар ставить діагноз, а не вгадує таблетку.** Не кидайся міняти все підряд у проді. **Збери симптоми → звузь зону → перевір по черзі ланки → знайди причину → лагодь.**

**Перші три питання:**
1. **Масштаб:** один стрім чи всі? (один → проблема джерела; всі → проблема інфраструктури).
2. **Коли почалось?** Що змінилось перед цим (деплой, конфіг, навантаження)?
3. **Що саме?** Замерзає (freeze), сипле (артефакти), розсинхрон, чорний екран?

### L2 — Middle
**Методичний обхід ланок (signal flow):**
```
[Джерело/студія] → [Capture] → [Encoder] → [Packaging] → [CDN/transport] → [Player]
       ↓ перевір по черзі, де ламається ↓
```
| Ланка | Що перевірити | Інструменти |
|---|---|---|
| Мережа | packet loss, jitter, bandwidth | `ping`, `mtr`, `iftop`, `tcpdump` |
| Capture | сигнал зі студії є? коректний? | моніторинг входу, SDI/NDI статус |
| Encoder | CPU, dropped frames, стан pipeline | `htop`, GStreamer `GST_DEBUG`, метрики |
| Packaging | сегменти створюються? валідні? | перевірити .m3u8/сегменти, логи |
| CDN | edge віддає? кеш? 4xx/5xx? | логи CDN, статус-коди |
| Player | що показує статистика плеєра | dev tools, player stats |

**Тактика ізоляції — «bisect pipeline»:** перевір вихід **кожної** ланки. Джерело добре, а вихід енкодера битий? → проблема в енкодері. Енкодер добрий, плеєр битий? → транспорт/CDN.

**Gotcha:** **не міняй наосліп у проді о 3 ночі.** Спершу — **дані** (логи, метрики, моніторинг), потім гіпотеза, потім **мінімальна** перевірена зміна. І комунікація: повідом команду/чергового.

### L3 — Senior
**Системний фреймворк діагностики (як думає senior):**
1. **Scope & timeline:** масштаб (один/всі/регіон) + коли почалось + correlation із подіями (деплой, сплеск трафіку, оновлення обладнання). «Що змінилось» — 80% інцидентів.
2. **Observability спершу** (блок Python — три стовпи): **метрики** (event loop lag, CPU, dropped frames, bitrate, buffer level), **логи** (помилки pipeline, GStreamer bus), **трейси** (шлях кадру через систему). Дашборди → де аномалія.
3. **Шарова локалізація:** мережа (jitter/loss — питання 10) → енкодер (CPU/GIL/pipeline стан — питання 1, 4) → packaging (GOP/сегменти — питання 5) → CDN → плеєр.
4. **Гіпотеза → перевірка → висновок**, не «міняй і дивись».

**Типові причини «freeze/артефакти» у live (доменне знання):**
- **Packet loss/jitter** на UDP → пропущені пакети → артефакти (питання 10). Перевір мережу між студією й енкодером.
- **Encoder перевантажений** (CPU 100%, event loop lag — питання 1) → dropped frames → freeze. Сплеск роздільності/бітрейту?
- **Пошкоджений I-frame** → весь GOP сипле, поки не прийде наступний ключовий (питання 5).
- **Buffer underrun** на плеєрі → freeze (мережа глядача або CDN не встигає).
- **Clock drift / розсинхрон** аудіо-відео.
- **Витік ресурсів** (memory leak за години роботи — тому «о 3 ночі», після годин аптайму).

**Senior-якості, які перевіряють:**
- **Спокій і метод** під тиском (production down).
- **Комунікація** (інформувати, ескалювати, документувати).
- **Безпечні зміни** (rollback-план, не ламати ще більше).
- **Post-mortem** після — корінна причина, щоб не повторилось (blameless).

**Як відповідати на інтерв'ю:** покажи **структуру** («спершу масштаб і таймлайн → метрики/логи → шарова ізоляція → гіпотеза → мінімальна зміна → post-mortem»), назви конкретні інструменти й типові причини. Інтерв'юер хоче побачити **системне мислення під тиском**, а не «я б подивився логи».

> ⚠️ Конкретні інструменти моніторингу залежать від стеку компанії. Головне — методологія; інструменти підлаштуєш.

## 7. Performance optimization обробки кадрів

### L1 — Junior
**Аналогія — спершу заміряй, потім ремонтуй.** Не вгадуй, **де** повільно — **виміряй профайлером**. Часто вузьке місце не там, де здається. Знайшов гаряче місце → прискорюй саме його.

**Головне правило:** profile first, optimize second.

### L2 — Middle
```python
# 1. Профілювання — знайти bottleneck
import cProfile
cProfile.run("process_video(frames)")   # де витрачається час

# py-spy — профайлер без зміни коду (для прода!)
# $ py-spy top --pid <PID>
# $ py-spy record -o profile.svg --pid <PID>
```

**Типові оптимізації обробки відео:**
| Проблема | Рішення |
|---|---|
| Копіювання великих кадрів | **zero-copy** (`memoryview`, buffer protocol) |
| Python-цикл по пікселях | **numpy** (векторизація, C, відпускає GIL) |
| CPU-важкий енкодинг | винести в C / GStreamer / процеси |
| Per-frame Python overhead | батчинг, мінімум Python на кадр |

```python
# ❌ Python-цикл по пікселях — катастрофічно повільно
for y in range(h):
    for x in range(w):
        frame[y][x] = transform(frame[y][x])

# ✅ numpy — векторно, у C, відпускає GIL
frame = frame * 1.2          # одна C-операція над усім масивом
```

**Gotcha:** копіювання кадру 1920×1080×3 = ~6MB **на кожен кадр × 30fps** = 180MB/с зайвого копіювання. **Zero-copy** (передавати посилання на буфер, не копію) — найважливіша оптимізація у відео.

### L3 — Senior
**Ієрархія оптимізації (від найбільшого ефекту):**
1. **Алгоритм/архітектура** — не роби зайвого: уникай зайвого transcode (remux дешевше, питання 2), обробляй лише потрібне, кешуй.
2. **Делегування в C** — гарячий шлях у нативний код (GStreamer, PyAV, numpy, OpenCV, Cython) → відпускає GIL (питання 1), на порядки швидше за Python-цикли.
3. **Zero-copy** — `memoryview`, buffer protocol, `SharedMemory` між процесами; уникати `.copy()`, зайвих конвертацій формату.
4. **Паралелізм** — `multiprocessing`/потоки з C-бібліотеками для багатьох ядер.
5. **Мікрооптимізації** — лише після профілювання, лише на гарячому шляху.

**Чому профілювання обов'язкове:** Big-O й інтуїція брешуть про **константи й реальні bottleneck'и**. У відео часто вузьке місце — **пам'ять/копіювання й конвертація форматів**, а не «алгоритм». `py-spy` (sampling profiler без overhead) — стандарт для прода (можна профілювати живий процес).

**Трейдоф:** оптимізація коштує часу/складності/читабельності — оптимізуй **те, що профайлер показав гарячим**, не «про запас» (premature optimization). Винесення в C прискорює, але ускладнює дебаг/збірку. Senior балансує: 80% часу зазвичай у 20% коду — туди й зусилля.

**Метрика для відео-сервісу:** не лише CPU, а **frames dropped**, **encode latency per frame**, **event loop lag** (питання 1, 6). Якщо енкодер не встигає за реальним часом (30fps = 33мс/кадр) — стрім «відстає» або дропає кадри.

> ⚠️ Конкретні бібліотеки (PyAV vs OpenCV vs raw GStreamer) і їх перформанс залежать від задачі — бенчмаркай на своїх даних.

## 8. WebRTC: як встановлюється з'єднання

### L1 — Junior
**Аналогія — двоє хочуть зустрітись, але не знають адрес одне одного.** Потрібен «посередник», щоб обмінятись контактами (**signaling**), спосіб дізнатись свою публічну адресу за NAT (**STUN**), і запасний «кур'єр», якщо напряму не вийде (**TURN**).

WebRTC дає **прямий (P2P) real-time** медіа-канал між браузерами/клієнтами з низькою затримкою.

### L2 — Middle
**Кроки встановлення з'єднання:**
```
1. SIGNALING (ти реалізуєш сам, зазвичай WebSocket):
   A створює OFFER (SDP) → через сервер → B
   B створює ANSWER (SDP) → через сервер → A

2. ICE (пошук шляху):
   обидва збирають "candidates" (можливі адреси) і обмінюються ними
   - STUN: "яка моя публічна IP:port за NAT?"
   - TURN: relay-сервер, якщо прямий шлях неможливий

3. Перевірка з'єднання → DTLS handshake (шифрування) → 
   SRTP медіа тече (зашифровано)
```

**Компоненти:**
| Термін | Роль |
|---|---|
| **Signaling** | обмін SDP (offer/answer) — **поза** WebRTC, твоя відповідальність |
| **SDP** | опис медіа (кодеки, роздільність, мережа) |
| **ICE** | фреймворк пошуку найкращого шляху |
| **STUN** | дізнатись свій публічний IP:port (дешево) |
| **TURN** | relay медіа, коли P2P неможливий (дорого — весь трафік через сервер) |

**Gotcha:** **signaling — НЕ частина WebRTC.** WebRTC дає медіа-транспорт, але **як** клієнти обміняються SDP — реалізуєш ти (WebSocket, HTTP). Це часта плутанина.

### L3 — Senior
**Чому стільки складності — NAT traversal:** більшість пристроїв за NAT (немає прямої публічної адреси). STUN дозволяє дізнатись зовнішню адресу й пробити «дірку» (hole punching) для P2P. Якщо NAT суворий (symmetric) — P2P неможливий → **TURN relay** (весь медіа-трафік через сервер, дорого по смузі). ~10-20% з'єднань потребують TURN.

**Для streaming-сервера (вакансія) — SFU vs MCU:**
- **SFU (Selective Forwarding Unit)** — **пересилає** потоки без перекодування. Масштабованіше (CPU-дешево), кожен отримує потоки інших. Стандарт для WebRTC-стримінгу на багатьох глядачів.
- **MCU (Multipoint Control Unit)** — **змішує** потоки в один (CPU-важко, але клієнт отримує один потік). Для слабких клієнтів.
- Чистий **P2P** не масштабується (N² з'єднань) → для стримінгу потрібен **SFU** на сервері.

**Трейдоф WebRTC для стримінгу (питання 3):** sub-second латентність (UDP/SRTP, без сегментної буферизації) ↔ складність (NAT, TURN-витрати, SFU-каскади для масштабу, немає простого CDN-кешу). Для мільйонів глядачів WebRTC дорогий — звідси гібриди (WebRTC для інтерактиву + HLS для масовості).

**Безпека:** WebRTC **завжди шифрований** (DTLS-SRTP обов'язково) — на відміну від RTMP/HLS (опціонально). Плюс для приватності, мінус — overhead.

> ⚠️ WebRTC — глибока тема (ICE-стани, codec negotiation, simulcast, bandwidth estimation). Деталі звіряй з webrtc.org / специфікаціями. Для серверної частини дивись медіасервери (mediasoup, Janus, Pion) — вони реалізують SFU.

## 9. Web player + Media Source Extensions (MSE)

### L1 — Junior
**Аналогія — годувати плеєр шматками з рук.** Простий `<video src="file.mp4">` грає **один готовий файл**. Для **адаптивного стримінгу** (різна якість, live-сегменти) браузеру треба **подавати медіа шматками з JavaScript** — це робить **MSE (Media Source Extensions)**.

```html
<video id="player" controls></video>
<!-- JS через MSE подає сегменти у <video> -->
```

### L2 — Middle
**Як працює адаптивний веб-плеєр:**
```
.m3u8/.mpd манифест → плеєр читає список сегментів і якостей
   → тягне сегменти (fMP4/TS) по HTTP
   → подає у SourceBuffer через MSE
   → <video> декодує й показує
   → ABR: перемикає якість залежно від смуги/буфера
```

**Бібліотеки:**
| Інструмент | Для чого |
|---|---|
| **hls.js** | відтворення HLS у браузерах (крім Safari — там нативно) через MSE |
| **dash.js** | відтворення MPEG-DASH через MSE |
| **video.js / Shaka Player** | універсальні плеєри (HLS+DASH+DRM) |
| **WebRTC** (`RTCPeerConnection`) | low-latency плеєр (без MSE) |

```javascript
// Спрощено: hls.js під'єднує HLS до <video> через MSE
import Hls from "hls.js";
const video = document.getElementById("player");
if (Hls.isSupported()) {
  const hls = new Hls();
  hls.loadSource("https://cdn.example.com/stream.m3u8");
  hls.attachMedia(video);          // подає сегменти у video через MSE
} else if (video.canPlayType("application/vnd.apple.mpegurl")) {
  video.src = "https://cdn.example.com/stream.m3u8";  // Safari — нативний HLS
}
```

**Gotcha:** браузери (крім Safari/iOS) **не** підтримують HLS нативно в `<video src>` → потрібен **hls.js + MSE**. Safari підтримує HLS нативно, але **не** має MSE на iPhone (історично) — звідси фрагментація.

### L3 — Senior
**MSE — серце сучасного веб-стримінгу:** JS-API, що дозволяє програмно наповнювати медіа-буфер (`SourceBuffer`) сегментами. Без нього неможливі адаптивний бітрейт, live, DVR у браузері. hls.js/dash.js — це, по суті, **JS-реалізації стримінг-логіки поверх MSE** (парсинг манифесту, завантаження сегментів, ABR, керування буфером).

**ABR (Adaptive Bitrate) — ключовий алгоритм плеєра:**
- Кодек видає **кілька рендицій** (bitrate ladder: 240p/480p/720p/1080p — питання 2).
- Плеєр **вимірює смугу й рівень буфера** → перемикає якість (низька смуга → нижча якість, щоб не було freeze).
- Трейдоф: агресивне перемикання (плавно, але стрибки якості) vs стабільне (краща якість, але ризик буферизації). Алгоритми: buffer-based, throughput-based, гібридні.

**Трейдоф вибору технології плеєра:**
- **MSE + hls.js/dash.js** — масштабовано (HTTP/CDN), ABR, широка підтримка ↔ латентність 6-30с (питання 3).
- **WebRTC-плеєр** (`RTCPeerConnection`, без MSE) — sub-second латентність ↔ складність, погана масштабованість, дорого.
- **Native HLS** (Safari) — просто, але обмежений контроль.

**Додаткові senior-теми:** **EME (Encrypted Media Extensions)** для DRM (Widevine/FairPlay/PlayReady), **LL-HLS/LL-DASH** для зниження латентності через MSE, buffer management (скільки тримати наперед), error recovery (перепідключення на збоях). Для «create a web player» (вакансія) — знати MSE-модель і коли потрібен WebRTC замість HLS.

> ⚠️ Підтримка MSE/HLS/кодеків різниться між браузерами/платформами (особливо iOS Safari) і змінюється. Звіряй матриці сумісності (caniuse) і docs hls.js/dash.js.

## 10. UDP vs TCP для live-відео

### L1 — Junior
**Аналогія — лист рекомендований vs звичайний.**
- **TCP** — рекомендований лист: гарантовано дійде, але якщо загубився — **перешлють** (чекаємо → затримка).
- **UDP** — звичайний лист: швидко, але якщо загубився — **пропав** (не чекаємо).

Для **live-відео** часто краще UDP: «**краще втратити один кадр, ніж чекати** його й відстати від реального часу».

### L2 — Middle
| | TCP | UDP |
|---|---|---|
| Надійність | гарантована (retransmit) | немає |
| Порядок | гарантований | немає |
| Швидкість/latency | повільніше (підтвердження) | швидко |
| Втрати | перешле (→ затримка) | втратить (→ артефакт) |
| Для відео | VOD, HLS/DASH (HTTP) | real-time (WebRTC, RTP, SRT) |

**Чому UDP для real-time:**
```
TCP: пакет загубився → СТОП, чекаємо перешле → всі наступні чекають
     (head-of-line blocking) → відстаємо від live → ростуча затримка

UDP: пакет загубився → пропускаємо, граємо далі → невеликий артефакт,
     але лишаємось у реальному часі
```

**Gotcha:** **head-of-line blocking** у TCP — головна причина його непридатності для low-latency live: один втрачений пакет блокує **всі наступні** (вони чекають перешле), хоча вони вже прийшли. UDP цього не має.

### L3 — Senior
**Jitter і jitter buffer:**
- **Jitter** — нерівномірність прибуття пакетів (один за 10мс, наступний за 50мс). Псує плавність.
- **Jitter buffer** — невеликий буфер на приймачі, що **згладжує** нерівномірність (накопичує трохи й віддає рівномірно). Трейдоф: більший буфер = плавніше, але **+латентність**. Класичний баланс live.

**Packet loss — як борються (поверх UDP):**
- **FEC (Forward Error Correction)** — надлишкові пакети дозволяють відновити втрачені без перезапиту (ціна — +смуга).
- **NACK/retransmit** — вибірковий перезапит **важливих** пакетів (WebRTC) — гібрид.
- **SRT/RIST** — додають **надійність поверх UDP** (ARQ-перезапит у межах latency-вікна) → надійність TCP + контроль латентності UDP. Для contribution по поганих мережах (питання 3).
- **Concealment** — декодер «маскує» втрату (повторює/інтерполює).

**Фундаментальний трейдоф (senior-суть):** **latency ↔ reliability ↔ quality** — не можна максимізувати все.
- TCP/HTTP (HLS) — надійність + якість ціною латентності (буферизація). Для масової VOD/live, де 10с затримки ок.
- UDP (WebRTC/RTP) — низька латентність ціною можливих втрат/артефактів. Для інтерактиву.
- SRT — компроміс: надійність по UDP у межах заданого latency-бюджету. Для contribution.

**Чому це фундамент ролі:** вибір транспорту визначає латентність і стійкість стриму; розуміння jitter/loss — основа **troubleshooting** (питання 6: «артефакти» → packet loss на UDP; «freeze/затримка» → retransmit на TCP або великий jitter buffer). Це зв'язує мережу, протоколи й діагностику воєдино.

> ⚠️ Деталі FEC/NACK/SRT-механізмів — глибокі й залежать від реалізації. RTP/RTCP, SRT — окремі специфікації; звіряй при потребі.

## ✅ Підсумок
Усі 10 питань розкрито за Принципом Сходів. Наскрізні зв'язки:
- **GIL/C-делегування** (1) → **performance** (7) → **troubleshooting** (6).
- **Codecs/containers** (2) → **GOP/frames** (5) → **encoding-сервіс** (GStreamer, 4).
- **Протоколи** (3) → **UDP/TCP** (10) → **WebRTC** (8) → **плеєр** (9).
- **Troubleshooting** (6) інтегрує все — мережа, енкодер, packaging, CDN, плеєр.

> **Найважливіше для цієї ролі:** питання **6 (troubleshooting)** і доменний фундамент (**2, 3, 5**). GStreamer (**4**) — пріоритет для практичної підготовки, якщо новий.

Скажи, якщо хочеш: глибше розкрити якесь питання, додати ще питань (system design, event handlers, Linux/Docker, алгоритми), або зробити **mock-інтерв'ю** по цих темах.

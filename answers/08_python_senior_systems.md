## 94. Що таке `@property`?

### L1 — Junior
**Аналогія — спідометр.** Ти дивишся на стрілку як на звичайний показник, але за нею **обчислення** (оберти коліс → швидкість). `@property` робить так, що **метод виглядає як атрибут**: звертаєшся `obj.area`, а під ним виконується код.

```python
class Circle:
    def __init__(self, radius: float) -> None:
        self.radius = radius

    @property
    def area(self) -> float:          # викликається як атрибут, не метод
        return 3.14159 * self.radius ** 2

c = Circle(5)
print(c.area)                          # 78.5 — без дужок! не c.area()
```

### L2 — Middle
Три частини: getter (`@property`), setter, deleter:
```python
class Temperature:
    def __init__(self, celsius: float) -> None:
        self._celsius = celsius

    @property
    def celsius(self) -> float:               # getter
        return self._celsius

    @celsius.setter
    def celsius(self, value: float) -> None:  # setter + ВАЛІДАЦІЯ
        if value < -273.15:
            raise ValueError("нижче абсолютного нуля")
        self._celsius = value

    @property
    def fahrenheit(self) -> float:            # обчислюване, read-only
        return self._celsius * 9 / 5 + 32

t = Temperature(25)
t.celsius = 30          # викликає setter (з валідацією)
t.fahrenheit            # 86.0 — обчислюється на льоту
t.fahrenheit = 100      # AttributeError — нема setter (read-only)
```

**Головні юзкейси:**
1. **Обчислювані атрибути** (area, fahrenheit) — не зберігаємо, рахуємо.
2. **Валідація** при присвоєнні (setter).
3. **Read-only** атрибути (лише getter).
4. **Зворотна сумісність** — перетворити публічний атрибут на property **без зміни API** (код, що читав `obj.x`, працює далі).

**Gotcha:** `@property` — причина, чому Python **не пише getter'и/setter'и наперед** (як Java). Починаєш із простого `self.x`, а **коли** знадобиться логіка — робиш property, не ламаючи зовнішній код. Тому «гетери про запас» у Python — антипатерн (YAGNI, блок 1 п.2).

### L3 — Senior
`@property` — це **дескриптор даних** (data descriptor, п.98): клас `property` реалізує `__get__`, `__set__`, `__delete__`. Коли ти пишеш `@property`, створюється екземпляр `property` на **рівні класу**, і доступ `obj.area` йде через дескрипторний протокол (`__getattribute__` → знаходить data descriptor → викликає `__get__`).

**Чому це data descriptor і чому важливо:** data descriptor має **пріоритет над `instance.__dict__`** (п.98). Тому `obj.area = ...` без setter падає (а не тихо створює атрибут екземпляра) — property «перехоплює» і читання, і запис.

**Трейдоф:**
- **+** Інкапсуляція без бойлерплейту, обчислювані/валідовані атрибути, еволюція API без поломки.
- **−** **Прихована вартість:** `obj.area` виглядає дешево (як атрибут), але може ховати дорогі обчислення/I/O → читач не очікує. Property з важкою логікою порушує «принцип найменшого здивування». Для дорогих обчислень — `functools.cached_property` (рахує раз, кешує) або явний метод `compute_area()`, що сигналізує вартість.

**`cached_property` (блок 6, п.50):** обчислює при першому доступі, зберігає в `instance.__dict__` (тому це **non-data** descriptor — інстансний dict його потім «перекриває» → наступні доступи безкоштовні). Тонкий нюанс протоколу дескрипторів на практиці.

> ⚠️ Точна механіка `property` як data descriptor — звіряй з Descriptor HowTo Guide (блок 6 п.35).

## 95. Як запустити код на Python паралельно?

### L1 — Junior
**Аналогія — більше працівників на задачу.** Є кілька способів «робити одночасно»: кілька рук одного працівника (потоки), кілька окремих працівників (процеси), або один спритний працівник, що не простоює (async). *(Threads/processes — блок 2 п.30; async — блок 6 п.73.)*

**Головне правило (через GIL):**
- **CPU-bound** (рахунок) → **процеси**.
- **I/O-bound** (мережа/диск) → **потоки** або **async**.

### L2 — Middle
**1. `concurrent.futures` — найзручніший інтерфейс:**
```python
from concurrent.futures import ProcessPoolExecutor, ThreadPoolExecutor

# CPU-bound → процеси (справжній паралелізм, обхід GIL)
with ProcessPoolExecutor() as ex:
    results = list(ex.map(heavy_compute, data))

# I/O-bound → потоки
with ThreadPoolExecutor(max_workers=20) as ex:
    results = list(ex.map(fetch_url, urls))
```

**2. `multiprocessing` напряму:**
```python
from multiprocessing import Pool
with Pool(4) as p:
    results = p.map(cpu_task, items)
```

**3. `asyncio` — масовий I/O в одному потоці (блок 6, п.73):**
```python
results = await asyncio.gather(*[fetch(u) for u in urls])
```

**Карта вибору:**
| Задача | Інструмент |
|---|---|
| CPU-bound, локально | `ProcessPoolExecutor` / `multiprocessing` |
| I/O-bound, помірно | `ThreadPoolExecutor` |
| I/O-bound, тисячі з'єднань | `asyncio` |
| CPU-bound, розподілено | Celery / Dask / Ray |
| Числові масиви | NumPy (векторизація, C без GIL) |

**Gotcha:** `multiprocessing` **серіалізує** аргументи й результати через **pickle** → не всі об'єкти передаються (lambda, локальні функції, відкриті сокети — ні), і великі дані дорого копіювати. Старт процесу теж дорогий.

### L3 — Senior
**Чому процеси для CPU (блок 2, п.30):** GIL дозволяє одному потоку виконувати байткод одночасно → потоки **не** паралелять CPU. Кожен процес має **свій інтерпретатор і свій GIL** → справжній паралелізм, ціною: дорогий старт, окрема пам'ять, IPC через pickle.

**Обхід GIL без процесів:**
1. **C-розширення відпускають GIL:** NumPy, що рахує над масивом, звільняє GIL на час C-обчислення → потоки **можуть** паралелити числовий код (блок 7, п.93). Те саме — `Py_BEGIN_ALLOW_THREADS` у власних розширеннях.
2. **Блокуючий I/O відпускає GIL** → потоки паралелять очікування мережі/диску.
3. **Cython/Numba** з `nogil`-секціями.

**Трейдофи моделей паралелізму:**
| Модель | Паралелізм CPU | Пам'ять | Overhead | Передача даних |
|---|---|---|---|---|
| threading | ❌ (GIL) | спільна | низький | прямий |
| multiprocessing | ✅ | окрема | високий старт | pickle/IPC |
| asyncio | ❌ | спільна | дуже низький | прямий |
| розподілено (Celery/Ray) | ✅ | окрема | мережа | серіалізація |

**Розподілений паралелізм (масштаб понад одну машину):**
- **Celery** — черга задач, воркери на багатьох машинах (блок 5, п.67).
- **Dask / Ray** — паралельні/розподілені обчислення (DataFrames, ML).
- Тут паралелізм обмежений мережею й серіалізацією, але масштабується горизонтально.

**Сучасне (важливо):** Python 3.13 **free-threaded** (PEP 703, no-GIL) обіцяє справжній потоковий паралелізм CPU без процесів; 3.12 **субінтерпретатори** (PEP 684) з ізольованим GIL — проміжний крок. Це може змінити дефолт «CPU → процеси» у майбутньому.

**Senior-стратегія:** спершу **виміряй**, що bottleneck (CPU чи I/O) — від цього модель. Часто реальна відповідь — **NumPy-векторизація** (CPU без процесів) або **async** (I/O), а не наївне «накидати потоків».

> ⚠️ Статус free-threaded/субінтерпретаторів і поведінка `nogil` швидко змінюються між версіями. Звіряй із release notes.

## 96. Як працювати зі stdlib?

### L1 — Junior
**Аналогія — вбудований набір інструментів.** Python іде «з батарейками» (*batteries included*): величезна стандартна бібліотека на всі базові задачі — не треба нічого ставити. Перш ніж тягнути сторонній пакет — глянь, чи це вже є в stdlib.

```python
import json, datetime, pathlib, collections   # усе вбудоване
```

### L2 — Middle
**Карта найважливіших модулів (блок 2, п.50):**
| Сфера | Модулі |
|---|---|
| Дані/колекції | `collections`, `itertools`, `functools`, `dataclasses`, `enum`, `heapq`, `bisect` |
| Текст/формат | `json`, `csv`, `re`, `string`, `textwrap` |
| Файли/ОС | `pathlib`, `os`, `sys`, `shutil`, `tempfile`, `glob` |
| Час | `datetime`, `time`, `zoneinfo`, `calendar` |
| Мережа/веб | `urllib`, `http`, `socket`, `asyncio` |
| Конкурентність | `threading`, `multiprocessing`, `concurrent.futures`, `asyncio`, `queue` |
| Тести/дебаг | `unittest`, `logging`, `pdb`, `traceback`, `typing` |
| Математика | `math`, `statistics`, `decimal`, `fractions`, `random` |

**Як орієнтуватись:**
1. **Знай категорії** (не всі функції) — пам'ятай, що задача «парсинг дат» → `datetime`, «лічильник» → `collections.Counter`.
2. **`dir(module)` + `help(obj)`** — introspection (блок 2, п.27) прямо в REPL.
3. **docs.python.org** — офіційна документація з прикладами.

**Gotcha:** stdlib стабільна, але **не завжди найзручніша/найшвидша**. `requests` зручніший за `urllib`, `httpx` — async. `pendulum`/`arrow` приємніші за голий `datetime`. Stdlib — дефолт, але не догма.

### L3 — Senior
**Трейдоф stdlib vs сторонні (блок 2, п.40):**
- **stdlib:** нуль залежностей, гарантована сумісність із версією Python, нема supply-chain ризику (блок 1, п.11), завжди доступна. Ціна — інколи незграбний API, повільніший розвиток (прив'язаний до релізів Python).
- **Сторонні:** зручність, швидкість, фічі ціною ваги, ризику й сумісності.

**Правило Senior:** для **бібліотек/публічних пакетів** — мінімізуй залежності (кожна — твоя проблема й проблема користувачів), тяжій до stdlib. Для **застосунків** — бери зручність (requests/httpx), де вона того варта.

**stdlib як еталон якості Python:** читання сорсів stdlib (`Lib/` у CPython) — спосіб вчити ідіоматичний Python і патерни (як написані `collections`, `functools`, `contextlib`). Багато stdlib-модулів — приклади чистого API-дизайну.

**Підводний камінь — депрекації:** stdlib еволюціонує (видалили `distutils` у 3.12, `asyncore`, `imp`). Senior стежить за **deprecation warnings** і «What's New» при апгрейді версії — те, що було в stdlib, може зникнути.

**Що варто знати «напам'ять» (гарячі):** `collections` (Counter/defaultdict/deque), `itertools` (chain/groupby/islice), `functools` (lru_cache/partial/wraps/reduce), `pathlib`, `dataclasses`, `contextlib`, `typing`. Це покриває 80% щоденних задач без сторонніх пакетів.

> ⚠️ Склад stdlib змінюється між версіями (додавання/видалення модулів). Звіряй з docs «What's New» для цільової версії.

## 97. Які задачі вирішували метакласами?

### L1 — Junior
*(Метакласи заклали в блоці 2, п.33: метаклас — «клас, що створює класи», за замовчуванням `type`.)*

**Аналогія — конвеєр, що штампує форми за шаблоном.** Метаклас втручається **в момент створення класу** й може автоматично щось зробити: зареєструвати клас, додати методи, перевірити структуру. Це «магія» під ORM і фреймворками.

### L2 — Middle
**Реальні задачі, що вирішують метакласами:**

**1. Реєстрація класів (auto-registry) — плагіни, серіалізатори:**
```python
class PluginMeta(type):
    registry: dict[str, type] = {}
    def __new__(mcs, name, bases, ns):
        cls = super().__new__(mcs, name, bases, ns)
        if bases:                          # не реєструємо базовий
            PluginMeta.registry[name] = cls
        return cls

class Plugin(metaclass=PluginMeta): ...
class CsvExporter(Plugin): ...             # авто-реєструється
# PluginMeta.registry["CsvExporter"] доступний
```

**2. ORM (Django/SQLAlchemy):** метаклас читає атрибути-`Field`, будує мапінг на таблицю, додає менеджери.
**3. ABC (`abc.ABCMeta`):** перевіряє реалізацію абстрактних методів при інстанціюванні.
**4. Singleton (блок 2, п.33):** `__call__` метакласу контролює створення.
**5. Валідація структури класу:** перевірити, що клас має потрібні атрибути/методи на етапі визначення.
**6. Enum (`EnumMeta`):** перетворює атрибути на members.

**Gotcha (Tim Peters, блок 2 п.33):** *«Якщо вагаєшся, чи потрібен метаклас — не потрібен.»* 99% задач вирішуються простіше: `__init_subclass__`, декоратори класів, дескриптори.

### L3 — Senior
**Сучасна альтернатива метакласам — PEP 487 (`__init_subclass__`, `__set_name__`):** більшість того, для чого раніше брали метаклас, тепер роблять **простіше**:

```python
class Plugin:
    registry: dict[str, type] = {}
    def __init_subclass__(cls, **kwargs):     # хук при створенні підкласу
        super().__init_subclass__(**kwargs)
        Plugin.registry[cls.__name__] = cls   # та сама реєстрація, без метакласу

class CsvExporter(Plugin): ...                 # авто-реєструється
```

`__init_subclass__` викликається при **створенні підкласу**, `__set_name__` — коли дескриптор присвоюється атрибуту класу (знає своє ім'я). Разом покривають реєстрацію, валідацію, конфігурацію підкласів — **без** складності метакласів.

**Коли метаклас **справді** потрібен (рідко):**
- Треба контролювати **створення екземплярів** (`__call__` метакласу) — Singleton, кеш, пул.
- Треба змінити **сам процес** побудови класу (namespace, MRO) глибше, ніж дозволяє `__init_subclass__`.
- Уникнути **конфлікту метакласів** при складній множинній спадковості (усі бази мають сумісний метаклас).

**Трейдоф:** метаклас — найпотужніший інструмент метапрограмування ↔ **найскладніший для читання** (магія при визначенні класу, неочевидний потік), конфлікти метакласів при множинному наслідуванні, важко дебажити. Senior використовує **сходи простоти**: звичайний клас → декоратор класу → `__init_subclass__`/дескриптор → **метаклас лише в крайньому разі** (коли пишеш фреймворк/ORM).

> ⚠️ Семантика `__init_subclass__`/`__set_name__`/порядку викликів — звіряй з PEP 487 і Data model docs.

## 98. Що таке дескриптори?

### L1 — Junior
**Аналогія — розумна розетка замість дроту.** Звичайний атрибут — це прямий дріт (поклав значення — взяв значення). **Дескриптор** — «розумна розетка»: при кожному доступі/записі спрацьовує **твій код** (перевірка, обчислення, логування). `@property` (п.94) — це дескриптор.

### L2 — Middle
Дескриптор — об'єкт, що реалізує хоч один із методів протоколу:
- `__get__(self, obj, objtype)` — при **читанні** атрибута.
- `__set__(self, obj, value)` — при **записі**.
- `__delete__(self, obj)` — при `del`.

```python
class Positive:                       # переиспользуваний дескриптор-валідатор
    def __set_name__(self, owner, name):
        self._name = f"_{name}"       # знає ім'я атрибута (PEP 487)

    def __get__(self, obj, objtype=None):
        if obj is None:
            return self
        return getattr(obj, self._name)

    def __set__(self, obj, value):
        if value <= 0:
            raise ValueError(f"{self._name} має бути > 0")
        setattr(obj, self._name, value)

class Product:
    price = Positive()                # валідація переиспользується
    quantity = Positive()
    def __init__(self, price, quantity):
        self.price = price            # викликає Positive.__set__
        self.quantity = quantity

p = Product(10, 5)
p.price = -1                          # ValueError
```

**Два види:**
- **Data descriptor** — має `__set__` або `__delete__` (як `property`).
- **Non-data descriptor** — лише `__get__` (як звичайні функції/методи).

**Gotcha — пріоритет пошуку атрибутів:**
**data descriptor > `instance.__dict__` > non-data descriptor > `__getattr__`**.
Тому `property` (data) перекриває інстансний атрибут, а метод (non-data) може бути «затінений» атрибутом екземпляра.

### L3 — Senior
**Дескриптори — фундамент під половиною Python** (часто недооцінений). Через них реалізовані:
- **Методи** — функція є non-data descriptor; `obj.method` викликає `function.__get__`, що повертає **bound method** (прив'язує `self`). Ось чому методи «знають» свій екземпляр (блок 2, п.35).
- **`@property`** (п.94), **`@classmethod`/`@staticmethod`** (блок 2, п.35) — усі дескриптори з різним `__get__`.
- **`__slots__`** — member descriptors за фіксованим offset (блок 6, п.79).
- **`functools.cached_property`** — non-data descriptor, що кешує в `__dict__`.
- ORM-поля (Django/SQLAlchemy) — дескриптори, що мапляться на колонки.

**Як це працює в `__getattribute__`:** доступ `obj.x` → `type(obj).__getattribute__` шукає `x` у `type(obj).__mro__`; якщо знайдене — **data descriptor** → викликає `__get__` (пріоритет над `__dict__`); інакше дивиться `obj.__dict__`; інакше — **non-data descriptor** `__get__`; інакше `__getattr__`. Ця послідовність пояснює **весь** механізм атрибутів Python.

**Чому пріоритет саме такий:** data descriptor (property з валідацією) **має** перехоплювати запис → інакше `obj.x = ...` обійшов би валідацію через `__dict__`. Non-data (метод) навпаки — дозволяє екземпляру перекрити (рідко, але можливо). Це продуманий дизайн, не випадковість.

**Трейдоф:** дескриптори — потужне **переиспользування** логіки доступу (один `Positive()` валідує багато полів, DRY) ↔ непрямість і складність (доступ до атрибута таємно виконує код). Для разового випадку — `property` (простіше); для **переиспользуваної** логіки на багато атрибутів/класів — власний дескриптор. Це інструмент авторів фреймворків.

> ⚠️ Точний порядок у `__getattribute__` і взаємодія з `__getattr__`/`__set_name__` — звіряй з Descriptor HowTo Guide та Data model (це серце моделі об'єктів Python).

## 99. Знання інших мов, крім Python (як подавати + порівняння)

### L1 — Junior
*(Це питання про **твій досвід** — головне правило: «жодних галюцинацій», не вигадуй досвіду, якого нема.)*

**Як відповідати чесно й сильно:** назви те, з чим **реально** працював, навіть мало, і **що з того виніс**. Формула: *«Основна — Python. Дотично — [мова]: робив [що], звідти зрозумів [концепцію]»*. Чесне «знаю на рівні читання» сильніше за блеф, який розвалиться на уточненні.

### L2 — Middle
**Мови, корисні Python-розробнику, і що кожна дає:**
| Мова | Що дає розумінню | Де перетинається з Python |
|---|---|---|
| **C** | пам'ять, вказівники, як влаштований CPython | C-розширення, `ctypes`, розуміння GIL/refcount |
| **Go** | конкурентність (goroutines), простота, компіляція | мікросервіси, заміна Python там, де треба швидкість+concurrency |
| **Rust** | пам'ять без GC, ownership, безпека | **pydantic-core, ruff, uv, polars** написані на Rust! |
| **JavaScript/TS** | async-модель, frontend | повний стек, контракт API (блок 7, п.89) |
| **Java/C#** | статична типізація, JVM, справжні потоки | enterprise, порівняння моделей ООП/паралелізму |
| **SQL** | декларативність, теорія множин | щодня (блоки 3, 7) |

**Як демонструвати «знання» без багаторічного досвіду:**
- Порівнюй **концепції**: «у Go конкурентність через goroutines+channels, у Python — asyncio/GIL; Go дає справжній паралелізм компіляцією, Python — кооперативність».
- Покажи **трансфер знань**: «з Rust зрозумів ownership → у Python свідоміше ставлюсь до спільних мутабельних посилань (блок 6, п.80)».

**Gotcha:** інтерв'юер легко ловить блеф уточненням («а як у Go з дженериками?»). **Чесні межі** + здатність **порівнювати концепції** цінніші за заявлений, але порожній «досвід».

### L3 — Senior
**Поліглотизм як інженерна зрілість:** Senior цінується не за кількість мов, а за **розуміння парадигм і трейдофів**, що переносяться:
- **Моделі пам'яті** (п.102): C (ручна) → Python (refcount+GC) → Rust (ownership) → Java/Go (tracing GC). Знання спектру → свідомі рішення про продуктивність.
- **Моделі конкурентності:** GIL Python vs goroutines Go vs async Node vs потоки Java vs ownership-безпека Rust. Розумієш, **чому** Python обрав GIL і коли брати інше (блоки 2, 6).
- **Системи типів:** динамічний Python vs gradual (TS/mypy) vs статичний строгий (Rust/Haskell). Трейдоф гнучкість ↔ безпека (блок 6, п.81).

**Коли Senior тягнеться за іншою мовою:**
- **Rust/Go/C** для гарячих ділянок, де Python впирається в GIL/швидкість (переписати hot path як розширення — pydantic-core робить саме це).
- **Right tool for the job:** Python для бізнес-логіки/ML/скриптів, Go/Rust для high-throughput інфраструктури, SQL для даних.

**Чесна позиція (правило CLAUDE.md):** на інтерв'ю про мови, яких **не знаєш глибоко**, кажи прямо: *«поверхнево, можу читати / порівняти концептуально, продакшн-досвіду нема»*. Це **підвищує** довіру до решти відповідей. Видавати начитане за досвід — найгірша стратегія: один уточнюючий питання — і вся співбесіда під сумнівом.

> ⚠️ Конкретику чужих мов (синтаксис, остання семантика Go/Rust) перевіряй — не моя core-зона. Сила тут — у **порівнянні концепцій** з Python, а не в деталях.

## 100. Технологічні особливості реалізації розподілених систем

### L1 — Junior
**Аналогія — координація команди в різних містах.** Коли система розкидана по багатьох машинах (блок 5, п.65–66), з'являються проблеми, яких нема на одній: зв'язок рветься, годинники розходяться, хтось «не почув» повідомлення. Розподілені системи — це **мистецтво працювати, коли частина системи будь-коли може впасти або відстати**.

### L2 — Middle
**Ключові проблеми й рішення:**
| Проблема | Суть | Підхід |
|---|---|---|
| **Consistency** | усі бачать однакові дані? | strong vs eventual |
| **Доступність** при збоях | працює, коли вузол впав? | реплікація, redundancy |
| **Партиціювання** | дані не влазять в один вузол | sharding (блок 5, п.66) |
| **Координація** | хто головний / хто вирішує | consensus (Raft) |
| **Надійність повідомлень** | доставили? один раз? | at-least-once + ідемпотентність |
| **Розподілені транзакції** | атомарність на кількох сервісах | saga, 2PC |

**CAP-теорема (блоки 1, 7):** при мережевому розриві (**P**) обираєш **C** (consistency) або **A** (availability). Не можна мати все троє одночасно при partition.

**Ідемпотентність — основа надійності (блок 1, п.7):** мережа ненадійна → клієнт повторює запит, не знаючи, чи дійшов перший. Операції мають бути **ідемпотентними** (повтор = той самий ефект) або дедуплікуватись за **idempotency key**.

**Gotcha — «exactly-once» майже міф:** у розподілених системах гарантують **at-least-once** (можливі дублікати → роби ідемпотентно) або **at-most-once** (можливі втрати). «Exactly-once» досягається лише як at-least-once **+ ідемпотентний консюмер/дедуплікація**, не на рівні мережі.

### L3 — Senior
**Моделі узгодженості (спектр, не бінарно):**
- **Strong / Linearizable** — усі бачать останній запис миттєво (як одна машина). Дорого, потребує координації.
- **Eventual** — згодом усі зійдуться (блок 7, п.84). Дешево, висока доступність.
- **Causal** — зберігає причинно-наслідковий порядок (відповідь не з'явиться раніше за питання).
- **Read-your-writes** — бачиш власні зміни (проблема replication lag, блок 5 п.66).

**Consensus — Raft/Paxos:** як група вузлів **домовляється** про спільне рішення (хто лідер, який порядок записів у лог) попри збої. **Raft** (зрозуміліший за Paxos): leader election + log replication + safety. Лежить під etcd, Consul, Kafka (контролер), розподіленими БД. Працює, поки жива **більшість** (quorum, N/2+1) → стійкість до меншості збоїв.

**Реплікація — трейдофи:**
- **Leader-follower** (PostgreSQL): прості записи в лідера, читання з реплік. Лаг → stale reads. Failover при падінні лідера.
- **Multi-leader:** запис у кілька → конфлікти (треба resolution).
- **Leaderless (Dynamo/Cassandra):** quorum reads/writes (R+W>N → перетин), eventual consistency, гнучка доступність.

**Розподілені транзакції (блок 5, п.65):**
- **2PC (two-phase commit)** — координатор + prepare/commit. Атомарно, але **блокуюче** (координатор-SPOF, повисання при збої).
- **Saga** — ланцюг локальних транзакцій + **компенсуючі** дії при відкаті. Не атомарно, але стійко й масштабовано → стандарт для мікросервісів.

**Логічні годинники:** фізичні годинники розсинхронізовані → порядок подій визначають **Lamport timestamps / vector clocks** (причинність без єдиного годинника). Основа для causal consistency, виявлення конфліктів.

**Failure detection:** heartbeats + timeouts; **split-brain** (дві частини вважають себе головними при розриві) → лікується quorum/fencing. **Неможливо** відрізнити «вузол впав» від «вузол повільний» (FLP impossibility) → завжди компроміс timeout'ів.

**Senior-висновок:** розподілені системи — це **управління частковими відмовами**. Фундамент: «мережа ненадійна, годинники брешуть, вузли падають» → проєктуй з ідемпотентністю, таймаутами, retry+backoff, circuit breakers (блок 5), і **свідомо** обирай точку на спектрі consistency↔availability під бізнес-інваріант.

> ⚠️ Це оглядова мапа величезної дисципліни (Kleppmann, *DDIA* — еталон). Гарантії конкретних систем (Kafka, Cassandra) перевіряй у їх docs — деталі критичні й відрізняються.

## 101. Низькорівневі особливості мов і фреймворків

### L1 — Junior
**Аналогія — що під капотом авто.** Можна водити, не знаючи двигуна, але інженер розуміє, **чому** машина поводиться так: чому Python повільніший за C, чому NumPy швидкий, чому async масштабує I/O. Низькорівневі особливості — це **«чому»** за поведінкою.

### L2 — Middle
**Низькорівневі речі, що реально впливають (зведення ниток із усіх блоків):**
| Особливість | Наслідок | Блок |
|---|---|---|
| **Байт-код + інтерпретація** | Python повільніший за компільовані | 2, п.24 |
| **GIL** | потоки не паралелять CPU | 2/6, п.30/73 |
| **Refcount + GC** | overhead на посилання, паузи | 6, п.77 |
| **Усе — об'єкт `PyObject`** | пам'ять ~28 байт на int | 6, п.75 |
| **Дескриптори/MRO** | магія атрибутів, lookup-вартість | 8, п.98 |
| **C-розширення відпускають GIL** | NumPy паралелить, швидкий | 7/8, п.93/95 |
| **Локальність кешу** | масиви C > розкидані об'єкти | 1/7, п.3/93 |

```python
# Чому NumPy швидкий (низькорівнево): C-масив + SIMD + nogil
arr * 2          # один C-цикл vs мільйон інтерпретованих кроків
```

**Як фреймворки досягають швидкості (низькорівнево):**
- **Pydantic v2** — ядро на **Rust** (pydantic-core) → валідація в рази швидша (блок 6, п.74).
- **uvloop** — event loop на **libuv/Cython** (блок 6, п.73).
- **FastAPI** — на **introspection анотацій** (блок 2/6, п.27/81), не на рантайм-рефлексії щоразу.
- **ruff/uv** — на **Rust** замість Python (блок 6, п.83).

**Gotcha:** «низькорівневе знання» на інтерв'ю — не зубрити асемблер, а **пояснити причину поведінки**: чому список `O(1)` на append але `O(n)` на pop(0) (динамічний масив, блок 2 п.46), чому `is` працює для малих int (кеш, блок 2 п.28).

### L3 — Senior
**Спектр «близькості до заліза» (навіщо це Senior):**
1. **CPU/кеш:** локальність даних (cache lines ~64 байти) → чому суцільні масиви (NumPy) б'ють розкидані по heap об'єкти (блок 1, п.3). Cache miss коштує ~100 циклів.
2. **Пам'ять:** stack (дешево, авто) vs heap (дорого, GC); фрагментація; pymalloc (блок 6, п.75).
3. **Інтерпретація:** байт-код, eval loop, adaptive specialization (3.11+, блок 2 п.24).
4. **FFI / zero-copy:** `memoryview`, buffer protocol, `ctypes`/`cffi` — Python спілкується з C **без копіювання** даних (NumPy, БД-драйвери на C).

**Зведення: чому Python повільний і коли це неважливо:**
- Повільний через: інтерпретацію байт-коду, динамічну типізацію (lookup типу на кожну операцію), `PyObject`-overhead, refcount, GIL.
- **Неважливо**, коли bottleneck — I/O (БД/мережа домінують над CPU застосунку) або коли важке винесено в C (NumPy/Rust-ядра). Тому Python царює в web/ML/data: «клей» повільний, але «м'язи» (C/Rust) швидкі.

**Низькорівневі важелі Senior:**
- Гарячий CPU-код → **Cython/Rust-розширення/NumPy** (обхід інтерпретації+GIL).
- Пам'ять → `__slots__` (блок 6 п.79), генератори (блок 2 п.32), `array`/NumPy замість list of objects.
- Латентність → профілювати (`cProfile`, `py-spy`), знайти реальний low-level затик, не вгадувати.

**Трейдоф абстракції ↔ контроль:** Python обрав **високу абстракцію** (продуктивність розробки, безпека пам'яті) ціною контролю/швидкості. Senior знає, **де абстракція протікає** (performance-critical шлях) і **спускається** на рівень нижче точково — не переписує все на C, а лише hot path (правило: 80% часу в 20% коду).

> ⚠️ Низькорівневі деталі (adaptive interpreter, layout об'єктів, buffer protocol) — специфіка реалізації/версії CPython. Звіряй із dev-docs CPython перед дослівним цитуванням.

## 102. Способи та методи керування пам'яттю

### L1 — Junior
**Аналогія — прибирання в готелі.** Кімнату (пам'ять) треба звільнити, коли гість виїхав. Питання — **хто і коли** це робить:
- **Вручну** (C) — гість сам прибирає й здає ключ (легко забути → сміття/конфлікти).
- **Автоматично за лічильником** (Python) — щойно останній вийшов, прибирають.
- **Обхідник-прибиральник** (Java/Go) — періодично шукає порожні кімнати.

*(Пам'ять Python детально — блок 6, п.75/77.)*

### L2 — Middle
**Основні методи (спектр від ручного до автоматичного):**
| Метод | Хто/коли звільняє | Мови |
|---|---|---|
| **Manual** | програміст (`malloc`/`free`) | C |
| **RAII** | деструктор при виході зі scope | C++ |
| **Reference counting** | при refcount→0 (миттєво) | CPython, Swift |
| **Tracing GC** | збирач періодично (mark-sweep) | Java, Go, JS, C# |
| **Ownership** | компілятор (нема рантайм-GC) | Rust |

**Python (блок 6, п.77):** **refcount (основний)** + **generational cyclic GC** (для циклів).

```python
import gc
gc.collect()        # примусовий збір циклів
gc.disable()        # вимкнути циклічний GC (refcount лишається)
```

**Gotcha:** ручне керування (C) — джерело найгірших багів: **memory leak** (забув звільнити), **use-after-free** (звернувся до звільненого), **double free**. Автоматичне (Python/Java) прибирає цей клас багів ціною overhead/пауз.

### L3 — Senior
**Трейдофи методів (фундаментальний інженерний вибір):**

**1. Manual (C/malloc-free):**
- **+** Повний контроль, нуль overhead, детерміновано.
- **−** Цілий клас багів (leak/UAF/double-free), важко, небезпечно.

**2. Reference counting (CPython, блок 6 п.77):**
- **+** Детерміноване звільнення (миттєво при 0), пам'ять рано вільна, прості C-розширення.
- **−** Overhead на **кожну** операцію з посиланням (inc/dec), **не ловить цикли**, non-atomic → **причина GIL** (блок 2 п.30).

**3. Tracing GC (Java/Go) — варіанти алгоритмів:**
- **Mark-Sweep** — позначити досяжне, звільнити решту.
- **Generational** — молоді об'єкти збираються частіше (більшість вмирає молодими, блок 6 п.77).
- **Copying / Compacting** — переміщує живі об'єкти, усуває фрагментацію.
- **+** Ловить цикли, нема overhead на присвоєння, вільний паралелізм (нема GIL).
- **−** **Паузи** (stop-the-world), недетермінований момент звільнення, більше пам'яті (працює з запасом).

**4. RAII (C++):** звільнення прив'язане до **scope** (деструктор при виході) → детерміновано, без GC. Аналог у Python — **контекстні менеджери** (`with`, блок 2 п.36) для ресурсів (файли, з'єднання), бо `__del__` ненадійний.

**5. Ownership / borrow checker (Rust) — найцікавіше:** компілятор **на етапі компіляції** доводить, що кожне значення має одного власника, і вставляє звільнення автоматично → **нема рантайм-GC, нема overhead, нема leak/UAF**. Ціна — складніша модель для програміста (borrow checker «бореться» з тобою). Це чому Rust обрали для pydantic-core/ruff/polars (блок 6) — швидкість C + безпека без GC.

**Чому CPython обрав refcount+GC, а не tracing (трейдоф):** детермінованість (об'єкт вільний одразу — добре для ресурсів/кешу) + **простота C-API** (розширення роблять Py_INCREF/DECREF) важливіші за вартість GIL. Tracing-GC дав би вільний паралелізм, але ускладнив би C-розширення й прибрав детермінованість. Це історичне рішення, що тягне за собою GIL — і саме його намагається змінити free-threaded Python (PEP 703, блок 6 п.75).

**Практичні важелі в Python:**
- `with` для детермінованого звільнення ресурсів (не `__del__`).
- `gc.disable()` на короткоживучих процесах без циклів (прибрати паузи).
- `gc.freeze()` після старту (оптимізація copy-on-write у gunicorn-форках, блок 6 п.77).
- `__slots__`, генератори, `weakref` (розірвати цикли/кеш без утримання) — точкове керування пам'яттю.

> ⚠️ Деталі GC-алгоритмів (Java G1/ZGC, Go concurrent GC, CPython generational) специфічні й еволюціонують. Звіряй із docs конкретного рантайму.

## 63. Орієнтація в *nix, скрипти/автоматизація

### L1 — Junior
**Аналогія — майстерня з інструментами.** *nix (Linux/macOS) — це майстерня, де кожна команда — окремий інструмент (`ls`, `grep`, `cat`), а **pipe (`|`)** — конвеєр, що передає результат одного інструмента наступному. Замість рутини руками — пишеш **скрипт**, який робить це за тебе.

```bash
# «Знайди всі .log файли, відбери з помилками, порахуй»
find . -name "*.log" | xargs grep "ERROR" | wc -l
```

### L2 — Middle
**Базовий арсенал:**
| Категорія | Команди |
|---|---|
| Файли/навігація | `ls cd pwd cp mv rm mkdir find` |
| Текст | `cat less head tail grep sed awk sort uniq wc` |
| Процеси | `ps top htop kill` |
| Мережа | `curl wget ss netstat ssh scp` |
| Права | `chmod chown` |
| Пакети/диск | `apt/yum/brew df du` |

**Скрипт автоматизації (bash):**
```bash
#!/usr/bin/env bash
set -euo pipefail        # критично! строгий режим

BACKUP_DIR="/backups"
DB_NAME="mydb"
DATE=$(date +%Y-%m-%d)

pg_dump "$DB_NAME" | gzip > "$BACKUP_DIR/${DB_NAME}_${DATE}.sql.gz"
find "$BACKUP_DIR" -name "*.sql.gz" -mtime +7 -delete   # чистимо старі
echo "Backup done: ${DB_NAME}_${DATE}"
```

**`set -euo pipefail` — must have:**
- `-e` — падати при першій помилці.
- `-u` — помилка на невизначеній змінній.
- `-o pipefail` — помилка в pipe не «ковтається».

**Коли bash, коли Python:** короткі pipe з утиліт → bash. Логіка з умовами, парсингом JSON, API → **Python** (читабельніше, тестованіше). Правило: bash «розпух» більше ~50 рядків → переписуй на Python.

**Gotcha:** не цитований `$VAR` ламається на пробілах/спецсимволах → **завжди** `"$VAR"`. `rm -rf "$DIR/"` з порожнім `$DIR` = `rm -rf /` — тому `set -u` рятує життя.

### L3 — Senior
**Філософія Unix** (Doug McIlroy): «роби одне діло добре; програми, що працюють разом; текст як універсальний інтерфейс». Pipe — це **композиція** маленьких інструментів, кожен з яких — фільтр `stdin → stdout`. Це той самий принцип, що й функціональна композиція в коді (блок 1, п.2).

**Процеси й стрімінг:** `cat huge.log | grep X` не вантажить файл у пам'ять — дані течуть **потоково** через pipe-буфер ядра (`O(1)` пам'яті, як генератори в Python). Тому Unix-конвеєр обробляє файли, більші за RAM.

**Exit codes — основа автоматизації:** кожна команда повертає код (`0` = успіх, ≠0 = помилка), доступний у `$?`. На цьому тримаються `&&`, `||`, `set -e`, CI-пайплайни. Senior пише скрипти, що **коректно проганяють exit code** нагору (інакше CI «зелений» при реальній помилці).

**Трейдоф bash vs Python для автоматизації:** bash — нульовий старт, ідеальний для glue з утиліт ↔ погана обробка помилок, немає структур даних, крихкий на edge-cases. Python — тестованість, типи, бібліотеки ↔ потрібен інтерпретатор/venv. Для критичної автоматизації (деплой, бекапи) Senior тяжіє до Python/Ansible, лишаючи bash для тонкого glue.

> ⚠️ Поведінка `sed`/`awk`/`date` відрізняється на GNU (Linux) vs BSD (macOS). Скрипти для прода тестуй на цільовій ОС.

## 64. Що таке багатопотоковість?

### L1 — Junior
**Аналогія — кухар із кількома руками.** Багатопотоковість (multithreading) — коли програма робить **кілька справ ніби одночасно**: поки одна рука чекає, доки закипить вода (I/O), інша ріже овочі. Не простоюємо в очікуванні.

### L2 — Middle
Спершу — **три рівні «одночасності»** (часто плутають):
| Поняття | Що це |
|---|---|
| **Concurrency** (конкурентність) | задачі **чергуються**, прогресують разом (можливо на 1 ядрі) |
| **Parallelism** (паралелізм) | задачі **реально одночасно** на різних ядрах |
| **Multithreading** | кілька потоків у межах одного процесу, спільна пам'ять |

```python
from concurrent.futures import ThreadPoolExecutor
import requests

def fetch(url: str) -> int:
    return len(requests.get(url, timeout=5).content)

# Потоки добре для I/O — поки один чекає мережу, інші працюють
with ThreadPoolExecutor(max_workers=10) as ex:
    sizes = list(ex.map(fetch, urls))
```

**У Python — головне правило (через GIL, блок 2 п.30):**
- **I/O-bound** (мережа, диск, БД) → **threads** або **asyncio** ✅
- **CPU-bound** (обчислення) → **processes** (`multiprocessing`), бо GIL не дає потокам рахувати паралельно.

**Проблеми спільної пам'яті:** race conditions, deadlocks. Захист — **locks**:
```python
import threading
lock = threading.Lock()
counter = 0
def increment():
    global counter
    with lock:              # критична секція — лише один потік
        counter += 1
```

**Gotcha:** `counter += 1` **не атомарна** (read-modify-write) → без lock дві нитки можуть загубити інкремент (race condition).

### L3 — Senior
**GIL** дозволяє лише одному потоку виконувати байткод одночасно → потоки в CPython дають **конкурентність, але не паралелізм CPU**. Виграш на I/O — бо блокуючі syscalls **відпускають GIL** (нитка чекає мережу — інші працюють).

**Три моделі конкурентності в Python і трейдофи:**
| Модель | Для чого | Ціна |
|---|---|---|
| **threading** | I/O, мало задач | GIL, race conditions, overhead потоків ОС |
| **multiprocessing** | CPU-bound | дорогий старт, IPC (pickle), окрема пам'ять |
| **asyncio** | масовий I/O (10k+ з'єднань) | кооперативність, «зараза» async, складніший дебаг |

**Asyncio vs threads для I/O:** потоки — **прееемптивні** (ОС перемикає будь-коли → race conditions, але код «звичайний»). Asyncio — **кооперативний** (перемикання лише на `await` → менше гонок, але один блокуючий виклик вішає весь loop). Потоки масштабуються до сотень (кожен — стек ~MB у ядрі); asyncio — до десятків тисяч (корутини дешеві).

**Класичні помилки конкурентності:**
- **Race condition** — результат залежить від таймінгу (немає синхронізації спільного стану).
- **Deadlock** — потоки взаємно чекають замки (A тримає L1 чекає L2, B навпаки). Профілактика: завжди бери замки в одному порядку.
- **Starvation / livelock** — потік ніколи не отримує ресурс.

**Сучасне:** Python 3.13 free-threaded (PEP 703, no-GIL) обіцяє справжній паралелізм потоків — але код, не готовий до гонок, зламається (раніше GIL «випадково» захищав).

> ⚠️ Статус free-threaded build і реальна атомарність операцій — деталі версії CPython; звіряй з release notes.

## 65. Що таке архітектура вебсервісів?

### L1 — Junior
**Аналогія — пристрій ресторану.** Є зал (інтерфейс), офіціанти (API), кухня (бізнес-логіка), склад (база даних). Архітектура — це **як ти розділив систему на частини й як вони спілкуються**, щоб усе не перетворилось на хаос.

### L2 — Middle
**Шарувата (layered) архітектура — класика:**
```
┌─────────────────────────────┐
│ Presentation (API / routes)  │  ← приймає HTTP, валідує вхід
├─────────────────────────────┤
│ Business Logic (services)    │  ← правила, юзкейси
├─────────────────────────────┤
│ Data Access (repositories)   │  ← робота з БД (ORM/SQL)
├─────────────────────────────┤
│ Database                     │
└─────────────────────────────┘
```
Кожен шар знає лише про сусідній знизу → **separation of concerns** (блок 1, п.2).

**Monolith vs Microservices:**
| | Monolith | Microservices |
|---|---|---|
| Структура | один деплой | багато незалежних сервісів |
| Зв'язок | виклики функцій | мережа (HTTP/gRPC/черги) |
| Деплой | весь застосунок | кожен окремо |
| Складність | проста на старті | складна (розподілена) |
| Масштабування | весь моноліт | окремі сервіси |

**Best practice:** **починай з моноліту** (модульного), розбивай на мікросервіси **коли є реальний біль** (команди впираються одна в одну, частини масштабуються по-різному). «Microservices first» для стартапу — часто передчасна оптимізація.

**Gotcha:** мікросервіси не «спрощують» — вони **обмінюють** складність коду на складність **розподіленої системи** (мережеві збої, eventual consistency, distributed tracing). Conway's Law: архітектура відображає структуру команд.

### L3 — Senior
**Clean / Hexagonal Architecture (Ports & Adapters):** бізнес-логіка в центрі **не залежить** від фреймворку/БД/HTTP. Залежності спрямовані **всередину** (Dependency Inversion, блок 1 п.2): домен визначає інтерфейси (ports), а БД/HTTP — це adapters, що їх реалізують. Виграш: домен тестується без БД/мережі, інфраструктуру можна замінити.

**Stateless як фундамент масштабування:** app-сервери мають бути **stateless** (стан — у БД/Redis), тоді load balancer кидає запит на будь-який інстанс, і горизонтальне масштабування тривіальне (блок 1, п.8). Стан у пам'яті процесу (сесія в RAM) ламає це.

**Ключові архітектурні трейдофи:**
- **Coupling ↔ autonomy:** мікросервіси автономні, але мережеві виклики ненадійні (потрібні retry, circuit breaker, timeout).
- **Consistency ↔ availability** (CAP): розподілені БД змушують вибирати (блок 1, п.12). Мікросервіси часто живуть з eventual consistency + saga-патернами замість розподілених транзакцій.
- **Синхронна ↔ асинхронна комунікація:** HTTP (просто, але tight coupling, каскадні збої) vs черги/події (decoupling, стійкість, але складніше відстежувати, eventual consistency).

Senior обирає архітектуру під **організацію й навантаження**, а не за модою. Модульний моноліт із чіткими межами часто кращий за передчасні мікросервіси.

## 66. Як працює сучасний навантажений вебзастосунок (Twitter/Instagram)

### L1 — Junior
**Аналогія — велика пошта в годину пік.** Один поштар не впорається з мільйонами листів. Тому: багато поштарів (сервери), сортувальні центри (балансувальники), готові копії популярного ближче до людей (кеш/CDN), а складні задачі — в чергу на потім (фонові воркери).

### L2 — Middle
**Приблизна архітектура (намалюй це на співбесіді):**
```
[Клієнти]
    │
[CDN] ──── статика (фото, відео, JS) ближче до користувача
    │
[DNS / Load Balancer]  ← розподіляє трафік
    │
┌───┴────┬─────────┐
│ App 1  │ App 2 ...│  ← STATELESS app-сервери (горизонтальне масштабування)
└───┬────┴─────────┘
    │
    ├──[Cache: Redis]   ← гарячі дані, сесії (O(1), розвантажує БД)
    │
    ├──[Database]
    │     ├─ Primary (запис)
    │     └─ Read Replicas (читання) ← read/write split
    │     └─ Sharding (розбиття за user_id)
    │
    ├──[Message Queue: Kafka/RabbitMQ]  ← асинхронні задачі
    │       └──[Workers: Celery]  ← фон: розсилки, обробка фото, стрічка
    │
    ├──[Object Storage: S3]  ← фото/відео
    └──[Search: Elasticsearch]  ← пошук
```

**Ключові ідеї масштабу:**
- **CDN** — статику віддають edge-сервери поряд із юзером (низька латентність).
- **Load Balancer** — розкидає трафік по stateless app-серверах.
- **Кеш (Redis)** — гарячі дані з пам'яті (`O(1)`), щоб не бити БД щоразу.
- **Read replicas** — читання масштабується реплікацією (90% трафіку — читання).
- **Sharding** — дані розбиті по серверах (за `user_id`), бо одна БД не вміщає все.
- **Черги + воркери** — важке (обробка відео, розсилка) виноситься з request-response у фон.

**Gotcha — генерація стрічки (timeline), класичне питання:**
- **Fan-out on write** (push): при публікації твіт одразу розкладається в кеші стрічок усіх підписників. Читання стрічки — швидке (`O(1)`), але запис «зірки» з 100М фоловерів — дорогий.
- **Fan-out on read** (pull): стрічка збирається в момент запиту. Дешевий запис, дорогі читання.
- **Гібрид** (реальність Twitter): push для звичайних, pull для «зірок» (celebrities) → змішують при читанні.

### L3 — Senior
**Чому read replicas + кеш:** соцмережі мають співвідношення read:write ~ 100:1. Тому масштабують **читання** (репліки, кеш), а запис лишається вузьким місцем (один primary). Звідси sharding для масштабування **запису**.

**Реплікація — трейдоф consistency:** асинхронна реплікація дає **replication lag** → читач з репліки може побачити застарілі дані (read-your-writes проблема: запостив — не бачу свій пост). Рішення: читати власні свіжі дані з primary, решту — з реплік.

**Кеш — найскладніше («there are only two hard things…»):**
- **Інвалідація:** як оновити/викинути застаріле. Стратегії: TTL, write-through, cache-aside.
- **Thundering herd / cache stampede:** кеш протух → тисячі запитів б'ють у БД одночасно. Захист: locks, request coalescing, stale-while-revalidate.
- **Hot keys:** один популярний ключ перевантажує один Redis-вузол.

**Sharding — трейдофи:** розбиття за `user_id` масштабує запис, але ламає крос-шардові запити (JOIN між шардами дорогий) і ребалансування при додаванні вузлів (consistent hashing пом'якшує). Розподілені транзакції майже неможливі → eventual consistency + saga.

**Backpressure і черги:** черга (Kafka) **поглинає сплески** навантаження й **розв'язує** продюсера від консюмера. Якщо воркери не встигають — черга росте (буфер), а не падає система. Це фундамент стійкості до піків.

**CAP на практиці:** при мережевому розриві соцмережі обирають **availability** (краще показати трохи застарілу стрічку, ніж помилку) → AP-системи з eventual consistency. Гроші/інвентар обирають consistency (CP).

> ⚠️ Це **спрощена** еталонна архітектура для співбесіди. Реальні Twitter/Instagram значно складніші й змінюються; деталі (Manhattan, Haystack тощо) — окрема тема. Не видавай це за точний пристрій конкретної компанії.

## 67. Що потрібно для сервісу середнього розміру (redis/celery/кеш/логи/метрики)

### L1 — Junior
**Аналогія — обладнання кафе, що виросло.** Поки кафе мале — вистачає однієї людини. Виросло — потрібні: холодильник для готового (кеш), кухар на замовлення «з собою» на потім (черга задач), журнал скарг (логи), лічильники відвідувачів (метрики). Це **операційна інфраструктура** дорослого сервісу.

### L2 — Middle
**Стек зрілого сервісу:**
| Компонент | Інструмент | Навіщо |
|---|---|---|
| **Кеш** | Redis / Memcached | розвантажити БД, гарячі дані `O(1)` |
| **Черга задач** | Celery + Redis/RabbitMQ | фонові задачі (email, звіти, обробка) |
| **Логування** | structured logs (JSON) → ELK / Loki | дебаг, аудит, розслідування інцидентів |
| **Метрики** | Prometheus + Grafana | RPS, латентність, помилки, ресурси |
| **Трейсинг** | OpenTelemetry / Jaeger | шлях запиту через сервіси |
| **Алерти** | Alertmanager / PagerDuty | сповіщення при проблемах |
| **Error tracking** | Sentry | агрегація винятків зі стектрейсами |

**Celery — винесення важкого з request:**
```python
from celery import Celery
app = Celery("tasks", broker="redis://localhost:6379")

@app.task
def send_email(user_id: int) -> None:
    ...                              # повільне — у фоні

# У хендлері: не блокуємо відповідь користувачу
send_email.delay(user.id)            # ставимо в чергу, відповідаємо одразу
```

**Кешування (cache-aside):**
```python
def get_user(user_id: int) -> dict:
    if cached := redis.get(f"user:{user_id}"):
        return json.loads(cached)            # hit
    user = db.fetch_user(user_id)            # miss → БД
    redis.setex(f"user:{user_id}", 300, json.dumps(user))  # TTL 5 хв
    return user
```

**Gotcha:** **не логуй секрети** (паролі, токени, карти) — це витік і порушення (GDPR/PCI). Логуй **структуровано** (JSON з полями), а не рядки — інакше неможливо шукати/агрегувати.

### L3 — Senior
**Три стовпи observability:** **Logs** (що сталося — дискретні події), **Metrics** (скільки/як швидко — числові ряди), **Traces** (де саме й чому повільно — шлях запиту). Разом дають відповідь не лише «що зламалось», а «**чому й де**». Логи дорогі в обсязі, метрики дешеві й агреговані, трейси — для діагностики латентності в розподілених системах.

**Метрики, що моніторять (RED / USE):**
- **RED** (для сервісів): **R**ate (RPS), **E**rrors (% помилок), **D**uration (латентність, p50/p95/**p99**).
- **USE** (для ресурсів): **U**tilization, **S**aturation, **E**rrors.
- **p99, не середнє:** середня латентність бреше — 1% повільних запитів (p99) б'є по UX найбільше. Senior дивиться на хвости розподілу.

**Celery / черги — трейдофи:** виносять важке з request-response (швидка відповідь користувачу) ↔ додають складність (брокер, моніторинг черг, idempotency задач, retry, dead-letter queue). Задача може виконатись **двічі** (at-least-once delivery) → задачі мають бути **ідемпотентними**.

**Кеш — інвалідація:** TTL (просто, але stale-вікно) vs write-through (свіжо, але складніше) vs event-based invalidation. Трейдоф consistency ↔ складність. Кеш покращує латентність і розвантажує БД ціною ризику застарілих даних і нової точки відмови.

**Graceful degradation:** зрілий сервіс **деградує, а не падає** — Redis недоступний → читаємо з БД (повільніше, але працює); рекомендації впали → показуємо популярне. Circuit breaker запобігає каскадним збоям.

> ⚠️ Конкретні гарантії доставки Celery/брокерів і семантика retry залежать від конфігу (acks_late, broker). Звіряй з docs перед побудовою на ідемпотентність.

## 68. Як написати, задеплоїти й підтримувати (мікро)сервіс?

### L1 — Junior
**Аналогія — життєвий цикл страви:** придумати рецепт (написати код) → приготувати на тестовій кухні (CI/тести) → подати в зал (деплой) → стежити за відгуками й поповнювати продукти (моніторинг, підтримка).

### L2 — Middle
**Повний цикл:**

**1. Написати:**
```python
# FastAPI-мікросервіс із health-check
from fastapi import FastAPI
app = FastAPI()

@app.get("/health")          # для load balancer / k8s probes
def health() -> dict:
    return {"status": "ok"}
```
+ тести (`pytest`), типи (`mypy`), лінт (`ruff`), конфіг через **env vars** (12-factor).

**2. Контейнеризувати (Docker):**
```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY pyproject.toml uv.lock ./
RUN pip install uv && uv sync --frozen
COPY . .
CMD ["uv", "run", "uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

**3. CI/CD (GitHub Actions):**
```
push → lint + test + build image → push to registry → deploy
```

**4. Деплой:** Kubernetes / ECS / Cloud Run. Стратегії: **rolling update**, **blue-green**, **canary**.

**5. Підтримувати:** health-checks, метрики (п.67), логи, алерти, on-call.

**Gotcha:** **конфіг — через env vars, не хардкод** (12-factor app). Секрети — у secret manager (Vault/K8s secrets), **не в git**. Образ має бути **immutable** — той самий артефакт проходить dev→staging→prod (не перезбирати на кожному середовищі).

### L3 — Senior
**12-Factor App** — канон для cloud-native: конфіг в оточенні, stateless процеси, логи в stdout (не в файли), dev/prod parity, явні залежності. Це робить сервіс **переносним і масштабованим**.

**Deployment-стратегії (трейдофи):**
| Стратегія | Як | Трейдоф |
|---|---|---|
| **Rolling** | поступова заміна інстансів | простий, але дві версії co-existують |
| **Blue-Green** | паралельне середовище, перемикання | миттєвий rollback, але ×2 ресурси |
| **Canary** | 5% трафіку на нову версію | раннє виявлення проблем, складніше |

**Backward compatibility — критично для мікросервісів:** rolling deploy і незалежні релізи означають, що **старі й нові версії працюють одночасно**. Тому зміни API/схеми БД мають бути **сумісними назад** (expand-contract: спочатку додай нове, потім міграй, потім прибери старе — ніколи не ламай за один реліз).

**Health checks розрізняй:** **liveness** (чи живий процес — перезапустити, якщо ні) vs **readiness** (чи готовий приймати трафік — не слати запити, поки прогрівається/немає БД). Плутання валить деплой.

**SLO/SLA/error budget:** Senior визначає **SLO** (ціль надійності, напр. 99.9%) → error budget (дозволені 0.1% збоїв) керує балансом «нові фічі vs стабільність». Вичерпали бюджет — фокус на надійність.

**Трейдоф мікросервіс vs моноліт у підтримці:** незалежний деплой і масштабування ↔ розподілений дебаг, distributed tracing, версіонування контрактів, мережеві збої. Не починай із мікросервісів без потреби (п.65).

> ⚠️ Конкретика k8s probes/деплою залежить від платформи. Це загальна карта; звіряй із docs твого оркестратора.

## 69. Як масштабувати API?

### L1 — Junior
**Аналогія — черга в банку.** Якщо черга росте: 1) посадити більше касирів (більше серверів), 2) дати касиру потужніший комп'ютер (більший сервер), 3) готувати популярні документи наперед (кеш). Масштабування — це **прибрати вузьке місце**.

### L2 — Middle
**Вертикальне vs горизонтальне:**
| | Вертикальне (scale up) | Горизонтальне (scale out) |
|---|---|---|
| Як | потужніший сервер (CPU/RAM) | більше серверів |
| Межа | стеля заліза | майже безмежно |
| Складність | проста | потрібен LB + stateless |
| Відмовостійкість | SPOF | є резерв |

**Карта масштабування API:**
1. **Stateless app + Load Balancer** → горизонтальне масштабування (основа).
2. **Кешування** (Redis, CDN, HTTP-кеш) → менше роботи на запит.
3. **БД:** read replicas (читання), sharding (запис), індекси (п.48), connection pooling.
4. **Async/черги** → виносимо важке з request (п.67).
5. **Rate limiting** → захист від перевантаження/abuse.
6. **Пагінація** → не віддавати тисячі рядків за раз.

**Gotcha:** **БД — найчастіше вузьке місце**, не app-сервери. Додавати app-інстанси безглуздо, якщо вони всі впираються в одну перевантажену БД. Спершу профілюй, **де** затик (часто — повільний запит без індексу або N+1).

### L3 — Senior
**Спочатку виміряй, потім масштабуй:** «premature scaling» — корінь зла. Профілюй (p99 латентності, slow query log, APM) → знайди реальний bottleneck → усунь точково. Часто один індекс або кеш дає більше, ніж подвоєння серверів.

**Закон масштабованості (Universal Scalability Law / Amdahl):** додавання вузлів дає **спадну віддачу** через **серіалізацію** (спільні ресурси) і **когерентність** (координація між вузлами). Тому stateless + мінімум спільного стану — ключ; спільна БД лишається точкою серіалізації.

**Stateless — передумова горизонтального масштабування:** будь-який стан у пам'яті інстанса (локальна сесія, кеш у процесі) ламає LB-розподіл. Стан → Redis/БД. Sticky sessions — милиця, що обмежує балансування.

**Rate limiting — алгоритми (трейдофи):** token bucket (дозволяє сплески), leaky bucket (згладжує), sliding window (точніше, дорожче). У розподіленому API ліміт зберігають у Redis (спільний лічильник). Захищає від abuse і каскадних збоїв.

**Трейдофи масштабування:**
- Кеш — швидкість ↔ stale-дані + інвалідація (п.66).
- Sharding — масштабує запис ↔ ламає крос-шардові запити/транзакції.
- Async — пропускна здатність ↔ eventual consistency + складність.
- Horizontal — еластичність ↔ розподілена складність (LB, service discovery, distributed state).

Senior масштабує **за даними профілювання**, ітеративно усуваючи поточний bottleneck, а не «про запас».

## 70. Як проводити Code Review?

### L1 — Junior
**Аналогія — вичитка тексту колегою перед публікацією.** Свіже око ловить помилки, які автор «замилив». Code review — взаємна перевірка коду перед злиттям: ловимо баги рано, ділимось знаннями, тримаємо єдиний стиль.

### L2 — Middle
**На що дивитись (пріоритет згори вниз):**
1. **Коректність** — чи робить код те, що треба? Edge cases, помилки.
2. **Безпека** — SQLi/XSS, секрети в коді, валідація вводу (блок 1, п.11).
3. **Тести** — є? покривають логіку й краї?
4. **Читабельність** — зрозуміють інші? імена, структура.
5. **Дизайн** — чи вписується в архітектуру, чи не дублює.
6. **Продуктивність** — N+1, зайві цикли `O(n²)` (лише якщо реально важливо).
7. **Стиль** — на це є лінтери, не витрачай review.

**Best practices автору:**
- **Малі PR** (<400 рядків) — великі рев'ю погано (увага падає).
- Зрозумілий опис: *що, навіщо, як тестувалось*.
- Self-review перед надсиланням.

**Best practices рев'юеру:**
- **Коментуй код, не людину**: «цей запит без індексу» замість «ти не подумав».
- Розрізняй **блокери** і **nitpicks** (познач: `nit:` — необов'язкове).
- Хвали хороше, не лише критикуй.
- Питання замість наказів: «а що, якщо тут прийде `None`?»

**Gotcha:** автоматизуй усе, що можна (лінт, формат, типи, тести — в CI), щоб **рев'ю було про логіку й дизайн**, а не про пробіли й крапки з комою. Суперечки про стиль = марнування на боті, яке мав робити `ruff`/`black`.

### L3 — Senior
**Code review — це передусім про людей і знання, не лише про баги.** Цілі: (1) якість, (2) **поширення знань** (junior учиться, bus factor росте), (3) колективна відповідальність за код, (4) консистентність кодбази. Дослідження (SmartBear, Google) показують: review ловить дефекти **дешевше**, ніж QA/прод, але ефективність падає після ~400 рядків / 60 хв.

**Що Senior шукає, чого не бачить молодший рев'юер:**
- **Що НЕ написано:** відсутній edge case, неопрацьована помилка, гонка даних, відсутній тест на регрес.
- **Системні наслідки:** як зміна впливає на інші сервіси, зворотна сумісність API/схеми (п.68), міграції без даунтайму.
- **Приховані трейдофи:** передчасна абстракція (over-engineering), tight coupling, порушення меж шарів.

**Культурні трейдофи:**
- **Строгість ↔ швидкість:** надто прискіпливе рев'ю гальмує доставку й демотивує; надто м'яке — пропускає борг. Баланс через чіткі стандарти + автоматизацію.
- **Блокуюче ↔ неблокуюче:** не блокуй PR на суб'єктивних дрібницях (`nit:`); блокуй на коректності/безпеці.
- **Async ↔ pair:** для складного — краще парне програмування/дзвінок, ніж 30 коментарів туди-сюди.

Google's Code Review standard: **апрувити, коли PR покращує здоров'я кодбази в цілому**, навіть якщо не ідеальний — «не дай ідеальному стати ворогом кращого». Це запобігає нескінченним циклам.

## 71. Що таке Абстрактна Фабрика, як реалізувати й навіщо?

### L1 — Junior
**Аналогія — меблева фабрика під стиль.** Замовляєш «вікторіанський набір» — отримуєш вікторіанські стілець, стіл, диван, що пасують один одному. Замовляєш «модерн» — узгоджений набір у стилі модерн. **Abstract Factory** створює **сімейства пов'язаних об'єктів**, гарантуючи їх сумісність, не вказуючи конкретні класи.

### L2 — Middle
**Abstract Factory** — creational-патерн (GoF): інтерфейс для створення **сімейства** об'єктів без прив'язки до конкретних класів.

```python
from abc import ABC, abstractmethod

# --- Продукти (інтерфейси) ---
class Button(ABC):
    @abstractmethod
    def render(self) -> str: ...

class Checkbox(ABC):
    @abstractmethod
    def render(self) -> str: ...

# --- Конкретні продукти (два сімейства) ---
class WinButton(Button):
    def render(self) -> str: return "[Windows Button]"
class WinCheckbox(Checkbox):
    def render(self) -> str: return "[Windows Checkbox]"

class MacButton(Button):
    def render(self) -> str: return "(Mac Button)"
class MacCheckbox(Checkbox):
    def render(self) -> str: return "(Mac Checkbox)"

# --- Абстрактна фабрика ---
class GUIFactory(ABC):
    @abstractmethod
    def create_button(self) -> Button: ...
    @abstractmethod
    def create_checkbox(self) -> Checkbox: ...

class WinFactory(GUIFactory):
    def create_button(self) -> Button: return WinButton()
    def create_checkbox(self) -> Checkbox: return WinCheckbox()

class MacFactory(GUIFactory):
    def create_button(self) -> Button: return MacButton()
    def create_checkbox(self) -> Checkbox: return MacCheckbox()

# --- Клієнт працює лише з абстракціями ---
def build_ui(factory: GUIFactory) -> None:
    print(factory.create_button().render())
    print(factory.create_checkbox().render())

build_ui(WinFactory())   # узгоджене сімейство Windows
build_ui(MacFactory())   # узгоджене сімейство Mac
```

**Навіщо:**
- Гарантувати, що об'єкти **сумісні** (не змішаєш Win-кнопку з Mac-чекбоксом).
- Легко додати нове сімейство (Linux) — нова фабрика, клієнт не змінюється (**Open/Closed**).
- Ізолювати клієнта від конкретних класів.

**Gotcha — Factory Method vs Abstract Factory:** Factory Method створює **один** продукт (один метод), Abstract Factory — **сімейство пов'язаних** (кілька методів). Часто плутають.

### L3 — Senior
Abstract Factory реалізує **Dependency Inversion** (блок 1, п.2): клієнт залежить від абстракції `GUIFactory`, а не від `WinButton`. Додати Linux-сімейство — `O(1)` зміна (нова фабрика), клієнтський код **не чіпаємо** (Open/Closed).

**Трейдоф — і чому в Python часто простіше:** GoF-патерн виник у Java/C++, де нема first-class functions і duck typing. У Python те саме часто роблять **легше**:
- Фабрика — просто **функція** або `dict` маппінг (`{"win": WinFactory, "mac": MacFactory}[os]`).
- Duck typing прибирає потребу в явних ABC-інтерфейсах (хоча `Protocol`/ABC корисні для mypy).

```python
# Pythonic-варіант — реєстр замість ієрархії класів
FACTORIES = {"windows": WinFactory, "mac": MacFactory}
factory = FACTORIES[platform]()
```

**Коли патерн виправданий:** реально кілька **взаємозамінних сімейств** з кількома продуктами кожне (крос-платформенний UI, різні БД-бекенди, dev/prod провайдери). Коли сімейство одне або продукт один — це **over-engineering** (patternitis, блок 2 п.16). Класична пастка: розгорнути повну ієрархію фабрик там, де вистачило б функції чи `dict`.

**Трейдоф загалом:** гнучкість і розширюваність (нові сімейства без зміни клієнта) ↔ багато класів/бойлерплейту, рівень непрямості, складніше читати. Вводь, коли **вже є** потреба в кількох сімействах, а не «про запас» (YAGNI).

## 72. Що таке цикломатична складність?

### L1 — Junior
**Аналогія — кількість розвилок на дорозі.** Чим більше `if`, `for`, `while`, `and/or` у функції — тим більше різних шляхів, якими може піти виконання, тим важче все це втримати в голові й протестувати. **Цикломатична складність** рахує ці «розвилки».

```python
def f(x):           # складність 1 (один шлях)
    return x + 1

def g(x):           # складність 2 (два шляхи: if / else)
    if x > 0:
        return "pos"
    return "neg"
```

### L2 — Middle
**Cyclomatic Complexity (CC)** — метрика Томаса Маккейба: кількість **незалежних шляхів** через код. Практична формула: **CC = 1 + кількість точок розгалуження**.

Точки, що +1: `if`, `elif`, `for`, `while`, `and`, `or`, `except`, `case`, тернарний.

```python
def categorize(user):                  # старт: 1
    if user.age < 18:                  # +1 = 2
        return "minor"
    elif user.age < 65:                # +1 = 3
        if user.is_premium and user.active:   # +2 (and) = 5
            return "premium_adult"
        return "adult"
    else:                              # else не додає
        return "senior"
# CC = 5 → 5 незалежних шляхів → мінімум 5 тестів для повного покриття
```

**Орієнтири (McCabe):**
| CC | Оцінка |
|---|---|
| 1–5 | проста, легко тестувати ✅ |
| 6–10 | помірна, прийнятно |
| 11–20 | складна, рефакторити |
| 20+ | дуже ризикова, важко тестувати/підтримувати 🔴 |

**Інструменти Python:** `radon cc file.py`, `ruff` (правило `C901` — `max-complexity`), `flake8-mccabe`.

**Gotcha:** CC = **нижня межа кількості тест-кейсів** для покриття всіх гілок. Висока CC → багато шляхів → багато тестів → вища ймовірність багів. Тому її тримають у CI як «запобіжник».

### L3 — Senior
**Формальне визначення (теорія графів):** CC = `E − N + 2P`, де `E` — ребра графа потоку керування (control flow graph), `N` — вузли, `P` — кількість зв'язних компонентів (зазвичай 1). Метрика вимірює кількість **лінійно незалежних шляхів** у CFG.

**Чому це важливо інженерно:**
- **Тестованість:** CC = мінімальна кількість тестів для **branch coverage** (покрити кожну гілку).
- **Підтримуваність:** висока CC корелює з дефектами (емпірично) — більше шляхів = більше місць для помилки й важче утримати в робочій пам'яті (людина тримає ~7±2 елементи).
- **Зниження CC** = розбити функцію (extract method), замінити вкладені `if` на early return / guard clauses, поліморфізм/`dict`-диспетчеризацію замість довгих `if/elif`.

**Трейдоф і критика метрики:**
- CC — **необхідна, але не достатня**: проста `match/case` на 15 варіантів має високу CC, але читається легко; хитра функція з CC=4 і прихованими побічними ефектами може бути гіршою. Метрика не бачить **семантичної** складності.
- **Cognitive Complexity** (SonarSource) — сучасна альтернатива: штрафує **вкладеність** (nested if важчий за послідовні), краще відображає реальну важкість читання для людини.
- Сліпе гнатися за низькою CC → штучне дроблення на крихітні функції, що **погіршує** читабельність (over-fragmentation).

Senior використовує CC як **сигнал** («ця функція розрослась — глянь»), а не як догму. Поєднує з code review (п.70): метрика підсвічує кандидатів на рефакторинг, рішення — за людиною.

> ⚠️ Різні інструменти рахують CC трохи по-різному (чи рахувати `and/or`, `else`, `case`). Звіряй методологію конкретного тулу (radon vs mccabe vs Sonar).

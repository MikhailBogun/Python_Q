## 24. Python — мова, що інтерпретується чи компілюється?

### L1 — Junior
**Аналогія:** перекладач на переговорах. Компільована мова (C) — це коли всю книгу переклали наперед і віддали готовий том. Інтерпретована — перекладач сидить поруч і перекладає речення за реченням «на льоту».

**Правильна відповідь:** Python — **і те, і те**. Спочатку код **компілюється** в проміжний **байт-код**, який потім **інтерпретується** віртуальною машиною.

### L2 — Middle
Точна послідовність у **CPython** (стандартна реалізація):
1. Вихідний `.py` → **компілюється** в **байт-код** (інструкції для PVM).
2. Байт-код кешується у `__pycache__/*.pyc` (щоб не компілювати щоразу).
3. **PVM (Python Virtual Machine)** інтерпретує байт-код інструкція за інструкцією.

```python
import dis
def add(a, b): return a + b
dis.dis(add)   # покаже байт-код: LOAD_FAST, BINARY_OP, RETURN_VALUE
```

**Gotcha:** часта помилка на інтерв'ю — назвати Python «чисто інтерпретованою». Коректно: «компілюється в байт-код, який виконується інтерпретатором CPython». `.pyc` — це і є той скомпільований байт-код, а не машинний код.

### L3 — Senior
**Реалізація ≠ мова.** «Python» — це специфікація; компіляція/інтерпретація залежить від реалізації:
- **CPython** — компіляція в байт-код + інтерпретація (eval loop у `ceval.c`).
- **PyPy** — **JIT** (Just-In-Time): «гарячі» ділянки компілюються в нативний машинний код у рантаймі → у рази швидше на циклах.
- **Cython / Nuitka** — компіляція в C/нативний код заздалегідь.

**Трейдоф байт-коду:** переносимість і простота реалізації ↔ швидкість. Байт-код не прив'язаний до CPU, але інтерпретація додає overhead на кожну інструкцію.

**Сучасний контекст:** CPython 3.11+ отримав **adaptive specializing interpreter** (PEP 659, «спеціалізація» гарячих інструкцій) і прискорення; обговорюється/розвивається JIT.

> ⚠️ Деталі adaptive interpreter та статус вбудованого JIT швидко змінюються між версіями — звіряй з release notes конкретної версії CPython.

## 25. Змінні (mutable) та незмінні (immutable) типи

### L1 — Junior
**Аналогія:**
- **Immutable (незмінні)** — написане кулькою на папері. Хочеш «змінити» — береш новий аркуш.
- **Mutable (змінні)** — написане на дошці, стираєш і дописуєш на тій самій дошці.

| Immutable | Mutable |
|---|---|
| `int, float, bool, complex` | `list` |
| `str, bytes` | `dict` |
| `tuple, frozenset` | `set, bytearray` |
| `None` | свої класи (за замовч.) |

### L2 — Middle
```python
x = 5
print(id(x))
x += 1          # створюється НОВИЙ об'єкт, не змінюється старий
print(id(x))    # інший id

lst = [1, 2]
print(id(lst))
lst.append(3)   # той самий об'єкт мутується
print(id(lst))  # ТОЙ САМИЙ id
```

**Класичний gotcha — mutable default argument:**
```python
def bad(item, target: list = []):   # ❌ список створюється ОДИН раз
    target.append(item)
    return target

bad(1)  # [1]
bad(2)  # [1, 2]  ← сюрприз! спільний список

def good(item, target: list | None = None):  # ✅
    if target is None:
        target = []
    target.append(item)
    return target
```

**Чому важливо:** immutable можна безпечно шарити (немає побічних ефектів), вони **хешовані** → можуть бути ключами dict / елементами set. Mutable — ні.

### L3 — Senior
**Під капотом:** змінна в Python — це **ім'я-посилання** на об'єкт у купі (heap), а не «комірка пам'яті». Присвоєння — це **rebinding** імені, а не зміна значення.

- **Immutable** дозволяють **кешування/інтернування** (small ints −5..256, інтерн рядків) — безпечно ділити один об'єкт.
- **Hashability:** `__hash__` має бути стабільним протягом життя об'єкта. Mutable типи навмисно `__hash__ = None`, бо зміна вмісту зламала б їх позицію в хеш-таблиці.

**Трейдоф:** immutability дає **thread-safety** (нічого не змінюється — нема гонок) і передбачуваність ↔ ціна — алокація нових об'єктів при кожній «зміні» (тому конкатенація рядків у циклі — `O(n²)`, краще `"".join(...)`).

**Підступ із `tuple`:** tuple незмінний, але якщо містить mutable елемент — вміст того елемента можна змінити, і тоді `tuple` стане нехешованим у момент хешування:
```python
t = (1, [2, 3])
t[1].append(4)     # дозволено! t = (1, [2, 3, 4])
hash(t)            # TypeError: unhashable
```

## 26. Область видимості змінних (scope)

### L1 — Junior
**Аналогія — пошук речі вдома:** спершу шукаєш у своїй кімнаті (Local), потім у кімнаті батьків (Enclosing), потім у спільній вітальні (Global), нарешті — у коморі будинку (Built-in). Знайшов — зупинився.

Це правило **LEGB**: **L**ocal → **E**nclosing → **G**lobal → **B**uilt-in.

### L2 — Middle
```python
x = "global"

def outer():
    x = "enclosing"
    def inner():
        x = "local"
        print(x)        # local
    inner()
    print(x)            # enclosing

outer()
print(x)                # global
print(len)              # built-in
```

**Ключові слова:**
```python
counter = 0
def increment():
    global counter      # дозволяє ПИСАТИ в глобальну
    counter += 1

def outer():
    n = 0
    def inner():
        nonlocal n      # дозволяє ПИСАТИ в enclosing
        n += 1
    inner()
```

**Gotcha:** присвоєння **створює локальну змінну**. Якщо читаєш глобальну, а потім присвоюєш — `UnboundLocalError`:
```python
count = 5
def f():
    print(count)   # ❌ UnboundLocalError
    count = 10     # бо count тут локальна для всієї функції
```

### L3 — Senior
**Scope визначається на етапі компіляції**, не в рантаймі. Компілятор сканує тіло функції: чи є **присвоєння** імені → воно стає `LOAD_FAST/STORE_FAST` (локальне, швидкий доступ через масив, а не словник). Інакше — `LOAD_GLOBAL`.

Тому `UnboundLocalError` виникає **до** виконання — ім'я класифіковане як локальне ще при компіляції функції.

**Локальні швидші за глобальні:** локальні зберігаються у **масиві** `fast locals` (доступ за індексом, `O(1)` без хешування), а глобальні — у **словнику модуля** (`globals()`, хеш-lookup). Звідси відома мікрооптимізація: винести `len` у локальну змінну в гарячому циклі.

**Немає block scope:** у Python `if/for/while` **не створюють** свою область — змінна циклу «витікає» назовні (див. питання 33). Scope створюють лише **функції, класи, модулі та comprehensions** (у Py3 у comprehensions своя область).

> ⚠️ Класи мають особливий scope (тіло класу не є enclosing для методів) — деталі звіряй з Language Reference (Naming and binding).

## 27. Що таке introspection?

### L1 — Junior
**Аналогія:** рентген. Introspection — здатність програми «просвітити» сама себе під час роботи: подивитись на об'єкт і спитати «хто ти, якого типу, які в тебе методи й атрибути?» — не знаючи цього наперед.

### L2 — Middle
Основні інструменти:
```python
type(obj)              # тип об'єкта
isinstance(obj, int)   # чи є екземпляром
id(obj)                # ідентичність (адреса)
dir(obj)               # усі атрибути/методи
hasattr(obj, "name")   # чи є атрибут
getattr(obj, "name", default)  # дістати атрибут динамічно
obj.__dict__           # словник атрибутів екземпляра
callable(obj)          # чи можна викликати
vars(obj)              # = obj.__dict__
```

Модуль **`inspect`** — потужніше:
```python
import inspect
inspect.signature(func)     # сигнатура
inspect.getsource(func)     # вихідний код
inspect.getmembers(obj)     # усі члени
inspect.isclass(obj)
```

**Де застосовується:** ORM (читає поля моделі), серіалізатори (Pydantic, DRF), DI-контейнери, дебагери, `pytest` (знаходить тести за іменами), документатори.

**Gotcha:** introspection ≠ метапрограмування. Introspection — **читати** структуру; метапрограмування (метакласи, `setattr`) — **змінювати** її.

### L3 — Senior
Introspection можлива тому, що в Python **усе є об'єктом `PyObject`**, і метадані не стираються (на відміну від, скажімо, type erasure дженериків у Java). Кожен об'єкт несе вказівник на свій тип (`ob_type`), кожен тип — `__mro__`, `__dict__`, дескриптори.

**Трейдоф:** ця «самоописовість» — джерело гнучкості Python (фреймворки на ній тримаються), але й **накладних витрат**: кожен атрибут — це lookup у словнику/MRO, що повільніше за статичну розкладку полів у C/Rust. Це частина того, чому Python повільніший за компільовані мови.

`__slots__` обмежує introspection (прибирає `__dict__`) заради економії пам'яті й швидшого доступу — приклад того, як Senior свідомо **жертвує гнучкістю** заради продуктивності.

## 28. Різниця між `is` і `==`

### L1 — Junior
**Аналогія — близнюки:**
- `==` — «ви однаково виглядаєте?» (рівність **значень**).
- `is` — «ви та сама людина?» (це **той самий об'єкт** у пам'яті).

```python
a = [1, 2, 3]
b = [1, 2, 3]
a == b   # True  — значення однакові
a is b   # False — різні об'єкти
```

**Правило:** `is` використовуй **лише** з `None`, `True`, `False`. Для решти — `==`.

### L2 — Middle
- `==` викликає `__eq__` (можна перевизначити).
- `is` порівнює `id()` (адреси), не перевизначається.

```python
x = None
if x is None:     # ✅ канонічно
    ...
if x == None:     # ❌ працює, але стиль поганий (і повільніше)
```

**Класичний gotcha — кешування малих int і інтернування:**
```python
a = 256
b = 256
a is b      # True  — малі int кешуються

a = 257
b = 257
a is b      # False (часто!) — поза кешем
```

**Чому небезпечно:** покладатись на `is` для чисел/рядків — це покладатись на **деталь реалізації CPython**, яка не гарантована.

### L3 — Senior
**Під капотом CPython:**
- **Small integer cache**: інти від **−5 до 256** — це заздалегідь створені синглтони (масив `small_ints`). Тому `is` для них «випадково» працює.
- **String interning**: рядки, схожі на ідентифікатори (літерали, імена), інтернуються компілятором → один об'єкт. `sys.intern()` робить це вручну (оптимізація для частих порівнянь рядків — порівняння за вказівником замість посимвольного).

**`is` — це `O(1)`** (порівняння вказівників), `==` для контейнерів — `O(n)` (поелементно). Тому `is None` трохи швидший за `== None` — але це не головна причина, головна — **коректність**: `__eq__` можна перевизначити так, що `obj == None` поверне щось дивне; `is None` надійний завжди.

**Трейдоф кешування:** економія пам'яті й швидкі порівняння для частих малих значень ↔ непередбачувана поведінка `is`, що плутає новачків. CPython обрав економію, але задокументував `is` як «identity», не «equality».

> ⚠️ Точні межі кешу (−5..256) і правила інтернування — деталі CPython, можуть відрізнятися між версіями/реалізаціями. Не покладайся на `is` для значень у проді.

## 29. Різниця між `__init__()` і `__new__()`

### L1 — Junior
**Аналогія — народження й виховання:**
- `__new__` — **створює** об'єкт (народжує, виділяє пам'ять). Повертає новонароджений об'єкт.
- `__init__` — **налаштовує** вже створений об'єкт (дає ім'я, виховує). Нічого не повертає.

Спочатку `__new__` (створення) → потім `__init__` (ініціалізація).

### L2 — Middle
```python
class Point:
    def __new__(cls, *args, **kwargs):
        print("1. __new__ — створюємо")
        instance = super().__new__(cls)   # реальне створення
        return instance

    def __init__(self, x, y):
        print("2. __init__ — заповнюємо")
        self.x = x
        self.y = y

p = Point(1, 2)  # __new__ → __init__
```

- `__new__` — **статичний метод** (отримує `cls`), має **повернути** екземпляр.
- `__init__` — отримує вже створений `self`, **повертає `None`**.

**Коли чіпати `__new__` (рідко):**
- Subclassing immutable типів (`int`, `str`, `tuple`) — їх не можна змінити в `__init__`.
- Singleton, object pooling, кешування екземплярів.

```python
class PositiveInt(int):
    def __new__(cls, value):
        if value < 0:
            raise ValueError("must be positive")
        return super().__new__(cls, value)   # int незмінний → лише тут
```

**Gotcha:** якщо `__new__` повертає об'єкт **не того класу**, `__init__` **не викликається**.

### L3 — Senior
Виклик `Point(1, 2)` — це насправді `type.__call__(Point, 1, 2)`, який всередині робить приблизно:
```python
obj = cls.__new__(cls, *args, **kwargs)
if isinstance(obj, cls):
    cls.__init__(obj, *args, **kwargs)
return obj
```

Тобто **метаклас `type` керує процесом** через `__call__`. Це пояснює, чому `__init__` не викликається, якщо `__new__` повернув чужий тип — перевірка `isinstance`.

**Трейдоф розділення на два методи:** дозволяє відокремити **алокацію** від **ініціалізації**. Для більшості класів `__new__` не чіпають (успадковують від `object`) — це деталь, яка «не заважає». Але вона критична для immutable-сабкласів і метапрограмування (синглтони, ABC, ORM).

> ⚠️ Точна послідовність у `type.__call__` — деталь реалізації CPython; концептуально стабільна, але звіряй з docs (Data model → object.\_\_new\_\_).

## 30. Різниця між потоками (threads) та процесами (processes)

### L1 — Junior
**Аналогія — офіс:**
- **Процес** — окремий офіс зі своїм приміщенням, меблями, документами. Ізольований.
- **Потоки** — співробітники в **одному** офісі: спільні документи й простір, але працюють паралельно.

Процеси **не діляться** пам'яттю, потоки — **діляться**.

### L2 — Middle
| | Threads (`threading`) | Processes (`multiprocessing`) |
|---|---|---|
| Пам'ять | спільна | ізольована |
| Створення | дешеве | дороге |
| Обмін даними | прямий (спільні об'єкти) | IPC (pickle, pipe, queue) |
| Краш | валить увесь процес | ізольований |
| Паралелізм CPU | **обмежений GIL** | справжній |

**Головне правило вибору в CPython:**
- **I/O-bound** (мережа, диск, БД) → **threads** або **async**. Поки потік чекає відповіді, GIL відпускається, інші працюють.
- **CPU-bound** (обчислення, обробка зображень) → **processes**. Бо GIL не дає потокам рахувати паралельно.

```python
# CPU-bound → процеси
from multiprocessing import Pool
with Pool(4) as p:
    results = p.map(heavy_compute, data)

# I/O-bound → потоки/async
from concurrent.futures import ThreadPoolExecutor
with ThreadPoolExecutor() as ex:
    ex.map(fetch_url, urls)
```

**Gotcha:** «потоки в Python дають паралелізм CPU» — **хибно** через GIL. Для рахунку — процеси.

### L3 — Senior
**GIL (Global Interpreter Lock)** — м'ютекс, що дозволяє **лише одному потоку** виконувати Python-байткод одночасно. Причина існування: спрощення керування пам'яттю — **reference counting** не атомарний, GIL захищає лічильники посилань від гонок без дорогого блокування на кожен об'єкт.

Наслідок: на CPU-bound потоки **не прискорюють** (лише перемикаються), а інколи **сповільнюють** (overhead перемикання + боротьба за GIL). На I/O-bound — прискорюють, бо блокуючі syscalls **відпускають GIL**.

**Процеси** обходять GIL (у кожного свій інтерпретатор і свій GIL), ціною: дорогий старт, серіалізація (pickle) для передачі даних, окрема пам'ять.

**Сучасне (важливо):**
- **Python 3.12+** — субінтерпретатори з ізольованим GIL (PEP 684).
- **Python 3.13** — експериментальна **free-threaded** збірка (PEP 703, «no-GIL») — справжній паралелізм потоків без GIL.

**Трейдоф async vs threads для I/O:** `asyncio` — один потік, кооперативна багатозадачність (явні `await`), масштабується на десятки тисяч з'єднань без overhead потоків ОС, але вимагає неблокуючого коду «згори донизу» (один блокуючий виклик вішає всі корутини).

> ⚠️ Статус free-threaded build (3.13/3.14) активно змінюється — перевіряй актуальні release notes і PEP 703 перед продакшн-рішеннями.

## 31. Які є види імпорту?

### L1 — Junior
**Аналогія:** взяти інструмент зі складу. Можна взяти всю валізу (`import module`), один молоток (`from module import hammer`), або перейменувати під себе (`as`).

### L2 — Middle
```python
# 1. Імпорт усього модуля
import math
math.sqrt(4)

# 2. Імпорт конкретних імен
from math import sqrt, pi
sqrt(4)

# 3. Аліас (перейменування)
import numpy as np
from datetime import datetime as dt

# 4. Wildcard (НЕ рекомендовано)
from math import *      # засмічує namespace, незрозуміло звідки що

# 5. Абсолютний vs відносний
from myapp.utils import helper      # абсолютний (рекомендовано)
from .utils import helper           # відносний (1 крапка = поточний пакет)
from ..models import User           # 2 крапки = на рівень вище
```

**Best practices (PEP 8):** порядок груп — stdlib → third-party → local, розділені порожнім рядком. Абсолютні імпорти переважніші за відносні. `import *` — табу (окрім інтерактиву).

**Gotcha:** `from x import *` тягне лише імена з `__all__` (якщо визначено) або всі без `_` — і непомітно перекриває локальні імена.

### L3 — Senior
**Механіка імпорту:**
1. Перевірка `sys.modules` (**кеш**) — якщо вже імпортовано, береться звідти. Тому модуль виконується **один раз** за процес (модуль = синглтон).
2. Пошук через **finders/loaders** (`sys.meta_path`) по `sys.path`.
3. Виконання коду модуля → створення об'єкта модуля → запис у `sys.modules`.

**Lazy/conditional import** — імпорт усередині функції: відкладає завантаження, розриває цикли, прискорює старт CLI:
```python
def to_df(data):
    import pandas as pd   # важка залежність — лише коли реально треба
    return pd.DataFrame(data)
```

**Циклічні імпорти (gotcha Senior):** A імпортує B, B імпортує A. Оскільки модуль кладеться в `sys.modules` **до** завершення виконання, частково ініціалізований модуль може дати `ImportError`/`AttributeError`. Лікування: відкласти імпорт у функцію, або винести спільне в третій модуль, або `import module` замість `from module import name`.

**Трейдоф:** `from x import name` — коротше й трохи швидший доступ (ім'я в локальному namespace), але приховує походження й сприяє циклам; `import x` — багатослівніше, зате прозоро і стійко до циклів.

## 32. Що таке клас, ітератор, генератор?

### L1 — Junior
**Аналогії:**
- **Клас** — креслення дому. За одним кресленням будуєш багато домів (об'єктів).
- **Ітератор** — пульт із кнопкою «наступний»: дає елементи по одному, пам'ятає, де зупинився.
- **Генератор** — лінива «фабрика» ітераторів: проста функція з `yield`, що видає значення на вимогу, не зберігаючи все в пам'яті.

### L2 — Middle
```python
# КЛАС — шаблон для об'єктів
class Dog:
    def __init__(self, name: str) -> None:
        self.name = name

# ІТЕРАТОР — реалізує __iter__ і __next__
class Counter:
    def __init__(self, limit: int) -> None:
        self.limit = limit
        self.current = 0
    def __iter__(self) -> "Counter":
        return self
    def __next__(self) -> int:
        if self.current >= self.limit:
            raise StopIteration       # сигнал кінця
        self.current += 1
        return self.current

# ГЕНЕРАТОР — функція з yield (та сама логіка, 3 рядки)
def counter(limit: int):
    current = 0
    while current < limit:
        current += 1
        yield current                 # «віддати й заморозитись»

list(counter(3))   # [1, 2, 3]
```

**Зв'язок:** генератор — це **найпростіший спосіб створити ітератор**. Виклик генератор-функції повертає **generator object**, який уже є ітератором.

**Gotcha:** ітератор **одноразовий** — після вичерпання дає `StopIteration` назавжди. Щоб пройти знову — створи новий.

### L3 — Senior
**Iterator protocol** — це duck typing: `iter(obj)` шукає `__iter__`, `next(obj)` — `__next__`. Цикл `for` неявно викликає `iter()`, потім `next()` до `StopIteration`.

**Генератор під капотом:** функція з `yield` компілюється з прапором `CO_GENERATOR`. Її виклик не виконує тіло, а повертає об'єкт-генератор зі своїм **frame** (стек, локальні змінні). `yield` **зберігає стан frame** і повертає керування; `next()` відновлює виконання з того ж місця. Це **корутина** на рівні реалізації.

**Big-O / пам'ять — головна перевага:** генератор — **`O(1)` пам'яті** (lazy, по одному елементу), список — `O(n)` (усе одразу). Тому для великих/нескінченних потоків — генератори:
```python
sum(x*x for x in range(10**8))   # genexpr: майже без пам'яті
sum([x*x for x in range(10**8)]) # list: ~гігабайти RAM
```

**Трейдоф:** генератор економить пам'ять і дає lazy-обчислення (можна обірвати рано) ↔ одноразовість, немає `len()`/індексації, складніше дебажити (стан «застиглий»). `yield from` делегує підгенератору; `.send()/.throw()/.close()` роблять генератор повноцінною корутиною (основа старого `asyncio`).

## 33. Що таке метаклас і змінна циклу?

### L1 — Junior
**Метаклас:** якщо клас — креслення для об'єктів, то **метаклас — це креслення для класів** («хто створює самі класи»). У Python за замовчуванням це `type`.

**Змінна циклу:** ім'я, яке «крутиться» в `for x in ...` — приймає по черзі кожне значення.

### L2 — Middle
**Метакласи:**
```python
print(type(42))        # <class 'int'>
print(type(int))       # <class 'type'>  ← клас int сам є екземпляром type
print(type(type))      # <class 'type'>  ← type створює себе

# Кастомний метаклас
class Singleton(type):
    _instances: dict = {}
    def __call__(cls, *args, **kwargs):
        if cls not in cls._instances:
            cls._instances[cls] = super().__call__(*args, **kwargs)
        return cls._instances[cls]

class DB(metaclass=Singleton):
    pass

DB() is DB()   # True — завжди той самий екземпляр
```

**Де реально зустрічається:** Django ORM (`models.Model`), Pydantic, SQLAlchemy, ABC (`abc.ABCMeta`), enum.

**Змінна циклу — класичний gotcha (late binding):**
```python
funcs = [lambda: i for i in range(3)]
[f() for f in funcs]    # [2, 2, 2] — НЕ [0,1,2]!

# Фікс — зафіксувати значення через default-аргумент:
funcs = [lambda i=i: i for i in range(3)]
[f() for f in funcs]    # [0, 1, 2]
```

### L3 — Senior
**Метаклас керує створенням класу.** Коли інтерпретатор бачить `class C: ...`, він викликає `metaclass(name, bases, namespace)` → це створює об'єкт класу. Хуки: `__new__`/`__init__` метакласу (втручання при **визначенні** класу), `__call__` (втручання при **створенні екземплярів**, як у Singleton вище).

**Tim Peters:** *«Metaclasses are deeper magic than 99% of users should ever worry about. If you wonder whether you need them, you don't.»* На практиці те, що раніше робили метакласами, тепер часто роблять `__init_subclass__` і `__set_name__` (PEP 487) — простіше й достатньо.

**Змінна циклу — чому `[2,2,2]`:** замикання захоплює **змінну**, а не її значення. Усі лямбди посилаються на ту саму `i`; на момент виклику `i == 2` (останнє значення). Глибша причина — в Python **немає block scope**: `i` живе в області функції/модуля, а не тіла циклу, і **«витікає»** після циклу:
```python
for x in range(3):
    pass
print(x)   # 2 — x існує після циклу
```

**Трейдоф такого дизайну:** простота моделі scope (лише функції створюють область) ↔ пастка late-binding у замиканнях. Деякі мови (JS з `let`) дали block scope саме щоб це прибрати; Python лишив, але дав явні обхідні шляхи (default-аргумент, `functools.partial`).

> ⚠️ `__init_subclass__`/`__set_name__` (PEP 487) і деталі створення класів — звіряй з Data model docs.

## 34. Різниця між ітераторами та генераторами

### L1 — Junior
**Аналогія:** генератор — це **готовий заводський ітератор**. Ітератор можна зробити вручну (написати клас із `__next__`), а генератор — Python зробить його за тебе з простої функції з `yield`.

**Усі генератори — ітератори, але не всі ітератори — генератори.**

### L2 — Middle
| | Ітератор (клас) | Генератор |
|---|---|---|
| Як створити | клас з `__iter__`+`__next__` | функція з `yield` або genexpr |
| Код | багатослівний | компактний |
| Стан | вручну в атрибутах | автоматично (frame) |
| `StopIteration` | піднімаєш сам | Python сам при `return`/кінці |
| `.send()/.throw()` | немає | є |

```python
# Те саме, два способи:
class Squares:                      # ітератор
    def __init__(self, n): self.n, self.i = n, 0
    def __iter__(self): return self
    def __next__(self):
        if self.i >= self.n: raise StopIteration
        self.i += 1
        return (self.i - 1) ** 2

def squares(n):                     # генератор — коротше
    for i in range(n):
        yield i ** 2
```

**Gotcha:** генератор-**вираз** vs список:
```python
gen = (x*x for x in range(5))   # генератор (ліниво, дужки круглі)
lst = [x*x for x in range(5)]   # список (одразу, дужки квадратні)
```

### L3 — Senior
**Генератор — підмножина ітераторів** із додатковими можливостями: він **двосторонній** (`.send()` передає значення назад у `yield`, `.throw()` кидає виключення всередину, `.close()` зупиняє). Звичайний ітератор — лише однонаправлений `next()`.

**Під капотом** генератор зберігає **frame object** між викликами (заморожений стек), тоді як ручний ітератор тримає стан явно в `self.*`. Перший — менше коду й помилок, другий — повний контроль і можливість, наприклад, рестарту (`__iter__` повертає новий стан).

**Big-O однаковий** (both lazy, `O(1)` пам'яті на крок), але генератор зазвичай швидший і чистіший. Винятки, коли потрібен **клас-ітератор**: потрібен `__len__`, індексація, багаторазовий обхід (`__iter__` повертає свіжий ітератор щоразу — як у `list`), або складний стан з кількома методами.

**Трейдоф:** генератор — лаконічність і автоматичне керування станом ↔ менший контроль (не можна «перемотати», немає довільного доступу). Клас-ітератор — гнучкість ↔ бойлерплейт.

## 35. Різниця між `staticmethod` та `classmethod`

### L1 — Junior
**Аналогія — піцерія:**
- **`classmethod`** — рецепт, прив'язаний до **самої піцерії** (класу): «створи піцу за нашим фірмовим стандартом». Отримує `cls` — доступ до класу.
- **`staticmethod`** — просто корисна **формула** (перевести грами в унції), що логічно лежить у піцерії, але не залежить ні від конкретної піци, ні від піцерії. Не отримує ні `self`, ні `cls`.

### L2 — Middle
```python
class Pizza:
    def __init__(self, size: int, toppings: list[str]) -> None:
        self.size = size
        self.toppings = toppings

    # Звичайний — доступ до екземпляра (self)
    def describe(self) -> str:
        return f"{self.size}cm: {self.toppings}"

    # classmethod — альтернативний конструктор (factory)
    @classmethod
    def margherita(cls, size: int) -> "Pizza":
        return cls(size, ["tomato", "mozzarella"])

    # staticmethod — утиліта, не торкається стану
    @staticmethod
    def cm_to_inch(cm: int) -> float:
        return cm / 2.54

Pizza.margherita(30)        # фабрика — не треба знати інгредієнти
Pizza.cm_to_inch(30)        # просто функція в namespace класу
```

**Головні юзкейси:**
- `classmethod` → **alternative constructors** (`dict.fromkeys`, `datetime.fromtimestamp`), бо `cls` коректно працює з **наслідуванням** (поверне підклас).
- `staticmethod` → групування логічно пов'язаної функції без стану.

**Gotcha:** `classmethod` із `cls` поважає наслідування — `SubPizza.margherita()` створить `SubPizza`, а не `Pizza`. Якби захардкодити `Pizza(...)`, наслідування б зламалось.

### L3 — Senior
Обидва — **дескриптори** (`classmethod`/`staticmethod` об'єкти з `__get__`), що змінюють поведінку доступу до методу:
- `staticmethod.__get__` повертає **просто функцію** (нічого не зв'язує).
- `classmethod.__get__` повертає **bound method**, прив'язаний до **класу** (передає `cls` автоматично).

Це той самий дескрипторний протокол, що робить звичайні методи «bound» до екземпляра.

**Трейдоф / коли що:**
- Якщо метод **не використовує `self` і `cls`** і логічно міг би бути модульною функцією — часто **краще винести у функцію модуля** (Pythonic), а `staticmethod` лишити, коли важлива **приналежність до namespace класу** чи перевизначення в підкласах.
- `classmethod` незамінний для **поліморфних фабрик** і роботи з класовим станом (реєстри, лічильники).

> ⚠️ Деталі дескрипторного протоколу `__get__` для цих типів — звіряй з Descriptor HowTo Guide.

## 36. Як працюють декоратори й контекстні менеджери?

### L1 — Junior
**Аналогії:**
- **Декоратор** — подарункова обгортка. Подарунок (функція) той самий, але навколо додається обгортка (логування, перевірка прав, кеш) — без зміни самого подарунка.
- **Контекстний менеджер** — автоматичні двері: заходиш (`__enter__`) — відчиняються; виходиш (`__exit__`) — гарантовано зачиняються, навіть якщо ти впав. Це `with`.

### L2 — Middle
**Декоратор** — функція, що приймає функцію й повертає нову функцію:
```python
import functools, time

def timing(func):
    @functools.wraps(func)          # зберігає __name__, __doc__ оригіналу
    def wrapper(*args, **kwargs):
        start = time.perf_counter()
        result = func(*args, **kwargs)
        print(f"{func.__name__}: {time.perf_counter()-start:.4f}s")
        return result
    return wrapper

@timing                             # = slow = timing(slow)
def slow():
    time.sleep(1)
```

**Контекстний менеджер** — `__enter__`/`__exit__` або `@contextmanager`:
```python
# Спосіб 1 — клас
class Timer:
    def __enter__(self):
        self.start = time.perf_counter(); return self
    def __exit__(self, exc_type, exc_val, exc_tb):
        print(f"{time.perf_counter()-self.start:.4f}s")
        return False                # False → виключення проброситься далі

# Спосіб 2 — contextlib (коротше)
from contextlib import contextmanager
@contextmanager
def timer():
    start = time.perf_counter()
    try:
        yield                       # до yield = __enter__, після = __exit__
    finally:
        print(f"{time.perf_counter()-start:.4f}s")

with timer():
    slow()
```

**Gotcha:** забув `@functools.wraps` → втрачаєш `__name__`, `__doc__`, ламаєш introspection та інструменти, що на неї спираються. `__exit__` повертає `True` → **глушить** виключення (часто баг).

### L3 — Senior
**Декоратор — синтаксичний цукор:** `@d \n def f` ≡ `f = d(f)`. Виконується **на етапі визначення** функції (один раз при імпорті). Декоратор з аргументами — це **фабрика декораторів** (3 рівні вкладеності):
```python
def retry(times):                   # фабрика
    def decorator(func):            # декоратор
        @functools.wraps(func)
        def wrapper(*a, **kw):      # обгортка
            for _ in range(times): ...
        return wrapper
    return decorator
```

**Контекстний менеджер гарантує cleanup** через `__exit__`, який викликається **навіть при виключенні** (на відміну від простого коду після виклику). Аргументи `__exit__(exc_type, exc_val, exc_tb)` дають доступ до виключення → можна обробити/проковтнути.

**Трейдофи:**
- Декоратор: чисте розділення наскрізної логіки (cross-cutting: логи, кеш, auth) ↔ ускладнює стектрейс і дебаг (зайвий рівень `wrapper`), приховує що відбувається.
- `@contextmanager` (генератор) лаконічніший, але підтримує **один** `yield` і складніше з кількома виключеннями; клас-CM потужніший (стан, реентрабельність). `contextlib.ExitStack` — для динамічної кількості ресурсів.

## 37. Як працюють dict/list/set comprehensions?

### L1 — Junior
**Аналогія:** конвеєр. Замість того, щоб вручну в циклі складати елементи в коробку, ти описуєш **один рядок**: «візьми кожен X, переший його, поклади в коробку (список/множину/словник)».

```python
squares = [x*x for x in range(5)]            # list  → [0,1,4,9,16]
unique  = {x % 3 for x in range(10)}         # set   → {0,1,2}
mapping = {x: x*x for x in range(5)}          # dict  → {0:0, 1:1, ...}
```

### L2 — Middle
```python
# З умовою (фільтр)
evens = [x for x in range(10) if x % 2 == 0]

# З умовою-виразом (тернарний)
labels = ["even" if x % 2 == 0 else "odd" for x in range(5)]

# Вкладений
matrix = [[i*j for j in range(3)] for i in range(3)]

# dict comprehension — інверсія словника
inv = {v: k for k, v in original.items()}

# genexpr (ліниво, без побудови списку)
total = sum(x*x for x in range(1000))
```

**Best practices:** comprehension читабельніший і **швидший** за `for`+`append`. Але **не зловживати** — якщо логіка складна або вкладеність >2, звичайний цикл зрозуміліший (PEP 8: читабельність важливіша).

**Gotcha:** не пхай у comprehension побічні ефекти (`[print(x) for x in xs]`) — це антипатерн, для побічних дій є звичайний цикл.

### L3 — Senior
**Чому швидше за `for`+`append`:** comprehension має **спеціалізований байт-код** (`LIST_APPEND`/`SET_ADD`/`MAP_ADD`) і **не викликає** `list.append` як метод (немає накладних на attribute lookup + виклик функції на кожній ітерації). Економія — на пошуку методу й створенні frame.

**Власна область видимості (Py3):** comprehension виконується у **власному scope** (як прихована функція) — змінна циклу **не витікає** назовні (на відміну від звичайного `for`):
```python
[i for i in range(5)]
print(i)   # NameError — i не витекла (на відміну від for-циклу!)
```
Це навмисна зміна Py2→Py3, щоб comprehensions не засмічували простір імен.

**Трейдоф пам'яті:** list/set/dict comprehension будують **усю** колекцію в пам'яті (`O(n)`). Для великих/нескінченних потоків — **генератор-вираз** (`O(1)`), особливо коли результат одразу агрегується (`sum`, `any`, `max`) і повний список не потрібен.

## 38. Чи можна кілька декораторів на одну функцію?

### L1 — Junior
**Так.** Це як одягтися шарами: спершу футболка, зверху светр, зверху куртка. Декоратори накладаються один на одного.

```python
@auth_required
@log_calls
@cache
def get_data(): ...
```

### L2 — Middle
**Порядок застосування — знизу вгору** (найближчий до функції — перший):
```python
@a
@b
@c
def f(): ...

# Еквівалент:
f = a(b(c(f)))
```
Тобто `c` обгортає першим, `a` — останнім (зовнішній).

**Порядок виконання — зверху вниз** при виклику: зовнішній `a` спрацьовує першим, потім `b`, потім `c`, потім сама функція.

**Gotcha:** **порядок має значення.** `@auth` над `@cache` — спочатку перевіримо права, потім кеш. Якщо навпаки — віддамо кешований результат **до** перевірки прав = діра безпеки. Думай про порядок свідомо.

### L3 — Senior
Кожен декоратор додає **рівень обгортки** → стек викликів зростає, і без `functools.wraps` метадані «глибшої» функції втрачаються. На кожен виклик — прохід через усі `wrapper`'и (накладні витрати ростуть лінійно з кількістю декораторів).

**Трейдоф:** композиція наскрізних аспектів (auth + cache + log + retry) елегантна й декларативна ↔ важче дебажити (глибокий стектрейс із `wrapper`), порядок неочевидний, і помилка в порядку може бути критичною (безпека/коректність). Для складних комбінацій корисно логувати/документувати очікуваний порядок.

## 39. Чи можна створити декоратор з класу?

### L1 — Junior
**Так.** Якщо зробити об'єкт «викликабельним» (додати `__call__`), його можна використати як декоратор. Клас зручний, коли декоратору треба **пам'ятати стан** (наприклад, лічильник викликів).

### L2 — Middle
```python
import functools

class CountCalls:
    def __init__(self, func):
        functools.update_wrapper(self, func)   # аналог @wraps для класу
        self.func = func
        self.count = 0

    def __call__(self, *args, **kwargs):
        self.count += 1
        print(f"Виклик #{self.count} {self.func.__name__}")
        return self.func(*args, **kwargs)

@CountCalls
def greet(name: str) -> str:
    return f"Привіт, {name}"

greet("Іван")        # Виклик #1
greet("Марія")       # Виклик #2
print(greet.count)   # 2 — стан зберігається в об'єкті
```

**Коли клас кращий за функцію-декоратор:**
- Потрібен **стан** (лічильники, кеш, реєстр).
- Складна конфігурація (декоратор з багатьма параметрами).
- Декоратор має додаткові методи (`.reset()`, `.stats()`).

**Gotcha:** при декоруванні **методів** класовий декоратор ламає дескрипторний протокол (`self` не зв'язується автоматично) — треба реалізувати `__get__` або використати функцію-декоратор. Тому для методів частіше беруть функцію.

### L3 — Senior
Це працює через **callable-протокол**: `obj(...)` ≡ `type(obj).__call__(obj, ...)`. Об'єкт класу з `__call__` нічим не гірший за функцію в ролі декоратора.

**Дескрипторна проблема для методів:** функція є дескриптором (`function.__get__` створює bound method), а екземпляр класу-декоратора — ні. Тому `instance.decorated_method()` не передасть `instance` як `self`. Рішення: додати `__get__` у клас-декоратор (зробити його дескриптором) і повертати `functools.partial(self.__call__, obj)`.

**Окремо:** «декоратор з класу» ≠ «декорувати клас». Можна й **декорувати клас** (`@dataclass`, `@final`) — декоратор приймає клас і повертає (модифікований) клас.

**Трейдоф:** клас-декоратор — явний стан і розширюваність ↔ багатослівніший, проблеми з методами, трохи повільніший (екземпляр + lookup `__call__`). Для простих обгорток функція + closure лаконічніша; для stateful/конфігурованих — клас.

## 40. Основні популярні пакети

### L1 — Junior
**Аналогія:** інструменти в майстерні. Замість того щоб кувати свій молоток, береш перевірений зі стандартного набору спільноти (PyPI).

Топ для співбесіди:

- **`FastAPI` / `Django` / `Flask`** — веб.
- **`pydantic`** — валідація даних.
- **`SQLAlchemy`** — ORM.

```python
import requests

r.raise_for_status()
data = r.json()
```

**Gotcha:** `requests` **синхронний** і блокує. У async-коді (FastAPI) використовуй **`httpx`** з `await`, інакше заблокуєш event loop. Завжди став `timeout` — без нього запит може висіти вічно.

### L3 — Senior

**Трейдоф «батарейки» vs залежності:** stdlib (`urllib`, `unittest`, `json`) — нуль залежностей, але незручний API; сторонні (`requests`, `pytest`) — ергономіка ціною ваги й ризику. Для бібліотек-публікацій мінімізують залежності; для застосунків — беруть зручність.

## 41. Що таке lambda-функції?

### L1 — Junior
**Аналогія:** одноразова записка замість іменованого документа. **Lambda** — маленька **безіменна** функція в один вираз, яку зручно передати «тут і зараз».

```python
square = lambda x: x ** 2
square(5)                    # 25

sorted(words, key=lambda w: len(w))   # сортуємо за довжиною
```

### L2 — Middle
Lambda = `lambda аргументи: вираз`. Обмеження: **один вираз**, без `return`, без statements (немає `if/for/try` як блоків — лише тернарний вираз).

**Де доречно:** короткий `key`/callback у `sorted`, `map`, `filter`, `min/max`:
```python
sorted(users, key=lambda u: (u.age, u.name))
max(items, key=lambda i: i.price)
filter(lambda x: x > 0, numbers)
```

**Best practice (PEP 8):** **не присвоюй lambda змінній** — якщо потрібне ім'я, використовуй `def`:
```python
f = lambda x: x + 1     # ❌ PEP8: тоді вже def
def f(x): return x + 1  # ✅ є __name__, краще для дебагу/трейсу
```

**Gotcha:** lambda з замиканням ловить **той самий** late-binding баг, що й цикл (питання 33) — `[lambda: i for i in range(3)]` дасть `[2,2,2]`.

### L3 — Senior
Технічно lambda **нічим не відрізняється** від `def` усередині — це той самий об'єкт `function`, з тим самим байт-кодом, просто з `__name__ == "<lambda>"` і без можливості містити statements. Жодного приросту швидкості vs `def` немає.

**Трейдоф:** лаконічність для дрібних callback ↔ погіршення дебагу (анонімність у трейсбеку: `<lambda>`), неможливість документування (`__doc__`), спокуса втиснути складну логіку в один вираз. Часто читабельніша альтернатива — `operator.itemgetter/attrgetter` чи `functools.partial`:
```python
from operator import attrgetter
sorted(users, key=attrgetter("age"))   # швидше й ясніше за lambda
```

**Філософія:** Python навмисно тримає lambda «куцою» (один вираз) — Гвідо не хотів повноцінних анонімних функцій, щоб не заохочувати нечитабельний код. Це свідомий дизайн-трейдоф проти стилю функціональних мов.

## 42. Що означає `*args`, `**kwargs`?

### L1 — Junior
**Аналогія:** валіза без фіксованої кількості речей. `*args` — «скільки завгодно позиційних аргументів», `**kwargs` — «скільки завгодно іменованих аргументів». Функція приймає будь-яку кількість.

```python
def f(*args, **kwargs):
    print(args)      # кортеж позиційних
    print(kwargs)    # словник іменованих

f(1, 2, name="Іван", age=30)
# args = (1, 2)
# kwargs = {'name': 'Іван', 'age': 30}
```

### L2 — Middle
Дві ролі — **збір** (у визначенні) і **розпакування** (у виклику):
```python
# Збір
def total(*nums): return sum(nums)
total(1, 2, 3)                  # nums = (1,2,3)

# Розпакування (unpacking)
nums = [1, 2, 3]
total(*nums)                    # = total(1, 2, 3)

config = {"host": "localhost", "port": 5432}
connect(**config)               # = connect(host="localhost", port=5432)
```

**Типовий продакшн-патерн — проброс у батьківський метод / декоратор:**
```python
def wrapper(*args, **kwargs):
    return func(*args, **kwargs)     # прозоро передаємо все далі
```

**Порядок параметрів:** `def f(pos, /, normal, *args, kw_only, **kwargs)`. `/` — positional-only, `*args` — далі лише keyword-only.

**Gotcha:** імена `args`/`kwargs` — лише **конвенція**; працюють самі `*` і `**`. Але відступати від конвенції — заплутувати читача.

### L3 — Senior
`*` і `**` — це **оператори пакування/розпакування**. `*args` пакує надлишок позиційних у **`tuple`** (immutable), `**kwargs` — іменовані у **`dict`**.

**Розширене розпакування (PEP 448):** працює і поза викликами:
```python
merged = {**dict_a, **dict_b}          # злиття словників
combined = [*list_a, *list_b]          # конкатенація
first, *rest = [1, 2, 3, 4]            # rest = [2,3,4]
```

**Keyword-only аргументи (`*`)** — важливий інструмент API-дизайну: змушують викликати з іменем → читабельніше й стійкіше до перестановки:
```python
def create(name, *, admin=False):     # admin лише як create(name, admin=True)
    ...
```

**Трейдоф:** `*args/**kwargs` дають гнучкість (декоратори, проксі, наслідування) ↔ **втрата чіткої сигнатури** — IDE/`mypy` не підкажуть параметри, легше передати зайве/неправильне. Для публічного API краще явні параметри; `**kwargs` доречний для прозорого пробросу. Типізувати можна через `ParamSpec` (PEP 612) для збереження сигнатури в декораторах.

## 43. Що таке exceptions і `try-except`?

### L1 — Junior
**Аналогія:** подушка безпеки. Коли щось іде не так (ділення на нуль, файл не знайдено), програма не «розбивається» — спрацьовує захист (`except`), який це обробляє.

```python
try:
    result = 10 / x
except ZeroDivisionError:
    result = 0          # план Б замість падіння
```

### L2 — Middle
Повна форма:
```python
try:
    data = parse(file)
except FileNotFoundError as e:        # конкретне виключення
    log.error(f"Немає файлу: {e}")
    raise                             # проброс далі (re-raise)
except (ValueError, KeyError) as e:   # кілька типів
    handle(e)
else:                                 # якщо НЕ було виключення
    commit(data)
finally:                              # ЗАВЖДИ (очищення ресурсів)
    file.close()
```

**Best practices:**
- Лови **конкретні** виключення, не голий `except:` (проковтне навіть `KeyboardInterrupt`).
- Не глуши «німо» (`except: pass`) — мінімум залогуй.
- Власні виключення — наслідуй від `Exception`, будуй ієрархію (`class PaymentError(Exception)`).
- `raise ... from e` — зберігає ланцюг причин.

**Gotcha — EAFP vs LBYL:** Python тяжіє до **EAFP** («Easier to Ask Forgiveness than Permission») — спробуй і злови виключення, а не перевіряй наперед:
```python
# Pythonic (EAFP)
try:
    return d["key"]
except KeyError:
    return default

# Не-Pythonic (LBYL) — гонки в конкурентному коді
if "key" in d:
    return d["key"]
```

### L3 — Senior
**Виключення в Python — дешеві на «щасливому шляху»** (setup `try` майже безкоштовний у сучасному CPython із zero-cost exceptions), але **дорогі при спрацюванні** (розкрутка стеку, побудова traceback). Тому EAFP виграє, коли виключення — **рідкість**; якщо «помилка» очікувана часто (наприклад, перевірка в гарячому циклі) — дешевша явна перевірка.

**Винятки — частина потоку керування**, не лише «помилки»: `StopIteration` керує ітерацією, `GeneratorExit` — закриттям генератора, `KeyError`/`AttributeError` використовуються у dunder-протоколах.

**Сучасне:**
- **PEP 654 — Exception Groups** (`except*`, Python 3.11) — обробка кількох виключень одночасно (важливо для `asyncio.TaskGroup`).
- `raise X from Y` керує `__cause__`/`__context__` для chained tracebacks.

**Трейдоф EAFP:** менше гонок (TOCTOU — time-of-check-to-time-of-use), Pythonic, читабельно ↔ може приховати неочікувані виключення (надто широкий `except`), і дорого при частих спрацюваннях. Senior балансує за частотою «помилкового» шляху.

> ⚠️ «Zero-cost exceptions» і деталі вартості — оптимізації CPython, що залежать від версії; звіряй з release notes (3.11+).

## 44. Що таке PEP, які знаєте?

### L1 — Junior
**PEP** = **Python Enhancement Proposal** — офіційні документи, через які пропонують і фіксують зміни в Python. Це «закони» мови.

Найвідоміші:
- **PEP 8** — стиль коду.
- **PEP 20** — Zen of Python (`import this`).
- **PEP 484** — type hints (підказки типів).

### L2 — Middle
| PEP | Про що |
|---|---|
| **PEP 8** | стиль коду (відступи, імена, форматування) |
| **PEP 20** | Zen of Python — філософія («Explicit is better than implicit») |
| **PEP 257** | docstring conventions |
| **PEP 484** | type hints (`def f(x: int) -> str`) |
| **PEP 585** | `list[int]` замість `List[int]` (вбудовані дженерики) |
| **PEP 604** | `int \| None` замість `Optional[int]` |
| **PEP 572** | walrus `:=` (присвоєння у виразі) |
| **PEP 634** | structural pattern matching (`match/case`) |
| **PEP 557** | dataclasses |
| **PEP 612 / 695** | ParamSpec / новий синтаксис дженериків |

```python
# Сучасна типізація (484 + 585 + 604)
def find(items: list[int], target: int) -> int | None:
    ...
```

**Gotcha:** type hints **не перевіряються в рантаймі** (Python їх ігнорує при виконанні) — їх перевіряє **`mypy`/`pyright`** статично. Pydantic/FastAPI — виняток: вони **читають** анотації через introspection і валідують у рантаймі.

### L3 — Senior
**Категорії PEP:** Standards Track (зміни мови/stdlib), Informational (рекомендації, як PEP 20), Process (як PEP 1 — сам процес PEP). Кожен має статус: Draft → Accepted → Final / Rejected / Withdrawn.

**Type hints — глибше:** анотації зберігаються в `__annotations__` і за замовчуванням **обчислюються при визначенні** (eager). PEP 563 (`from __future__ import annotations`) робить їх **рядками** (lazy) — рятує від forward references і circular imports, але ламає рантайм-читачів типів (Pydantic). PEP 649 пропонує lazy-обчислення через дескриптори як компроміс.

**Трейдоф типізації:** gradual typing — поступове додавання типів без перепису ↔ типи в Python **не дають рантайм-гарантій** і не прискорюють CPython (на відміну від статичних мов). Виграш — інструменти (автокомпліт, рефакторинг, відлов помилок до запуску), документація. Це свідомий вибір Python: «опціональні типи без втрати динамічності».

> ⚠️ Статуси PEP 563/649 (поведінка анотацій за замовчуванням) змінювались між версіями — звіряй із docs для своєї версії перед тим, як покладатись.

## 45. Hello-world сервіс на фреймворку

### L1 — Junior
Найпростіший веб-сервіс — це програма, що на HTTP-запит віддає відповідь. На **FastAPI** (стек цього проєкту) — буквально кілька рядків.

### L2 — Middle
```python
# main.py
from fastapi import FastAPI
from pydantic import BaseModel

app = FastAPI()

class Item(BaseModel):
    name: str
    price: float

@app.get("/")
def root() -> dict[str, str]:
    return {"message": "Hello, World!"}

@app.get("/items/{item_id}")
def read_item(item_id: int, q: str | None = None) -> dict:
    return {"item_id": item_id, "q": q}

@app.post("/items")
def create_item(item: Item) -> Item:
    return item                # Pydantic сам валідує тіло й серіалізує
```

Запуск (як у CLAUDE.md проєкту):
```bash
uv run uvicorn main:app --reload
```
Авто-документація: `http://localhost:8000/docs` (Swagger UI, генерується з типів).

**Gotcha:** FastAPI **асинхронний** (ASGI). Якщо в ендпоінті робиш блокуючий I/O (`requests`, важкий CPU) у `async def` — заблокуєш event loop. Або роби ендпоінт `def` (FastAPI винесе в threadpool), або використовуй async-бібліотеки (`httpx`, `asyncpg`) з `await`.

### L3 — Senior
FastAPI стоїть на **Starlette** (ASGI-фреймворк) + **Pydantic** (валідація). `uvicorn` — ASGI-сервер (на `uvloop`+`httptools`). Уся «магія» — на **type hints**: FastAPI через introspection анотацій генерує валідацію, серіалізацію та OpenAPI-схему. Це приклад того, як питання 27 (introspection) і 44 (typing) працюють у проді.

**ASGI vs WGSI (трейдоф):** WSGI (Flask/Django classic) — синхронний, один запит = один потік/процес → масштабування потоками. ASGI (FastAPI) — асинхронний, тисячі одночасних I/O-з'єднань на одному процесі через event loop, ціною вимоги писати неблокуючий код наскрізно. Для I/O-важких API (проксі, агрегатори) ASGI виграє; для CPU-важких різниці мало (все одно треба воркери/процеси).

Для прода: `uvicorn` за `gunicorn` з кількома воркерами (`-w 4 -k uvicorn.workers.UvicornWorker`) — щоб задіяти кілька ядер (обхід GIL процесами, питання 30).

## 46. Типи даних; різниця list vs tuple, навіщо

### L1 — Junior
**Аналогія:**
- **`list`** — записник на спіралі: дописуй, викреслюй, перемішуй сторінки.
- **`tuple`** — табличка з вигравіруваним текстом: зафіксовано назавжди.

Обидва — впорядковані послідовності. Різниця: **list змінюваний (mutable), tuple — ні (immutable)**.

### L2 — Middle
Базові типи Python:
- **Числові:** `int`, `float`, `complex`, `bool`.
- **Послідовності:** `list`, `tuple`, `range`, `str`, `bytes`.
- **Множини:** `set`, `frozenset`.
- **Відображення:** `dict`.
- **None:** `NoneType`.

```python
lst = [1, 2, 3]
lst.append(4); lst[0] = 99      # ✅ можна

tpl = (1, 2, 3)
tpl[0] = 99                     # ❌ TypeError
```

**Навіщо обидва:**
| | list | tuple |
|---|---|---|
| Зміна | так | ні |
| Хешований | ні | так (якщо вміст hashable) |
| Ключ dict / елемент set | ні | **так** |
| Семантика | однорідна колекція | фіксований запис (record) |
| Пам'ять | більше | менше |

```python
point = (10, 20)                # tuple = «координати», фіксований запис
coords[(10, 20)] = "точка"      # tuple як ключ dict — list так не може
```

**Gotcha:** tuple з одного елемента — `(1,)`, **не** `(1)` (це просто число в дужках).

### L3 — Senior
**Під капотом (CPython):**
- `tuple` — **фіксований** масив вказівників, розмір відомий при створенні → менше пам'яті, без overhead на ріст.
- `list` — **динамічний** масив із **over-allocation**: тримає запас місткості, тому `append` амортизовано `O(1)` (інколи реалокація з копіюванням, але рідко). Це класична динамічна-масив структура, **не** зв'язний список.

**Чому tuple хешований, а list — ні:** хеш має бути стабільним; mutable list міг би змінитись і «загубитись» у хеш-таблиці. Тому `list.__hash__ = None`. tuple хешується від вмісту (тому містить mutable → стає нехешованим, питання 25).

**Big-O однаковий** для спільних операцій (індексація `O(1)`, пошук `O(n)`). Tuple трохи швидший на створення/ітерацію й компактніший — CPython навіть кешує дрібні tuple (free list).

**Трейдоф / семантика:** обирай tuple коли дані **фіксовані й гетерогенні** (запис: `(x, y)`, рядок БД) — immutability документує намір і дає безпеку/хешованість. list — коли колекція **однорідна й змінна**. Для іменованих записів — `NamedTuple`/`dataclass(frozen=True)` (читабельніше за позиційний tuple).

## 47. Як використовувати вбудовані колекції (list, set, dict)?

### L1 — Junior
**Аналогія:**
- **`list`** — черга в магазині: порядок важливий, дублікати можливі.
- **`set`** — мішок унікальних значків: порядку нема, дублікатів нема, швидко перевірити «чи є».
- **`dict`** — телефонна книга: за іменем (ключем) миттєво знаходиш номер (значення).

### L2 — Middle
```python
# LIST — впорядкований, з дублікатами
nums = [3, 1, 2, 1]
nums.append(5); nums.sort(); nums[0]          # ріст, сортування, індекс
nums = [x*2 for x in nums]                    # comprehension

# SET — унікальні, операції множин
a, b = {1, 2, 3}, {2, 3, 4}
a & b        # перетин {2,3}
a | b        # об'єднання {1,2,3,4}
a - b        # різниця {1}
3 in a       # O(1) перевірка наявності

# DICT — ключ→значення
user = {"name": "Іван", "age": 30}
user.get("email", "—")            # безпечний доступ із дефолтом
user.setdefault("roles", []).append("admin")
for k, v in user.items(): ...
```

**Best practices:**
- Перевірка «чи є елемент» серед багатьох → **`set`/`dict`** (`O(1)`), не `list` (`O(n)`).
- Дедуплікація: `list(set(items))` (якщо порядок неважливий) або `dict.fromkeys(items)` (зберігає порядок).
- `collections.Counter`, `defaultdict` — для підрахунків і групувань (питання 50).

**Gotcha:** `set` і `dict` **не зберігають довільні (non-hashable) ключі** — лише immutable (tuple так, list ні). `dict` зберігає порядок вставки (з Py3.7), `set` — **ні** (порядок недетермінований).

### L3 — Senior
**Big-O — головний критерій вибору:**
| Операція | list | set/dict |
|---|---|---|
| Доступ за індексом | `O(1)` | — |
| Пошук значення (`in`) | `O(n)` | `O(1)` середній |
| Вставка в кінець | `O(1)` аморт. | `O(1)` середній |
| Вставка на початок | `O(n)` | — |
| Видалення за значенням | `O(n)` | `O(1)` |

`set`/`dict` — **хеш-таблиці** (питання 48): `O(1)` ціною додаткової пам'яті й вимоги хешованості. `list` — динамічний масив: дешева пам'ять і порядок ціною `O(n)` пошуку.

**Класична оптимізація:** заміна `if x in big_list` (`O(n)`) на `if x in big_set` (`O(1)`) у циклі перетворює `O(n²)` на `O(n)`. Це найчастіша «безкоштовна» оптимізація на code review.

**Трейдоф:** для черг з обох кінців `list.pop(0)` — `O(n)`; бери **`collections.deque`** (`O(1)` з обох боків). Для впорядкованого пошуку діапазонів — структури на кшталт `sortedcontainers` (бінарне дерево, `O(log n)`), яких нема в stdlib.

## 48. Складність доступу до елементів dict

### L1 — Junior
**Аналогія — гардероб із номерками.** Ти даєш номерок (ключ) → гардеробник одразу йде до потрібної комірки, **не перебираючи всі**. Тому доступ до `dict[key]` — **миттєвий**, незалежно від розміру: **O(1)**.

### L2 — Middle
- Доступ, вставка, видалення за ключем — **O(1) у середньому** (амортизовано).
- Це працює завдяки **хеш-таблиці**: ключ → `hash(key)` → індекс у внутрішньому масиві → одразу комірка.

```python
d = {"a": 1, "b": 2}
d["a"]          # O(1): hash("a") → індекс → значення
"a" in d        # O(1): те саме
```

**Вимога:** ключ має бути **hashable** (immutable: str, int, tuple). Тому list/dict/set не можуть бути ключами.

**Gotcha:** O(1) — **середній** випадок. Гірший — **O(n)** при масових колізіях хешів (рідко, але можливо). Тому формально кажуть «O(1) amortized average».

### L3 — Senior
**Як працює хеш-таблиця CPython:**
1. `hash(key)` → велике ціле.
2. Молодші біти → індекс у масиві слотів.
3. **Колізія** (два ключі в один слот) → CPython використовує **open addressing** (відкриту адресацію з перебором проб), **не** ланцюжки. Шукає наступний вільний слот за псевдовипадковою послідовністю.
4. **Resize**: коли заповнення ~2/3 — масив росте (зазвичай ×4 для малих), елементи **рехешуються**. Це робить окремі вставки інколи дорогими, але **амортизовано O(1)**.

**Чому гірший випадок O(n):** якщо всі ключі колізують в один слот (патологічні хеші або hash-flooding атака), пошук вироджується в лінійний перебір. Для рядків CPython використовує **рандомізований seed** (`PYTHONHASHSEED`) проти hash-DoS-атак.

**Compact dict (з Py3.6):** dict зберігає **порядок вставки** і економить ~20–25% пам'яті за рахунок розділення на щільний масив записів + масив індексів. Порядок вставки став **гарантією мови** з Py3.7.

**Трейдоф:** O(1)-доступ ціною: (а) пам'ять (хеш-таблиця тримає запас), (б) ключі мають бути hashable, (в) хеш має бути добре розподіленим. Для впорядкованого доступу за діапазоном dict не годиться (там потрібні дерева, O(log n)).

> ⚠️ Деталі resize-фактора, стратегії проб і compact-layout — внутрішня кухня CPython, що змінювалась між версіями. Звіряй з `Objects/dictobject.c` / docs перед тим, як цитувати точні числа.

## 49. Як створюється об'єкт; навіщо `__new__` і `__init__`?

### L1 — Junior
*(Поглиблення питання 29.)* Створення об'єкта — два кроки:
1. **`__new__`** — «виліпити» порожній об'єкт (виділити пам'ять).
2. **`__init__`** — «наповнити» його даними (`self.x = ...`).

Виклик `User("Іван")` запускає обидва автоматично.

### L2 — Middle
```python
class User:
    def __new__(cls, *args, **kwargs):
        print("крок 1: створення")
        return super().__new__(cls)       # повертає новий екземпляр

    def __init__(self, name: str) -> None:
        print("крок 2: ініціалізація")
        self.name = name                  # заповнюємо стан

u = User("Іван")
```

**Навіщо розділені:**
- `__init__` — **майже завжди** саме його ти й пишеш (звичайна ініціалізація).
- `__new__` — рідко, лише для: immutable-сабкласів (`int`, `str`, `tuple`), синглтонів, кешу/пулу об'єктів, метапрограмування.

**Gotcha:** в `__init__` **не можна** змінити immutable-базу (`int`/`str`) — значення вже зафіксоване в `__new__`. Тому валідацію/трансформацію таких типів роблять у `__new__` (приклад `PositiveInt` у питанні 29).

### L3 — Senior
Повний ланцюг створення: `User(...)` → `type.__call__(User, ...)` → `User.__new__` (алокація) → якщо результат `isinstance(cls)`, то `User.__init__` (ініціалізація) → повернення об'єкта. Керує цим **метаклас** (`type`) через `__call__`.

`object.__new__` робить власне виділення пам'яті під структуру `PyObject` (з `__dict__` або `__slots__`, лічильником посилань `ob_refcnt`, вказівником на тип `ob_type`). `__init__` лише заповнює вже виділену пам'ять.

**Трейдоф розділення:** дозволяє контролювати алокацію окремо від ініціалізації — критично для immutable-типів (де «змінити після створення» неможливо) і для патернів керування екземплярами (синглтон, flyweight, object pool). Ціна — концептуальна складність (два методи, неочевидний `__new__` як `@staticmethod`, що повертає об'єкт). Для 99% класів `__new__` не чіпають — успадковують від `object`.

## 50. Модуль `collections` та інші built-in модулі

### L1 — Junior
**`collections`** — набір «прокачаних» колекцій понад базові list/dict/set. Наприклад, словник, що сам рахує (`Counter`), або черга, швидка з обох кінців (`deque`).

### L2 — Middle
```python
from collections import Counter, defaultdict, deque, namedtuple, OrderedDict, ChainMap

# Counter — підрахунок
Counter("banana")              # {'a':3,'n':2,'b':1}
Counter(words).most_common(3)  # топ-3

# defaultdict — дефолт для відсутніх ключів (групування)
groups = defaultdict(list)
groups["fruits"].append("apple")   # не треба перевіряти наявність ключа

# deque — швидка черга/стек з обох кінців O(1)
dq = deque([1, 2, 3])
dq.appendleft(0); dq.popleft()     # O(1) (у list pop(0) — O(n))

# namedtuple — легкий immutable-запис з іменами
Point = namedtuple("Point", ["x", "y"])
p = Point(1, 2); p.x               # читабельніше за p[0]
```

**Інші ключові stdlib-модулі:**
| Модуль | Для чого |
|---|---|
| `itertools` | ліниві ітератори (`chain`, `groupby`, `product`, `combinations`) |
| `functools` | `lru_cache`, `reduce`, `partial`, `wraps`, `cached_property` |
| `datetime` | дати й час |
| `os` / `pathlib` | файли, шляхи (`Path` — сучасніше) |
| `json` | серіалізація |
| `re` | регулярки |
| `typing` | type hints |
| `dataclasses` | класи-записи без бойлерплейту |
| `enum` | перелічення |
| `logging` | логування |

**Gotcha:** `defaultdict` **створює** ключ при доступі (`d[missing]` додасть його). Інколи це небажано — тоді `dict.get(key, default)`.

### L3 — Senior
**`deque`** реалізована як **doubly-linked list блоків** → `O(1)` додавання/видалення з обох кінців, але `O(n)` доступ за індексом усередині (на відміну від `list` — `O(1)` індекс, `O(n)` вставка на початок). Вибір deque vs list — це **трейдоф «кінці vs середина»**.

**`functools.lru_cache`** — memoization-декоратор, що тримає хеш-таблицю результатів (`O(1)` lookup) ціною пам'яті; ключем є аргументи (мають бути hashable). `maxsize` балансує пам'ять vs hit-rate.

**`itertools` — ліниві, `O(1)` пам'яті:** обробка великих/нескінченних потоків без матеріалізації (`itertools.islice`, `count`, `cycle`). Реалізовані в C → швидші за ручні Python-цикли.

**Трейдоф spec-колекцій:** спеціалізована структура під задачу (Counter, deque, defaultdict) дає кращу складність і читабельність ↔ ще одна абстракція для розуміння. Senior знає **Big-O кожної** й обирає за патерном доступу, а не за звичкою «скрізь dict/list».

> ⚠️ Внутрішня реалізація `deque` (блоки) і `lru_cache` — деталі CPython; концептуально стабільні, точні гарантії звіряй з docs.

## 51. Що таке шаблонізатор і базові операції в ньому?

### L1 — Junior
**Аналогія — бланк із пропусками.** Шаблонізатор (template engine) — це HTML-«бланк» зі спецмісцями, куди сервер підставляє дані: ім'я користувача, дату, список товарів. Один шаблон → багато сторінок із різними даними.

У Python найпопулярніший — **Jinja2** (Flask, FastAPI), і вбудований у **Django**.

### L2 — Middle
**Базові операції (синтаксис Jinja2):**
```jinja
{# Виведення змінної (дані з серверного боку) #}
<h1>Привіт, {{ user.name }}!</h1>

{# Дата (передається з бекенду або форматується фільтром) #}
<p>Сьогодні: {{ now.strftime('%d.%m.%Y') }}</p>
<p>{{ created_at | default('—') }}</p>

{# Умови й цикли #}
{% if user.is_admin %}
  <a href="/admin">Адмінка</a>
{% endif %}

<ul>
{% for item in items %}
  <li>{{ loop.index }}. {{ item.title }} — {{ item.price }}</li>
{% endfor %}
</ul>

{# Фільтри — трансформація даних #}
{{ name | upper }}  {{ price | round(2) }}  {{ text | truncate(50) }}
```

**Об'єднання ділянок шаблона — успадкування й include:**
```jinja
{# base.html — каркас #}
<html><body>
  {% block content %}{% endblock %}
</body></html>

{# page.html — наповнення #}
{% extends "base.html" %}
{% block content %}<p>Контент сторінки</p>{% endblock %}

{# Включення фрагмента #}
{% include "navbar.html" %}
```

Рендеринг із Python:
```python
from jinja2 import Environment, FileSystemLoader
env = Environment(loader=FileSystemLoader("templates"))
html = env.get_template("page.html").render(user=user, items=items)
```

**Gotcha:** Jinja2 **за замовчуванням автоекранує** HTML (`autoescape`) → захист від XSS (питання 11). Не вимикай і не роби `| safe` на даних користувача — це відкриває XSS.

### L3 — Senior
Jinja2 **компілює** шаблон у **Python-байткод** (генерує функцію), не інтерпретує його щоразу як рядок → швидкий рендеринг + кешування скомпільованих шаблонів. Тобто шаблон проходить той самий шлях, що й код (питання 24).

**Архітектурний трейдоф SSR vs SPA:**
- **Server-Side Rendering** (шаблонізатор) — сервер віддає готовий HTML: швидкий перший рендер, гарне SEO, простіше ↔ навантаження на сервер, повне перезавантаження сторінок.
- **SPA + API** (React/Vue + JSON) — фронт рендерить сам: інтерактивність, розвантажений сервер ↔ гірше SEO «з коробки», складніше.
- **Логіка в шаблонах** — антипатерн: шаблон має лише **відображати**, не містити бізнес-логіки (separation of concerns, питання 2). Складні обчислення — на бекенді, у шаблон передавай готові дані.

**Безпека — головний інженерний аспект:** автоекранування контекстно-залежне (HTML ≠ JS ≠ URL). Сліпий `| safe` чи рядкова конкатенація HTML = XSS. Це місце, де шаблонізатор і безпека (питання 10, 11) перетинаються найгостріше.

## 52. Як Python працює з HTTP-сервером?

### L1 — Junior
**Аналогія — ресторан:** Python-код — це **кухар** (готує відповіді), але йому потрібен **офіціант** (веб-сервер), що приймає замовлення від клієнтів і носить страви. Python сам по собі не «слухає» порт — це робить сервер, який передає запити твоєму коду через стандартний інтерфейс.

### L2 — Middle
**Найпростіше (stdlib, для розробки/тестів):**
```python
from http.server import HTTPServer, BaseHTTPRequestHandler

class Handler(BaseHTTPRequestHandler):
    def do_GET(self):
        self.send_response(200)
        self.send_header("Content-Type", "text/plain")
        self.end_headers()
        self.wfile.write(b"Hello")

HTTPServer(("", 8000), Handler).serve_forever()
```

**У продакшні — через стандартні інтерфейси:**
- **WSGI** (синхронний) — Flask, Django classic. Сервер: **gunicorn**, **uWSGI**.
- **ASGI** (асинхронний) — FastAPI, Starlette. Сервер: **uvicorn**, **hypercorn**.

```
Інтернет → Nginx (reverse proxy) → gunicorn/uvicorn (WSGI/ASGI сервер) → твій Python-додаток
```

**Gotcha:** `http.server` і вбудований `uvicorn --reload` / `runserver` — **лише для розробки** (одно-потокові, без безпеки/масштабування). У проді — gunicorn/uvicorn за Nginx.

### L3 — Senior
**WSGI** (PEP 3333) — це **контракт**: сервер викликає `app(environ, start_response)` — простий callable, що отримує словник запиту й повертає ітерабельне тіло. Це розв'язує **сервер і фреймворк**: будь-який WSGI-сервер працює з будь-яким WSGI-фреймворком. Синхронний → один запит блокує воркер → масштабування **процесами/потоками** (обхід GIL, питання 30).

**ASGI** — async-наступник: `async def app(scope, receive, send)`. Підтримує **WebSocket, SSE, HTTP/2, lifespan**, тисячі одночасних I/O-з'єднань на event loop. Це відповідь на обмеження WSGI для real-time і high-concurrency I/O.

**Чому Nginx попереду:** Python-сервери погані в роздачі статики, TLS-термінації, буферизації повільних клієнтів (slowloris). Nginx бере це на себе, а gunicorn/uvicorn — лише динаміку. Класичний поділ відповідальностей.

**Трейдоф WSGI vs ASGI:** WSGI — простота, зрілість, достатньо для CRUD ↔ блокуючий, не вміє WebSocket. ASGI — конкурентність, real-time ↔ складніший, вимагає async наскрізно (один блокуючий виклик вішає loop), молодша екосистема. Senior обирає за профілем навантаження: CPU/класичний CRUD → WSGI цілком ок; багато повільного I/O / WebSocket → ASGI.

> ⚠️ Деталі WSGI/ASGI спеки — звіряй з PEP 3333 та ASGI specification перед реалізацією власного middleware.

## 53. Що відбувається при створенні віртуального середовища?

### L1 — Junior
**Аналогія — окрема пісочниця для кожного проєкту.** Без неї всі проєкти граються в одній пісочниці (глобальний Python) і конфліктують: проєкт А хоче Django 3, проєкт Б — Django 5. **Віртуальне середовище** дає кожному проєкту **свою ізольовану** пісочницю з власними пакетами.

```bash
python -m venv .venv
source .venv/bin/activate        # Linux/Mac
```

### L2 — Middle
Що фізично створюється (`python -m venv .venv`):
```
.venv/
├── bin/                 # (Scripts/ на Windows)
│   ├── python           # симлінк/копія інтерпретатора
│   ├── pip
│   └── activate         # скрипт активації
├── lib/python3.X/
│   └── site-packages/   # СЮДИ ставляться пакети проєкту (ізольовано)
└── pyvenv.cfg           # конфіг: версія, шлях до базового Python
```

**`activate` робить:**
- Додає `.venv/bin` на **початок `PATH`** → `python`/`pip` тепер з venv.
- Ставить `VIRTUAL_ENV`, міняє prompt `(.venv)`.

**Сучасні інструменти:** `uv` (цей проєкт), `poetry`, `pipenv` — керують venv + залежностями + lock-файлом автоматично:
```bash
uv sync          # створює .venv і ставить усе з lock
uv run python …  # запуск у venv без activate
```

**Gotcha:** забув активувати → `pip install` ставить **глобально** (засмічує систему). `uv run`/`poetry run` рятують від цього (не треба activate). `.venv` **не комітять** у git (`.gitignore`).

### L3 — Senior
**Механіка ізоляції** тримається на **`sys.path`** і `site` module. `pyvenv.cfg` каже інтерпретатору, де **базовий** Python (для stdlib) і вмикає `include-system-site-packages = false` → інтерпретатор будує `sys.path` так, що бачить **лише** `site-packages` свого venv, ігноруючи глобальні пакети. `python` у venv — це тонкий шар, що вказує на реальний інтерпретатор, але з іншим `sys.prefix`.

Тобто **ізоляція — це маніпуляція шляхами пошуку модулів**, а не контейнеризація. venv **не ізолює** саму версію Python (бере базову), системні C-бібліотеки, змінні оточення.

**Трейдоф рівнів ізоляції:**
- **venv** — легкий, швидкий, ізолює лише Python-пакети ↔ не ізолює версію Python/ОС-залежності.
- **pyenv** — ізолює **версію Python** (компілює різні), часто разом із venv.
- **Docker** — ізолює **всю ОС-середу** (системні бібліотеки, версію Python, оточення) ↔ важчий, повільніший старт. Для відтворюваності в проді/CI — Docker; для локальної розробки — venv.

**Lock-файли (`uv.lock`, `poetry.lock`)** додають **відтворюваність**: фіксують точні версії + хеші **всіх транзитивних** залежностей → «works on my machine» зникає. `requirements.txt` без pin — джерело недетермінованих білдів.

> ⚠️ Точна структура `pyvenv.cfg` і поведінка `site` різняться між версіями/платформами та інструментами (`virtualenv` vs stdlib `venv`); звіряй з docs модуля `venv`.

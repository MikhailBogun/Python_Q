## L1 — Junior: інтуїція

**Аналогія — рецепт на різну кількість гостей.** Питання не «скільки хвилин готувати», а «**як зросте час, якщо гостей удесятеро більше?**». Один салат — порізав і все (O(1)). Помити кожну тарілку — вдесятеро більше тарілок = вдесятеро довше (O(n)). Кожен гість цокається з кожним — натовп множиться катастрофічно (O(n²)).

**Big-O** ігнорує константи й дрібниці, дивиться лише на **темп зростання** при великих даних.

```python
def first(items: list) -> int:
    return items[0]              # O(1) — байдуже, 10 чи 10⁹ елементів

def total(items: list[int]) -> int:
    s = 0
    for x in items:              # O(n) — проходимо кожен раз
        s += x
    return s

def all_pairs(items: list) -> None:
    for a in items:              # n разів
        for b in items:          # × n разів
            print(a, b)          # = O(n²)
```

## L2 — Middle: класи складності, правила, space

### Ієрархія (від кращої до гіршої)
```
O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(n³) < O(2ⁿ) < O(n!)
```

| Big-O | Назва | При ×10 даних | Приклад |
|---|---|---|---|
| O(1) | константна | без змін | `dict[k]`, `list[i]`, `len()` |
| O(log n) | логарифмічна | +константа | binary search |
| O(n) | лінійна | ×10 | прохід списку |
| O(n log n) | лінеаритмічна | ×~13 | сортування |
| O(n²) | квадратична | ×100 | вкладені цикли |
| O(2ⁿ) | експоненційна | ×~1000 | наївний рекурсивний перебір |
| O(n!) | факторіальна | × мільйони | усі перестановки |

### Правила підрахунку
1. **Відкидай константи:** `O(2n)` → `O(n)`, `O(n/2)` → `O(n)`.
2. **Бери домінантний член:** `O(n² + n)` → `O(n²)`.
3. **Послідовні цикли — сума:** цикл n + цикл m = `O(n + m)`.
4. **Вкладені цикли — добуток:** цикл n × цикл m = `O(n·m)`.
5. **Різні входи — різні букви:** не `O(n²)`, а `O(n·m)`, якщо це **дві різні** колекції.

```python
def example(a: list, b: list) -> None:
    for x in a:          # O(n)
        print(x)
    for y in b:          # O(m)  → разом O(n + m), НЕ O(2n)
        print(y)
    for x in a:          # ┐
        for y in b:      # ┘ O(n·m) — різні входи!
            print(x, y)
```

### Space complexity (пам'ять) — не забувай!
Окрема метрика: **скільки додаткової пам'яті** алгоритм використовує (крім входу).
```python
def squares_list(n: int) -> list[int]:
    return [x*x for x in range(n)]      # O(n) пам'яті — зберігаємо всі

def squares_sum(n: int) -> int:
    return sum(x*x for x in range(n))   # O(1) пам'яті — генератор, по одному
```

### Best / Average / Worst
Складність залежить від даних. Зазвичай говорять про **worst case** (гарантія):
```python
def find(items: list, target) -> bool:
    for x in items:              # best: O(1) (перший), worst: O(n) (останній/немає)
        if x == target:
            return True
    return False
```

### Gotcha — приховані складності в Python
```python
if x in my_list:        # O(n)! — лінійний пошук у списку
if x in my_set:         # O(1)  — хеш
s = ""
for w in words:
    s += w              # O(n²)! рядок immutable → новий об'єкт щоразу
s = "".join(words)      # O(n)  — правильно

list.insert(0, x)       # O(n) — зсув усіх елементів
list.pop(0)             # O(n) — те саме; для черги бери deque
```

## L3 — Senior: нотації, рекурентності, Master theorem, amortized

### Три нотації (точно)
- **O (Big-O)** — верхня межа: «не гірше ніж» (worst case ceiling).
- **Ω (Omega)** — нижня межа: «не краще ніж».
- **Θ (Theta)** — точна: коли O і Ω збігаються.

На практиці кажуть «O(n log n)», але строго QuickSort — `Θ(n log n)` average і `O(n²)` worst. Senior розрізняє «O» від «реальний average».

### Складність рекурсивних алгоритмів — рекурентні співвідношення
Час рекурсії описується **рекурентністю** `T(n)`. Приклади:

```python
# T(n) = T(n-1) + O(1)  →  O(n)   (лінійна рекурсія)
def factorial(n):
    if n <= 1: return 1
    return n * factorial(n - 1)

# T(n) = 2·T(n-1) + O(1)  →  O(2ⁿ)  (двійкове розгалуження!)
def fib(n):
    if n < 2: return n
    return fib(n-1) + fib(n-2)

# T(n) = T(n/2) + O(1)  →  O(log n)  (ділимо навпіл)
def binary_search(arr, lo, hi, target):
    if lo > hi: return -1
    mid = (lo + hi) // 2
    if arr[mid] == target: return mid
    if arr[mid] < target: return binary_search(arr, mid+1, hi, target)
    return binary_search(arr, lo, mid-1, target)

# T(n) = 2·T(n/2) + O(n)  →  O(n log n)  (merge sort)
```

### Master Theorem (для «divide and conquer»)
Для рекурентностей виду **T(n) = a·T(n/b) + O(nᵈ)**, де:
- `a` — кількість підзадач,
- `b` — у скільки разів менша кожна,
- `d` — степінь роботи на «склеювання».

Порівнюємо `d` з `log_b(a)`:
| Випадок | Умова | Результат |
|---|---|---|
| 1 | d < log_b(a) | **O(n^log_b(a))** — домінує рекурсія |
| 2 | d = log_b(a) | **O(nᵈ · log n)** |
| 3 | d > log_b(a) | **O(nᵈ)** — домінує склеювання |

**Приклади:**
- **Merge sort:** `T(n)=2T(n/2)+O(n)` → a=2,b=2,d=1; log₂2=1=d → випадок 2 → **O(n log n)**. ✅
- **Binary search:** `T(n)=1·T(n/2)+O(1)` → a=1,b=2,d=0; log₂1=0=d → випадок 2 → **O(log n)**. ✅
- **Naive множення матриць (Strassen):** `T(n)=7T(n/2)+O(n²)` → log₂7≈2.81>2 → випадок 1 → **O(n^2.81)**.

### Amortized analysis (амортизована складність)
Усереднення дорогих операцій по **серії**. `list.append` — `O(1)` амортизовано:

```python
# CPython list: over-allocation. append зазвичай O(1), зрідка O(n) (реалокація+копія).
# Але по серії n операцій сумарна вартість O(n) → амортизовано O(1) на операцію.
lst = []
for i in range(n):
    lst.append(i)      # амортизовано O(1)
```

**Чому:** при зростанні масив подвоюється (≈×1.125 у CPython для великих). Реалокації стаються все рідше, їх сумарна вартість — геометрична сума `n + n/2 + n/4 + ... = O(n)`. Розподіливши на n append'ів → `O(1)` кожен.

Три методи амортизованого аналізу: **aggregate** (сумарно/n), **accounting** (банківський — «передплачуєш» дорогі операції), **potential** (функція потенціалу). Для інтерв'ю достатньо aggregate-інтуїції.

### Чому Big-O — про масштабованість, не швидкість
`O(n)` з великою константою може **програвати** `O(n log n)` на реальних `n`. Big-O ігнорує:
- **Константи** (`100n` vs `2n` — обидва O(n), але різниця ×50).
- **Кеш CPU** (локальність даних: масив б'є linked list при однаковому Big-O).
- **Дрібні n** (для n=10 O(n²) часто швидший за O(n log n) — менша константа; тому Timsort усередині використовує insertion sort).

**Висновок Senior:** Big-O каже, як код **деградує при рості** — це необхідно, але **недостатньо**. Для реальних рішень — Big-O **+** профілювання (`timeit`, `cProfile`, `py-spy`).

### Pseudo-polynomial (пастка)
Складність 0/1-рюкзака часто пишуть `O(n·W)`. Це виглядає поліноміально, але `W` — **значення**, а не розмір входу (його кодування — `log W` біт). Тому це **псевдополіноміальний** алгоритм — експоненційний від розміру входу. Нюанс для DP (файл 11).

## 🎯 Чекліст і типові питання інтерв'ю
- «Яка складність цього коду?» → рахуй цикли (правила вище), назви час **і** пам'ять.
- «Можеш швидше?» → шукай приховані O(n) (`in list`), заміни на hash/sort/two-pointers.
- «O(n log n) — звідки log?» → щоразу ділиш навпіл (sort/binary search) → Master theorem.
- «Різниця O і Θ?» → O — верхня межа, Θ — точна.
- «append це O(1)?» → амортизовано O(1) (over-allocation), worst-case окремого виклику O(n).

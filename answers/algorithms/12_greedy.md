# 12. Жадібні алгоритми (Greedy)

> «Бери локально найкраще і сподівайся, що вийде глобально найкраще.» Просто й швидко — **коли працює**. Головна навичка: довести, що жадібність дає оптимум (інакше потрібен DP, файл 11).

---

## L1 — Junior: інтуїція

**Аналогія — здача монетами.** Треба видати 87 грн мінімумом монет. Береш найбільшу, що влазить: 50 → 20 → 10 → 5 → 2. Щоразу **локально найкращий** вибір. Для звичайних номіналів це й дає глобальний оптимум.

**Greedy** = на кожному кроці робимо вибір, що **здається найкращим зараз**, ніколи не переглядаючи минулі рішення.

```python
def coin_change_greedy(amount: int, coins: list[int]) -> int:
    coins.sort(reverse=True)             # від найбільшої
    count = 0
    for coin in coins:
        count += amount // coin          # беремо максимум цієї монети
        amount %= coin
    return count                         # працює для КАНОНІЧНИХ номіналів
```

## L2 — Middle: класичні жадібні задачі

### 1. Activity Selection / Interval Scheduling — максимум несуперечливих
**Обрати максимум зустрічей, що не перетинаються.** Жадібний ключ: сортувати за **часом завершення**.
```python
def max_meetings(intervals: list[tuple[int, int]]) -> int:
    intervals.sort(key=lambda x: x[1])   # за часом ЗАВЕРШЕННЯ (не початку!)
    count = 0
    end = float("-inf")
    for start, finish in intervals:
        if start >= end:                 # не перетинається з попередньою
            count += 1
            end = finish
    return count
```
**Чому за завершенням:** хто закінчує раніше — лишає більше місця наступним. Це доводиться **exchange argument** (див. L3).

### 2. Jump Game — чи досяжний кінець (LC 55)
```python
def can_jump(nums: list[int]) -> bool:
    max_reach = 0
    for i, jump in enumerate(nums):
        if i > max_reach:                # сюди не дострибнути
            return False
        max_reach = max(max_reach, i + jump)   # найдалі досяжне
    return True
# O(n) — жадібно тримаємо найдальшу досяжну точку
```

### 3. Merge Intervals — об'єднати перетинні (LC 56)
```python
def merge_intervals(intervals: list[list[int]]) -> list[list[int]]:
    intervals.sort(key=lambda x: x[0])   # за початком
    merged: list[list[int]] = []
    for start, end in intervals:
        if merged and start <= merged[-1][1]:
            merged[-1][1] = max(merged[-1][1], end)   # розширити останній
        else:
            merged.append([start, end])
    return merged
```

### 4. Найбільше число з цифр (custom comparator)
```python
from functools import cmp_to_key

def largest_number(nums: list[int]) -> str:
    def compare(a: str, b: str) -> int:
        return (a + b < b + a) - (a + b > b + a)   # який порядок дає більше число
    strs = sorted(map(str, nums), key=cmp_to_key(compare))
    result = "".join(strs)
    return "0" if result[0] == "0" else result
# [3,30,34,5,9] → "9534330"
```

**Gotcha:** жадібний алгоритм майже завжди починається з **правильного сортування**. Обрати, **за яким ключем** сортувати (за завершенням? початком? відношенням?), — це 90% розв'язку.

## L3 — Senior: коли greedy працює, доказ, vs DP

### Дві умови коректності greedy
1. **Greedy choice property** — глобальний оптимум **містить** локально жадібний вибір (можна почати з нього).
2. **Optimal substructure** — після жадібного вибору залишок — це та сама задача меншого розміру (як у DP).

Якщо обидві є — greedy дає оптимум. Якщо лише друга (а перша ні) — потрібен **DP** (розглянути всі варіанти).

### Exchange argument — як доводять greedy
Класичний доказ: припустимо, є оптимальне рішення, що **не** збігається з жадібним. Покажемо, що можна **обміняти** елемент оптимуму на жадібний вибір **без погіршення**. Отже, жадібне теж оптимальне. Для Activity Selection: якщо оптимум бере зустріч, що закінчується пізніше за жадібну, заміняємо її на жадібну (що закінчується раніше) — решта рішення лишається валідною, кількість та сама. □

### Коли greedy ОБМАНЮЄ (контрприклади — критично!)
**Coin Change з «неканонічними» монетами:**
```python
# Монети [1, 3, 4], сума 6.
# Greedy: 4 + 1 + 1 = 3 монети.
# Оптимум (DP): 3 + 3 = 2 монети!  Greedy ПОМИЛЯЄТЬСЯ.
```
Тому загальний Coin Change — **DP** (файл 11), а не greedy. Greedy працює лише для канонічних систем (як справжні валюти).

**0/1 Knapsack:** greedy «за відношенням value/weight» **не** дає оптимум (предмет можна взяти лише цілим) → потрібен DP. Але **Fractional Knapsack** (можна брати частинами) — greedy за відношенням **працює** ідеально. Тонка межа!

### Greedy vs DP — головний трейдоф
| | Greedy | DP |
|---|---|---|
| Підхід | локальний вибір, без перегляду | розглядає всі варіанти |
| Швидкість | швидше (часто O(n log n) на сорті) | повільніше (O(n²), O(n·W)) |
| Пам'ять | O(1) зазвичай | O(n) / O(n²) таблиця |
| Гарантія оптимуму | **лише якщо доведено** | завжди (для коректної постановки) |
| Ризик | може дати неоптимум | — |

**Правило Senior:** greedy спокусливий (простий, швидкий), але **небезпечний без доказу**. На інтерв'ю: запропонував greedy → **спершу спробуй знайти контрприклад**. Не зміг + бачиш exchange-аргумент → застосовуй. Сумніваєшся → безпечніше DP (гарантований оптимум). Greedy без обґрунтування — класична пастка «здавалося правильним».

### Greedy у відомих алгоритмах
Багато «великих» алгоритмів — жадібні з доведеною коректністю:
- **Dijkstra** (файл 10) — жадібно бере найближчу вершину.
- **Kruskal/Prim MST** (файл 10) — жадібно беруть найлегше ребро.
- **Huffman coding** — жадібно зливає найрідші символи (оптимальне стиснення).

Вони працюють саме тому, що для них **доведено** greedy choice property. Це показує: greedy не «слабкий» — він потужний **там, де обґрунтований**.

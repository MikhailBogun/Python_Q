# Блок 1. Загальне з Computer Science та Web Development

> Формат кожної відповіді — **Принцип Сходів**:
> **L1 Junior** (аналогія + «коли використовуємо») → **L2 Middle** (продакшн, best practices, gotchas) → **L3 Senior** (під капотом, Big-O, трейдофи).

---

## 1. Що таке інженерія і процес розробки загалом?

### L1 — Junior
**Аналогія:** будівництво будинку. Програміст, який просто «кодить», — це робітник, що кладе цеглу. **Інженер** — це той, хто спочатку дивиться на ґрунт, рахує навантаження, малює креслення, обирає матеріали, а вже потім будує так, щоб будинок не впав через рік і його можна було добудувати.

**Інженерія ПЗ (Software Engineering)** — це **системний, дисциплінований підхід** до створення софту: не просто «написати код, що працює», а зробити його надійним, передбачуваним і таким, що його можна супроводжувати командою.

**Коли використовуємо:** завжди, коли код житиме довше за один день і його читатиме хтось крім тебе.

### L2 — Middle
Інженерія = **код + процес + якість + комунікація**. Програмування — це лише ~20% роботи інженера. Решта:
- **Requirements** — зрозуміти, що насправді потрібно бізнесу (а не що сказали).
- **Design** — продумати архітектуру, межі модулів, контракти (API).
- **Implementation** — написати читабельний, типізований код.
- **Verification** — тести, code review, CI.
- **Maintenance** — логи, моніторинг, рефакторинг, борг.

**Best practices:**
- **DRY / KISS / YAGNI** як щоденний компас.
- Код пишеться **для людини**, машина зрозуміє будь-що.
- Рішення документуються (ADR — Architecture Decision Records).

**Gotcha на інтерв'ю:** «розробка» ≠ «написання коду». Якщо на питання про процес ти відповідаєш лише про синтаксис — це сигнал junior-мислення. Senior говорить про **trade-offs, ризики, supportability**.

### L3 — Senior
Ключова теза інженерії (за Фредом Бруксом, *The Mythical Man-Month* / *No Silver Bullet*): більшість складності — **есенційна** (суть задачі), а не **акцидентальна** (інструменти). Тому інженерія — це передусім **управління складністю**.

**Архітектурні трейдофи, які інженер балансує постійно:**
- **Швидкість поставки ↔ технічний борг.**
- **Гнучкість ↔ простота** (over-engineering vs. жорсткість).
- **Performance ↔ читабельність.**
- **Consistency ↔ availability** (CAP-теорема в розподілених системах).

Інженерія відрізняється від ремесла тим, що рішення **обґрунтовуються** (вимірюваннями, бенчмарками, аналізом ризиків), а не беруться «бо так звик».

## 2. Які знаєте принципи програмування?

### L1 — Junior
**Аналогія:** кухня. Є прості правила гігієни, які роблять кухню робочою: не дублюй ножі по всіх шухлядах (DRY), не неси на стіл те, що сьогодні не готуєш (YAGNI), не ускладнюй рецепт без потреби (KISS).

Базовий набір:
- **DRY** — Don't Repeat Yourself (не дублюй знання/логіку).
- **KISS** — Keep It Simple, Stupid (найпростіше робоче рішення).
- **YAGNI** — You Aren't Gonna Need It (не пиши «про запас»).

### L2 — Middle
Розширений набір, який питають на інтерв'ю:

**SOLID** (для ООП):
- **S** — Single Responsibility (один клас = одна причина для зміни).
- **O** — Open/Closed (відкритий для розширення, закритий для модифікації).
- **L** — Liskov Substitution (підклас має заміняти базовий без сюрпризів).
- **I** — Interface Segregation (багато вузьких інтерфейсів кращі за один товстий).
- **D** — Dependency Inversion (залежимо від абстракцій, не від реалізацій).

Плюс:
- **Separation of Concerns** — розділення відповідальностей по шарах.
- **Composition over Inheritance** — композиція гнучкіша за наслідування.
- **Law of Demeter** — «не розмовляй з незнайомцями» (`a.b.c.d()` — поганий знак).
- **Fail Fast** — падай рано і голосно, а не тихо ковтай помилку.

**Gotcha:** SOLID легко перецитувати, але інтерв'юер просить **приклад порушення і виправлення**. Май у голові: «God-object порушує SRP, розбиваю на сервіси».

### L3 — Senior
Принципи — це **евристики, а не закони**. Senior знає, **коли їх порушити**:
- **DRY навпаки шкодить**, коли передчасно абстрагуєш дві випадково схожі речі — потім вони розходяться, і абстракція стає тягарем. Краще «WET» (write everything twice), поки не побачив третій реальний дублікат — *Rule of Three*.
- **SOLID** виріс з ООП 2000-х; у функціональному/data-oriented підході частина принципів (DIP) реалізується через функції вищого порядку, а не через інтерфейси.

Глибший шар — **coupling & cohesion** (зв'язність і зачеплення): мета будь-якого принципу зводиться до **low coupling, high cohesion**. Це фундамент, з якого виводяться SOLID, SoC і Law of Demeter.

## 3. Чим відрізняються процедурна та об'єктно-орієнтована парадигми?

### L1 — Junior
**Аналогія:**
- **Процедурна** — кулінарна книга: список інструкцій зверху вниз. Дані (продукти) окремо, дії (кроки) окремо.
- **ООП** — кухонні прилади: тостер «знає», як робити тости, і сам тримає в собі свій стан (увімкнено/вимкнено). Дані + поведінка склеєні в один об'єкт.

**Коли:** процедурна — для коротких скриптів, лінійних задач. ООП — для великих систем з багатьма сутностями, що мають стан і поведінку.

### L2 — Middle
| | Процедурна | ООП |
|---|---|---|
| Одиниця | функція/процедура | об'єкт (дані+методи) |
| Дані | глобальні/передаються | інкапсульовані в об'єкті |
| Фокус | «що робимо» (дії) | «з чим робимо» (сутності) |
| Повторне використання | функції | класи, наслідування, композиція |
| Приклад у Python | модуль зі скриптом | `class User:` |

```python
# Процедурний стиль
def area(width: float, height: float) -> float:
    return width * height

# ООП стиль
class Rectangle:
    def __init__(self, width: float, height: float) -> None:
        self.width = width
        self.height = height

    def area(self) -> float:
        return self.width * self.height
```

**Gotcha:** Python — **мультипарадигмений**. Не треба все пхати в класи. Якщо клас має лише `__init__` і один метод без стану — це функція, переодягнена в клас (антипатерн). Senior обирає парадигму під задачу.

### L3 — Senior
У Python **усе є об'єктом** (`PyObject`), навіть функції, навіть класи (вони — екземпляри `type`). Тож «процедурний» Python — це теж об'єкти, просто ти не визначаєш власні класи.

**Трейдоф:** ООП дає **інкапсуляцію стану**, але вводить **приховану складність** — стан розмазується по об'єктах, з'являються побічні ефекти, складніше тестувати й розпаралелювати. Тому в багатопотокових/конкурентних системах часто тяжіють до **функціонального** стилю (незмінні дані, чисті функції) — менше станів = менше гонок даних.

Процедурний код часто **швидший і простіший для CPU-кеша** (дані лежать поруч, data-oriented design), тоді як ООП розкидає об'єкти по купі (heap) з рознесеними вказівниками — гірша локальність кешу. Для більшості бізнес-логіки це неважливо, для high-performance — критично.

## 4. Основні принципи ООП (інкапсуляція, наслідування, поліморфізм)

### L1 — Junior
**Аналогія — автомобіль:**
- **Інкапсуляція** — капот закритий. Ти крутиш кермо й тиснеш педаль, не знаючи, як працює двигун. Деталі сховані.
- **Наслідування** — «спортивне авто» — це «авто» з додатковими можливостями. Бере все базове + своє.
- **Поліморфізм** — педаль газу однакова в бензиновому й електро авто, але «під нею» різна реалізація. Один інтерфейс — різна поведінка.
- *(+ **Абстракція** — приладова панель показує лише потрібне, ховаючи складність.)*

### L2 — Middle
```python
class Animal:
    def __init__(self, name: str) -> None:
        self._name = name          # protected (конвенція)
        self.__id = id(self)       # private (name mangling)

    def speak(self) -> str:        # поліморфний інтерфейс
        raise NotImplementedError

class Dog(Animal):                 # наслідування
    def speak(self) -> str:
        return f"{self._name}: Гав"

class Cat(Animal):
    def speak(self) -> str:        # поліморфізм (override)
        return f"{self._name}: Мяу"

animals: list[Animal] = [Dog("Рекс"), Cat("Мурка")]
for a in animals:
    print(a.speak())               # один виклик — різна поведінка
```

**Інкапсуляція в Python — «джентльменська угода»:**
- `name` — public.
- `_name` — protected (конвенція, «не чіпай ззовні»).
- `__name` — name mangling → стає `_ClassName__name` (захист від випадкового перекриття в підкласах, **не** від доступу).

**Gotcha:** у Python **немає справжніх private**. `obj._ClassName__name` все одно доступний. Це філософія «we're all consenting adults».

### L3 — Senior
**Поліморфізм у Python — це duck typing**, а не статична перевірка типів. «Якщо крякає як качка — це качка». Інтерпретатору байдуже на тип, важлива наявність методу під час виконання (look-up в `__dict__` / MRO).

**Під капотом:**
- Доступ до атрибута `obj.attr` запускає `__getattribute__` → шукає в `type(obj).__mro__`, потім в `obj.__dict__`. Дескриптори (`property`, методи) обробляються тут.
- **Name mangling** (`__x`) робиться компілятором на етапі **парсингу байт-коду**, не в рантаймі.

**Трейдоф duck typing:** гнучкість і менше бойлерплейту ↔ помилки спливають лише в рантаймі. Тому в сучасному Python додають `typing.Protocol` — **структурна типізація** (статичний duck typing), яку перевіряє `mypy` без накладних витрат у рантаймі:

```python
from typing import Protocol

class Speaker(Protocol):
    def speak(self) -> str: ...
```

## 5. Що таке множинне наслідування?

### L1 — Junior
**Аналогія:** дитина успадковує риси і від мами, і від тата одночасно. **Множинне наслідування** — коли клас успадковується одразу від кількох батьківських класів і отримує можливості всіх.

```python
class Swimmer:
    def move(self) -> str: return "пливу"

class Flyer:
    def fly(self) -> str: return "лечу"

class Duck(Swimmer, Flyer):   # бере від обох
    pass
```

### L2 — Middle
Головна проблема — **Diamond Problem (ромб)**: клас D успадковує B і C, які обидва успадковують A. Чий метод викликати?

```
    A
   / \
  B   C
   \ /
    D
```

Python вирішує це через **MRO (Method Resolution Order)** — лінійний порядок пошуку методів. Дивимось так:

```python
print(Duck.__mro__)
# або
print(Duck.mro())
```

**Best practice:** для множинного наслідування використовують **міксини (mixins)** — невеликі класи з однією конкретною можливістю (`LoggerMixin`, `SerializableMixin`), які не мають власного стану й не призначені для самостійного інстанціювання.

**Gotcha з `super()`:** у множинному наслідуванні `super()` викликає **наступний клас у MRO**, а не «батька». Тому всі `__init__` мають акуратно передавати `**kwargs` далі по ланцюгу — інакше частина класів не проініціалізується (cooperative multiple inheritance).

### L3 — Senior
Python будує MRO за алгоритмом **C3-лінеаризації** (з'явився в Python 2.3). Він гарантує:
1. **Monotonicity** — порядок предків зберігається.
2. Дитина завжди перед батьками.
3. Порядок батьків у визначенні класу зберігається зліва направо.

C3 може **відмовитися будувати MRO** і кинути `TypeError`, якщо ієрархія суперечлива (неможливо лінеаризувати) — це фіча, що ловить криву архітектуру на старті.

`super()` без аргументів у Python 3 — це синтаксичний цукор: компілятор неявно передає `__class__` (через closure cell) і перший аргумент методу. `super()` повертає **проксі-об'єкт**, що шукає атрибут у MRO **починаючи з наступного після поточного класу**.

**Трейдоф:** множинне наслідування потужне, але робить потік виконання неочевидним. Більшість сучасних кодбаз обмежуються **міксинами + композицією**. Java/C# взагалі заборонили множинне наслідування класів (лише інтерфейси) саме через diamond problem. Python обрав «дозволити, але дисциплінувати через C3».

> ⚠️ Тут деталі реалізації C3 та closure cell для `super()` — перевір офіційну документацію (Python data model, `descrintro`) перед тим, як цитувати дослівно на співбесіді.

## 6. Шість етапів SDLC + різниця Agile vs Kanban

### L1 — Junior
**Аналогія:** будівництво будинку від ідеї до новосілля. SDLC (Software Development Life Cycle) — це конвеєр стадій, через які проходить продукт.

**6 класичних етапів:**
1. **Planning** — навіщо й кому це потрібно, ресурси, ризики.
2. **Requirements Analysis** — збір і фіксація вимог.
3. **Design** — архітектура, схема БД, інтерфейси.
4. **Implementation (Development)** — написання коду.
5. **Testing** — перевірка якості.
6. **Deployment & Maintenance** — реліз і подальша підтримка.

### L2 — Middle
**Agile** — це **філософія/набір цінностей** (Agile Manifesto, 2001): ітеративність, цінність робочого продукту, реакція на зміни. **Scrum, Kanban, XP** — це конкретні **фреймворки/методи** під цією філософією.

| | Scrum (як приклад Agile) | Kanban |
|---|---|---|
| Ритм | фіксовані спринти (1–4 тижні) | безперервний потік |
| Зміни | під час спринту не міняємо scope | можна додати в будь-який момент |
| Метрики | velocity (per sprint) | lead time, cycle time |
| Ролі | Scrum Master, PO, Team | ролей не нав'язує |
| Обмеження | обсяг спринту | **WIP limit** (ліміт задач «в роботі») |
| Дошка | очищується щоспринт | живе постійно |

**Gotcha на інтерв'ю:** часта помилка — протиставляти «Agile vs Kanban». Коректно: **Kanban — це один зі способів реалізувати Agile**. Протиставляти можна **Scrum vs Kanban**.

### L3 — Senior
Kanban виріс з **Lean / Toyota Production System** і теорії черг. Його суть — **обмеження WIP** (Work In Progress). Чому це критично: за **законом Літтла** (Little's Law) середній час проходу задачі = (кількість задач в роботі) / (пропускна здатність). Менше задач паралельно → коротший cycle time → швидший фідбек.

Scrum натомість дає **передбачуваність** (планування спринтами) ціною гнучкості.

**Трейдоф:** Scrum кращий, коли потрібні дедлайни й прогнозованість релізів; Kanban — для команд підтримки/інфраструктури з непередбачуваним потоком (інциденти, баги). На практиці часто **Scrumban** — гібрид.

Окремо: класичний 6-етапний SDLC у «водоспадному» (Waterfall) вигляді — послідовний, з повним проходом один раз. Agile робить ці **6 етапів циклічними**: кожен спринт проходить через design→dev→test у мініатюрі.

## 7. Методи HTTP-запитів і різниця між ними

### L1 — Junior
**Аналогія — поштові операції з документом:**
- **GET** — прочитати документ (нічого не змінює).
- **POST** — створити новий документ.
- **PUT** — повністю замінити документ.
- **PATCH** — виправити одне поле в документі.
- **DELETE** — викинути документ.

### L2 — Middle
| Метод | Призначення | Safe? | Idempotent? | Тіло |
|---|---|---|---|---|
| GET | отримати ресурс | ✅ | ✅ | ні |
| POST | створити / дія | ❌ | ❌ | так |
| PUT | замінити повністю | ❌ | ✅ | так |
| PATCH | часткове оновлення | ❌ | ❌* | так |
| DELETE | видалити | ❌ | ✅ | інколи |
| HEAD | як GET, але без тіла (тільки заголовки) | ✅ | ✅ | ні |
| OPTIONS | які методи дозволені (CORS preflight) | ✅ | ✅ | ні |

- **Safe** — не змінює стан сервера.
- **Idempotent** — N однакових запитів = той самий ефект, що й 1.

**Gotcha:** ключова пара на інтерв'ю — **POST vs PUT**. POST створює і **не ідемпотентний** (два POST = два записи). PUT ідемпотентний (двічі покласти той самий ресурс = один результат). PATCH формально **не гарантовано** ідемпотентний (`*` — залежить від реалізації, напр. `increment` не ідемпотентний).

### L3 — Senior
Ідемпотентність — не просто теорія, це **основа надійних розподілених систем**. Якщо мережа обірвалась і клієнт не знає, чи дійшов запит, він може **безпечно повторити** лише ідемпотентний метод. Для POST цю проблему вирішують **idempotency key** (унікальний заголовок, за яким сервер дедуплікує).

Семантика методів — це **контракт, а не примус**: технічно можна змінювати дані через GET, але це порушує очікування проксі/кешів/краулерів (вони вважають GET безпечним і можуть повторити/закешувати) → баги й діри безпеки.

**Кешування:** GET/HEAD кешуються за замовчуванням; POST/PUT/DELETE/PATCH — ні (вони «unsafe»). Це впливає на дизайн API й CDN.

## 8. Як виглядають HTTP request / response?

### L1 — Junior
**Аналогія — лист у конверті:** є **адреса й тип операції** (стартовий рядок), **службові позначки на конверті** (заголовки) і **сам лист усередині** (тіло). І запит, і відповідь мають цю структуру.

### L2 — Middle
**HTTP Request:**
```http
POST /api/users HTTP/1.1          ← стартовий рядок: метод + шлях + версія
Host: example.com                 ← заголовки (headers)
Content-Type: application/json
Authorization: Bearer eyJhb...
Content-Length: 27
                                  ← порожній рядок-роздільник
{"name": "Іван", "age": 30}       ← тіло (body)
```

**HTTP Response:**
```http
HTTP/1.1 201 Created              ← статус-рядок: версія + код + причина
Content-Type: application/json
Content-Length: 38

{"id": 1, "name": "Іван", "age": 30}
```

**Класи статус-кодів:**
- **1xx** — інформаційні.
- **2xx** — успіх (200 OK, 201 Created, 204 No Content).
- **3xx** — редиректи (301 Moved, 304 Not Modified).
- **4xx** — помилка клієнта (400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests).
- **5xx** — помилка сервера (500 Internal, 502 Bad Gateway, 503 Unavailable).

**Gotcha:** **401 vs 403**. 401 = «я не знаю, хто ти» (немає/невалідна автентифікація). 403 = «я знаю, хто ти, але тобі сюди не можна» (немає прав).

### L3 — Senior
HTTP — це **stateless текстовий протокол** поверх TCP (HTTP/1.1). Ключове:
- **HTTP/1.1** — текст, одне «in-flight» з'єднання → проблема head-of-line blocking; вирішується keep-alive і pipelining (рідко вмикають).
- **HTTP/2** — бінарний, **мультиплексування** потоків в одному TCP-з'єднанні, стиснення заголовків (HPACK).
- **HTTP/3** — поверх **QUIC (UDP)**, прибирає TCP head-of-line blocking на рівні транспорту.

**Stateless** — фундаментальний трейдоф: сервер не пам'ятає попередні запити → легко масштабувати горизонтально (будь-який інстанс обробить будь-який запит), але стан (хто залогінений) треба нести в кожному запиті → звідси cookies/токени (питання 9, 10).

## 9. Що таке авторизація і як вона працює?

### L1 — Junior
**Аналогія — нічний клуб:**
- **Автентифікація (Authentication, AuthN)** — фейс-контроль перевіряє паспорт: «ти справді той, за кого себе видаєш?»
- **Авторизація (Authorization, AuthZ)** — браслет VIP: «що тобі дозволено» (зайти в VIP-зону чи ні).

Спочатку AuthN (хто ти), потім AuthZ (що тобі можна).

### L2 — Middle
**Типові механізми:**
1. **Session-based** (stateful): після логіну сервер створює сесію, зберігає її (Redis/БД), віддає клієнту `session_id` у cookie. Кожен запит → сервер шукає сесію.
2. **Token-based / JWT** (stateless): сервер видає підписаний токен, клієнт носить його в `Authorization: Bearer <token>`. Сервер перевіряє підпис, **не звертаючись до сховища**.
3. **OAuth 2.0 / OIDC** — делегований доступ («Увійти через Google»). Видається access token, не пароль.

**Моделі прав:**
- **RBAC** — Role-Based (ролі: admin, user).
- **ABAC** — Attribute-Based (за атрибутами: відділ, час, регіон).

**Gotcha:** **не плутай AuthN і AuthZ** — улюблене питання. І: **JWT не можна «відкликати» легко** (він валідний до `exp`), тому для logout роблять короткий TTL + refresh token, або blacklist.

### L3 — Senior
**JWT під капотом** — три base64url-частини через крапку: `header.payload.signature`.
- Payload — **не зашифрований**, лише закодований (base64) → **не клади туди секрети**.
- Signature = `HMAC(header.payload, secret)` (симетрично, HS256) або RSA/ECDSA (асиметрично, RS256). Підпис гарантує **цілісність**, а не конфіденційність.

**Трейдофи stateful vs stateless:**
| | Session (stateful) | JWT (stateless) |
|---|---|---|
| Зберігання | сервер | клієнт |
| Масштабування | потрібен спільний store | не потрібен |
| Відкликання | миттєве (видалив сесію) | складне (живе до exp) |
| Розмір на запит | малий (id) | великий (весь токен) |

**Класична вразливість JWT:** `alg: none` — старі бібліотеки приймали токен без підпису. Завжди валідуй очікуваний алгоритм на сервері.

> ⚠️ Деталі підпису й валідації JWT — звіряйся з RFC 7519 та документацією твоєї бібліотеки (`PyJWT`), не покладайся на пам'ять у проді.

## 10. Що таке cookies?

### L1 — Junior
**Аналогія — гардеробний номерок.** HTTP «забудькуватий» (stateless): після кожного запиту сервер тебе забуває. Cookie — це номерок, який сервер дає браузеру, а браузер показує його при кожному наступному запиті, щоб сервер «згадав» тебе.

### L2 — Middle
Cookie — невеликий шматок даних, який сервер ставить заголовком `Set-Cookie`, а браузер автоматично надсилає назад у заголовку `Cookie`.

```http
Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Strict; Max-Age=3600; Path=/
```

**Критичні атрибути безпеки:**
- **HttpOnly** — JS не має доступу (`document.cookie`), захист від XSS-крадіжки.
- **Secure** — надсилається лише по HTTPS.
- **SameSite** (`Strict`/`Lax`/`None`) — захист від CSRF; контролює відправку cookie на крос-сайтові запити.
- **Max-Age / Expires** — час життя (session cookie без них живе до закриття браузера).
- **Domain / Path** — область дії.

**Gotcha:** cookie з токеном **без `HttpOnly`** = подарунок для XSS. А `SameSite=None` **обов'язково** з `Secure`.

### L3 — Senior
Cookie ↔ **CSRF (Cross-Site Request Forgery)**: оскільки браузер шле cookie **автоматично** на домен, зловмисний сайт може спровокувати запит до твого банку від імені залогіненого юзера. Захист: `SameSite`, anti-CSRF токени (double submit / synchronizer token).

**Трейдоф зберігання токена:**
- **Cookie (HttpOnly)** — захищено від XSS, але вразливо до CSRF (лікується SameSite).
- **localStorage** — імунний до CSRF (JS сам кладе в заголовок), але **вразливий до XSS** (будь-який інжектнутий скрипт прочитає).

Senior-висновок: для веб-сесій частіше обирають **HttpOnly+Secure+SameSite cookie**, бо XSS вважається небезпечнішим і ширшим вектором, ніж CSRF.

Обмеження: ~4KB на cookie, ліміт штук на домен → cookie не для зберігання даних, лише ідентифікаторів.

## 11. Що таке вебвразливість?

### L1 — Junior
**Аналогія:** незамкнені двері або вікно в будинку, через які може залізти злодій. **Вебвразливість** — слабке місце в застосунку, яке дозволяє атакувальнику зробити те, що не передбачалося: вкрасти дані, видати себе за іншого, зламати систему.

Орієнтир — **OWASP Top 10** (список найпоширеніших вразливостей).

### L2 — Middle
Найважливіші для інтерв'ю:

- **SQL Injection** — підстановка SQL через незекрановані вхідні дані. Лікування: **параметризовані запити / ORM**, ніколи не конкатенуй SQL рядками.
- **XSS (Cross-Site Scripting)** — інжект JS у сторінку. Лікування: екранування виводу, `Content-Security-Policy`, `HttpOnly` cookie.
- **CSRF** — підроблений запит від імені юзера. Лікування: `SameSite`, CSRF-токени.
- **Broken Access Control** — доступ до чужих ресурсів (`/orders/123` → міняю на `/orders/124` = IDOR). Лікування: перевірка прав на **кожен** ресурс на сервері.
- **Insecure Deserialization**, **SSRF**, **Security Misconfiguration**.

```python
# ❌ ВРАЗЛИВО до SQL injection
cursor.execute(f"SELECT * FROM users WHERE name = '{name}'")

# ✅ Параметризований запит
cursor.execute("SELECT * FROM users WHERE name = %s", (name,))
```

**Gotcha:** «валідація на фронті» — це UX, **не безпека**. Будь-яку перевірку дублюй на сервері, бо фронт під контролем атакувальника.

### L3 — Senior
Корінь більшості вразливостей — **змішування даних і коду / довіра до вводу**. SQLi, XSS, command injection — це все один клас: дані користувача потрапляють у контекст інтерпретації (SQL, HTML, shell). Принцип захисту єдиний: **separation of code and data** (параметризація, екранування під цільовий контекст).

**Defense in depth** — кілька шарів захисту, бо один завжди може впасти:
1. Валідація вводу (allowlist > denylist).
2. Параметризація на межі (БД, шаблонізатор).
3. Принцип найменших привілеїв (least privilege) для сервісів/БД-юзерів.
4. Моніторинг/WAF.

**Трейдоф:** абсолютної безпеки нема — є **управління ризиком**. Senior мислить **threat model**: хто атакувальник, що захищаємо, яка ціна злому vs ціна захисту.

## 12. Які знаєте класичні бази даних?

### L1 — Junior
**Аналогія:**
- **Реляційна (SQL)** — Excel із суворими зв'язаними таблицями: рядки, колонки, чіткі правила.
- **NoSQL** — комора з різними коробками: документи, ключ-значення, графи — гнучко, без жорсткої схеми.

### L2 — Middle
**Реляційні (SQL):** PostgreSQL, MySQL, SQLite, MS SQL Server, Oracle.
- Дані в таблицях зі зв'язками (FK), сувора схема, **транзакції ACID**, мова SQL.

**NoSQL** — 4 типи:
- **Document** — MongoDB (JSON-документи).
- **Key-Value** — Redis (кеш, сесії, черги).
- **Column-family** — Cassandra (величезні обсяги, запис).
- **Graph** — Neo4j (зв'язки: соцмережі, рекомендації).

| | SQL | NoSQL |
|---|---|---|
| Схема | жорстка | гнучка |
| Масштабування | вертикальне (в основному) | горизонтальне |
| Транзакції | сильні ACID | часто eventual consistency |
| Коли | складні зв'язки, гроші | великі обсяги, гнучкі дані |

**Gotcha:** «NoSQL швидший за SQL» — міф. Питання не в швидкості, а в **моделі даних і consistency-гарантіях**. PostgreSQL з JSONB часто закриває потреби, де поспішають брати Mongo.

### L3 — Senior
**ACID** (реляційні): Atomicity, Consistency, Isolation, Durability.
**Рівні ізоляції** (за SQL-стандартом, від слабшого до сильнішого): Read Uncommitted → Read Committed → Repeatable Read → Serializable. Кожен прибирає аномалії (dirty read, non-repeatable read, phantom read) ціною конкурентності.

**CAP-теорема** (розподілені): за наявності мережевого розриву (**P**) обираєш між **C** (consistency) і **A** (availability). Класичні SQL тяжіють до CP, багато NoSQL — до AP (eventual consistency).

**Під капотом індексів:** більшість SQL використовують **B-tree** → пошук/вставка `O(log n)`. Тому індекс на колонці перетворює `O(n)` full-scan на `O(log n)`. Redis у пам'яті дає `O(1)` за ключем (хеш-таблиця). Трейдоф індексу: швидкий read ↔ повільніший write + витрата місця.

> ⚠️ Назви рівнів ізоляції стандартні, але **їх реальна реалізація відрізняється** (PostgreSQL використовує MVCC і трактує деякі рівні строгіше за стандарт). Перевіряй документацію конкретної СУБД.

## 13. Як читати специфікацію мови (наприклад, PEP8)?

### L1 — Junior
**Аналогія:** PEP — це офіційні «закони» Python. **PEP8** — конкретний закон про **стиль коду** (як форматувати: відступи, імена, довжина рядка). Читати специфікацію = знати, де першоджерело правди, замість «мені здається».

### L2 — Middle
**PEP** = Python Enhancement Proposal. Категорії:
- **Standards Track** — зміни в мові (PEP 8 — стиль, PEP 484 — type hints, PEP 572 — walrus `:=`).
- **Informational** — рекомендації.
- **Process** — процеси спільноти.

**Ключове з PEP8:**
- 4 пробіли на відступ, max довжина рядка 79 (часто 88–120 у командах).
- `snake_case` для функцій/змінних, `PascalCase` для класів, `UPPER_CASE` для констант.
- 2 порожні рядки між функціями верхнього рівня.

**Best practice:** PEP8 **не зубрять руками** — його ензфорсять інструменти: `ruff`, `black`, `flake8`. У цьому проєкті — `ruff` (див. CLAUDE.md).

**Як читати специфікацію взагалі:**
1. Заголовок/Abstract — про що.
2. Motivation — навіщо.
3. Specification — формальні правила (тут суть).
4. Examples / Rationale — приклади й обґрунтування.
5. Backwards Compatibility — що ламає.

### L3 — Senior
PEP8 сам каже: **«A Foolish Consistency is the Hobgoblin of Little Minds»** — стиль не догма, узгодженість всередині проєкту важливіша за букву PEP. Команда може legально розширити лінію до 88/120 (як `black`).

Для глибокого розуміння мови читають не лише PEP, а:
- **Language Reference** (граматика, data model — `object.__dunder__`).
- **CPython-сорси** (`Objects/`, `Python/ceval.c`) — як це реально працює.
- **PEP-и фіч** (484 typing, 525 async gen, 634 structural pattern matching) — щоб розуміти **чому** мова така.

Senior читає специфікацію, щоб розуміти **rationale й trade-offs** автора (напр. чому GIL існує — PEP-обговорення), а не лише синтаксис.

## 14. Як відбувається взаємодія клієнта і сервера?

### L1 — Junior
**Аналогія — ресторан:** ти (**клієнт/браузер**) робиш замовлення офіціанту (**запит**), кухня (**сервер**) готує й офіціант приносить страву (**відповідь**). Ти не знаєш, як влаштована кухня — лише замовляєш і отримуєш.

### L2 — Middle
Повний шлях запиту `https://example.com`:
1. **DNS** — браузер питає у DNS IP-адресу домену.
2. **TCP-з'єднання** — three-way handshake (SYN → SYN-ACK → ACK).
3. **TLS handshake** — узгодження ключів шифрування (для HTTPS).
4. **HTTP Request** — браузер шле `GET / HTTP/1.1`.
5. Сервер обробляє (роутинг → бізнес-логіка → БД).
6. **HTTP Response** — статус + заголовки + тіло (HTML/JSON).
7. Браузер рендерить / клієнт парсить JSON.

Це **client-server модель**: клієнт ініціює, сервер відповідає, протокол stateless.

**Gotcha:** часта плутанина — «де виконується код». Frontend (JS) — у браузері клієнта; backend (Python) — на сервері. API — це **контракт** між ними (JSON по HTTP).

### L3 — Senior
**Latency складається з шарів**, і кожен — точка оптимізації:
- DNS lookup (кешується).
- TCP handshake — 1 RTT; TLS 1.2 — 2 RTT, **TLS 1.3 — 1 RTT** (швидше), 0-RTT для повторних.
- Передача даних (обмежена bandwidth + congestion control TCP).

**Альтернативи request-response** для real-time:
- **WebSocket** — повнодуплексний канал поверх одного TCP (чати, ігри).
- **Server-Sent Events (SSE)** — односторонній стрім сервер→клієнт.
- **Long polling** — старий хак до WebSocket.

**Трейдоф:** stateless HTTP легко масштабувати (load balancer кидає запит на будь-який інстанс), але для постійного зв'язку (push) доводиться тримати **stateful** з'єднання (WebSocket), що ускладнює масштабування (sticky sessions, окремий шар pub/sub типу Redis).

## 15. Які є підходи до проєктування API?

### L1 — Junior
**Аналогія — меню в ресторані.** API — це меню: список того, що можна замовити в сервера, і в якому форматі. Хороше меню зрозуміле без пояснень офіціанта.

Основні стилі: **REST**, **GraphQL**, **gRPC**, старий **SOAP/RPC**.

### L2 — Middle
**REST** (найпоширеніший):
- Ресурси як URL-іменники: `/users`, `/users/1/orders`.
- Дії — через HTTP-методи (GET/POST/PUT/DELETE), не в URL (`/getUsers` — антипатерн).
- Stateless, JSON, статус-коди за призначенням.
- Версіонування: `/api/v1/...`.

**GraphQL:**
- Один endpoint, клієнт сам описує, які поля хоче (немає over/under-fetching).
- Сильна типізована схема.

**gRPC:**
- Бінарний (Protobuf) поверх HTTP/2, дуже швидкий, для **сервіс-сервіс** комунікації (мікросервіси).

| | REST | GraphQL | gRPC |
|---|---|---|---|
| Формат | JSON | JSON | Protobuf (бінарний) |
| Endpoints | багато | один | методи |
| Over-fetching | можливе | вирішено | — |
| Де | публічні API | складні фронти | внутрішні сервіси |

**Best practices REST:** консистентні імена (множина: `/users`), пагінація, фільтрація через query (`?status=active`), правильні статус-коди, ідемпотентність (питання 7), HATEOAS (опційно).

**Gotcha:** дієслова в REST-URL (`/createUser`) — найчастіша помилка дизайну. Дія = HTTP-метод, URL = ресурс.

### L3 — Senior
**Design-first vs Code-first:**
- **Design-first** — спочатку контракт (**OpenAPI/Swagger** spec, `.proto`), потім код. Дозволяє паралельну роботу фронту й бекенду, генерацію клієнтів/моків.
- **Code-first** — код породжує спеку (FastAPI генерує OpenAPI з Pydantic-моделей автоматично).

**Трейдофи:**
- **REST** — простота, кешованість (HTTP-кеш), всюди підтримується ↔ over/under-fetching, багато round-trip'ів.
- **GraphQL** — гнучкість клієнта ↔ складність кешування (усе POST), ризик дорогих запитів (N+1, треба DataLoader, query depth limit).
- **gRPC** — швидкість, контракт, стрімінг ↔ не для браузера напряму (треба grpc-web), бінарний формат важче дебажити.

Senior обирає за **контекстом споживача**: публічне партнерське API → REST/OpenAPI; складний продуктовий фронт із багатьма джерелами → GraphQL; high-throughput внутрішня комунікація → gRPC.

## 16. Як використовувати патерни програмування?

### L1 — Junior
**Аналогія:** патерни — це **перевірені рецепти** для типових задач. Як шеф не вигадує щоразу, як зварити бульйон, так і розробник не вигадує заново, як створювати об'єкти чи сповіщати підписників — бере готовий перевірений підхід (**GoF — Gang of Four patterns**).

### L2 — Middle
Три групи (GoF):
- **Creational** (створення): Singleton, Factory, Builder.
- **Structural** (структура): Adapter, Decorator, Facade, Proxy.
- **Behavioral** (поведінка): Strategy, Observer, Iterator, Command.

**У Python багато патернів «вбудовані»** і не потребують класичної реалізації:
- **Iterator** — протокол `__iter__/__next__`, генератори.
- **Decorator** — синтаксис `@decorator` (хоча це не зовсім GoF-decorator).
- **Strategy** — просто передай функцію (first-class functions).
- **Singleton** — модуль уже синглтон; часто роблять через модульну змінну, а не клас.

```python
# Strategy в Python — без класів, просто функція
def process(data: list[int], strategy: Callable[[list[int]], int]) -> int:
    return strategy(data)

process([1, 2, 3], sum)
process([1, 2, 3], max)
```

**Gotcha:** **«patternitis»** — натягувати важкий Java-патерн там, де Python робить це однією функцією. Це antipattern. Senior знає патерн, але застосовує **найлегше рішення**.

### L3 — Senior
Патерни — це **словник для комунікації**, а не самоціль. Сказати «тут Observer» команді швидше, ніж описувати реалізацію. Але:

> «Design patterns are signs of weakness in a programming language» — частково правда: багато GoF-патернів існують, щоб обійти обмеження Java/C++ (відсутність функцій вищого порядку, метакласів). У Python із first-class functions, duck typing, метакласами й дескрипторами половина патернів **розчиняється в мові**.

**Трейдоф:** патерн додає **рівень непрямості (indirection)** — гнучкість ↔ складність розуміння. Правило: вводь патерн, **коли вже є реальний біль** (зміни постійно ламають код), а не «про запас» (YAGNI). Передчасний патерн = over-engineering.

Senior також знає **антипатерни** (God Object, Spaghetti, Golden Hammer) і **архітектурні патерни** вищого рівня (MVC, Repository, CQRS, Event Sourcing), які важливіші за GoF у реальній роботі.

## 17. Що таке Acceptance Testing і навіщо він?

### L1 — Junior
**Аналогія:** приймання квартири від забудовника. Ти не перевіряєш марку цементу (це робив інженер) — ти перевіряєш, що **двері відчиняються, вода тече, світло є**, тобто що квартира **придатна для життя** з погляду майбутнього мешканця.

**Acceptance Testing** перевіряє, що система робить те, що **потрібно бізнесу/користувачу**, а не «чи правильно працює функція».

### L2 — Middle
- Перевіряє відповідність **бізнес-вимогам / acceptance criteria** (часто з user story: «As a user I want… so that…»).
- Часто пишуть **QA + бізнес/PO**, не лише розробники.
- **UAT (User Acceptance Testing)** — фінальна перевірка реальними користувачами перед релізом.
- Формалізують через **BDD** (Given-When-Then), інструменти: Cucumber, `behave`, `pytest-bdd`.

```gherkin
Feature: Логін
  Scenario: Успішний вхід
    Given користувач зареєстрований
    When він вводить правильні логін і пароль
    Then він потрапляє на дашборд
```

**Навіщо:** ловить розрив між «код працює» і «продукт вирішує задачу». Unit-тести можуть бути зелені, а фіча — не та, що просили.

### L3 — Senior
Місце в **піраміді тестування** — вершина: повільні, дорогі, крихкі, але перевіряють **end-to-end бізнес-цінність**. Тому їх **мало** (одиниці критичних сценаріїв), на відміну від багатьох unit-тестів унизу.

**Трейдоф:** acceptance/E2E дають найбільше впевненості в «продукт працює цілісно», але:
- повільні (підіймають всю систему),
- flaky (мережа, тайминги, зовнішні сервіси),
- дорогі в підтримці.

Стратегія Senior: критичні **happy paths** покрити acceptance-тестами, а граничні випадки й логіку — дешевшими unit-тестами нижче. BDD-сценарії додатково слугують **живою документацією** вимог.

## 18. Модульні, інтеграційні, API-тести

### L1 — Junior
**Аналогія — складання авто:**
- **Unit (модульні)** — перевіряємо кожну деталь окремо: чи крутиться це колесо?
- **Integration (інтеграційні)** — чи працюють деталі **разом**: чи передає коробка передач крутний момент на колеса?
- **API-тести** — натискаємо на педалі ззовні й дивимось, чи їде авто, не залазячи під капот.

### L2 — Middle
| Тип | Що перевіряє | Залежності | Швидкість |
|---|---|---|---|
| **Unit** | одну функцію/клас ізольовано | мокаються | дуже швидко |
| **Integration** | взаємодію кількох компонентів (код + БД, два сервіси) | реальні (БД, черга) | повільніше |
| **API / E2E** | поведінку через HTTP-ендпоінти | вся система | найповільніше |

```python
# Unit — ізольовано, з моком БД
def test_calculate_discount() -> None:
    assert calculate_discount(price=100, percent=10) == 90

# API-тест (FastAPI)
def test_create_user(client: TestClient) -> None:
    resp = client.post("/users", json={"name": "Іван"})
    assert resp.status_code == 201
    assert resp.json()["name"] == "Іван"
```

**Gotcha:** «unit-тест, що ходить у реальну БД» — це вже **integration-тест**. Unit за визначенням **ізольований** (мок/стаб зовнішніх залежностей).

### L3 — Senior
**Піраміда тестування** (Mike Cohn): багато unit → менше integration → ще менше E2E. Чому така форма — **трейдоф вартість/швидкість/впевненість**:
- Unit: дешеві, швидкі, точно локалізують баг, але не ловлять помилки інтеграції.
- E2E: ловлять реальні баги взаємодії, але повільні, flaky, погано локалізують («щось зламалось десь»).

**Антипатерни форми:**
- **Ice cream cone** (перевернута піраміда) — багато E2E, мало unit → повільний flaky CI.
- **Hourglass** — багато unit і E2E, провал по integration.

Senior балансує: тестова стратегія = функція від ризику й вартості. Для мікросервісів важливі **contract tests** (Pact) — перевірка, що сервіси не зламали контракт один одному, без підняття всього E2E.

## 19. Як писати unit-тести?

### L1 — Junior
**Аналогія:** перевірка калькулятора. Натиснув `2 + 2`, очікуєш `4`, звіряєш. Unit-тест — це автоматизована перевірка: «дав вхід → отримав очікуваний вихід».

Структура **AAA**:
- **Arrange** — підготуй дані.
- **Act** — виклич те, що тестуєш.
- **Assert** — звір результат.

### L2 — Middle
```python
import pytest

def divide(a: float, b: float) -> float:
    if b == 0:
        raise ValueError("division by zero")
    return a / b

# Happy path
def test_divide_returns_quotient() -> None:
    # Arrange
    a, b = 10, 2
    # Act
    result = divide(a, b)
    # Assert
    assert result == 5

# Edge case / помилка
def test_divide_by_zero_raises() -> None:
    with pytest.raises(ValueError, match="division by zero"):
        divide(10, 0)

# Параметризація — багато кейсів одним тестом
@pytest.mark.parametrize("a, b, expected", [
    (10, 2, 5),
    (9, 3, 3),
    (-6, 2, -3),
])
def test_divide_cases(a: float, b: float, expected: float) -> None:
    assert divide(a, b) == expected
```

**Best practices:**
- **Один тест — одна поведінка** (одна логічна перевірка).
- Назва описує **що і за яких умов**: `test_divide_by_zero_raises`.
- Тестуй **happy path + edge cases + помилки** (порожній список, 0, None, межі).
- **Fixtures** (`@pytest.fixture`) для спільної підготовки.
- **Mock** зовнішніх залежностей (`unittest.mock`, `monkeypatch`).

**Gotcha:** тест **без `assert`** — найгірший: він зелений, але нічого не перевіряє (false confidence).

### L3 — Senior
**Що робить тест якісним (FIRST):**
- **Fast** — мілісекунди (інакше їх перестають ганяти).
- **Isolated/Independent** — не залежать від порядку й один від одного (жодного спільного стану).
- **Repeatable** — той самий результат завжди (без `datetime.now()`, random, мережі — їх мокають).
- **Self-validating** — pass/fail без ручної перевірки очима.
- **Timely** — пишуться вчасно (ідеально — TDD: red→green→refactor).

**Тестуй поведінку, а не реалізацію.** Тест, прив'язаний до внутрішніх викликів (over-mocking), ламається на кожному рефакторингу, хоча поведінка не змінилась → tests as a maintenance burden.

**Трейдоф mocking:** мок ізолює й прискорює, але **відв'язує від реальності** — можна мокнути так, що тест зелений, а інтеграція падає. Правило: мокай **межі системи** (мережа, час, зовнішнє API), не мокай **власну логіку**.

**Coverage — необхідна, але не достатня умова:** 100% покриття ≠ відсутність багів (можна виконати рядок, не перевіривши результат). Корисніший орієнтир — **mutation testing** (`mutmut`): чи ловлять тести реально внесені баги.

## 20. Best practices автотестів

### L1 — Junior
**Аналогія:** хороша інструкція до тесту має бути такою, щоб через рік **ти сам зрозумів**, що й навіщо перевіряється, без розшифровки. Тести — теж код, і їх теж читають люди.

Коротко: тести мають бути **читабельні, незалежні, надійні, швидкі**.

### L2 — Middle
**Чек-лист best practices:**
1. **Зрозумілі назви** — `test_<що>_<умова>_<очікування>`.
2. **AAA / Given-When-Given** структура.
3. **Незалежність** — будь-який тест запускається сам по собі й у будь-якому порядку.
4. **Детермінованість** — ніякого `random`, `now()`, реальної мережі без моку → **freeze time** (`freezegun`), seed для random.
5. **Одна поведінка на тест.**
6. **Тестуй поведінку, не приватні методи.**
7. **Не дублюй** — fixtures, factories (`factory_boy`), параметризація.
8. **Тести в CI** — зелений pipeline як умова мержу.
9. **Покривай баги тестом** — знайшов баг → спочатку тест, що його відтворює (regression test).

**Gotcha:** **flaky-тести** (то зелені, то червоні) — гірше за відсутність тестів: команда перестає їм довіряти й ігнорує червоний CI. Flaky треба чинити або видаляти негайно.

### L3 — Senior
**Test smells (антипатерни), які впізнає Senior:**
- **Fragile tests** — ламаються від будь-якої зміни коду (over-coupling до реалізації).
- **Slow tests** — їх перестають ганяти локально.
- **Mystery guest** — тест залежить від зовнішніх даних/файлів, прихованих від читача.
- **Test interdependence** — тест B працює лише після A.
- **Assertion roulette** — багато assert без повідомлень → незрозуміло, який упав.

**Стратегічні трейдофи:**
- **Тести як документація** ↔ тести як тягар: добре названі тести описують контракт; over-mocked — лише гальмують рефакторинг.
- **Швидкість CI** ↔ повнота: розшаровують (fast unit на кожен пуш, повільні E2E — на nightly/pre-merge).
- **DRY у тестах має межу:** надмірні абстракції в тестах знижують читабельність — тут інколи **DAMP > DRY** (Descriptive And Meaningful Phrases): трохи дублювання заради ясності кожного тесту окремо.

Головний принцип: **тест має падати з однієї причини й чітко казати, що зламалось.**

## 21. Базові команди системи контролю версій

### L1 — Junior
**Аналогія:** VCS (система контролю версій) — це «машина часу» + «журнал змін» для коду. Можна повернутися до будь-якої збереженої точки й побачити, хто, що й коли змінив. Найпопулярніша — **Git**.

### L2 — Middle
Базовий цикл роботи:
```bash
git init                    # створити репозиторій
git clone <url>             # склонувати існуючий

git status                  # що змінилось
git add <file> / git add .  # додати в staging
git commit -m "message"     # зафіксувати зміни

git pull                    # забрати зміни з remote
git push                    # відправити свої

git branch <name>           # створити гілку
git checkout <name>         # перемкнутись (або: git switch <name>)
git checkout -b <name>      # створити + перемкнутись
git merge <name>            # злити гілку

git log                     # історія комітів
git diff                    # переглянути зміни
```

**Три зони Git:**
- **Working Directory** — твої файли.
- **Staging Area (Index)** — що піде в наступний коміт (`git add`).
- **Repository** — зафіксована історія (`git commit`).

**Gotcha:** `git add` ≠ збереження. Дані фіксуються лише `commit`. А `commit` ≠ публікація — на сервер потрапляє лише після `push`.

### L3 — Senior
Git — **розподілена** VCS (DVCS): на відміну від централізованих (SVN, CVS), **кожен клон містить повну історію**. Тому commit/log/diff/branch працюють **локально й миттєво**, без мережі.

**Під капотом** Git — це **контентно-адресований сховище** (key-value):
- Усе ідентифікується **SHA-1/SHA-256 хешем** вмісту.
- 4 типи об'єктів: **blob** (вміст файлу), **tree** (директорія), **commit** (знімок + метадані + посилання на батьків), **tag**.
- **Коміт — це знімок (snapshot) усього дерева**, а не diff (diff обчислюється на льоту). Це ключова відмінність від SVN.
- **Гілка — це просто рухомий вказівник** (файл з хешем коміта в `.git/refs/heads/`), тому створення гілки в Git — `O(1)`, майже безкоштовне.

> ⚠️ Git переходить з SHA-1 на SHA-256; внутрішня модель об'єктів стабільна, але деталі звіряй з Pro Git book / git internals docs.

## 22. Як використовувати Git?

### L1 — Junior
**Аналогія:** Git — як збереження в грі. Перед складним босом (ризикованою зміною) робиш save (commit), щоб у разі провалу повернутись. А branch — це окрема «лінія збереження», де можна експериментувати, не псуючи основну гру.

Типовий день:
```bash
git switch -c feature/login   # нова гілка під задачу
# ...пишеш код...
git add .
git commit -m "feat: add login form"
git push -u origin feature/login
# відкриваєш Pull Request
```

### L2 — Middle
**Feature Branch Workflow** (стандарт у командах):
1. Гілка від `main` під кожну задачу (`feature/...`, `fix/...`).
2. Невеликі атомарні коміти.
3. **Pull/Merge Request** → code review → CI.
4. Merge у `main`.

**Conventional Commits** (читабельна історія):
```
feat: add user registration
fix: correct discount calculation
refactor: extract payment service
docs: update README
```

Корисні команди:
```bash
git stash                   # сховати незакомічене, щоб перемкнутись
git rebase main             # «перенести» гілку поверх свіжого main
git cherry-pick <hash>      # взяти один коміт з іншої гілки
git revert <hash>           # безпечно скасувати коміт (новим комітом)
git reset --soft HEAD~1     # відкотити коміт, лишивши зміни
```

**Gotcha:** **`reset` vs `revert`**. `revert` створює **новий** коміт-антагоніст (безпечно для спільної гілки). `reset` **переписує історію** (небезпечно на спільній гілці — зламає колегам).

### L3 — Senior
**Merge vs Rebase** — головний архітектурний вибір історії:
- **Merge** — зберігає реальну історію з merge-комітом (нелінійна, «правдива», але «кошлата»).
- **Rebase** — переписує коміти поверх цілі → **лінійна чиста історія**, але **змінює хеші** (нові об'єкти).

**Золоте правило rebase:** ніколи не роби rebase **опублікованої спільної гілки** — переписані хеші розсинхронять усіх, хто на ній працює. Rebase — лише для **своєї локальної** ще не запушеної гілки (або з `--force-with-lease` і повним розумінням наслідків).

**`git reflog`** — рятувальний круг Senior: Git тримає журнал, **куди вказував HEAD**, тож майже будь-яку «втрачену» гілку/коміт після reset чи rebase можна відновити (об'єкти живуть до garbage collection).

**Трейдоф стратегій merge у PR:**
- **Merge commit** — повна історія, видно контекст гілки.
- **Squash** — один чистий коміт на фічу (чистий `main`, але втрата гранулярності).
- **Rebase-merge** — лінійна історія без merge-комітів.
Команда обирає одну політику; це впливає на читабельність `git log` і `git bisect` (пошук комітa, що вніс баг, бінарним пошуком — `O(log n)` комітів).

## 23. Різниця між хешуванням і шифруванням

### L1 — Junior
**Аналогія:**
- **Шифрування (encryption)** — сейф із ключем. Поклав документ, замкнув, **ключем можна відкрити назад**. Двостороннє.
- **Хешування (hashing)** — м'ясорубка. Закинув м'ясо → отримав фарш. **Назад фарш у шматок не зібрати.** Одностороннє.

**Коли:** шифрування — коли треба **повернути** дані (передати й розшифрувати). Хешування — коли назад **не треба** (перевірка паролів, цілісність файлів).

### L2 — Middle
| | Хешування | Шифрування |
|---|---|---|
| Напрямок | одностороннє | двостороннє |
| Ключ | не потрібен | потрібен |
| Вихід | фіксована довжина | змінна |
| Призначення | цілісність, паролі | конфіденційність |
| Приклади | SHA-256, bcrypt, Argon2 | AES, RSA, ChaCha20 |

**Паролі → лише хешування** (ти не маєш могти «розшифрувати» пароль користувача). Але:
- **Не голий SHA-256** для паролів! Він **занадто швидкий** → перебір мільярдів за секунду.
- Використовуй **повільні** алгоритми, створені для паролів: **bcrypt, scrypt, Argon2**.
- Завжди із **сіллю (salt)** — унікальний випадковий префікс на кожен пароль → ламає rainbow tables і робить однакові паролі різними хешами.

```python
import bcrypt

# Хешуємо при реєстрації
hashed = bcrypt.hashpw(b"my_password", bcrypt.gensalt())

# Перевіряємо при логіні (НЕ розшифровуємо!)
bcrypt.checkpw(b"my_password", hashed)  # True/False
```

**Gotcha:** «зашифрувати пароль» — червоний прапор на інтерв'ю. Паролі **хешують**, не шифрують. Якщо БД можна розшифрувати — витік ключа = витік усіх паролів.

### L3 — Senior
**Три категорії, які часто плутають:**
1. **Encryption** — конфіденційність, оборотне з ключем (symmetric: AES; asymmetric: RSA).
2. **Hashing** — цілісність/ідентифікація, необоротне.
3. **Encoding** (Base64) — це **взагалі не безпека**, лише перетворення формату, оборотне без ключа.

**Властивості криптографічного хешу:**
- **Deterministic** — той самий вхід → той самий хеш.
- **Pre-image resistance** — за хешем не відновити вхід.
- **Collision resistance** — складно знайти два входи з однаковим хешем (тому MD5/SHA-1 **скомпрометовані** для безпеки — знайдено колізії; SHA-1 зламано у 2017, SHATTERED).
- **Avalanche effect** — зміна 1 біта входу змінює ~половину бітів виходу.

**Чому для паролів важлива «повільність» (work factor):** bcrypt/Argon2 мають **налаштовуваний cost** — підвищуючи його з ростом заліза, ти тримаєш вартість брутфорсу високою роками. Argon2 додатково **memory-hard** (вимагає багато RAM) → стійкий до атак на GPU/ASIC, де паралелізм дешевий, а пам'ять — ні.

**Симетричне vs асиметричне шифрування (трейдоф):**
- Symmetric (AES) — швидке, але треба **безпечно обмінятись ключем**.
- Asymmetric (RSA/ECC) — вирішує обмін ключами (public/private), але повільне.
- На практиці — **гібрид** (TLS): asymmetric для обміну сесійним ключем, потім symmetric для самих даних. Це і є питання 14 (TLS handshake) на криптографічному рівні.

> ⚠️ Статуси зламів (MD5, SHA-1) і рекомендовані параметри (cost-factor Argon2) змінюються — звіряйся з актуальними OWASP Password Storage Cheat Sheet перед продакшн-рішеннями.

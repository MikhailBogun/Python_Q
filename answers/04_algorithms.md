## 59. Що таке алгоритми (Big-O notation)?

### L1 — Junior
**Аналогія — рецепт.** Алгоритм — це чітка покрокова інструкція, як з вхідних даних отримати результат. Як рецепт борщу: послідовність кроків, що гарантовано приводить до страви.

**Big-O** — це спосіб сказати, **як швидко зростає час роботи, коли даних стає більше**. Не «скільки секунд», а «у скільки разів повільніше стане, якщо даних удесятеро більше».

**Аналогія Big-O — пошук людини:**
- **O(1)** — подзвонити їй (миттєво, байдуже скільки людей у місті).
- **O(n)** — обійти всі квартири підряд (вдвічі більше будинків — вдвічі довше).
- **O(n²)** — кожен питає кожного (натовп множиться катастрофічно).

### L2 — Middle
**Big-O описує асимптотичну складність** — поведінку при великих `n`, відкидаючи константи й молодші члени (`3n² + 5n + 10` → `O(n²)`).

**Таблиця складностей (від кращої до гіршої):**
| Big-O | Назва | Приклад |
|---|---|---|
| `O(1)` | константна | доступ `dict[key]`, `list[i]` |
| `O(log n)` | логарифмічна | бінарний пошук, B-tree |
| `O(n)` | лінійна | прохід по списку, `x in list` |
| `O(n log n)` | лінійно-логарифмічна | ефективні сортування |
| `O(n²)` | квадратична | вкладені цикли, bubble sort |
| `O(2ⁿ)` | експоненційна | наївний рекурсивний Фібоначчі |
| `O(n!)` | факторіальна | перебір усіх перестановок |

```python
# O(1) — час не залежить від n
def first(items: list) -> int:
    return items[0]

# O(n) — один прохід
def total(items: list[int]) -> int:
    s = 0
    for x in items:        # n ітерацій
        s += x
    return s

# O(n²) — вкладені цикли
def has_duplicate(items: list) -> bool:
    for i in range(len(items)):
        for j in range(i + 1, len(items)):   # n × n
            if items[i] == items[j]:
                return True
    return False
```

**Складність буває за часом (time) і пам'яттю (space).** Часто між ними **трейдоф**: можна пришвидшити ціною пам'яті (кеш, хеш-таблиця).

**Gotcha:** Big-O — це **верхня межа для великих n**. На малих даних `O(n²)`-алгоритм може бути швидшим за `O(n log n)` через менші константи. Тому Python для сортування дрібних шматків використовує insertion sort усередині Timsort.

### L3 — Senior
**Три нотації** (Senior не плутає):
- **O (Big-O)** — верхня межа (worst case ceiling): «не гірше ніж».
- **Ω (Omega)** — нижня межа: «не краще ніж».
- **Θ (Theta)** — точна межа (і зверху, і знизу): коли O і Ω збігаються.

На практиці кажуть «O», але мають на увазі різні випадки: **best / average / worst**. Наприклад, QuickSort — `Θ(n log n)` в середньому, але `O(n²)` у гіршому.

**Amortized analysis** — усереднення дорогих операцій по серії. `list.append` — `O(1)` амортизовано: зрідка реалокація `O(n)`, але розподілена на багато дешевих вставок (блок 2, п.46).

**Чому відкидаємо константи:** Big-O описує **масштабованість**, а не абсолютну швидкість. `O(n)` алгоритм з великою константою може програвати `O(n log n)` на реальних даних. Тому Senior **профілює** (вимірює), а не лише рахує Big-O — Big-O каже, як код **деградує при рості**, а не яка його швидкість тут і зараз.

**Space complexity не менш важлива:** `O(n)` пам'яті може вбити сервіс на великих даних навіть при гарному часі. Генератори (блок 2, п.32) — приклад зниження space до `O(1)`.

> ⚠️ Big-O — математична абстракція; реальна швидкість залежить від кеша CPU, локальності пам'яті, констант. Завжди підтверджуй бенчмарком (`timeit`, профайлер).

## 60. Базові алгоритми сортування

### L1 — Junior
**Аналогія — впорядкувати колоду карт.** Є різні способи: перекладати сусідні (bubble), щоразу шукати найменшу (selection), вставляти кожну на місце (insertion), ділити колоду навпіл і зливати (merge). Усі дають відсортований результат, але **різною ціною**.

### L2 — Middle
**Порівняльна таблиця** (head-to-head — улюблене на інтерв'ю):
| Алгоритм | Best | Average | Worst | Space | Stable? | In-place? |
|---|---|---|---|---|---|---|
| **Bubble** | O(n) | O(n²) | O(n²) | O(1) | ✅ | ✅ |
| **Selection** | O(n²) | O(n²) | O(n²) | O(1) | ❌ | ✅ |
| **Insertion** | O(n) | O(n²) | O(n²) | O(1) | ✅ | ✅ |
| **Merge** | O(n log n) | O(n log n) | O(n log n) | O(n) | ✅ | ❌ |
| **Quick** | O(n log n) | O(n log n) | O(n²) | O(log n) | ❌ | ✅ |
| **Heap** | O(n log n) | O(n log n) | O(n log n) | O(1) | ❌ | ✅ |
| **Timsort** (Python) | O(n) | O(n log n) | O(n log n) | O(n) | ✅ | ❌ |

**Два терміни, які питають:**
- **Stable (стабільне)** — зберігає відносний порядок рівних елементів. Критично при сортуванні за кількома ключами (спершу за іменем, потім за віком — стабільність зберігає сортування за іменем усередині однакового віку).
- **In-place** — `O(1)` додаткової пам'яті (сортує «на місці», не копіює масив).

```python
# Insertion sort — простий, швидкий на майже відсортованих даних
def insertion_sort(arr: list[int]) -> None:
    for i in range(1, len(arr)):
        key = arr[i]
        j = i - 1
        while j >= 0 and arr[j] > key:   # зсуваємо більші вправо
            arr[j + 1] = arr[j]
            j -= 1
        arr[j + 1] = key
```

**Best practice:** у реальному коді **не пишуть свій сорт** — використовують `sorted()` / `list.sort()` (Timsort, дуже оптимізований). Власні реалізації — лише для співбесід і навчання.

**Gotcha:** `list.sort()` сортує **на місці** (повертає `None`!), `sorted()` повертає **новий** список. Часта помилка: `x = mylist.sort()` → `x is None`.

### L3 — Senior
**Дві сім'ї сортувань:**
1. **Comparison-based** (bubble, merge, quick, heap) — порівнюють пари елементів. **Теоретична нижня межа — `Ω(n log n)`** (доказ через decision tree, п.62). Швидше за `n log n` порівняннями **неможливо**.
2. **Non-comparison** (counting, radix, bucket) — використовують структуру даних (цілі в діапазоні) → можуть `O(n)`, обходячи межу (п.62).

**Чому різні алгоритми за однакового `O(n log n)`:**
- **Merge** — стабільний, гарантований `n log n`, але `O(n)` пам'яті → для зовнішнього сортування (дані не влазять у RAM) і linked lists.
- **Quick** — `O(n²)` у гіршому (поганий pivot), зате **найкраща локальність кешу** й мала константа → найшвидший на практиці in-memory. Захист від `O(n²)`: рандомізований pivot / median-of-three / introsort (перехід на heap при деградації).
- **Heap** — гарантований `n log n` + `O(1)` пам'ять, але погана локальність кешу (стрибки по масиву) → повільніший за quick на практиці.

**Трейдоф stable vs in-place:** зазвичай не можна мати обидва дешево. Merge — stable, але не in-place. Quick/Heap — in-place, але не stable. Це фундаментальний компроміс пам'ять ↔ збереження порядку.

**Чому Python обрав Timsort:** реальні дані часто **частково відсортовані**. Timsort знаходить уже відсортовані «runs» і зливає їх → `O(n)` на майже впорядкованих даних, гарантований `O(n log n)` у гіршому, стабільний. Деталі — п.62/нижче.

> ⚠️ Стабільність і складність конкретних реалізацій (особливо QuickSort pivot-стратегії) різняться. Звіряй з джерелом реалізації.

## 61. Що таке Bubble Sort і як він працює?

### L1 — Junior
**Аналогія — бульбашки повітря у воді.** Найбільші елементи «спливають» нагору. Проходиш масив, порівнюєш **сусідів**, і якщо лівий більший за правого — міняєш місцями. За один прохід найбільший елемент «спливає» в кінець. Повторюєш, поки все не впорядкується.

### L2 — Middle
```python
def bubble_sort(arr: list[int]) -> None:
    n = len(arr)
    for i in range(n):
        swapped = False                       # оптимізація: чи були обміни
        # після i проходів останні i елементів уже на місці
        for j in range(0, n - i - 1):
            if arr[j] > arr[j + 1]:            # сусіди не в порядку
                arr[j], arr[j + 1] = arr[j + 1], arr[j]   # swap
                swapped = True
        if not swapped:                        # уже відсортовано → вихід
            break
```

**Трасування** `[5, 1, 4, 2]`:
```
Прохід 1: [1,4,2,5]  (5 сплив у кінець)
Прохід 2: [1,2,4,5]  (4 на місці)
Прохід 3: [1,2,4,5]  (без обмінів → swapped=False → стоп)
```

**Складність:**
- **Worst/Average:** `O(n²)` — вкладені цикли.
- **Best:** `O(n)` — якщо вже відсортовано (з прапором `swapped`, один прохід без обмінів).
- **Space:** `O(1)`, **stable**, **in-place**.

**Gotcha:** без оптимізації `swapped` bubble sort завжди `O(n²)`, навіть на відсортованому масиві. Прапор раннього виходу робить best case `O(n)` — це класичне уточнення, яке хоче почути інтерв'юер.

### L3 — Senior
Bubble sort **майже ніколи не використовують у проді** — він педагогічний. Навіть серед `O(n²)`-алгоритмів він **найгірший на практиці**: робить найбільше **обмінів (swaps)**, а swap дорожчий за порівняння (3 присвоєння). Insertion sort на тих самих даних робить менше записів і кращий до майже-відсортованих даних, тому його використовують реально (всередині Timsort для малих шматків).

**Чому `O(n²)`:** кількість порівнянь = `(n-1) + (n-2) + ... + 1 = n(n-1)/2` ≈ `n²/2` → відкидаємо константу → `O(n²)`.

**Єдина ніша bubble:** виявлення, чи масив **уже відсортований**, за `O(n)` (один прохід з `swapped`). Але для цього достатньо простого проходу-перевірки.

**Трейдоф:** простота розуміння й реалізації ↔ катастрофічна продуктивність на великих даних і максимум записів у пам'ять (погано для flash/SSD з обмеженим ресурсом запису, погана локальність). Senior знає bubble sort, щоб **пояснити, чому його не варто застосовувати**, і назвати кращу альтернативу (insertion для малих/майже-відсортованих, Timsort у реальності).

## 62. Що таке лінійна складність сортування?

### L1 — Junior
**Аналогія — розкласти пошту по скриньках за номерами квартир.** Якщо знаєш, що номери від 1 до 100, не треба порівнювати листи між собою — просто кидаєш кожен у свою скриньку за номером. Один прохід → відсортовано. Це **`O(n)`** — лінійно.

Але є умова: **значення мають бути обмеженими цілими** (номери квартир), інакше так не вийде.

### L2 — Middle
Лінійне сортування `O(n)` дають **non-comparison sorts** — вони не порівнюють елементи між собою, а використовують самі значення як індекси.

**Counting Sort** — рахуємо, скільки разів зустрічається кожне значення:
```python
def counting_sort(arr: list[int]) -> list[int]:
    if not arr:
        return []
    k = max(arr)                          # діапазон значень [0..k]
    counts = [0] * (k + 1)
    for x in arr:                         # O(n): рахуємо входження
        counts[x] += 1
    result = []
    for value, cnt in enumerate(counts):  # O(k): відновлюємо порядок
        result.extend([value] * cnt)
    return result                         # разом O(n + k)
```

**Три лінійні алгоритми:**
| Алгоритм | Складність | Умова |
|---|---|---|
| **Counting Sort** | `O(n + k)` | цілі в обмеженому діапазоні `[0..k]` |
| **Radix Sort** | `O(d·(n + k))` | цілі/рядки фіксованої довжини `d` розрядів |
| **Bucket Sort** | `O(n)` середній | рівномірний розподіл на `[0,1)` |

**Gotcha — «лінійне» з зірочкою:** Counting Sort `O(n + k)`. Якщо діапазон `k` **величезний** (напр. сортуємо 10 чисел від 0 до 1 млрд), то `k >> n` і алгоритм стає `O(k)` — гірше за `O(n log n)`! Лінійність працює **лише коли `k = O(n)`** (діапазон порівнянний з кількістю).

### L3 — Senior
**Чому `O(n log n)` — стеля для comparison sort (фундамент!):**
Будь-яке сортування порівняннями можна зобразити **decision tree** (деревом рішень): кожен вузол — порівняння `a < b`, кожен лист — одна з можливих перестановок. Для `n` елементів існує **`n!` перестановок** → дерево має ≥ `n!` листків. Висота бінарного дерева з `n!` листками ≥ `log₂(n!)`. За формулою Стірлінга `log₂(n!) ≈ n log n`. Висота дерева = кількість порівнянь у гіршому випадку → **будь-яке порівняльне сортування робить `Ω(n log n)` порівнянь**. Швидше — неможливо.

**Як лінійні сорти обходять цю межу:** вони **не порівнюють елементи** — використовують значення безпосередньо як індекси/розряди. Тому теорема decision tree до них не застосовна. Ціна — **обмеження на дані** (цілі/обмежений діапазон) і **додаткова пам'ять** (`O(n + k)`).

**Radix Sort** — застосовує stable counting sort порозрядно (LSD — від молодшого розряду). `O(d·(n + k))`, де `d` — кількість розрядів. Для 32-бітних int `d` фіксоване → формально `O(n)`, але константа велика → на практиці виграє лише на спеціальних даних (великі масиви цілих/рядків).

**Трейдофи лінійних сортувань:**
- **+** Обходять `n log n` межу → `O(n)` за сприятливих умов.
- **−** Працюють лише на **обмежених типах** (цілі, рядки фіксованої довжини), не на довільних об'єктах з компаратором.
- **−** Витрачають `O(n + k)` пам'яті.
- **−** Залежність від `k`: при великому діапазоні гірше за comparison sort.

**Чому Python використовує Timsort (`O(n log n)`), а не лінійне:** `sorted()` має сортувати **будь-що** з `__lt__` (рядки, кортежі, об'єкти) → потрібен **загальний** comparison sort. Лінійні працюють лише на вузькому класі даних. Timsort — найкращий **загальний** вибір: стабільний, `O(n)` на впорядкованих ділянках (галопінг по runs), гарантований `O(n log n)` у гіршому. Universal beats specialized, коли треба сортувати довільні типи.

> ⚠️ Точні константи Radix/Bucket і їх перевага залежать від розподілу даних і `k`. Доказ нижньої межі `n log n` стосується **лише** comparison-моделі — звіряй формулювання з підручником (CLRS, Sorting Lower Bounds).

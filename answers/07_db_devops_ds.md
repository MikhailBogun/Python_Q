## 84. Різниця між SQL і NoSQL?

### L1 — Junior
**Аналогія:**
- **SQL** — анкета з суворими графами: усі рядки однакової структури, зв'язки чіткі (Excel із правилами).
- **NoSQL** — коробка з нотатками довільної форми: кожен запис може мати свій набір полів (гнучко).

SQL — **схема жорстка, зв'язки сильні**. NoSQL — **схема гнучка, масштаб горизонтальний**. *(Базу заклали в блоці 1, п.12 — тут поглиблюємо.)*

### L2 — Middle
| | SQL (реляційні) | NoSQL |
|---|---|---|
| Модель | таблиці + зв'язки | документи/ключ-значення/граф/колонки |
| Схема | жорстка (schema-on-write) | гнучка (schema-on-read) |
| Транзакції | сильні **ACID** | часто **BASE** / eventual consistency |
| Масштабування | вертикальне (історично) | горизонтальне (вбудоване) |
| Запити | потужний SQL, JOIN | прості за ключем, JOIN зазвичай нема |
| Приклади | PostgreSQL, MySQL | MongoDB, Redis, Cassandra, Neo4j |

**4 типи NoSQL (блок 1, п.12):** Document (MongoDB), Key-Value (Redis), Column-family (Cassandra), Graph (Neo4j).

**Коли що:**
- **SQL** — складні зв'язки, цілісність критична (фінанси, замовлення), нефіксований обсяг запитів.
- **NoSQL** — величезні обсяги, гнучка/змінна структура, відомі патерни доступу (кеш, сесії, логи, IoT).

**Gotcha:** «NoSQL швидший» — міф (блок 1, п.12). Питання не у швидкості, а в **моделі даних і consistency**. PostgreSQL із **JSONB** часто закриває потреби, де поспішають брати Mongo — отримуєш гнучкість документів **і** ACID/JOIN.

### L3 — Senior
**ACID vs BASE — фундаментальний трейдоф:**
- **ACID** (SQL): Atomicity, Consistency, Isolation, Durability → сильні гарантії, ціна — складніше масштабувати запис горизонтально.
- **BASE** (багато NoSQL): **B**asically **A**vailable, **S**oft state, **E**ventual consistency → жертвуємо миттєвою узгодженістю заради доступності й масштабу.

Це випливає з **CAP-теореми** (блок 1, п.12): при мережевому розриві обираєш C або A. SQL зазвичай CP, AP-NoSQL (Cassandra, Dynamo) — A.

**Чому NoSQL масштабує запис краще:** відмова від JOIN і строгих транзакцій дозволяє **шардинг без координації** між вузлами. SQL із крос-шардовими JOIN/транзакціями впирається в розподілену координацію (дорого). NoSQL проєктують під **денормалізацію** — дані дублюються так, щоб один запит брав усе з одного вузла.

**Трейдоф моделювання:** SQL — нормалізація, гнучкі ad-hoc запити (схему придумав раз, питаєш як завгодно). NoSQL — **query-first design**: спершу знаєш патерни доступу, **потім** моделюєш дані під них (зміна патерну = переробка моделі). Тому NoSQL чудовий при відомому навантаженні й поганий при змінних аналітичних запитах.

**Сучасна реальність:** межа розмивається — **NewSQL** (CockroachDB, Spanner) дає ACID + горизонтальний масштаб; PostgreSQL JSONB дає документи в реляційці. Senior не обирає «SQL чи NoSQL» догматично, а під **конкретні гарантії + патерни доступу + обсяг**.

> ⚠️ Гарантії конкретних NoSQL (рівні узгодженості, транзакційність) сильно різняться між продуктами й версіями. Звіряй з docs продукту.

## 85. Як оптимізувати SQL-запити?

### L1 — Junior
**Аналогія — пошук книги в бібліотеці.** Без каталогу (індексу) бігаєш повз кожну полицю (`O(n)`). З каталогом — одразу до потрібної (`O(log n)`). Оптимізація запитів — це передусім **додати індекси** й **не тягнути зайве**.

### L2 — Middle
**Чек-лист оптимізації (за частотою впливу):**
1. **`EXPLAIN ANALYZE`** — спочатку **подивись план**, не вгадуй. Шукай `Seq Scan` на великих таблицях.
2. **Індекси** на колонках у `WHERE`, `JOIN ON`, `ORDER BY` (п.87).
3. **Не `SELECT *`** — тягни лише потрібні колонки (менше I/O, можливий covering index).
4. **Усунь N+1** (блок 3, п.57; блок 6, п.76) — JOIN / eager loading замість циклу запитів.
5. **Пагінація** — `LIMIT/OFFSET`, краще **keyset pagination** (`WHERE id > last_id`) для глибоких сторінок.
6. **Фільтруй рано** (`WHERE` до агрегації, блок 3 п.56).
7. **Уникай функцій на індексованій колонці** в `WHERE`: `WHERE LOWER(email)=...` ламає індекс (потрібен expression index).
8. **Connection pooling** (блок 3, п.54).

```sql
-- ❌ повільно: seq scan + зайві колонки
SELECT * FROM orders WHERE user_id = 42;

-- ✅ індекс + лише потрібне
CREATE INDEX idx_orders_user ON orders(user_id);
SELECT id, amount FROM orders WHERE user_id = 42;
```

**Gotcha:** індекс **не безкоштовний** — пришвидшує читання, але **сповільнює запис** (INSERT/UPDATE оновлюють індекс) і їсть місце. Не індексуй усе підряд — лише колонки реальних запитів.

### L3 — Senior
**Читання плану — головна навичка:** `EXPLAIN ANALYZE` показує обраний планувальником шлях і **реальний** час. Червоні прапори:
- **Seq Scan** на великій таблиці у вибірковому запиті → бракує індексу.
- **Nested Loop** з великою внутрішньою таблицею без індексу → `O(n×m)` (блок 3, п.57).
- **Rows estimated ≫ actual** → застаріла статистика → `ANALYZE` / `VACUUM`.
- **Sort** із `disk` → не вистачає `work_mem`.

**Чому функція ламає індекс:** B-tree індексує **значення колонки**, а `LOWER(email)` — інше значення → планувальник не може використати індекс. Рішення: **expression index** `CREATE INDEX ON users(LOWER(email))` або зберігати нормалізовано.

**Selectivity і чому планувальник ігнорує індекс:** якщо запит повертає **велику частку** таблиці (низька селективність, напр. `WHERE active = true` де 90% активні), **seq scan дешевший** за index scan (випадковий доступ по індексу + heap fetch дорожчий за послідовне читання). Планувальник це знає зі статистики — тому індекс на низькоселективній колонці марний.

**Covering index (index-only scan):** якщо індекс містить **усі** потрібні колонки (`INCLUDE`), БД відповідає **з індексу, не торкаючись таблиці** → набагато швидше. Класична оптимізація гарячих запитів.

**Трейдофи оптимізації:**
- Індекси: read ↔ write + місце.
- Денормалізація: швидке читання ↔ дублювання + складніший запис (блок 3, п.57).
- Кеш (Redis): латентність ↔ stale-дані (блок 5, п.66).
- Матеріалізовані view: швидка аналітика ↔ оновлення/свіжість.

**Послідовність Senior:** виміряй (`EXPLAIN`, slow query log) → знайди конкретний bottleneck → точкове виправлення (індекс/переписати запит) → переміряй. Не «оптимізуй усе підряд» (блок 5, п.69).

> ⚠️ Поведінка планувальника специфічна для СУБД і залежить від статистики/конфігу. Завжди перевіряй на **реальних даних** через `EXPLAIN ANALYZE`.

## 86. Рівні ізоляції транзакцій

### L1 — Junior
**Аналогія — кілька касирів і один склад.** Якщо двоє одночасно міняють залишок товару, можуть «не побачити» зміни одне одного й зіпсувати облік. **Рівні ізоляції** визначають, **наскільки сильно** транзакції відгороджені одна від одної. *(Механіку транзакцій заклали в блоці 3, п.55 — тут фокус саме на рівнях.)*

### L2 — Middle
**4 рівні (ANSI SQL) і аномалії, які вони прибирають:**
| Рівень | Dirty read | Non-repeatable read | Phantom read |
|---|---|---|---|
| **Read Uncommitted** | ✅ можливий | ✅ | ✅ |
| **Read Committed** | ❌ | ✅ | ✅ |
| **Repeatable Read** | ❌ | ❌ | ✅* |
| **Serializable** | ❌ | ❌ | ❌ |

**Аномалії (must know):**
- **Dirty read** — читаю незакомічені чужі зміни (які можуть відкотитись).
- **Non-repeatable read** — двічі читаю той самий рядок у транзакції, значення **різні** (хтось закомітив UPDATE між ними).
- **Phantom read** — двічі роблю той самий `WHERE`, з'явились/зникли **рядки** (хтось вставив/видалив).
- **Lost update** — двоє читають, міняють, пишуть → одна зміна затирає іншу.

```python
# Явне встановлення рівня (psycopg)
conn.isolation_level = "SERIALIZABLE"
```

**Gotcha:** чим вищий рівень — тим **менше конкурентність** (більше блокувань/відкатів). Не став Serializable «про всяк випадок» — лише де бізнес-інваріант справді цього вимагає.

### L3 — Senior
**PostgreSQL-специфіка (критично знати, блок 3 п.55):**
- Дефолт — **Read Committed**.
- **Read Uncommitted у PG = Read Committed** — брудного читання немає **ніколи** (завдяки MVCC).
- **Repeatable Read у PG** прибирає й phantom reads (строгіше за стандарт) через snapshot-ізоляцію.
- **Serializable у PG** = **SSI** (Serializable Snapshot Isolation) — детектує конфлікти й кидає `serialization_failure` → транзакцію треба **повторити** (retry-логіка обов'язкова).

**MVCC — чому PG не має dirty read:** кожна транзакція бачить **знімок** (snapshot) БД на момент старту; замість блокувань на читання тримаються **версії рядків**. Читачі не блокують письменників → висока конкурентність (блок 3, п.55).

**Lost update — окрема історія:** не входить у класичну таблицю ANSI, але важлива. Read Committed її **не** запобігає. Рішення:
- **Песимістично:** `SELECT ... FOR UPDATE` (захопити рядок).
- **Оптимістично:** version-колонка + `UPDATE ... WHERE version = N` (для low contention).
- **Атомарно в БД:** `UPDATE accounts SET balance = balance - 100` (не читати-рахувати-писати в коді).

**Трейдоф ізоляція ↔ throughput:** Serializable дає коректність будь-якого бізнес-інваріанту, але ціна — serialization failures + retry + нижча пропускна здатність. Read Committed — максимальна конкурентність, але вимагає **явного** захисту критичних інваріантів (FOR UPDATE / атомарні UPDATE). Senior обирає рівень **точково** під конкретну транзакцію, не глобально.

> ⚠️ Реалізація рівнів **сильно** різниться між СУБД (PG ≠ MySQL/InnoDB ≠ Oracle ≠ SQL Server). Назви однакові, поведінка — ні. Звіряй з docs конкретної бази.

## 87. Які є види індексів?

### L1 — Junior
**Аналогія — різні покажчики в книзі.** Алфавітний покажчик (B-tree) — для пошуку за словом і діапазоном. Колонтитул (hash) — миттєво знайти точну сторінку. Різні задачі — різні типи покажчиків.

Найголовніший і дефолтний — **B-tree**.

### L2 — Middle
**Типи індексів (PostgreSQL):**
| Тип | Для чого | Складність |
|---|---|---|
| **B-tree** | дефолт: рівність + діапазони + сортування (`<`, `>`, `BETWEEN`, `ORDER BY`) | `O(log n)` |
| **Hash** | лише рівність (`=`) | `O(1)` |
| **GIN** | масиви, **JSONB**, full-text (багато значень в одній комірці) | — |
| **GiST** | геодані, діапазони, nearest-neighbor | — |
| **BRIN** | величезні таблиці з природним порядком (час-ряди, логи) | компактний |

**За структурою/властивостями:**
- **Composite (multicolumn)** — індекс на кількох колонках `(a, b)`. **Порядок важливий!**
- **Unique** — гарантує унікальність + індексує.
- **Partial** — лише частина рядків: `CREATE INDEX ... WHERE active = true`.
- **Covering** — `INCLUDE (cols)` → index-only scan (п.85).
- **Expression** — на виразі: `CREATE INDEX ON users(LOWER(email))`.

```sql
CREATE INDEX idx_orders_user_status ON orders(user_id, status);   -- composite
CREATE INDEX idx_active ON users(email) WHERE active;             -- partial
```

**Gotcha — leftmost prefix rule:** composite `(user_id, status)` працює для запитів за `user_id` або `user_id + status`, але **не** за самим `status` (без `user_id`). Порядок колонок у composite-індексі критичний.

### L3 — Senior
**B-tree — чому дефолт і `O(log n)`:** збалансоване дерево, листя зв'язані → ефективні і точковий пошук, і **діапазонне сканування** (range scan), і сортований вихід (`ORDER BY` без окремого sort). Універсальний → 90% випадків.

**Hash vs B-tree:** hash дає `O(1)` на рівність, але **не вміє діапазони/сортування** і історично не журналювався (не crash-safe до PG 10). На практиці B-tree майже завжди обирають — `O(log n)` різниця мізерна, а функціонал ширший.

**GIN (Generalized Inverted Index)** — інвертований індекс: ключ → список рядків, що його містять. Для **композитних значень** (масив, jsonb, tokens тексту) — індексує **кожен елемент**. Тому пошук «документи, що містять слово X» або «jsonb з ключем Y» — швидкий. Ціна: дорожче оновлення (один рядок → багато індексних записів).

**BRIN (Block Range Index)** — зберігає лише **min/max по блоках** даних. Крихітний (KB на величезну таблицю), але працює лише коли дані **фізично впорядковані** за колонкою (час-ряди, append-only логи). Трейдоф: мізерна пам'ять ↔ груба точність (відсікає блоки, не рядки).

**Clustered vs non-clustered:**
- **Clustered** (InnoDB PK, MS SQL) — рядки **фізично впорядковані** за індексом → одна таблиця = один clustered index, дуже швидкий range за ним.
- **Non-clustered** (PostgreSQL усі індекси — окремі структури з вказівниками на heap) → потрібен extra fetch у таблицю (крім index-only scan).

**Трейдофи індексів загалом:**
- Читання ↔ запис (кожен INSERT/UPDATE оновлює всі індекси).
- Швидкість ↔ місце на диску + пам'ять (індекси кешуються в RAM).
- **Over-indexing** — антипатерн: купа невикористаних індексів гальмує запис і їсть місце, нічого не даючи. Аналізуй `pg_stat_user_indexes` — викидай невикористані.
- Селективність вирішує користь (п.85): індекс на низькоселективній колонці марний.

> ⚠️ Доступні типи й їх поведінка специфічні для СУБД (PG має GIN/GiST/BRIN; MySQL — переважно B-tree/hash/fulltext). Звіряй з docs.

## 88. Точкові питання вибору БД і рушіїв

### L1 — Junior
**Аналогія — вибір транспорту під задачу.** Велосипед, авто, вантажівка, літак — кожен під свій випадок. Немає «найкращої БД», є **підхожа під конкретні вимоги**: обсяг, тип даних, патерни доступу, гарантії.

### L2 — Middle
**Реляційні — PostgreSQL vs MySQL (класичне порівняння):**
| | PostgreSQL | MySQL |
|---|---|---|
| Філософія | фічастий, строгий SQL-стандарт | простий, популярний |
| Типи | багаті: JSONB, arrays, PostGIS, custom | базові |
| Конкурентність | MVCC, складні запити | швидкі прості читання |
| Розширюваність | висока (extensions) | обмежена |
| Коли | складна логіка, аналітика, GIS, JSON | простий CRUD, читання-важкі, LAMP |

**Рушії MySQL — InnoDB vs MyISAM:**
- **InnoDB** (дефолт): ACID, **транзакції**, row-level locking, foreign keys, crash recovery. **Бери це.**
- **MyISAM** (легасі): без транзакцій/FK, table-level locking, історично швидке читання. Майже не використовують у новому коді.

**Інші:**
- **SQLite** — вбудована, single-file, serverless. Для застосунків, тестів, прототипів, мобільних.
- **Redis** — in-memory key-value: кеш, сесії, черги, rate limit (блок 5).
- **ClickHouse** — column-store для аналітики (OLAP).

**Gotcha:** «вибрати рушій» у MySQL — це майже завжди **InnoDB**. MyISAM без транзакцій = втрата даних при збої. Питання про InnoDB/MyISAM на інтерв'ю перевіряє, чи знаєш, чому транзакції/ACID важливі.

### L3 — Senior
**OLTP vs OLAP — головна вісь вибору:**
- **OLTP** (Online Transaction Processing) — багато дрібних транзакцій (замовлення, платежі). **Row-store** (PostgreSQL, MySQL): рядок лежить разом → швидко прочитати/оновити один запис.
- **OLAP** (Online Analytical Processing) — аналітика по мільйонах рядків (агрегації, звіти). **Column-store** (ClickHouse, Redshift, BigQuery): колонка лежить разом → читаєш лише потрібні колонки, чудове стиснення, векторна обробка.

Чому column-store виграє в аналітиці: `SELECT AVG(amount)` читає **одну колонку** з диска (не всі), однотипні дані стискаються в рази, SIMD-обробка. Row-store читав би цілі рядки.

**Спеціалізовані БД під модель даних:**
| Потреба | БД |
|---|---|
| Зв'язки/графи (соцмережі, рекомендації) | Neo4j (graph) |
| Час-ряди (метрики, IoT) | TimescaleDB, InfluxDB |
| Повнотекстовий пошук | Elasticsearch |
| Кеш/ефемерне | Redis |
| Гео | PostGIS |

**Критерії вибору Senior (не «модно», а):**
1. **Модель даних** — реляційна / документна / граф / час-ряди.
2. **Гарантії** — потрібен ACID чи прийнятна eventual consistency (CAP, п.84).
3. **Патерни доступу** — точкові читання vs аналітика vs пошук.
4. **Масштаб** — обсяг, read/write ratio, географія.
5. **Операційна зрілість** — досвід команди, екосистема, хостинг.

**Дефолт-рекомендація:** для більшості бекендів **PostgreSQL** — найбезпечніший вибір (ACID, JSONB для документів, повнотекст, PostGIS, розширення). Спеціалізовану БД додають як **доповнення** під конкретну потребу (Redis для кешу, Elastic для пошуку), а не замість. **Polyglot persistence** — різні БД під різні задачі в одній системі — але кожна нова БД = операційна вартість.

> ⚠️ Характеристики продуктів швидко змінюються (MySQL наздоганяє по JSON/CTE, PG по реплікації). Звіряй з актуальними docs/бенчмарками під свій кейс.

## 89. Frontend: сучасний JS (Babel, Webpack, TS, ES)

### L1 — Junior
*(Чесно: це frontend-стек, я бекенд-орієнтований, але терміни поясню чітко.)*

**Аналогія — конвеєр пакування товару для вітрини:**
- **ES (ECMAScript)** — сам «товар» (стандарт мови JavaScript).
- **TS (TypeScript)** — товар із етикетками-типами (надбудова над JS із типами).
- **Babel** — «перекладач» нового товару на стару мову, щоб старі браузери зрозуміли.
- **Webpack** — «пакувальник», що збирає всі шматки (модулі, стилі, картинки) в один акуратний бандл для відправки.

### L2 — Middle
| Термін | Що це |
|---|---|
| **ECMAScript (ES)** | стандарт JS. **ES6 / ES2015** — поворотний: `let/const`, arrow functions, класи, модулі, Promise, destructuring. Далі ES2016... ES2023 щороку |
| **Babel** | **транспілер**: сучасний JS/JSX → сумісний зі старими браузерами JS |
| **Webpack** | **бандлер**: збирає модулі + ассети в оптимізовані бандли (мінімізація, tree-shaking, code splitting) |
| **TypeScript (TS)** | **типізована надбудова** над JS, компілюється в JS; статичні типи (аналог Python type hints, але **обов'язкова компіляція**) |

```typescript
// TypeScript — типи перевіряються при компіляції
function greet(name: string): string {
    return `Привіт, ${name}`;
}
```

**Сучасний контекст:** на зміну Webpack приходять швидші **Vite**, **esbuild**, **Rollup** (на Go/Rust, миттєвий dev-сервер). Babel частково витісняється вбудованою підтримкою сучасного JS + esbuild/SWC.

**Gotcha:** **TS-типи стираються при компіляції** (type erasure) — у рантаймі їх немає, лишається чистий JS. На відміну від Pydantic у Python, TS **не** валідує дані в рантаймі (для цього — zod/io-ts). Це часта плутанина.

### L3 — Senior
**Паралель із Python (що корисно бекендеру):**
- **TS ↔ Python type hints + mypy:** обидва — gradual typing, статична перевірка до запуску, не впливає на рантайм. Різниця: TS компіляція **обов'язкова** (браузер не виконує TS), Python виконує `.py` напряму; TS робить **type erasure**, Python зберігає `__annotations__` (доступні через introspection — звідси Pydantic, блок 6 п.81).
- **Babel ↔ компіляція в байт-код:** транспіляція (новий синтаксис → старий) концептуально як крок компіляції; обидва — про сумісність.
- **Webpack ↔ упаковка залежностей:** бандлінг для браузера ↔ venv/Docker для Python (різні рівні, спільна ідея — відтворюване середовище виконання).

**Архітектурний трейдоф SSR vs SPA (перетин із блоком 2, п.51):**
- **SPA** (React/Vue, бандл через Webpack/Vite) — багатий інтерактив, розвантажений сервер ↔ гірше SEO «з коробки», важкий initial load, складність стану.
- **SSR/SSG** (Next.js, шаблонізатор) — швидкий перший рендер, SEO ↔ навантаження на сервер.

**Чесна позиція Senior на інтерв'ю:** якщо роль бекендова — варто **знати ці терміни й межу відповідальності** (що таке бандл, навіщо транспіляція, що TS ≈ типи), розуміти контракт API між фронтом і бекендом (JSON, CORS, auth — блоки 1, 2), але не вдавати глибокого фронтенд-досвіду, якого немає (правило «жодних галюцинацій»). Чесне «фронт знаю на рівні взаємодії, глибина — бекенд» — сильніше за блеф.

> ⚠️ Frontend-тулінг змінюється **дуже** швидко (Webpack→Vite, Babel→SWC/esbuild). Це не моя core-зона — для актуальної frontend-глибини звіряйся зі спеціалістом/docs.

## 90. DevOps: Docker, Kubernetes, Kibana

### L1 — Junior
**Аналогія — контейнерні перевезення.**
- **Docker-контейнер** — стандартний контейнер, у який пакуєш застосунок з усім потрібним (код, бібліотеки, залежності). «Працює в мене → працює всюди».
- **Kubernetes (K8s)** — порт із кранами й диспетчером, що керує тисячами контейнерів: де поставити, скільки копій, що робити, якщо контейнер впав.
- **Kibana** — приладова панель для перегляду логів і графіків.

### L2 — Middle
**Docker (блок 6, п.68):** контейнеризація — легка ізоляція на рівні процесів (не повна VM). `Dockerfile` → `image` (шаблон) → `container` (запущений екземпляр).

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY . .
RUN pip install uv && uv sync
CMD ["uv", "run", "uvicorn", "main:app", "--host", "0.0.0.0"]
```

**Терміни Kubernetes (must know):**
| Термін | Що це |
|---|---|
| **Cluster** | весь набір машин під керуванням K8s |
| **Node** | одна машина-воркер (VM/фізична), де крутяться поди |
| **Pod** | **найменша одиниця** деплою: 1+ контейнерів зі спільною мережею/сховищем |
| **Deployment** | керує **репліками** подів: масштабування, **rolling update**, self-healing (впав под → підніме новий) |
| **Service** | **стабільна мережева точка** + балансування на поди (поди ефемерні, IP змінюються) |

**Kibana** — частина стеку **ELK** (**E**lasticsearch + **L**ogstash + **K**ibana): візуалізація логів/метрик, дашборди, пошук по логах, алерти. Бекенд — Elasticsearch (блок 5, п.67).

**Gotcha:** **Pod ефемерний** — може бути вбитий/переміщений будь-коли (нода впала, rolling update). Тому: (1) звертайся до подів через **Service**, не за IP; (2) стан — **не** в поді (stateless, блок 5 п.65), а в зовнішній БД/volume.

### L3 — Senior
**Контейнер ≠ VM (важлива відмінність):** контейнер шарить **ядро хоста**, ізоляція через **namespaces** (PID, network, mount, user) + **cgroups** (ліміти CPU/RAM) Linux. Тому контейнер **легкий** (старт мілісекунди, мегабайти) проти VM (повна ОС, секунди, гігабайти). Трейдоф: слабша ізоляція (спільне ядро → ширша attack surface), ніж у VM.

**K8s — декларативна модель (ключова ідея):** ти описуєш **бажаний стан** (YAML: «хочу 3 репліки»), а K8s через **reconciliation loop** постійно зводить **фактичний** стан до бажаного (под впав → створює новий; оновлення → rolling). Це **declarative, not imperative** — ти кажеш «що», не «як». Звідси self-healing і auto-scaling.

**Чому Service потрібен:** поди ефемерні (IP змінюються при перестворенні), тож пряме звернення ненадійне. Service дає **стабільний віртуальний IP/DNS** + L4-балансування на здорові поди (через readiness probes, блок 6 п.68). Типи: ClusterIP (внутрішній), NodePort, LoadBalancer (зовнішній), + **Ingress** для L7-роутингу (HTTP-маршрути, TLS).

**Трейдофи K8s:**
- **+** Self-healing, auto-scaling, rolling deploy, портативність, декларативність.
- **−** **Величезна складність** — networking, storage, RBAC, operators. Для маленького проєкту K8s — over-engineering. Альтернативи: managed (ECS, Cloud Run, Fly.io), де оркестрація схована.

**Senior-позиція:** K8s виправданий при **багатьох сервісах + потребі в авто-масштабуванні/самовідновленні + командою, що його тягне**. Для одного-двох сервісів — простіший PaaS. «K8s бо модно» — класична передчасна складність (блоки 5, п.65/68).

**ELK трейдоф:** потужний пошук/візуалізація логів ↔ Elasticsearch ресурсожерливий і складний в експлуатації. Легші альтернативи: Grafana **Loki** (логи), якщо повнотекстовий пошук не критичний.

> ⚠️ K8s-абстракції й DevOps-екосистема глибокі й швидко еволюціонують. Це оглядовий рівень; для продакшн-конфігів звіряйся з docs і DevOps-спеціалістом.

## 91. Часова складність алгоритму (time complexity)

### L1 — Junior
**Аналогія — час очікування залежно від черги.** Часова складність відповідає на питання: **як зросте час роботи, коли даних побільшає?** Не «скільки секунд», а «у скільки разів повільніше при ×10 даних». *(Big-O детально розклали в блоці 4, п.59 — тут фокус саме на time complexity.)*

### L2 — Middle
**Time complexity** — кількість **операцій** як функція розміру входу `n`, у нотації Big-O (блок 4):
| Big-O | При ×10 даних | Приклад |
|---|---|---|
| `O(1)` | без змін | `dict[key]`, `list[i]` |
| `O(log n)` | +трохи | бінарний пошук |
| `O(n)` | ×10 | прохід списку |
| `O(n log n)` | ×~13 | сортування |
| `O(n²)` | ×100 | вкладені цикли |
| `O(2ⁿ)` | катастрофа | наївний рекурсивний перебір |

```python
def find(items: list[int], target: int) -> bool:
    for x in items:           # O(n) — час росте лінійно з len(items)
        if x == target:
            return True
    return False

def find_set(items: set[int], target: int) -> bool:
    return target in items    # O(1) — час не залежить від розміру
```

**Time vs Space (блок 4, п.59):** часова складність — про **час** (операції), просторова — про **пам'ять**. Часто трейдоф: пришвидшити час ціною пам'яті (хеш-таблиця, кеш, мемоізація).

**Gotcha:** best / average / worst різняться. Пошук у `list` — `O(n)` worst, але якщо елемент перший — `O(1)`. На інтерв'ю зазвичай мають на увазі **worst case** (або amortized average для `dict`/`list.append`).

### L3 — Senior
**Time complexity — про масштабованість, не швидкість (блок 4, п.59):** `O(n)` з великою константою може програвати `O(n log n)` на реальних `n`. Big-O каже, як код **деградує при рості**, а не яка абсолютна швидкість. Тому Senior **профілює** + рахує Big-O разом.

**Класична оптимізація через зміну складності (найчастіша на code review):**
```python
# O(n²): для кожного елемента шукаємо в списку O(n)
for x in items_a:
    if x in items_b:          # O(n) пошук × n = O(n²)
        ...
# O(n): пошук у set — O(1)
b_set = set(items_b)
for x in items_a:
    if x in b_set:            # O(1) × n = O(n)
        ...
```
Заміна `list`-пошуку на `set`/`dict` перетворює `O(n²)`→`O(n)` — «безкоштовний» виграш (блок 2, п.47).

**Amortized analysis (блок 4, п.59):** `list.append` — `O(1)` амортизовано (зрідка реалокація `O(n)`, розподілена на серію). `dict`/`set` доступ — `O(1)` амортизовано average, `O(n)` worst при колізіях (блок 2, п.48).

**Trade-off time ↔ space (фундаментальний):**
- **Мемоізація** (`functools.lru_cache`, блок 2 п.50): час ↓ ціною пам'яті ↑. Наївний Фібоначчі `O(2ⁿ)` → з кешем `O(n)`.
- **Хеш-таблиця**: `O(1)` пошук ціною додаткової пам'яті vs `O(n)` у списку.
- **Препроцесинг/індекс**: одноразовий `O(n log n)` на побудову → багато `O(log n)` запитів (як індекс БД, п.87).

Senior мислить складністю **на гарячих шляхах** (часто викликаний код, великі `n`), не догматично всюди — для `n=10` різниця `O(n)` vs `O(n²)` неважлива (premature optimization).

## 92. Поглиблений Linux: сервер, пакети, середовище, операції

### L1 — Junior
**Аналогія — керування віддаленим складом по телефону.** Ти не стоїш фізично на сервері — підключаєшся **по SSH** (захищений «телефон»), і далі командами керуєш: ставиш інструменти (пакети), налаштовуєш робоче місце (середовище), виконуєш роботу (запускаєш, дивишся логи). *(База команд — блок 5, п.63.)*

### L2 — Middle
**1. Підключення до сервера (SSH):**
```bash
ssh user@192.168.1.10                 # вхід за паролем
ssh -i ~/.ssh/key.pem user@host       # за ключем (безпечніше)
scp file.txt user@host:/path/         # копіювати файл на сервер
rsync -avz ./dir/ user@host:/path/    # синхронізація (ефективніше)
```
Зручність — `~/.ssh/config`:
```
Host prod
    HostName 192.168.1.10
    User deploy
    IdentityFile ~/.ssh/key.pem
```
→ потім просто `ssh prod`.

**2. Пакети (залежить від дистрибутиву):**
```bash
sudo apt update && sudo apt install nginx     # Debian/Ubuntu
sudo dnf install nginx                         # Fedora/RHEL
sudo systemctl start nginx                     # керування сервісом
sudo systemctl enable nginx                    # автозапуск
sudo systemctl status nginx
```

**3. Середовище:**
```bash
export DATABASE_URL="postgresql://..."         # env var (сесія)
# постійно — у ~/.bashrc / ~/.profile / /etc/environment
echo $PATH                                      # де шукаються команди
```

**4. Операції/діагностика:**
```bash
ps aux | grep python        # процеси
htop                        # моніторинг ресурсів
journalctl -u nginx -f      # логи сервісу (стрімінг)
df -h ; du -sh *            # диск
ss -tulpn                   # відкриті порти
tail -f /var/log/app.log    # стежити за логом
```

**Gotcha:** не працюй під `root` без потреби (**least privilege**, блок 1 п.11) — використовуй `sudo` точково. SSH **за ключем, не паролем** (вимкни password auth на проді). Ніколи не клади приватні ключі/секрети в git.

### L3 — Senior
**SSH під капотом — асиметрична криптографія (блок 1, п.23):** автентифікація за ключем використовує пару public/private. Сервер шифрує виклик публічним ключем — лише власник приватного може відповісти. Безпечніше за пароль (не передається секрет, стійко до брутфорсу). Далі сесія переходить на **симетричне** шифрування (як TLS, блок 1 п.14) — гібридна схема.

**Процеси й сигнали:** `kill` шле **сигнал** (не обов'язково вбиває): `SIGTERM` (15, чемна зупинка — дай процесу прибратись), `SIGKILL` (9, примусово — не перехоплюється), `SIGHUP` (часто reload конфігу). Graceful shutdown сервісу (блок 6, п.73 — закрити з'єднання, дочекати запити) тримається на коректній обробці `SIGTERM` — критично для zero-downtime деплою (блок 5, п.68).

**systemd — менеджер сервісів:** тримає процеси живими (`Restart=always`), керує залежностями, збирає логи (`journald`), ізолює (cgroups — як Docker, п.90). Production-сервіс під systemd, а не «запустив у терміналі».

**Файлова модель Unix:** «усе — файл» (пристрої, сокети, процеси в `/proc`). Права `rwx` для user/group/other (`chmod 644/755`), ownership (`chown`). Розуміння цього — основа безпеки сервера (хто що може читати/виконувати).

**Трейдоф ручне адміністрування vs IaC:** ручні `ssh + apt + правка конфігів` — швидко для одного сервера ↔ **не відтворювано, не масштабовано, дрейф конфігурації** (snowflake servers). Senior автоматизує: **Ansible/Terraform** (Infrastructure as Code) — декларативно, версіоновано, відтворювано (як K8s-підхід, п.90). Або взагалі **immutable infrastructure** — не правиш сервер, а перебудовуєш образ (Docker, блок 6 п.68). Ручний SSH на прод — для діагностики, не для змін.

> ⚠️ Команди різняться між дистрибутивами (apt/dnf/yum/pacman) і версіями (systemd vs init). Звіряй з docs цільової ОС.

## 93. Data Science: NumPy, Pandas та обробка/візуалізація даних

### L1 — Junior
**Аналогія — суперкалькулятор для таблиць.**
- **NumPy** — швидкі операції над **масивами чисел** (множ усі числа одразу, без циклу).
- **Pandas** — «Excel у Python»: таблиці (DataFrame) з фільтрами, групуванням, об'єднанням.
- **Matplotlib/Seaborn** — малювання графіків із цих даних.

### L2 — Middle
```python
import numpy as np
import pandas as pd

# NumPy — векторні операції (без циклів!)
arr = np.array([1, 2, 3, 4])
arr * 2                      # [2,4,6,8] — одразу над усім масивом
arr[arr > 2]                 # [3,4] — булеве маскування
arr.mean(), arr.sum()        # агрегації

# Pandas — таблиці
df = pd.read_csv("sales.csv")
df.head()                    # перші рядки
df[df["amount"] > 100]       # фільтр
df.groupby("category")["amount"].sum()    # групування+агрегація (як SQL GROUP BY)
df.merge(other, on="user_id")             # JOIN
df["amount"].fillna(0)                    # обробка пропусків
df["total"] = df["price"] * df["qty"]     # нова колонка
```

```python
# Візуалізація
import matplotlib.pyplot as plt
import seaborn as sns
sns.histplot(df["amount"])                # розподіл
df.groupby("month")["sales"].sum().plot() # тренд
plt.show()
```

**Best practices:**
- **Векторизуй** — операції над масивом/колонкою цілком, **не** Python-цикл по рядках (`df.apply` з Python-циклом — повільно).
- `df.groupby/merge/pivot_table` замість ручної логіки (це SQL-операції з блоку 3, п.56–57).

**Gotcha — `SettingWithCopyWarning`:** `df[df.x > 0]["y"] = 5` може писати в **копію**, а не оригінал → зміна губиться. Використовуй `.loc[]`: `df.loc[df.x > 0, "y"] = 5`.

### L3 — Senior
**Чому NumPy швидкий — векторизація + C (фундамент):** Python-цикл по числах повільний (кожен `int` — об'єкт `PyObject` ~28 байт, інтерпретація байткоду на кожен крок, блок 6 п.75). NumPy зберігає дані в **суцільному C-масиві однотипних чисел** (не Python-об'єктів) і виконує операції в **скомпільованому C/SIMD** циклі без GIL-overhead на елемент. Результат — у 10–100× швидше за цикл.

```python
# ❌ Python-цикл: ~секунди на млн елементів
result = [x * 2 for x in data]
# ✅ NumPy: ~мілісекунди — один C-виклик над усім масивом
result = arr * 2
```

**Broadcasting** — NumPy «розтягує» масиви різної форми під спільну без копіювання → векторні операції над різнорозмірними даними ефективно. Потужно, але джерело хитрих багів (мовчазне розтягування не туди).

**Локальність пам'яті й dtype:** NumPy масив — **щільний** блок пам'яті фіксованого `dtype` (`int64`, `float32`) → ідеальна локальність кешу CPU (блок 1, п.3 — data-oriented). Звідси й економія пам'яті: `float32` vs Python `float` (об'єкт). Вибір `dtype` — трейдоф точність ↔ пам'ять/швидкість.

**Pandas — трейдофи:**
- Зручність (SQL-подібні операції в пам'яті) ↔ **усе в RAM** (`O(n)` пам'яті — великі дані не влазять).
- Для даних більших за RAM → **Polars** (Rust, lazy, багатопотоковий, обходить GIL), **Dask** (паралельний/розподілений), або **виносити агрегацію в БД** (блок 3, п.56 — рахуй у базі, тягни результат).
- `.apply()` з Python-функцією губить векторизацію (повертається до повільного циклу) — уникай на гарячих шляхах.

**Сучасний контекст:** **Polars** (Rust) витісняє Pandas на великих даних — lazy evaluation (оптимізує план як SQL-планувальник, п.85), справжня багатопотоковість (немає GIL-обмеження, блок 2 п.30), краща пам'ять. Тренд той самий, що ruff/uv (блок 6, п.83) — Rust-інструменти за продуктивність.

**Чесна позиція:** DS — окрема глибока спеціалізація (статистика, ML, feature engineering). Бекендеру варто впевнено володіти NumPy/Pandas для обробки даних і розуміти **чому векторизація швидка**, але повноцінний DS/ML — суміжна роль зі своєю глибиною.

> ⚠️ DS-екосистема велика й рухлива (Pandas→Polars, нові ML-бібліотеки). Для глибокого DS/ML звіряйся з профільними джерелами — це не core-бекенд.

## 73. Async Python: як працює, навіщо, що під капотом?

### L1 — Junior
**Аналогія — один офіціант на 10 столиків.** Він не стоїть біля столу, чекаючи, поки гості оберуть страву. Прийняв замовлення в одного → пішов до іншого → повернувся, коли перший готовий. Один працівник обслуговує багатьох, **не простоюючи в очікуванні**.

Async потрібен для **I/O-bound** задач: поки чекаємо мережу/диск/БД, робимо інше.

```python
import asyncio

async def fetch(name: str) -> str:
    await asyncio.sleep(1)        # «чекаємо» (не блокуємо інших)
    return f"{name} готово"

async def main():
    # Три задачі «одночасно» — разом ~1с, а не 3с
    results = await asyncio.gather(fetch("A"), fetch("B"), fetch("C"))
    print(results)

asyncio.run(main())
```

### L2 — Middle
**Будівельні блоки:**
- **`async def`** — створює **корутину** (coroutine).
- **`await`** — точка призупинення: «віддай керування loop'у, поки це не готове».
- **Event loop** — планувальник, що крутить корутини: одна на `await` віддає керування — loop запускає наступну готову.
- **Task** — корутина, загорнута для конкурентного виконання (`asyncio.create_task`).

```python
async def main():
    # create_task — запускає конкурентно ОДРАЗУ
    task1 = asyncio.create_task(fetch("A"))
    task2 = asyncio.create_task(fetch("B"))
    await task1
    await task2

# 3.11+ — структурована конкурентність
async def main2():
    async with asyncio.TaskGroup() as tg:    # чекає всі, коректно скасовує
        tg.create_task(fetch("A"))
        tg.create_task(fetch("B"))
```

**Коли async, коли ні:**
- ✅ I/O-bound, багато з'єднань (API-проксі, веб-сервери, скрапінг, чати).
- ❌ CPU-bound — async **не допоможе** (один потік), потрібні процеси (блок 2, п.30).

**Gotcha #1 — «зараза» async:** один **блокуючий** виклик (`requests.get`, `time.sleep`, важкий CPU) у корутині **вішає весь event loop** — усі інші корутини стоять. Використовуй async-бібліотеки (`httpx`, `asyncpg`, `asyncio.sleep`) або виноси блокуюче в `asyncio.to_thread()`/executor.

**Gotcha #2:** виклик `fetch("A")` **без `await`** не виконує корутину — лише створює об'єкт (і `RuntimeWarning: coroutine was never awaited`).

### L3 — Senior
**Під капотом async — це генератори/корутини.** Історично `asyncio` будувався на **generator-based coroutines** (`yield from`). `await` — концептуальний нащадок `yield`: він **призупиняє** виконання, зберігаючи **frame** (стек, локальні), і повертає керування loop'у — точно як `yield` у генераторі (блок 2, п.32). Сучасні `async def` — окремий тип об'єкта (`coroutine`), але механіка призупинення/відновлення через frame — та сама.

**Event loop — це цикл навколо мультиплексора I/O** (`select`/`epoll`/`kqueue`):
1. Корутина доходить до `await` на I/O → реєструє fd у селекторі, віддає керування.
2. Loop питає ОС: «які fd готові?» (`epoll` — `O(1)` на готову подію).
3. Готовий fd → loop відновлює відповідну корутину з місця `await`.

Тобто **немає магії потоків** — один потік, кооперативне перемикання на `await`, а очікування десятків тисяч сокетів делегується **ядру ОС** через `epoll`.

**Cooperative vs preemptive (фундаментальний трейдоф):**
- **Threads** — прееемптивні: ОС перемикає будь-коли → потрібні locks (race conditions), але дорогі (стек ~MB, syscall на перемикання). Масштаб — сотні.
- **Asyncio** — кооперативне: перемикання **лише** на явному `await` → менше гонок (між `await` код атомарний), корутини дешеві (kB). Масштаб — десятки тисяч. Ціна: один «жадібний» (блокуючий) виклик вішає всіх.

**`uvloop`** замінює стандартний loop на реалізацію поверх **libuv** (як у Node.js) → у 2–4 рази швидше (Cython, оптимізований epoll). FastAPI/uvicorn використовують його.

**Future vs Task:** `Future` — низькорівневий «обіцяний результат» (placeholder); `Task` — `Future`, що **обгортає й планує корутину** в loop. `gather`/`TaskGroup` координують багато Task.

> ⚠️ Внутрішня реалізація корутин (`send`/`throw`/`StopIteration` як механізм повернення значення) — деталі CPython, що еволюціонували. Звіряй з docs (`asyncio`, PEP 492/525) перед дослівним цитуванням.

## 74. Порівняти асинхронні web-фреймворки

### L1 — Junior
Усі async-фреймворки роблять одне: дозволяють серверу обслуговувати **багато одночасних з'єднань одним потоком** (на event loop). Різниця — у зручності, швидкості й «батарейках» (що йде з коробки).

### L2 — Middle
| Фреймворк | Рівень | Особливості |
|---|---|---|
| **FastAPI** | high-level | типи→валідація (Pydantic), авто-OpenAPI, DI. Дефолт для API |
| **Starlette** | low-level (ASGI toolkit) | основа FastAPI; легкий, без валідації |
| **Litestar** | high-level | альтернатива FastAPI, batteries-included |
| **aiohttp** | mid | і клієнт, і сервер; зріла |
| **Sanic** | mid/high | акцент на швидкість |
| **Tornado** | старий | був async до asyncio (власний loop) |
| **Django (ASGI)** | full-stack | async views з 3.1, але ORM довго був sync |
| **Quart** | mid | «async Flask» (Flask-сумісний API) |

```python
# FastAPI — типи дають валідацію + доку безкоштовно
from fastapi import FastAPI
app = FastAPI()

@app.get("/users/{uid}")
async def get_user(uid: int) -> dict:    # uid авто-валідується як int
    user = await db.fetch_user(uid)      # async БД
    return user
```

**Як обрати:**
- **REST API з валідацією/докою** → **FastAPI** (або Litestar).
- **Мінімалізм/власний контроль** → Starlette.
- **HTTP-клієнт + сервер в одному** → aiohttp.
- **Уже на Django** → Django ASGI (поступова async-міграція).

**Gotcha:** усі вони — лише **ASGI-застосунки**; їх запускає ASGI-сервер (**uvicorn**/hypercorn), а не вони самі (блок 2, п.52). І async-фреймворк не врятує, якщо всередині блокуючий sync-код (п.73).

### L3 — Senior
**ASGI — спільний контракт** (блок 2, п.52): усі ці фреймворки реалізують `async def app(scope, receive, send)`, тому взаємозамінні на рівні сервера й middleware. Різниця — у шарі **над** ASGI: валідація, роутинг, DI.

**Трейдоф «батарейки vs контроль»:**
- **FastAPI/Litestar** — продуктивність розробки (валідація, серіалізація, OpenAPI, DI з коробки) ↔ більше «магії» (залежність від Pydantic, неявна поведінка), overhead валідації на кожен запит.
- **Starlette/aiohttp** — повний контроль, менший overhead, менше залежностей ↔ усе пишеш сам (валідація, схеми).

**Продуктивність — нюанс:** «бенчмарки RPS» оманливі. Для реального API вузьке місце — **БД і зовнішні I/O**, а не фреймворк. Різниця FastAPI vs Starlette в проді зазвичай губиться на тлі latency БД. Pydantic v2 (ядро на Rust) прибрав головний overhead FastAPI — валідацію.

**Зрілість екосистеми як критерій Senior:** FastAPI/Starlette мають велику спільноту, інтеграції, документацію → менше ризику. Швидший, але нішевий фреймворк може коштувати дорожче в підтримці. Вибір — це й **bus factor**, не лише RPS.

**Async ORM — реальний біль:** довго async-фреймворки страждали від відсутності зрілого async ORM. Тепер: SQLAlchemy 2.0 (async), Tortoise, SQLModel (поверх SQLAlchemy). Це частина вибору стека — фреймворк сам по собі async, але весь шлях даних має бути async (п.73).

> ⚠️ Екосистема async-фреймворків швидко рухається; конкретні переваги/недоліки версій звіряй з актуальними docs і бенчмарками на **своєму** навантаженні.

## 75. Що таке модель пам'яті Python?

### L1 — Junior
**Аналогія — склад із комірками й комірником.** Усі дані (об'єкти) лежать на складі (**heap**). Змінні — це **бирки** з адресами комірок, а не самі комірки. Спеціальний комірник (memory manager) сам виділяє місце й прибирає те, на що ніхто не дивиться.

Головне: у Python **усе — об'єкт у купі**, а змінна — лише **посилання** на нього.

### L2 — Middle
**Ключові факти моделі пам'яті CPython:**
1. **Усе — об'єкт** (`PyObject`) у heap: числа, рядки, функції, класи.
2. **Змінна = ім'я-посилання** (не комірка пам'яті). Присвоєння прив'язує ім'я до об'єкта (rebinding).
3. **Керування пам'яттю автоматичне:** reference counting + cyclic GC (п.77).
4. **Кешування дрібних об'єктів:** малі int (−5..256), інтерн рядків, free lists (блок 2, п.28).

```python
a = [1, 2, 3]
b = a              # b і a — два імені, ОДИН об'єкт
b.append(4)
print(a)           # [1,2,3,4] — той самий список!
print(a is b)      # True

c = a.copy()       # новий об'єкт
print(a is c)      # False
```

**Стек vs купа:** виклики функцій і локальні **посилання** живуть у стеку фреймів; самі **об'єкти** — завжди в heap. Тому «велике значення» не копіюється при передачі (п.80).

**Gotcha — shallow vs deep copy:**
```python
import copy
nested = [[1, 2], [3, 4]]
shallow = copy.copy(nested)        # верхній рівень новий, вкладені — спільні
deep = copy.deepcopy(nested)       # усе нове рекурсивно
shallow[0].append(99)
print(nested)                      # [[1,2,99],[3,4]] — змінилось! (shallow)
```

### L3 — Senior
**Трирівневий аллокатор CPython** (для дрібних об'єктів — pymalloc):
```
Arena (256 KB, від ОС через mmap/malloc)
  └─ Pool (4 KB, один розмірний клас)
       └─ Block (8..512 байт, під один об'єкт)
```
- Об'єкти **≤ 512 байт** ідуть через **pymalloc** (свій аллокатор) — швидше за `malloc`, менше фрагментації, бо CPython створює/видаляє мільйони дрібних об'єктів.
- Об'єкти **> 512 байт** — напряму через системний `malloc`.
- Free lists: дрібні int/float/tuple/list кешуються в списках вільних об'єктів → перевикористання без re-алокації.

**Кожен `PyObject` несе overhead:** `ob_refcnt` (лічильник посилань) + `ob_type` (вказівник на тип) — мінімум ~16 байт навіть на «голий» об'єкт. Тому `int` у Python — ~28 байт, а не 8. Це ціна «усе — об'єкт» і причина, чому Python пам'яте-затратний (numpy зберігає числа в C-масивах, обходячи це).

**Модель пам'яті в сенсі конкурентності:** історично Python **не мав** формальної memory model (як Java JMM). На практиці **GIL** + bytecode-атомарність окремих операцій давали неявні гарантії видимості. Free-threaded build (PEP 703) **потребує** формальнішої моделі (порядок, видимість між потоками). Це активна тема.

**Трейдоф:** автоматичне керування пам'яттю (refcount + GC) прибирає цілий клас багів (use-after-free, витоки) ↔ overhead на кожну операцію з посиланнями (inc/dec refcount), пам'ять на метадані об'єкта, паузи GC. Це свідомий вибір Python на користь безпеки й простоти проти продуктивності/контролю (C/Rust).

> ⚠️ Розміри arena/pool/block і поріг 512 байт — деталі реалізації CPython, що змінювалися між версіями (`Objects/obmalloc.c`). Звіряй перед цитуванням точних чисел.

## 76. Що таке SQLAlchemy (Core та ORM) і альтернативи?

### L1 — Junior
**Аналогія — перекладач між світами.** Python мислить об'єктами, БД — таблицями. **SQLAlchemy** — перекладач: ти працюєш з Python-об'єктами, а він генерує SQL і мапить рядки назад в об'єкти.

Має **два рівні**: **Core** (будівник SQL) і **ORM** (об'єкти ↔ рядки).

### L2 — Middle
**SQLAlchemy Core** — Query Builder, SQL виражений Python-об'єктами (без рядків, без повного ORM):
```python
from sqlalchemy import create_engine, Table, Column, Integer, String, MetaData, select

engine = create_engine("postgresql+psycopg://user:pass@localhost/db")
meta = MetaData()
users = Table("users", meta, Column("id", Integer, primary_key=True),
              Column("name", String))

stmt = select(users).where(users.c.name == "Іван")   # типобезпечно
with engine.connect() as conn:
    rows = conn.execute(stmt).fetchall()
```

**SQLAlchemy ORM** — об'єкти, що мапляться на таблиці:
```python
from sqlalchemy.orm import DeclarativeBase, Mapped, mapped_column, Session

class Base(DeclarativeBase): ...

class User(Base):
    __tablename__ = "users"
    id: Mapped[int] = mapped_column(primary_key=True)
    name: Mapped[str]

with Session(engine) as session:
    user = session.get(User, 1)          # рядок → об'єкт User
    user.name = "Новий"                  # зміна відстежується
    session.commit()                     # автоматичний UPDATE
```

**Альтернативи:**
| ORM | Особливості |
|---|---|
| **Django ORM** | вбудований у Django, простий, sync (async росте) |
| **SQLModel** | SQLAlchemy + Pydantic (автор FastAPI), типи |
| **Tortoise ORM** | async-native, Django-подібний API |
| **Peewee** | легкий, простий |
| **raw psycopg/asyncpg** | без ORM, максимум контролю (блок 3, п.58) |

**Gotcha — N+1 (головна пастка ORM):** ітерація по об'єктах із lazy-завантаженням пов'язаних → 1 запит + N запитів. Лікування: `selectinload`/`joinedload` (eager loading).
```python
# ❌ N+1: один запит на users + по запиту на orders кожного
for user in session.query(User).all():
    print(user.orders)        # lazy → окремий SELECT щоразу
# ✅ один-два запити
session.query(User).options(selectinload(User.orders)).all()
```

### L3 — Senior
**ORM реалізує патерни Data Mapper + Unit of Work + Identity Map:**
- **Unit of Work:** Session **відстежує зміни** об'єктів і робить usі INSERT/UPDATE/DELETE **разом** на `commit()` (одна транзакція, оптимізований порядок).
- **Identity Map:** один рядок БД = один Python-об'єкт у межах сесії (`session.get(User, 1)` двічі → той самий об'єкт). Запобігає неузгодженості.
- **Lazy loading** через дескриптори: доступ до `user.orders` тригерить SQL (звідси N+1).

Усе це працює завдяки **introspection і дескрипторам** (блок 2, п.27, 35).

**Core vs ORM — трейдоф:**
- **ORM** — продуктивність розробки, безпека, патерни ↔ overhead (трекінг, мапінг), «магія», N+1, складно з нетривіальним SQL.
- **Core** — контроль над SQL, типобезпека, без overhead ORM-трекінгу ↔ більше коду, сам мапиш результати.

Senior-стратегія (як у блоці 3): **ORM для CRUD**, **Core/raw для складних/гарячих запитів** (аналітика, bulk, оптимізовані JOIN). SQLAlchemy унікальна тим, що дає **обидва рівні в одній бібліотеці** — можна спускатись до Core там, де ORM незручний.

**Async (SQLAlchemy 2.0):** `AsyncSession` + `asyncpg`. Нюанс: lazy loading в async **не працює прозоро** (не можна тригерити I/O на доступі до атрибута поза await) → eager loading стає **обов'язковим**, не опціональним.

> ⚠️ API SQLAlchemy суттєво змінився у 2.0 (типізований `Mapped`, нова семантика). Звіряй із docs версії, що використовуєш.

## 77. Garbage Collection і reference counting

### L1 — Junior
**Аналогія — прибиральник за лічильником гостей.** Біля кожної кімнати (об'єкта) висить лічильник: скільки людей зараз нею користується. Останній вийшов (лічильник = 0) → кімнату одразу прибирають (звільняють пам'ять). Окремий обхідник час від часу шукає замкнені кімнати, де гості заблокували одне одного (цикли).

### L2 — Middle
CPython має **два механізми** разом:

**1. Reference counting (основний):** кожен об'єкт має лічильник посилань `ob_refcnt`. Нове посилання → `+1`, видалення → `−1`. Дійшов до `0` → **негайно** звільняється.
```python
import sys
a = [1, 2, 3]
sys.getrefcount(a)        # ~2 (a + тимчасове в getrefcount)
b = a
sys.getrefcount(a)        # ~3
del b                     # −1
```

**2. Cyclic GC (додатковий):** ловить **циклічні посилання**, які refcount сам не звільнить:
```python
a = {}
b = {}
a["b"] = b
b["a"] = a          # цикл: refcount ніколи не стане 0
del a, b            # об'єкти недосяжні, але refcount=1 → refcount їх НЕ звільнить
# Їх прибере циклічний GC
```

**Модуль `gc`:**
```python
import gc
gc.collect()         # примусовий збір
gc.disable()         # вимкнути циклічний GC (refcount лишається)
```

**Gotcha:** `__del__` + цикли історично конфліктували (GC не знав порядку звільнення). З Python 3.4 (PEP 442) це здебільшого вирішено, але **не покладайся на `__del__`** для критичного cleanup — використовуй контекстні менеджери (`with`).

### L3 — Senior
**Reference counting — трейдофи:**
- **+** Детермінованість: об'єкт звільняється **миттєво** при досягненні 0 (передбачувано, без пауз).
- **+** Пам'ять звільняється рано (добре для кешу).
- **−** Overhead на **кожну** операцію з посиланням (inc/dec на присвоєння, передачу аргументів).
- **−** **Не ловить цикли** (звідси потреба в другому механізмі).
- **−** **Несумісність з вільним паралелізмом:** non-atomic refcount — головна причина існування GIL (блок 2, п.30). Атомарні inc/dec на кожну операцію вбили б продуктивність.

**Циклічний GC — generational (покоління):**
Базується на **generational hypothesis**: «більшість об'єктів вмирають молодими». Три покоління (0, 1, 2):
- Нові об'єкти → gen 0, збирається часто (дешево).
- Пережив збір → переходить у старше покоління, збирається рідше.
Алгоритм відстежує лише **контейнери** (що можуть утворювати цикли: list, dict, об'єкти), обчислює «ефективний refcount» усередині групи й знаходить недосяжні цикли.

**Чому два механізми, а не один (як у Java):** refcount дає детермінованість і простоту, але не ловить цикли; tracing-GC (як JVM) ловить усе, але має паузи й недетермінований момент звільнення. CPython узяв **гібрид**: refcount для 99% (швидко, детерміновано) + tracing лише для циклів (рідко).

**Трейдоф vs суто tracing-GC (Java/Go):**
- CPython: детерміноване звільнення, прості C-розширення (refcount API), але GIL і overhead на refcount.
- Tracing-GC: вільний паралелізм, без overhead на присвоєння, але паузи (stop-the-world) і недетермінований момент звільнення.

**Практика:** для high-throughput сервісів інколи `gc.disable()` (якщо немає циклів) прибирає паузи; `gc.freeze()` після старту виносить довговічні об'єкти з-під збору (оптимізація форк-моделі gunicorn — менше copy-on-write).

> ⚠️ Деталі generational GC (пороги, threshold, алгоритм виявлення циклів) — реалізація CPython, змінювалася (3.12 уніфікувала деякі речі). Звіряй із docs `gc` і `Modules/gcmodule.c`.

## 78. Як працює thread locals?

### L1 — Junior
**Аналогія — особиста шафка в роздягальні.** У спільній роздягальні (процесі) кожен працівник (потік) має **свою шафку** з однаковим номером, але **різним вмістом**. `threading.local()` — це така шафка: одне ім'я змінної, але **в кожного потоку своє значення**.

### L2 — Middle
```python
import threading

local_data = threading.local()

def worker(name: str):
    local_data.value = name        # кожен потік пише СВОЄ
    process()

def process():
    print(local_data.value)        # бачить значення СВОГО потоку

threading.Thread(target=worker, args=("A",)).start()
threading.Thread(target=worker, args=("B",)).start()
# Потік A бачить "A", потік B бачить "B" — не змішуються
```

**Навіщо:** передати «контекст» через багато функцій **без явного прокидання аргументів**, ізолювавши потоки:
- З'єднання з БД на потік.
- Контекст HTTP-запиту (поточний user, request id) — Flask так робить (`g`, `request`).
- Уникнути race conditions без locks (кожен має своє, ділити нема чого).

**Gotcha:** thread locals **не працюють для asyncio** — там багато корутин в **одному** потоці, тож вони ділили б один `threading.local`. Для async є **`contextvars`** (PEP 567) — контекст на корутину/задачу:
```python
from contextvars import ContextVar
request_id: ContextVar[str] = ContextVar("request_id")
request_id.set("abc-123")           # ізольовано на корутину
```

### L3 — Senior
**Механізм:** `threading.local` зберігає дані, **прив'язані до ідентичності потоку**. Доступ до атрибута під капотом дивиться на поточний потік і повертає його приватний словник. Реалізація гарантує, що кожен потік бачить **власний** простір атрибутів того ж об'єкта.

**Трейдоф thread-local як патерн:**
- **+** Прибирає «прокидання» контексту через десятки сигнатур (request, user, db-conn), уникає гонок без locks.
- **−** Це фактично **глобальний стан** (приховані залежності) → ускладнює тестування й розуміння (функція таємно читає `local_data`), порушує явність («explicit is better than implicit»). Може маскувати погану архітектуру.

**`contextvars` — наступник для async (важливо):** thread-local прив'язаний до **потоку**, а async — один потік, багато задач. `ContextVar` прив'язаний до **контексту виконання**, який `asyncio` коректно копіює/ізолює на кожен Task → значення «течуть» з корутиною й не змішуються між конкурентними задачами. `contextvars` працює і для потоків, і для async → сучасний універсальний інструмент. Saved context копіюється при `create_task`.

**Senior-висновок:** thread locals/contextvars — потужний інструмент для cross-cutting контексту (трейсинг, request scope), але це **неявний глобальний стан** — застосовуй свідомо, для інфраструктурних речей (logging context, tracing), а не для бізнес-логіки, де явні аргументи чистіші.

> ⚠️ Точна семантика копіювання контексту між Task/потоками (`copy_context`) — звіряй з docs `contextvars`/`asyncio`.

## 79. Що таке `__slots__`?

### L1 — Junior
**Аналогія — валіза з фіксованими відділеннями замість мішка.** Звичайний об'єкт носить «мішок» (`__dict__`), куди можна докинути будь-який атрибут — гнучко, але важко. `__slots__` — це валіза з **наперед визначеними** відділеннями: лише задані атрибути, зате **легша й швидша**.

### L2 — Middle
```python
class PointDict:                  # звичайний — має __dict__
    def __init__(self, x, y):
        self.x = x
        self.y = y

class PointSlots:                 # зі __slots__ — без __dict__
    __slots__ = ("x", "y")
    def __init__(self, x, y):
        self.x = x
        self.y = y

p = PointSlots(1, 2)
p.x = 10                          # ✅ дозволені атрибути
p.z = 5                          # ❌ AttributeError — нема в __slots__
```

**Що дає `__slots__`:**
- **Економія пам'яті** (немає `__dict__` на екземпляр) — суттєво при **мільйонах** об'єктів.
- **Швидший доступ** до атрибутів (через слот-дескриптор, не хеш-lookup у словнику).
- **Захист від «опечаток»** (не можна випадково створити новий атрибут).

**Коли застосовувати:** багато екземплярів одного класу (точки, частинки, записи, вузли дерева, рядки даних). Не варто для класів з кількома екземплярами — економія неварта втрати гнучкості.

**Gotcha:** з `__slots__` **не можна** динамічно додавати атрибути й (за замовчуванням) немає `__weakref__` (об'єкт не можна слабко посилатись/кешувати через `weakref`). Щоб повернути — додай у slots: `__slots__ = ("x", "y", "__weakref__", "__dict__")` (але `__dict__` зводить нанівець економію).

### L3 — Senior
**Під капотом:** без `__slots__` атрибути зберігаються в `instance.__dict__` (хеш-таблиця на **кожен** об'єкт — мінімум ~100+ байт overhead). Зі `__slots__` CPython резервує **фіксовані слоти** в самій структурі об'єкта й створює **member descriptors** (один на клас, не на екземпляр) для доступу до них за зміщенням (offset) — як поля в C-структурі.

Тому виграш подвійний: **пам'ять** (немає per-instance dict) і **швидкість** (доступ за фіксованим offset через дескриптор, без хешування ключа).

**Складнощі наслідування (трейдоф):**
- Якщо **будь-який** клас в ієрархії не визначає `__slots__` — нащадки знову отримують `__dict__` (економія втрачена).
- Множинне наслідування двох класів зі `__slots__` з непорожніми слотами → конфлікт layout (`TypeError`).
- `__slots__` не успадковуються «автоматично» в сенсі складання — кожен клас оголошує свої.

**Трейдоф загалом:**
- **+** Пам'ять (×на кількість екземплярів) і швидкість доступу.
- **−** Втрата динамічності (немає monkey-patching атрибутів), складніше наслідування, конфлікти з кодом, що чекає `__dict__` (деякі серіалізатори, mixin'и).

**Сучасна альтернатива:** `@dataclass(slots=True)` (Python 3.10+) — генерує `__slots__` автоматично, поєднуючи зручність dataclass і економію. Це Pythonic-спосіб отримати слоти без ручного дублювання імен полів.

**Senior-правило:** `__slots__` — це **оптимізація пам'яті за профілем**, не дефолт. Застосовуй, коли профайлер показав, що мільйони об'єктів цього класу їдять RAM. Передчасне додавання заради «швидкості» — мікрооптимізація, що коштує гнучкості.

> ⚠️ Взаємодія `__slots__` з наслідуванням, `__weakref__`, pickle має нюанси per-version. Звіряй з docs (Data model → `__slots__`).

## 80. Як передаються аргументи функцій (by value / by reference)?

### L1 — Junior
**Ні те, ні те точно.** У Python — **«передача за посиланням на об'єкт»** (call by object reference / call by sharing). Передається **посилання** на об'єкт (не копія об'єкта), але **саме посилання — за значенням** (перепризначення параметра не змінює змінну зовні).

**Просте правило:**
- **Mutable** (list, dict) **змінив всередині** → зміна **видна** ззовні.
- **Перепризначив** параметр (`x = ...`) → ззовні **не видно**.

### L2 — Middle
```python
def mutate(lst: list) -> None:
    lst.append(4)            # МУТАЦІЯ об'єкта → видно ззовні

def reassign(lst: list) -> None:
    lst = [9, 9, 9]          # ПЕРЕПРИЗНАЧЕННЯ імені → НЕ видно ззовні

data = [1, 2, 3]
mutate(data)
print(data)                  # [1,2,3,4] ← змінилось

reassign(data)
print(data)                  # [1,2,3,4] ← НЕ змінилось (локальне rebinding)
```

**Чому immutable «ніби by value»:**
```python
def increment(n: int) -> None:
    n += 1                   # int незмінний → створює НОВИЙ об'єкт, локально

x = 5
increment(x)
print(x)                     # 5 ← незмінне, бо n+=1 — це rebinding
```

**Gotcha — повернення модифікованих значень:** оскільки immutable «не змінити на місці», функції їх **повертають**:
```python
def add_one(n: int) -> int:
    return n + 1             # повертаємо новий, не мутуємо
x = add_one(x)
```

### L3 — Senior
**Точна модель:** змінні Python — **імена в namespace**, прив'язані до об'єктів у heap (п.75). При виклику параметр стає **новим іменем**, прив'язаним до **того самого об'єкта**, що й аргумент (передається посилання-значення). Звідси:
- **Перепризначення** параметра (`x = ...`) міняє лише **локальну прив'язку** → зовнішнє ім'я недоторкане.
- **Мутація** об'єкта (`x.append`, `x[0]=...`) міняє **спільний об'єкт** → видно скрізь, де є посилання.

Тому термінологія «by value/by reference» з C/C++ **некоректна** для Python. Точні назви: **call by sharing** (Liskov), **call by object reference**, **call by assignment**. Поведінка = «mutable видно, rebinding ні», і вона **повністю виводиться** з моделі імен+об'єктів.

**Наслідки для дизайну:**
- **Mutable default argument** (блок 2, п.25) — той самий механізм: дефолт-об'єкт створюється раз і шариться.
- **Захист від небажаної мутації:** приймай дані, які не маєш міняти, як immutable (tuple), або копіюй (`list(items)`), або документуй контракт. Функція, що тихо мутує переданий list — джерело багів (порушення принципу найменшого здивування).
- **Продуктивність:** великий об'єкт **не копіюється** при передачі (`O(1)`, передається лише посилання) — на відміну від pass-by-value у C, де копіюється структура. Це робить передачу аргументів дешевою незалежно від розміру об'єкта.

**Трейдоф моделі:** єдина проста модель (імена→об'єкти, refcount) для всього ↔ плутанина для тих, хто мислить категоріями C; приховані мутації через спільні посилання — клас багів, що лікується дисципліною (immutability, копії, типи).

## 81. Що таке type annotation?

### L1 — Junior
**Аналогія — підпис на коробці «скло, верх».** Type annotation — це **підказка про тип**: «ця функція очікує `int` і повертає `str`». Python її **не примушує** (коробку все одно можна впустити), але вона допомагає людям і інструментам не помилитися.

```python
def greet(name: str) -> str:
    return f"Привіт, {name}"

age: int = 30
```

### L2 — Middle
```python
from typing import Optional, Callable

# Сучасний синтаксис (3.10+)
def find(items: list[int], target: int) -> int | None:   # | замість Optional
    ...

scores: dict[str, float] = {}
handler: Callable[[int, str], bool]                       # функція як тип

# Узагальнення
from typing import TypeVar
T = TypeVar("T")
def first(items: list[T]) -> T:
    return items[0]
```

**Ключове:** анотації **не перевіряються в рантаймі** Python'ом — їх перевіряє **статичний** аналізатор (`mypy`, `pyright`) **до запуску**.

**Винятки — рантайм-читачі анотацій:** Pydantic/FastAPI/SQLModel **читають** анотації через introspection (блок 2, п.27) і **валідують у рантаймі**:
```python
from pydantic import BaseModel
class User(BaseModel):
    name: str
    age: int
User(name="Іван", age="ні")    # ValidationError — Pydantic перевіряє в рантаймі
```

**Gotcha:** `def f(x: int)` сам по собі **не** заборонить `f("рядок")` — Python виконає це без помилки. Перевірку дає лише `mypy`/IDE. Анотації — документація + інструмент, **не** рантайм-гарантія (якщо це не Pydantic).

### L3 — Senior
**Gradual typing** (PEP 484) — філософія Python: типи **опціональні й поступові**, можна додавати в існуючий код частинами. Це компроміс між динамічністю (гнучкість, швидкість прототипування) і статичною безпекою (відлов помилок до запуску, рефакторинг, автокомпліт).

**Зберігання й обчислення анотацій:** лежать у `__annotations__` (функції/класу/модуля). За замовчуванням обчислюються **eagerly** при визначенні. `from __future__ import annotations` (PEP 563) робить їх **рядками** (lazy) → рятує від forward references і циклічних імпортів, **але** ламає рантайм-читачів (Pydantic мусить `get_type_hints()` резолвити). PEP 649 пропонує lazy-обчислення через descriptor як компроміс — активна еволюція.

**Типи НЕ прискорюють CPython** (на відміну від статичних мов, де тип → оптимізація розкладки/інструкцій). CPython анотації в рантаймі **ігнорує** для виконання. Виграш суто **інструментальний**: статичний аналіз, IDE, документація, рантайм-валідація сторонніми бібліотеками.

**Просунутий typing (Senior-арсенал):** `Protocol` (структурна типізація / статичний duck typing, блок 2 п.4), `Generic`/`TypeVar` (узагальнення), `ParamSpec` (PEP 612, збереження сигнатури в декораторах, блок 2 п.42), `Literal`, `TypedDict`, `overload`, `Self`, `@final`.

**Трейдоф:** типізація коштує часу написання й може дати хибне відчуття безпеки (типи не ловлять логічні баги, лише типові) ↔ масштабовність великих кодбаз, безпечний рефакторинг, самодокументованість. Для бібліотек/великих команд — must; для дрібних скриптів — overhead. Senior додає типи **на межах** (публічні API, складні структури), не догматично всюди.

> ⚠️ Поведінка анотацій за замовчуванням (eager vs lazy, PEP 563/649) різниться між версіями. Звіряй із docs для своєї версії, особливо якщо використовуєш Pydantic.

## 82. Навіщо нижні підкреслення в іменах?

### L1 — Junior
**Аналогія — таблички на дверях.** Підкреслення — це знаки «службове, не для відвідувачів». `_name` — «це внутрішнє, ззовні не чіпай». `__name__` — «це системне, належить Python».

### L2 — Middle
**П'ять випадків (знати всі):**
| Запис | Назва | Сенс |
|---|---|---|
| `_name` | single leading | **protected** за конвенцією: внутрішнє, не імпортується `from x import *` |
| `name_` | single trailing | уникнути конфлікту з ключовим словом: `class_`, `type_`, `id_` |
| `__name` | double leading | **name mangling** → стає `_ClassName__name` |
| `__name__` | dunder | системні/магічні: `__init__`, `__name__` — **не вигадуй свої** |
| `_` | single | «викидна» змінна, або останній результат у REPL |

```python
class BankAccount:
    def __init__(self, balance):
        self.balance = balance       # public
        self._pin = "0000"           # protected (конвенція)
        self.__secret = "key"        # mangled → _BankAccount__secret

    def _internal_helper(self): ...  # «не для зовнішнього використання»

acc = BankAccount(100)
acc._pin                             # доступно (конвенція, не заборона)
acc.__secret                         # AttributeError
acc._BankAccount__secret             # ось так доступно (mangling не захист!)

for _ in range(3):                   # _ — значення не потрібне
    print("hi")
first, _, third = (1, 2, 3)          # ігноруємо середнє
```

**Gotcha:** у Python **немає справжніх private** (блок 1, п.4). `_` і `__` — конвенції й mangling, **не** механізм захисту доступу. `__secret` доступний як `_ClassName__secret`. Філософія: «we're all consenting adults».

### L3 — Senior
**Name mangling (`__name`) — навіщо насправді:** це **не** «приватність», а **захист від випадкового перекриття** атрибута в підкласах. Компілятор переписує `__x` у `_ClassName__x` на етапі компіляції → підклас із власним `__x` не зіткнеться з батьківським (у кожного свій mangled-простір). Корисно для базових класів/міксинів, що не хочуть, щоб нащадки випадково затерли їх внутрішні атрибути.

```python
class Base:
    def __init__(self):
        self.__id = 1          # _Base__id
class Derived(Base):
    def __init__(self):
        super().__init__()
        self.__id = 2          # _Derived__id — НЕ конфліктує з базовим
```

**Dunder (`__x__`) — зарезервований простір протоколів:** Python резервує `__*__` за собою (data model: `__init__`, `__eq__`, `__iter__`, `__enter__`...). Не створюй власні dunder-імена — майбутня версія мови може ввести таке ім'я й зламати твій код. Dunder'и **не** піддаються mangling (подвійний leading **і** trailing).

**`_` множинні ролі (Senior знає всі):** throwaway-змінна, останній результат в інтерактивному REPL (`sys.displayhook`), `gettext` для i18n (`_("text")`), розділювач у числових літералах (`1_000_000`). Контекст вирішує.

**Трейдоф «конвенції замість enforcement»:** Python обрав **домовленості** (underscore) замість жорстких `private`/`protected` (Java/C++). Плюс — гнучкість (можна дістатись внутрішнього для дебагу/моків/хаків), менше бойлерплейту. Мінус — немає компіляторної гарантії інкапсуляції, дисципліна тримається на культурі. Це узгоджується із загальною філософією Python: довіра до програміста > примус.

> ⚠️ Деталі mangling (коли застосовується, межі) — звіряй з Language Reference (Identifiers / Private name mangling).

## 83. Статичні аналізатори: Flake8, Pylint, Radon

### L1 — Junior
**Аналогія — автоматичний коректор перед здачею.** Статичний аналізатор читає код **не запускаючи** його й вказує на проблеми: неакуратний стиль, ймовірні баги, занадто складні функції. Як перевірка орфографії, але для коду.

«Статичний» = аналізує **текст коду**, без виконання (на відміну від тестів, що запускають).

### L2 — Middle
| Інструмент | Що робить |
|---|---|
| **Flake8** | стиль (PEP8) + логічні помилки (невикористані імпорти/змінні) + складність. Швидкий лінтер |
| **Pylint** | глибший аналіз: помилки, code smells, конвенції, рефакторинг-поради. Прискіпливіший, повільніший |
| **Radon** | метрики: цикломатична складність (блок 5, п.72), maintainability index, Halstead |
| **mypy / pyright** | перевірка **типів** (анотацій, п.81) |
| **Ruff** | усе-в-одному на Rust: замінює Flake8+isort+частину pylint, у ~100× швидший |
| **Bandit** | пошук **вразливостей** безпеки (блок 1, п.11) |
| **Black** | автоформатування (не аналіз — форматер) |

```bash
flake8 myfile.py              # стиль + базові помилки
pylint myfile.py              # глибокий аналіз (оцінка з 10)
radon cc myfile.py -a         # цикломатична складність
mypy myfile.py                # типи
ruff check .                  # швидкий лінт (цей проєкт)
```

**Класифікація (важливо не плутати):**
- **Лінтери** (Flake8, Pylint, Ruff) — стиль + ймовірні баги.
- **Type checkers** (mypy, pyright) — узгодженість типів.
- **Форматери** (Black, `ruff format`) — стиль автоматично.
- **Метрики** (Radon) — складність/підтримуваність.
- **Security** (Bandit) — вразливості.

**Gotcha:** статичний аналіз ловить **синтаксичні/типові/стильові** проблеми, але **не** логічні баги («рахує неправильну формулу») — для цього потрібні **тести** (блок 1, п.18–20). Лінт і тести **доповнюють**, не замінюють одне одного.

### L3 — Senior
**Як працює статичний аналіз:** інструмент парсить код в **AST** (Abstract Syntax Tree) і обходить його, застосовуючи правила. Деякі (mypy, pylint) роблять **data/control flow analysis** — відстежують типи й шляхи виконання без запуску. Принципова межа — **теорема Райса / undecidability**: статично неможливо довести довільну властивість програми загалом → завжди є **false positives** (хибні тривоги) і **false negatives** (пропущене).

**Трейдоф глибина ↔ шум:**
- **Pylint** — глибокий, ловить більше, але багато false positives → потребує налаштування (`.pylintrc`, `# pylint: disable`). Розчаровує командою «надто бубонить».
- **Flake8/Ruff** — швидші, менше шуму, фокус на високовпевнених проблемах → легше прийняти всією командою.
Senior калібрує суворість: занадто строго → ігнорують і вимикають; занадто м'яко → пропускає борг.

**Місце в інженерному процесі:**
- **Швидкий зворотний зв'язок:** лінт/типи в IDE + pre-commit hook → ловлять до коміту (найдешевше).
- **Gate в CI:** лінт+типи+тести як умова мержу (блок 1, п.20) → не дають влити борг.
- **Автоматизація звільняє code review** для логіки/дизайну (блок 5, п.70) — машина бере стиль/типи, людина — суть.

**Maintainability Index (Radon)** — композитна метрика з cyclomatic complexity + Halstead volume + LOC. Senior ставиться до неї як до **сигналу** (підсвітити кандидатів на рефакторинг), не догми — метрика не бачить семантики (блок 5, п.72).

**Сучасний трейдоф — консолідація на Ruff:** історично стек = flake8 + isort + pydocstyle + частина pylint (повільно, багато конфігів). Ruff (Rust) об'єднує їх із ~100× швидкістю → миттєвий лінт навіть на великих кодбазах, один конфіг. Цей проєкт використовує саме Ruff (CLAUDE.md). Це приклад загального тренду: швидкі Rust-інструменти (ruff, uv) витісняють фрагментований Python-тулінг.

> ⚠️ Набори правил і дефолти інструментів швидко змінюються (особливо Ruff активно розвивається). Звіряй актуальні правила з docs конкретної версії.

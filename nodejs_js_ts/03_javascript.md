## 21. Що таке асинхронність і асинхронний код?

### L1 — Junior
**Аналогія — замовити каву й не стояти біля бариста.** Синхронно: стоїш і чекаєш, поки зварять (нічого іншого не робиш). Асинхронно: зробив замовлення, тобі дали номерок, ти **йдеш робити інше**, а коли кава готова — тебе покличуть.

**Асинхронний код** не блокує виконання, чекаючи довгу операцію (мережа, файл, таймер) — він «йде далі», а результат обробляє згодом (через callback/Promise).

```javascript
console.log("1");
setTimeout(() => console.log("3"), 1000); // не блокує — піде далі
console.log("2");
// Вивід: 1, 2, 3 (а не 1, [пауза], 3, 2)
```

### L2 — Middle
**Еволюція асинхронності в JS:**
```javascript
// 1. Callbacks (старий стиль → "callback hell")
fs.readFile("a.txt", (err, data) => {
  if (err) return handle(err);
  process(data, (err, result) => { /* вкладеність росте */ });
});

// 2. Promises (плоскіше, ланцюги)
fetch("/api/user")
  .then((res) => res.json())
  .then((user) => console.log(user))
  .catch((err) => console.error(err));

// 3. async/await (синтаксичний цукор над Promises — найчистіше)
async function getUser() {
  try {
    const res = await fetch("/api/user"); // "чекаємо" без блокування
    return await res.json();
  } catch (err) {
    console.error(err);
  }
}
```

**Gotcha:** `await` не блокує **потік** — він призупиняє лише поточну async-функцію, віддаючи керування event loop (блок 1, п.13). Інший код продовжує виконуватись.

### L3 — Senior
Асинхронність у JS тримається на **event loop** (блок 1, п.13): довгі операції делегуються середовищу (libuv у Node / браузерним API), а callback стає в чергу й виконується, коли потік вільний.

**`async/await` під капотом — це Promises + генератори.** `await` концептуально «призупиняє» функцію, зберігаючи її стан, і відновлює, коли Promise виконано (microtask, блок 1 п.10). `async`-функція **завжди повертає Promise**.

**Трейдоф:** асинхронна модель дає масову конкурентність I/O на одному потоці (блок 1, п.5) ↔ складніша ментальна модель, «зараза» async (одна async-функція тягне async по всьому ланцюгу), пастки порядку виконання. `async/await` повернув читабельність (лінійний код), але приховав, що під ним — черги мікрозадач.

**Конкурентність ≠ паралелізм:** `await` у циклі послідовний; для паралельного — `Promise.all` (питання 36):
```javascript
// ❌ Послідовно — сумарний час
for (const url of urls) await fetch(url);
// ✅ Паралельно — час найдовшого
await Promise.all(urls.map((url) => fetch(url)));
```

## 22. `var`, `let`, `const` — відмінність

### L1 — Junior
| | `var` | `let` | `const` |
|---|---|---|---|
| Область | функція | **блок** `{}` | **блок** `{}` |
| Переприсвоєння | так | так | **ні** |
| Повторне оголошення | так | ні | ні |

**Правило: завжди `const`, `let` — коли значення змінюється, `var` — ніколи.**

```javascript
const PI = 3.14;     // не зміниться
let count = 0;       // зміниться: count++
var old = 1;         // legacy — не використовуй
```

### L2 — Middle
**Чому `const` за замовчуванням, навіть якщо не плануєш міняти:**
- **Сигнал наміру** — читач бачить «це не зміниться», менше когнітивного навантаження.
- **Захист від випадкового переприсвоєння** — компілятор/рантайм впіймає помилку.
- **Менше багів** — більшість змінних насправді не повинні мінятися.

**Важливо — `const` не означає «незмінний об'єкт»:**
```javascript
const arr = [1, 2];
arr.push(3);        // ✅ можна — міняємо ВМІСТ, не прив'язку
arr = [4, 5];       // ❌ TypeError — переприсвоєння прив'язки заборонено

const obj = { a: 1 };
obj.a = 2;          // ✅ можна
```
`const` забороняє переприсвоювати **прив'язку (binding)**, а не мутувати об'єкт. Для незмінності об'єкта — `Object.freeze()`.

**Gotcha — `var` і блокова область:**
```javascript
for (var i = 0; i < 3; i++) {}
console.log(i);     // 3 — var "витікає" з циклу!
for (let j = 0; j < 3; j++) {}
console.log(j);     // ReferenceError — let обмежений блоком
```

### L3 — Senior
**Hoisting (підняття) і TDZ — головна різниця під капотом:**
- **`var`** піднімається й **ініціалізується `undefined`** — доступ до неї до оголошення дає `undefined` (не помилку).
- **`let`/`const`** теж піднімаються, але **не ініціалізуються** — потрапляють у **TDZ (Temporal Dead Zone)**: доступ до них до рядка оголошення → `ReferenceError`.

```javascript
console.log(a);  // undefined (var піднята й = undefined)
var a = 1;

console.log(b);  // ReferenceError (b у TDZ)
let b = 2;
```

**Класична пастка `var` у замиканнях (чому ввели `let`):**
```javascript
for (var i = 0; i < 3; i++) {
  setTimeout(() => console.log(i)); // 3, 3, 3 — одна спільна i
}
for (let j = 0; j < 3; j++) {
  setTimeout(() => console.log(j)); // 0, 1, 2 — нова j на кожну ітерацію!
}
```
`let` створює **нову прив'язку на кожну ітерацію** циклу — тому замикання захоплюють різні значення. Це прямий аналог late-binding пастки в Python (там навпаки — немає block scope).

**Трейдоф:** `var` (function scope + hoisting) — джерело багів, лишився для зворотної сумісності. `let`/`const` (block scope + TDZ) — передбачуваніші, ловлять помилки раніше. Senior: `const` усюди → `let` для лічильників/акумуляторів → `var` ніколи в новому коді.

## 23. Як відкласти виконання функції на конкретний час?

### L1 — Junior
**`setTimeout(fn, ms)`** — виконати функцію один раз через задану кількість мілісекунд.

```javascript
setTimeout(() => console.log("через 2 секунди"), 2000);
```

### L2 — Middle
```javascript
// Один раз через затримку
const id = setTimeout(() => console.log("раз"), 1000);
clearTimeout(id);            // скасувати

// Повторювати кожні N мс
const intId = setInterval(() => console.log("тік"), 1000);
clearInterval(intId);        // зупинити

// Node-специфічні (блок 1, п.13):
setImmediate(() => console.log("у фазі check"));  // після I/O
queueMicrotask(() => console.log("microtask"));   // ще раніше за таймери
process.nextTick(() => console.log("найвищий пріоритет")); // Node
```

**Gotcha:** `setTimeout(fn, 0)` **не** виконується «миттєво» — `0` означає «якнайшвидше, але **після** поточного синхронного коду й усіх microtasks» (блок 1, п.10). Мінімальна реальна затримка в браузері ~4мс для вкладених таймерів.

### L3 — Senior
`setTimeout` — це **macrotask** (блок 1, п.10), що стає в чергу таймерів event loop. Затримка `ms` — це **мінімум**, не гарантія: якщо потік зайнятий (синхронний код, інші callbacks), таймер спрацює **пізніше** за `ms`. Точність таймерів не гарантована — для точного таймінгу (анімації) використовують `requestAnimationFrame` (браузер) чи `performance.now()`.

**Дебаунс/тротлінг** — практичне застосування відкладеного виконання:
```javascript
// Debounce — виконати лише після паузи (пошук при введенні)
function debounce(fn, delay) {
  let timer;
  return (...args) => {
    clearTimeout(timer);
    timer = setTimeout(() => fn(...args), delay);
  };
}
```
**Трейдоф:** таймери прості, але не точні (залежать від завантаженості циклу) і легко витікають (`setInterval` без `clearInterval` працює вічно, тримаючи замикання в пам'яті — memory leak). Senior завжди очищає таймери в cleanup.

## 24. Способи оголошення функції

### L1 — Junior
```javascript
// 1. Function declaration (оголошення)
function add(a, b) { return a + b; }

// 2. Function expression (вираз — функція як значення)
const add2 = function (a, b) { return a + b; };

// 3. Arrow function (стрілкова — найкоротша)
const add3 = (a, b) => a + b;
```

### L2 — Middle
**Повний список:**
```javascript
// Method shorthand (в об'єкті/класі)
const obj = { greet() { return "hi"; } };

// Async-функції
async function fetchData() { /* await ... */ }
const fetchData2 = async () => { /* ... */ };

// Generator
function* gen() { yield 1; yield 2; }

// Function constructor (майже не використовується — eval-подібний)
const sum = new Function("a", "b", "return a + b");
```

**Arrow vs звичайна — головна відмінність:** arrow **не має власного `this`** (бере з оточення, питання 32), не має `arguments`, не може бути конструктором (`new`).

**Gotcha:** arrow для методів об'єкта, що використовують `this`, — пастка:
```javascript
const obj = {
  name: "Іван",
  greet: () => `Привіт, ${this.name}`, // ❌ this не obj! (lexical)
  greet2() { return `Привіт, ${this.name}`; }, // ✅ this = obj
};
```

### L3 — Senior
**Ключові розрізнення під капотом:**
- **Hoisting:** function declaration піднімається **повністю** (можна викликати до оголошення); function expression / arrow — лише прив'язка (`const`/`let` у TDZ, питання 22, 27).
- **`this`:** звичайні функції мають **динамічний `this`** (залежить від виклику); arrow — **лексичний** (фіксований із місця визначення, питання 32).
- **`arguments`:** є у звичайних, немає в arrow (там — rest `...args`).
- **Конструктор:** звичайні можна `new`; arrow — ні (немає `[[Construct]]`).

**Трейдоф вибору:** arrow — лаконічність + правильний `this` для callbacks (не треба `.bind`) ↔ непридатність для методів зі своїм `this`, конструкторів, generators. Звичайні declaration — hoisting (зручно для top-level утиліт) + повний набір можливостей ↔ динамічний `this` як джерело багів. Сучасна практика: arrow для callbacks/коротких функцій, method shorthand для методів, declaration для іменованих утиліт верхнього рівня.

## 25. Що таке анонімна функція?

### L1 — Junior
**Функція без імені.** Часто передається «на місці» як аргумент (callback) або присвоюється змінній.

```javascript
setTimeout(function () { console.log("привіт"); }, 1000); // анонімна
[1, 2, 3].map((x) => x * 2);  // arrow — теж анонімна
```

### L2 — Middle
Анонімні функції — норма для **callbacks** (event handlers, array methods, промісів):
```javascript
arr.filter((x) => x > 10);          // anonymous arrow
button.addEventListener("click", function () { /* ... */ });
promise.then((data) => process(data));
```

**Gotcha:** анонімні функції погано видно в **стектрейсі** (`<anonymous>`) → важче дебажити. Тому для складної логіки краще **іменовані** функції — ім'я з'явиться в трейсі.

### L3 — Senior
**Іменовані function expressions (NFE)** — компроміс: ім'я для дебагу, але видиме лише всередині функції (для рекурсії):
```javascript
const factorial = function fact(n) {     // "fact" видиме лише всередині
  return n <= 1 ? 1 : n * fact(n - 1);   // рекурсія через внутрішнє ім'я
};
```

**Трейдоф:** анонімні — лаконічні для одноразових callbacks ↔ погана діагностика (стектрейс), неможливість рекурсії/перевикористання. Сучасні рушії часто **виводять ім'я** з присвоєння (`const f = () => {}` → `f.name === "f"`), що пом'якшує проблему трейсу для змінних, але не для inline-callbacks. Senior: anonymous для тривіальних callbacks, named — для логіки, яку доведеться дебажити.

## 26. Функція, що самовикликається (IIFE)

### L1 — Junior
**IIFE (Immediately Invoked Function Expression)** — функція, яку **оголошують і одразу викликають**.

```javascript
(function () {
  console.log("виконуюсь одразу");
})();

(() => console.log("те саме, arrow"))();
```

### L2 — Middle
**Навіщо (історично):** до ES6 (модулів і `let`/`const`) IIFE створювали **приватну область видимості**, щоб не засмічувати глобальний простір:
```javascript
const counter = (function () {
  let count = 0;                    // приватна — ззовні недоступна
  return {
    increment: () => ++count,
    get: () => count,
  };
})();
counter.increment(); // 1 — count захований у замиканні
```

**Async IIFE** — запустити `await` на верхньому рівні (до підтримки top-level await):
```javascript
(async () => {
  const data = await fetch("/api");
})();
```

**Gotcha:** дужки навколо функції роблять її **виразом** (інакше `function () {}()` — синтаксична помилка, бо парситься як declaration). `(function(){})()` або `(function(){}())`.

### L3 — Senior
IIFE реалізує **module pattern** через **замикання** (closure): змінні всередині IIFE недосяжні ззовні, але повернений об'єкт має до них доступ → інкапсуляція приватного стану в мові без `private`.

**Трейдоф / сучасний контекст:** IIFE був головним інструментом ізоляції до **ES Modules** (`import`/`export`), які тепер дають справжню модульну область **нативно**. Сьогодні IIFE рідкісний — його витіснили модулі (приватність на рівні файлу) і `let`/`const` (block scope). Лишився для: top-level await у старих середовищах, негайної ініціалізації з приватним станом, бандлерів (UMD-обгортки). Senior знає IIFE як **історичний патерн** і впізнає його в legacy/bundled коді.

## 27. Function expression vs function declaration

### L1 — Junior
```javascript
// Declaration — самостійна інструкція
function add(a, b) { return a + b; }

// Expression — функція як ЗНАЧЕННЯ (присвоєна змінній)
const add = function (a, b) { return a + b; };
```
**Головна різниця — hoisting:** declaration можна викликати **до** оголошення, expression — ні.

### L2 — Middle
```javascript
sayHi();              // ✅ працює — declaration піднята повністю
function sayHi() { console.log("hi"); }

sayBye();             // ❌ ReferenceError / TypeError — expression не піднята
const sayBye = function () { console.log("bye"); };
```

**Чому:**
- **Declaration** піднімається **разом із тілом** → доступна в усій області з самого початку.
- **Expression** — піднімається лише змінна (`const sayBye` у TDZ, питання 22) → тіло присвоюється тільки на рядку оголошення.

**Gotcha:** це впливає на структуру коду — declaration дозволяє «використати спочатку, оголосити нижче» (зручно для читабельності top-level), expression змушує оголошувати перед використанням.

### L3 — Senior
**Технічно:** declaration створює прив'язку у **scope** на етапі компіляції (до виконання). Expression — це звичайне присвоєння значення-функції в рантаймі.

**Трейдоф і стиль:**
- **Declaration:** hoisting (гнучкий порядок коду), завжди іменована (кращий стектрейс) ↔ можна випадково перевизначити, «магія» доступності до оголошення.
- **Expression (особливо `const fn = () => {}`):** передбачуваний потік (треба оголосити перед використанням), блокова область, не переприсвоюється ↔ немає hoisting.

Багато сучасних styleguide (Airbnb) **рекомендують function expressions через `const`** — за передбачуваність і консистентність (усе — `const`, немає hoisting-сюрпризів). Але declaration лишається доречним для top-level утиліт, де hoisting робить код читабельнішим (головна логіка зверху, хелпери знизу).

## 28. Масив чисел → лише числа понад 10

### L1 — Junior
**`Array.prototype.filter()`** — повертає **новий** масив з елементами, що проходять умову.

```javascript
const nums = [5, 12, 8, 25, 3, 15];
const big = nums.filter((x) => x > 10);
// [12, 25, 15]
```

### L2 — Middle
`filter` — один із **функціональних методів масиву** (не мутує оригінал, повертає новий):
```javascript
nums.filter((x) => x > 10);          // вибірка за умовою
nums.map((x) => x * 2);              // трансформація кожного
nums.reduce((sum, x) => sum + x, 0); // згортка в одне значення
nums.find((x) => x > 10);            // ПЕРШИЙ, що підходить (не масив)
nums.some((x) => x > 10);            // чи є хоч один (bool)
nums.every((x) => x > 10);          // чи всі (bool)
```

**Gotcha:** `filter` повертає **новий** масив, не мутує оригінал (immutable-стиль). `nums` лишається незмінним. Якщо потрібен лише один елемент — `find` (поверне елемент, не масив).

### L3 — Senior
**Чейнінг (композиція) — сила функціональних методів:**
```javascript
const result = nums
  .filter((x) => x > 10)     // [12, 25, 15]
  .map((x) => x * 2)         // [24, 50, 30]
  .reduce((a, b) => a + b);  // 104
```
**Трейдоф vs цикл:** `filter/map/reduce` — декларативні, читабельні, immutable, композиційні ↔ кожен метод створює **проміжний масив** (3 проходи + 2 тимчасові масиви для прикладу вище) — для **дуже великих** даних цикл за один прохід ефективніший за пам'яттю/часом. Для більшості випадків читабельність переважає; для гарячих шляхів із мільйонами елементів — один `for...of` або генератори/ітератори. Це той самий трейдоф, що comprehensions vs генератори в Python.

## 29. Як видалити елемент масиву та об'єкта?

### L1 — Junior
```javascript
// Масив — splice (за індексом, МУТУЄ)
const arr = [1, 2, 3, 4];
arr.splice(1, 1);        // видалити 1 елемент з індексу 1 → [1, 3, 4]

// Об'єкт — delete
const obj = { a: 1, b: 2 };
delete obj.a;            // { b: 2 }
```

### L2 — Middle
**Масив — мутуючі vs немутуючі способи:**
```javascript
// МУТУЮЧІ (міняють оригінал)
arr.splice(index, 1);    // за індексом
arr.pop();               // останній
arr.shift();             // перший

// НЕМУТУЮЧІ (новий масив — immutable-стиль, краще для React/Redux)
const without = arr.filter((x) => x !== valueToRemove);
const sliced = [...arr.slice(0, i), ...arr.slice(i + 1)];
```

**Об'єкт:**
```javascript
delete obj.key;                            // мутує
const { key, ...rest } = obj;              // rest — новий об'єкт без key (immutable)
```

**Gotcha:** `delete arr[1]` для масиву — **антипатерн**: лишає «дірку» (`empty`), не змінює `length` (`[1, empty, 3]`). Для масивів — `splice`/`filter`, `delete` — лише для об'єктів.

### L3 — Senior
**`delete` і продуктивність:** `delete obj.key` змінює **форму об'єкта** → V8 «деоптимізує» його (переводить з fast hidden class у slow dictionary mode, блок 1 п.7). Тому в гарячому коді частий `delete` шкодить перформансу — інколи краще `obj.key = undefined` (зберігає форму) або immutable-копія через destructuring.

**Immutable vs mutable (архітектурний трейдоф):**
- **Мутуючі** (`splice`, `delete`) — ефективні за пам'яттю (на місці) ↔ побічні ефекти, проблеми зі спільними посиланнями, ламають change detection у React/Redux.
- **Немутуючі** (`filter`, spread, destructuring) — передбачуваність, чистота, дружні до reactive-фреймворків ↔ алокація нових об'єктів (overhead для великих структур).

Сучасний фронтенд (React/Redux) тяжіє до **immutable** оновлень. Для великих структур — бібліотеки структурного шерингу (Immer, Immutable.js), що дають immutable-семантику без повного копіювання.

## 30. Для чого тип `void`?

### L1 — Junior
**Два різні `void` у світі JS/TS:**
- **JS-оператор `void`** — обчислює вираз і **завжди повертає `undefined`**.
- **TS-тип `void`** — означає «функція **нічого** не повертає».

```javascript
void 0;                  // undefined (JS-оператор)
```
```typescript
function log(msg: string): void {  // TS: нічого не повертає
  console.log(msg);
}
```

### L2 — Middle
**TS-тип `void`** — для функцій без значущого повернення:
```typescript
function saveUser(user: User): void {
  db.save(user);         // нічого не return (або return; без значення)
}
```

**JS-оператор `void`** — рідкісні застосування:
```javascript
// Гарантовано отримати undefined (історично: undefined можна було перевизначити)
void 0;
// javascript:void(0) — посилання, що нічого не робить (legacy HTML)
// Arrow, що "викидає" повернене значення (щоб не повертати з callback)
button.onclick = () => void doSomething();
```

**Gotcha:** `void` ≠ `undefined` ≠ `never`. `void` — «не дивись на повернене»; `undefined` — конкретне значення/тип; `never` — функція **ніколи не завершується нормально** (кидає або вічний цикл).

### L3 — Senior
**`void` vs `undefined` vs `never` (TS-нюанс):**
- **`void`** — «повернене значення ігнорується». Функція може фактично повертати `undefined`, але контракт каже «не покладайся на це».
- **`undefined`** — конкретний тип з одним значенням.
- **`never`** — недосяжний тип: функція, що завжди кидає (`throw`) або зациклюється; порожній union. Використовується для exhaustiveness checking.

**Тонкість `void` у callbacks (часта пастка TS):** функція типу `() => void` **дозволяє** повертати значення — TS навмисно це допускає, щоб callbacks могли повертати щось, що ігнорується:
```typescript
const arr: number[] = [];
[1, 2, 3].forEach((x) => arr.push(x)); // push повертає number, але forEach очікує void — ОК
```
Якби `void` суворо вимагав `undefined`, такий поширений код не компілювався б. Це свідомий дизайн TS: `void` у позиції повернення callback означає «значення відкидається», а не «має бути undefined».

**Трейдоф:** `void` документує намір «без результату» (читабельність, контракт) ↔ це не жорстка гарантія в runtime (JS однаково поверне undefined). У TS це інструмент **типобезпеки й комунікації наміру**, не runtime-обмеження.

## 31. Де і для чого `super()`?

### L1 — Junior
**`super()`** викликає **конструктор батьківського класу** при наслідуванні. Потрібен у дочірньому класі **перед** використанням `this`.

```javascript
class Animal {
  constructor(name) { this.name = name; }
}
class Dog extends Animal {
  constructor(name, breed) {
    super(name);          // викликати конструктор Animal
    this.breed = breed;
  }
}
```

### L2 — Middle
**`super` має дві форми:**
```javascript
class Cat extends Animal {
  constructor(name) {
    super(name);                    // 1. super() — конструктор батька
  }
  speak() {
    super.speak();                  // 2. super.method() — метод батька
    console.log(`${this.name}: Мяу`);
  }
}
```

**Gotcha — `super()` обов'язковий перед `this`:**
```javascript
class Dog extends Animal {
  constructor(name) {
    this.name = name;   // ❌ ReferenceError — this до super()!
    super(name);
  }
}
```
У дочірньому класі `this` **не існує**, поки не викликано `super()` — бо саме батьківський конструктор «створює» `this`.

### L3 — Senior
**Чому `this` недоступний до `super()`:** у JS дочірній клас **не створює** власний `this` — він отримує його **від** батьківського конструктора (через ланцюг `super`). Поки `super()` не виконано, `this` у TDZ-подібному стані → доступ кидає `ReferenceError`. Це відрізняється від мов, де об'єкт алокується дочірнім класом.

**`super` у методах** — викликає метод батька через **прототипний ланцюг** (`[[HomeObject]]`): `super.method()` шукає `method` у прототипі **класу, де метод визначено**, не в прототипі `this`. Це дозволяє коректну поведінку при глибокому наслідуванні.

**Трейдоф / зв'язок з Python:** концептуально як `super().__init__()` у Python (блок Python п.5), але JS **жорсткіше** вимагає виклику перед `this`. JS-наслідування — одиничне (немає множинного, на відміну від Python MRO), тому `super` однозначний — завжди прямий батько в ланцюгу `extends`. Класи в JS — синтаксичний цукор над прототипним наслідуванням; `super` приховує роботу з `Object.getPrototypeOf`.

## 32. Для чого `this` і коли використовувати?

### L1 — Junior
**`this`** — посилання на **об'єкт, у контексті якого виконується функція**. «Хто викликав метод».

```javascript
const user = {
  name: "Іван",
  greet() { return `Привіт, ${this.name}`; }, // this = user
};
user.greet(); // "Привіт, Іван"
```

### L2 — Middle
**`this` визначається ТИМ, ЯК викликають функцію (call site), а не де оголошено:**
```javascript
const user = {
  name: "Іван",
  greet() { return this.name; },
};

user.greet();                    // this = user → "Іван"
const fn = user.greet;
fn();                            // this = undefined (strict) → помилка!
```

**Чотири правила `this`:**
1. **Method call** `obj.fn()` → `this = obj`.
2. **Plain call** `fn()` → `this = undefined` (strict) / `global` (non-strict).
3. **`new Fn()`** → `this` = новий об'єкт.
4. **`fn.call/apply/bind(obj)`** → `this = obj` (явно).

**Arrow-функції — виняток:** не мають власного `this`, беруть **лексично** (з оточення):
```javascript
const obj = {
  items: [1, 2],
  process() {
    this.items.forEach((x) => console.log(this.items)); // arrow → this = obj ✅
  },
};
```

**Gotcha:** втрата `this` при передачі методу як callback — класична пастка:
```javascript
button.addEventListener("click", user.greet);        // ❌ this ≠ user
button.addEventListener("click", () => user.greet()); // ✅ arrow зберігає
button.addEventListener("click", user.greet.bind(user)); // ✅ bind
```

### L3 — Senior
**`this` у JS — динамічний (late binding):** на відміну від `self` у Python (явний перший параметр), JS `this` визначається **в момент виклику** за правилами вище. Це гнучко (один метод можна «позичити» для іншого об'єкта через `call`), але джерело багів (втрата контексту).

**`call` / `apply` / `bind`:**
```javascript
fn.call(obj, arg1, arg2);    // викликати з this=obj, аргументи списком
fn.apply(obj, [arg1, arg2]); // те саме, аргументи масивом
const bound = fn.bind(obj);  // НОВА функція з назавжди прив'язаним this
```

**Arrow — лексичний `this` (ключове):** arrow «запам'ятовує» `this` з місця визначення й **ігнорує** правила виклику. Тому arrow ідеальний для callbacks усередині методів (зберігає `this` об'єкта), але **непридатний** для самих методів об'єкта (там `this` стане зовнішнім, питання 24).

**Трейдоф:** динамічний `this` дає гнучкість (method borrowing, узагальнені утиліти) ↔ непередбачуваність і клас багів «втрати контексту». Сучасна практика мінімізує залежність від `this`: arrow для збереження контексту, class fields (`greet = () => {}`) для авто-прив'язки методів, або взагалі функціональний стиль без `this`. У React класові компоненти потребували `.bind` у конструкторі — одна з причин переходу на хуки (функціональні компоненти без `this`).

> ⚠️ Поведінка `this` залежить від strict mode та контексту (модуль/функція/глобал). Деталі — звіряй зі специфікацією ECMAScript.

## 33. Що таке `NaN` і як його використати?

### L1 — Junior
**`NaN` (Not a Number)** — спеціальне значення «не число», результат некоректної математичної операції.

```javascript
0 / 0;              // NaN
parseInt("abc");    // NaN
Number("привіт");   // NaN
Math.sqrt(-1);      // NaN
```

### L2 — Middle
**Головна дивина — `NaN` не дорівнює сам собі:**
```javascript
NaN === NaN;        // false (!) — єдине значення в JS, нерівне собі
typeof NaN;         // "number" (так, NaN — це тип number)

// Тому перевіряти треба спеціально:
Number.isNaN(value);  // ✅ надійно — лише справжній NaN
isNaN(value);         // ⚠️ застарілий — спершу приводить до числа
```

**Gotcha — `isNaN` vs `Number.isNaN`:**
```javascript
isNaN("привіт");        // true — бо Number("привіт") = NaN (приведення!)
Number.isNaN("привіт"); // false — рядок не є NaN
Number.isNaN(NaN);      // true — лише справжній NaN
```
**Завжди `Number.isNaN`** — `isNaN` робить приховане приведення типу й дає хибні спрацювання.

### L3 — Senior
`NaN` походить зі стандарту **IEEE 754** (числа з плаваючою комою). Властивість «нерівне собі» — теж із IEEE 754: будь-яке порівняння з `NaN` (`<`, `>`, `===`) дає `false`. Це дозволяє **детектувати** `NaN` через `x !== x` (єдине значення, для якого це `true`) — старий трюк до `Number.isNaN`.

**Практичне застосування:** `NaN` сигналізує про **невдале перетворення/обчислення**. Senior перевіряє результати парсингу/арифметики:
```javascript
const n = Number(userInput);
if (Number.isNaN(n)) return handleInvalidInput();
```

**Трейдоф/пастка:** `NaN` **поширюється** (будь-яка арифметика з `NaN` дає `NaN`) — один поганий ввід отруює весь розрахунок, інколи непомітно (`NaN` у сумі/середньому). Тому валідація на межі (де дані входять у систему) критична. `NaN` — це JS-аналог «тихої помилки», що тягнеться далі замість раннього падіння (порушення fail-fast).

## 34. Що таке NPM? Аналоги?

### L1 — Junior
**NPM (Node Package Manager)** — менеджер пакетів для JS: встановлює сторонні бібліотеки з найбільшого у світі реєстру пакетів.

```bash
npm install express      # встановити пакет
npm install              # встановити все з package.json
```

### L2 — Middle
**NPM = CLI + реєстр + `package.json`:**
```json
{
  "dependencies": { "express": "^4.18.0" },
  "devDependencies": { "jest": "^29.0.0" },
  "scripts": { "start": "node index.js", "test": "jest" }
}
```
- `package.json` — маніфест (залежності, скрипти, метадані).
- `package-lock.json` — **точні** версії всього дерева (відтворюваність).
- `node_modules/` — встановлені пакети (не комітять).

**Аналоги (менеджери пакетів):**
| Менеджер | Особливість |
|---|---|
| **npm** | дефолтний, іде з Node |
| **Yarn** | був швидший/надійніший за старий npm; Yarn Berry (PnP) |
| **pnpm** | економить диск (символьні лінки, спільне сховище), швидкий |
| **Bun** | новий рантайм + менеджер, дуже швидкий |

**Gotcha:** завжди коміть **lock-файл** (`package-lock.json`/`yarn.lock`/`pnpm-lock.yaml`) — без нього колеги/CI отримають **різні** версії транзитивних залежностей → «works on my machine».

### L3 — Senior
*(Деталі переваг — питання 35.)* NPM-екосистема — і сила (найбільший реєстр, швидке прототипування), і ризик (**supply-chain атаки**: left-pad, event-stream, colors). Senior ставиться до кожної залежності як до **довіреного коду в проді**: аудит (`npm audit`), мінімізація дерева, lock-файли з хешами цілісності (integrity), фіксація версій для критичного, перевірка популярності/підтримки пакета перед додаванням.

**Трейдоф «багато дрібних пакетів» (JS-культура):** велика гранулярність (пакет на одну функцію) → швидке перевикористання ↔ роздуте дерево залежностей, вага `node_modules` (мем «найважчий об'єкт у Всесвіті»), широка attack surface, ризик зламаної транзитивної залежності. Тренд — консолідація (менше залежностей, вбудовані можливості Node, зрілі рантайми типу Bun/Deno зі стандартною бібліотекою).

## 35. NPM vs Yarn/PNPM — переваги й недоліки

### L1 — Junior
Усі три роблять одне — встановлюють пакети. Різниця — у **швидкості, використанні диска й надійності**.
- **npm** — стандартний, простий, іде з Node.
- **Yarn** — був швидший і надійніший (історично).
- **pnpm** — найекономніший за диском і дуже швидкий.

### L2 — Middle
| | npm | Yarn | pnpm |
|---|---|---|---|
| Іде з Node | ✅ | ні | ні |
| Швидкість | хороша (покращилась) | швидкий | **найшвидший** |
| Дисковий простір | дублює пакети | дублює | **спільне сховище** (економія) |
| Структура node_modules | плоска | плоска | **символьні лінки** (суворіша) |
| Lock-файл | package-lock.json | yarn.lock | pnpm-lock.yaml |
| Monorepo (workspaces) | ✅ | ✅ (сильний) | ✅ (сильний) |

**Ключова перевага pnpm:** зберігає кожну версію пакета **один раз** глобально, а в проєктах робить **жорсткі/символьні лінки** → економія гігабайтів диска при багатьох проєктах + швидша установка.

**Gotcha:** не змішуй менеджери в одному проєкті (різні lock-файли) — обери один. Перемикання вимагає видалити старий lock + `node_modules`.

### L3 — Senior
**Глибша різниця — структура `node_modules`:**
- **npm/Yarn (classic):** «плоский» (hoisted) `node_modules` — транзитивні залежності піднімаються нагору. Проблема — **phantom dependencies**: код може імпортувати пакет, який ти **не** оголосив (бо він випадково опинився в node_modules як чужа транзитивна залежність) → крихкість.
- **pnpm:** суворий, **немає** phantom dependencies — доступні лише явно оголошені залежності (через символьні лінки на ізольоване сховище). Це **коректніше**, але інколи ламає пакети, що покладались на hoisting.

**Трейдофи вибору:**
- **npm:** нуль налаштувань (іде з Node), найширша сумісність ↔ історично повільніший, дублювання диска, phantom deps.
- **Yarn:** зрілий, сильні workspaces, Yarn Berry (Plug'n'Play — без node_modules взагалі) ↔ PnP ламає сумісність із деякими інструментами, фрагментація версій (v1 vs Berry).
- **pnpm:** швидкість + економія диска + коректність (немає phantom deps) ↔ суворість інколи виявляє чужі баги, менша (хоч зростаюча) поширеність.

**Senior-вибір:** для нових проєктів/monorepo — **pnpm** (швидкість, диск, коректність) набирає популярність як дефолт. npm — безпечний стандарт без додаткових залежностей. Yarn Berry — для специфічних потреб (PnP, zero-installs). Головне — **узгодженість у команді** й закомічений lock-файл; конкретний інструмент вторинний.

> ⚠️ Бенчмарки швидкості й можливості менеджерів швидко змінюються (npm наздоганяє, Bun з'явився). Звіряй з актуальними даними під свій кейс.

## 36. Методи Promise API

### L1 — Junior
**Promise** — об'єкт, що представляє результат асинхронної операції (ще не готовий). Має три стани: **pending** (очікує) → **fulfilled** (успіх) / **rejected** (помилка).

```javascript
const p = fetch("/api");
p.then((res) => console.log(res))   // успіх
 .catch((err) => console.error(err)); // помилка
```

### L2 — Middle
**Комбінатори (для кількох промісів):**
```javascript
// Promise.all — чекає ВСІ; якщо хоч один впав → весь падає
const [a, b] = await Promise.all([fetch("/a"), fetch("/b")]);

// Promise.allSettled — чекає всі, НІКОЛИ не падає (повертає статуси)
const results = await Promise.allSettled([fetch("/a"), fetch("/b")]);
// [{status:"fulfilled", value}, {status:"rejected", reason}]

// Promise.race — перший, що ЗАВЕРШИВСЯ (успіх АБО помилка)
const fastest = await Promise.race([fetch("/a"), timeout(5000)]);

// Promise.any — перший УСПІШНИЙ (ігнорує помилки, поки є шанс)
const first = await Promise.any([fetch("/mirror1"), fetch("/mirror2")]);
```

| Метод | Завершується коли | Падає коли |
|---|---|---|
| `all` | усі fulfilled | **будь-який** rejected |
| `allSettled` | усі завершились | **ніколи** |
| `race` | перший завершився (будь-як) | перший — rejected |
| `any` | перший fulfilled | **усі** rejected (AggregateError) |

**Gotcha:** `Promise.all` — **fail-fast**: один reject валить усе, навіть якщо інші успішні. Потрібні всі результати незалежно від помилок → `allSettled`.

### L3 — Senior
**Коли який (вибір за семантикою):**
- **`all`** — потрібні **всі** результати, і помилка будь-якого критична (завантажити користувача + його замовлення + налаштування).
- **`allSettled`** — обробити що вийшло, помилки окремих ок (надіслати 100 листів — звіт по кожному).
- **`race`** — таймаут (гонка реального запиту з `setTimeout`-промісом), або найшвидше джерело.
- **`any`** — fallback/резервування (кілька дзеркал, бери перше робоче).

**Статичні фабрики:**
```javascript
Promise.resolve(value);  // одразу fulfilled
Promise.reject(err);     // одразу rejected
```

**Пастки Senior:**
- **Unhandled rejection:** проміс без `.catch`/`try-catch` → `unhandledRejection` (у Node може покласти процес). Завжди обробляй.
- **`Promise.all` втрачає часткові результати:** при fail-fast результати вже виконаних промісів губляться (хоча запити відбулись) — для cleanup це важливо.
- **Послідовно vs паралельно:** `await` у циклі — послідовно (повільно); `Promise.all(map)` — паралельно (питання 21). Але паралельність без ліміту може перевантажити (1000 одночасних запитів) → потрібен concurrency limit (`p-limit`).

**Трейдоф:** Promise API дає декларативну композицію асинхронності ↔ тонкощі обробки помилок (fail-fast vs settled), ризик unhandled rejection, неочевидний контроль паралельності. `async/await` (питання 21) — цукор над цим, але комбінатори (`all`/`allSettled`) лишаються незамінними для паралельних операцій.

## 37. Структура HTTP request/response

### L1 — Junior
**Аналогія — лист у конверті:** є адреса й тип операції (стартовий рядок), службові позначки (заголовки) і вміст (тіло).

*(Детально розкрито в основному блоці про HTTP — тут JS-специфіка.)*

### L2 — Middle
**Request:**
```http
POST /api/users HTTP/1.1          ← метод + шлях + версія
Host: example.com                 ← заголовки
Content-Type: application/json
Authorization: Bearer eyJ...
                                  ← порожній рядок
{"name": "Іван"}                  ← тіло
```
**Response:**
```http
HTTP/1.1 201 Created              ← версія + статус-код + причина
Content-Type: application/json

{"id": 1, "name": "Іван"}
```

**У JS (fetch):**
```javascript
const res = await fetch("/api/users", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ name: "Іван" }),
});
console.log(res.status);          // 201
const data = await res.json();    // тіло відповіді
```

**Статус-коди:** 2xx успіх, 3xx редирект, 4xx помилка клієнта (400, 401, 403, 404, 429), 5xx помилка сервера.

**Gotcha:** `fetch` **не кидає** помилку на 4xx/5xx (лише на мережевих збоях) — треба перевіряти `res.ok`/`res.status` вручну:
```javascript
if (!res.ok) throw new Error(`HTTP ${res.status}`);
```

### L3 — Senior
*(Глибина HTTP — в основному блоці: stateless, HTTP/1.1 vs 2 vs 3, idempotency, кешування.)* JS-специфіка:

**`fetch` нюанси (senior):**
- Не кидає на HTTP-помилках (лише network/CORS) — на відміну від axios.
- `res.json()`/`res.text()` можна викликати **лише раз** (тіло — стрим, споживається).
- CORS — браузерне обмеження (preflight OPTIONS для не-простих запитів).
- `AbortController` для скасування/таймауту:
```javascript
const ctrl = new AbortController();
setTimeout(() => ctrl.abort(), 5000);
await fetch(url, { signal: ctrl.signal });
```

**Трейдоф fetch vs бібліотеки (axios):** `fetch` вбудований (немає залежності), стандартний, стримовий ↔ багатослівніший (ручна перевірка `res.ok`, ручний JSON), немає вбудованих interceptors/таймауту. axios — зручніший API, авто-JSON, interceptors ↔ зайва залежність. Node 18+ має вбудований `fetch` — тренд до стандартизації.

## 38. Що таке `new Set()` і `new Map()`?

### L1 — Junior
- **`Set`** — колекція **унікальних** значень (без дублікатів).
- **`Map`** — колекція **пар ключ→значення** (як об'єкт, але ключем може бути **будь-що**).

```javascript
const set = new Set([1, 2, 2, 3]);   // {1, 2, 3} — дублікат прибрано
const map = new Map([["a", 1], ["b", 2]]);
map.get("a");                        // 1
```

### L2 — Middle
```javascript
// SET — унікальність
const s = new Set();
s.add(1); s.add(1);      // лише один
s.has(1);                // true — O(1) перевірка
s.delete(1);
s.size;                  // кількість
[...new Set(array)];     // дедуплікація масиву!

// MAP — ключ-значення
const m = new Map();
m.set("key", "val");
m.set(objKey, "val");    // ключ — об'єкт! (об'єкт так не може)
m.get("key");
m.has("key");
m.size;
for (const [k, v] of m) {}  // ітерація в порядку вставки
```

**Map vs звичайний об'єкт:**
| | Map | Object |
|---|---|---|
| Ключі | **будь-який тип** | лише string/symbol |
| Розмір | `.size` | `Object.keys().length` |
| Порядок | гарантований (вставки) | ~гарантований (з нюансами) |
| Ітерація | напряму (`for...of`) | через `Object.entries` |
| Прототип | чистий | має успадковані ключі |

**Gotcha:** `Set` порівнює об'єкти за **посиланням**, не за значенням: `new Set([{a:1}, {a:1}])` має **2** елементи (різні об'єкти).

### L3 — Senior
Обидва — **хеш-таблиці** (O(1) середній доступ/вставка/пошук), на відміну від масиву (O(n) пошук). Тому для частих перевірок «чи є елемент» серед багатьох — `Set` замість `array.includes()` перетворює O(n) на O(1) (класична оптимізація, як у Python set/dict).

**Коли Map замість Object (senior-критерії):**
- Ключі **не рядки** (об'єкти, числа без приведення).
- Часті додавання/видалення (Map оптимізований; видалення ключа Object деоптимізує форму, питання 29/блок 1 п.7).
- Потрібні розмір/ітерація/порядок «з коробки».
- Динамічні ключі без ризику колізії з прототипом (`__proto__`, `constructor`).

**WeakMap/WeakSet** — ключі тримаються **слабко** (не заважають garbage collection): для приватних даних об'єктів і кешів, що не повинні тримати об'єкти в пам'яті. Senior використовує для метаданих, прив'язаних до об'єктів, без витоку пам'яті.

**Трейдоф:** Set/Map — правильні структури для унікальності/мапінгу (семантика + O(1) + будь-які ключі) ↔ трохи більше пам'яті за масив/об'єкт, не серіалізуються напряму в JSON (треба `[...map]`). Object лишається для простих рядкових записів і JSON; Map — для «справжньої» мапи з довільними ключами.

## 39. `&&`, `||` та `??` — різниця

### L1 — Junior
- **`&&` (AND)** — повертає перший **falsy** або останній (усі truthy).
- **`||` (OR)** — повертає перший **truthy** або останній.
- **`??` (nullish coalescing)** — повертає праве, лише якщо ліве `null`/`undefined`.

```javascript
true && "далі";    // "далі"
false || "запасне"; // "запасне"
null ?? "дефолт";  // "дефолт"
```

### L2 — Middle
**`||` vs `??` — головна відмінність (часте питання):**
```javascript
0 || "дефолт";       // "дефолт" — бо 0 falsy
0 ?? "дефолт";       // 0 — бо 0 не null/undefined!

"" || "дефолт";      // "дефолт" — бо "" falsy
"" ?? "дефолт";      // "" — бо "" не nullish
```
`||` спрацьовує на **будь-яке falsy** (`0`, `""`, `false`, `NaN`, `null`, `undefined`). `??` — **лише** на `null`/`undefined`.

**Чому `??` важливий:** для дефолтів, коли `0`/`""`/`false` — **валідні** значення:
```javascript
function setVolume(level) {
  this.volume = level ?? 50;  // 0 → лишається 0 (валідний!)
  // level || 50 → 0 стало б 50 (БАГ!)
}
```

**Gotcha:** `||` для дефолтів — пастка, коли `0`/`""` легітимні. Для «значення відсутнє» (null/undefined) — `??`; для «значення хибне/порожнє» — `||`.

### L3 — Senior
**Short-circuit (коротке замикання) — обидва не обчислюють зайве:**
```javascript
user && user.name;           // якщо user falsy → не чіпає .name (захист)
config.timeout ?? DEFAULT;   // обчислює DEFAULT лише якщо timeout nullish
isReady && doSomething();    // умовний виклик без if
```
`&&` зупиняється на першому falsy, `||` — на першому truthy, `??` — на першому non-nullish. Праву частину **не обчислюють**, якщо ліва вже вирішила результат (важливо для побічних ефектів/дорогих викликів).

**Optional chaining `?.` + `??` (сучасний дует):**
```javascript
const city = user?.address?.city ?? "Невідомо";
// ?. — безпечний доступ (undefined замість помилки на null)
// ?? — дефолт, якщо результат nullish
```

**Gotcha синтаксису:** `??` не можна змішувати з `||`/`&&` без дужок (`a || b ?? c` — SyntaxError) — мова змушує явно групувати, щоб уникнути неоднозначності пріоритету.

**Трейдоф:** `??` і `?.` (ES2020) прибрали цілий клас багів із дефолтами й доступом до вкладених властивостей — раніше писали громіздкі перевірки (`a && a.b && a.b.c`) або помилково `|| default` на нулях. Senior використовує `??` для дефолтів (точна семантика «відсутнє»), `||` лише коли справді треба «будь-яке хибне», `?.` для безпечного доступу. Це підвищує і коректність, і читабельність.

> ⚠️ `??`, `?.` — ES2020+. У старих середовищах потрібен транспайлінг (Babel). У сучасному Node/браузерах — нативно.

# Блок 9. React — 20 питань з відповідями

> Формат — **Принцип Сходів**: L1 Junior → L2 Middle → L3 Senior.
> Сучасний React 18+ (функціональні компоненти + хуки). Окремо: **Hooks, Context API, Vite, Shadcn UI**.

---

## 1. Що таке React і навіщо?

### L1 — Junior
**Аналогія — конструктор LEGO для інтерфейсів.** React — це **бібліотека** для побудови UI з **компонентів** (переиспользуваних «цеглинок»). Ти описуєш, **що** показати (декларативно), а React сам оновлює екран.

```jsx
function Welcome() {
  return <h1>Привіт, світе!</h1>;   // компонент
}
```

### L2 — Middle
**Три ключові ідеї React:**
| Ідея | Пояснення |
|---|---|
| **Декларативність** | описуєш кінцевий стан UI, не кроки оновлення DOM |
| **Компонентність** | UI з переиспользуваних компонентів (композиція) |
| **Virtual DOM** | React рахує мінімальні зміни й ефективно оновлює реальний DOM (питання 5) |
| **Односторонній потік даних** | дані течуть згори вниз (props), передбачувано |

### L3 — Senior
React — **бібліотека** (не фреймворк): робить лише UI, решту (роутинг, стан, дані) добираєш сам (React Router, Redux/Zustand, React Query). **Декларативна модель:** ти описуєш `UI = f(state)`, React обчислює різницю й оновлює DOM (reconciliation, питання 5). 

**Трейдоф:** гнучкість (обираєш екосистему під себе) ↔ «fatigue» (треба збирати стек). vs Angular (повний фреймворк) / Vue (посередині). Ментальна модель `UI = f(state)` — фундамент: зміна стану → перерендер → React синхронізує DOM. Це прибирає ручну маніпуляцію DOM (jQuery-стиль) на користь передбачуваності.

## 2. Що таке JSX?

### L1 — Junior
**JSX** — синтаксис, що дозволяє писати «HTML усередині JavaScript».
```jsx
const element = <h1 className="title">Привіт, {name}!</h1>;
```
Це не рядок і не HTML — це JavaScript, який виглядає як розмітка.

### L2 — Middle
JSX **компілюється** в звичайні виклики JS:
```jsx
// JSX:
const el = <h1 className="title">Привіт</h1>;
// Компілюється (Babel/SWC) в:
const el = React.createElement("h1", { className: "title" }, "Привіт");
```
**Особливості:**
- `className` замість `class`, `htmlFor` замість `for` (бо `class`/`for` — зарезервовані в JS).
- `{вираз}` — вставити JS: `{name}`, `{2 + 2}`, `{items.map(...)}`.
- Один кореневий елемент (або `<>...</>` — Fragment).

### L3 — Senior
JSX — **синтаксичний цукор** над `React.createElement` (або `_jsx` у новому JSX-transform). Компілятор (Babel/SWC у Vite) перетворює його в об'єкти-описи елементів (**React elements** — легкі об'єкти, не DOM-вузли). Ці об'єкти → Virtual DOM → reconciliation → реальний DOM.

**Чому JSX, а не шаблони (як Angular/Vue):** JSX — це **повноцінний JS**, тому вся логіка (умови, цикли, функції) — рідною мовою, без спеціального шаблонного синтаксису. Трейдоф: змішування розмітки й логіки (критики кажуть «порушує separation») ↔ повна сила JS + типізація (TSX). JSX не обов'язковий (можна `createElement`), але стандарт де-факто.

## 3. Функціональні vs класові компоненти ⭐

### L1 — Junior
Два способи створити компонент:
```jsx
// Функціональний (сучасний стандарт)
function Button({ label }) {
  return <button>{label}</button>;
}

// Класовий (legacy)
class Button extends React.Component {
  render() { return <button>{this.props.label}</button>; }
}
```
**Сьогодні — функціональні + хуки.** Класові — legacy.

### L2 — Middle
| | Функціональні | Класові |
|---|---|---|
| Синтаксис | функція | `class extends Component` |
| Стан | `useState` (хук) | `this.state` |
| Lifecycle | `useEffect` (хук) | `componentDidMount` тощо |
| `this` | немає (простіше) | є (пастки з bind) |
| Сучасність | ✅ стандарт | legacy |

### L3 — Senior
**Хуки (2019, React 16.8) зробили функціональні компоненти повноцінними** — раніше клас був потрібен для стану/lifecycle. Тепер функціональні + хуки — стандарт: менше коду, немає `this`-пасток (JS-блок п.32), краще переиспользування логіки (custom hooks vs HOC/render-props).

**Чому перейшли:** класи мали проблеми — `this.bind`, розкидана логіка по lifecycle-методах (одна фіча в `componentDidMount` + `componentWillUnmount`), важке переиспользування stateful-логіки. Хуки групують логіку за **фічею**, не за lifecycle, і дають чисте переиспользування (custom hooks). Класи ще підтримуються (не видалять), але новий код — функціональний.

## 4. Props vs State ⭐

### L1 — Junior
- **Props** — дані, що **приходять ЗВЕРХУ** (від батька), **тільки для читання**.
- **State** — **внутрішні** дані компонента, які він **сам змінює** (і це викликає перерендер).

```jsx
function Counter({ start }) {          // start — PROP (від батька)
  const [count, setCount] = useState(start);   // count — STATE (внутрішній)
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
```

### L2 — Middle
| | Props | State |
|---|---|---|
| Джерело | батьківський компонент | усередині компонента |
| Змінюваність | **read-only** | змінюється (setState/useState) |
| Хто керує | батько | сам компонент |
| Перерендер при зміні | так (батько передав нове) | так (setState) |

**Gotcha:** props **не можна** мутувати (`props.x = 1` — заборонено). Односторонній потік: дані вниз через props, події вгору через callbacks.

### L3 — Senior
**Односторонній потік даних** — фундамент передбачуваності React: дані течуть **вниз** (props), зміни ініціюються **вгору** (callbacks від дитини до батька). Стан живе в **одному місці** (single source of truth), звідти передається вниз.

**Lifting state up** (питання 18) — якщо кілька компонентів ділять стан, він піднімається до спільного предка. **Трейдоф:** односторонній потік — передбачуваність, легкий дебаг (знаєш, звідки дані) ↔ prop drilling (передавати через багато рівнів) → лікується Context (питання 15) чи state-менеджером. Props = «конфігурація ззовні», state = «внутрішня пам'ять».

## 5. Virtual DOM і reconciliation ⭐

### L1 — Junior
**Virtual DOM** — «чернетка» UI в пам'яті. React малює нову чернетку при зміні стану, **порівнює** зі старою, і оновлює в реальному DOM **лише те, що змінилось** (не весь екран).

### L2 — Middle
```
Зміна стану → новий Virtual DOM (об'єкти в пам'яті)
   → DIFF зі старим (reconciliation) → знайти мінімальні зміни
   → оновити ТІЛЬКИ ці вузли в реальному DOM
```
**Чому:** пряма маніпуляція реальним DOM **повільна** (перерахунок layout/paint). Virtual DOM (JS-об'єкти) — швидкий; React батчить зміни й робить мінімум операцій із реальним DOM.

### L3 — Senior
**Reconciliation** — алгоритм порівняння двох дерев. React використовує **евристики** (повне дерево-порівняння було б `O(n³)`):
1. Різні типи елементів → замінити піддерево цілком.
2. Той самий тип → оновити атрибути, рекурсія в дітей.
3. **Keys** (питання 6) — для ефективного порівняння списків.

**Fiber (React 16+)** — переписана архітектура reconciliation: робота розбита на **одиниці (fibers)**, які можна **переривати/пріоритезувати** → плавніший UI (не блокує браузер довгим рендером). React 18 concurrent features (`useTransition`, Suspense) на цьому.

**Gotcha:** «Virtual DOM завжди швидший» — міф. Він дає **зручність** (декларативність) з **прийнятною** швидкістю; для екстремальної продуктивності інколи потрібна ручна оптимізація (memo, віртуалізація списків). Solid/Svelte взагалі відмовились від Virtual DOM (компіляція в прямі DOM-оновлення).

## 6. Keys у списках ⭐

### L1 — Junior
При рендері списку кожен елемент має **`key`** — унікальний ідентифікатор, щоб React знав, який є який.
```jsx
{users.map((user) => <li key={user.id}>{user.name}</li>)}
```

### L2 — Middle
**Навіщо key:** допомагає reconciliation (питання 5) відстежити, які елементи додались/видались/перемістились → ефективне оновлення.

**Gotcha — не використовуй index як key для динамічних списків:**
```jsx
{items.map((item, index) => <li key={index}>...</li>)}   // ❌ для змінних списків
{items.map((item) => <li key={item.id}>...</li>)}        // ✅ стабільний id
```
При видаленні/сортуванні index зсувається → React плутає елементи → баги (неправильний стан, input-и).

### L3 — Senior
**Key дає елементу стабільну ідентичність** між рендерами. Без стабільного key React може:
- Переиспользувати не той DOM-вузол (стан input прив'язується не до того елемента).
- Робити зайві перемонтування (втрата стану/фокусу).

**Чому index поганий:** при вставці на початок/сортуванні index кожного зсувається → React думає, що **всі** елементи змінились (хоча лише порядок) → неправильна асоціація стану. **Index ок лише** для статичних списків, що ніколи не міняються/сортуються. Правильний key — **стабільний унікальний id** з даних. Це часте питання й джерело хитрих багів зі станом у списках.

## 7. Controlled vs uncontrolled компоненти

### L1 — Junior
- **Controlled** — значення форми контролює **React state** (single source of truth).
- **Uncontrolled** — значення тримає **сам DOM**, React читає через ref.

```jsx
// Controlled — React керує значенням
const [value, setValue] = useState("");
<input value={value} onChange={(e) => setValue(e.target.value)} />

// Uncontrolled — DOM тримає значення
const ref = useRef();
<input ref={ref} />  // читаємо ref.current.value коли треба
```

### L2 — Middle
| | Controlled | Uncontrolled |
|---|---|---|
| Джерело істини | React state | DOM |
| Значення | `value` + `onChange` | `ref` |
| Валідація/трансформація | легко (у onChange) | важче |
| Дефолт | `value` | `defaultValue` |

**Best practice:** **controlled** за замовчуванням (React контролює → передбачувано, легко валідувати/трансформувати). Uncontrolled — для простих форм, інтеграції з не-React кодом, файлів (`<input type="file">` завжди uncontrolled).

### L3 — Senior
Controlled — узгоджується з `UI = f(state)`: значення завжди з state → передбачувано, можна валідувати/форматувати «на льоту», легко скинути/заповнити програмно. Ціна — ререндер на кожне натискання (зазвичай неважливо; для великих форм — оптимізують).

**Uncontrolled** — менше ререндерів, простіше для базових форм, але state «схований» у DOM (важче контролювати). **Трейдоф:** сучасні форм-бібліотеки (**React Hook Form**) використовують **uncontrolled + refs** саме для перформансу (менше ререндерів), даючи controlled-подібний API. Senior знає обидва: controlled для контролю, uncontrolled/RHF для перформансу великих форм.

## 8. Що таке Hooks + правила хуків ⭐

### L1 — Junior
**Hooks** — функції, що дозволяють функціональним компонентам мати **стан** і **побічні ефекти** (раніше — лише класи). Починаються з `use`: `useState`, `useEffect`.

**Правила хуків:**
1. Викликати **лише на верхньому рівні** (не в умовах/циклах).
2. Викликати **лише в React-компонентах** або власних хуках.

### L2 — Middle
```jsx
function Component() {
  // ✅ верхній рівень
  const [count, setCount] = useState(0);
  useEffect(() => { /* ... */ }, []);

  // ❌ НЕ можна в умові/циклі:
  if (x) {
    const [y] = useState(0);   // ПОРУШЕННЯ правил хуків!
  }
}
```
`eslint-plugin-react-hooks` автоматично ловить порушення.

### L3 — Senior
**Чому правила (під капотом):** React відстежує хуки за **порядком виклику** (не за іменем) — на кожному рендері хуки мають викликатись у **тому самому порядку**. Умовний виклик зсуває порядок → React прив'язує стан не до того хука → баги.

```
Рендер 1: useState(A), useState(B), useEffect(C)   ← порядок 1,2,3
Рендер 2: useState(A), useEffect(C)                ← якщо B пропущено умовою → плутанина!
```

**Правила забезпечують стабільний порядок** → React коректно асоціює стан/ефекти між рендерами. Хуки — це **композиція логіки** (custom hooks, питання 14) замість спадкової/HOC-складності класів. `exhaustive-deps` lint-правило — окрема важлива дисципліна (питання 10).

## 9. `useState` ⭐

### L1 — Junior
Додає **стан** у функціональний компонент.
```jsx
const [count, setCount] = useState(0);   // [значення, функція-сеттер], початкове 0
setCount(count + 1);                      // змінити → перерендер
```

### L2 — Middle
```jsx
// Functional update — коли нове залежить від старого
setCount(prev => prev + 1);   // ✅ безпечно (не залежить від застарілого count)

// Lazy initialization — важке початкове значення рахується раз
const [data, setData] = useState(() => expensiveInit());

// Об'єктний стан — заміняється ЦІЛКОМ (не мержиться, як у класах!)
const [user, setUser] = useState({ name: "", age: 0 });
setUser(prev => ({ ...prev, age: 30 }));   // треба спред вручну
```

**Gotcha:** `setState` **асинхронний/батчиться** — `count` не оновлюється миттєво в тому ж виклику:
```jsx
setCount(count + 1);
setCount(count + 1);   // обидва бачать старий count → +1, не +2!
setCount(p => p + 1);  // ✅ functional update → +2
```

### L3 — Senior
**setState тригерить перерендер** (React планує оновлення, батчить кілька setState в один рендер — React 18 автоматично батчить навіть у промісах/таймерах). **Functional update** (`prev => ...`) обов'язковий, коли нове значення залежить від старого (уникає stale closure — застарілого замикання).

**Stale closure** — часта пастка: замикання «запам'ятовує» значення на момент рендера. `setCount(count+1)` двічі використовує той самий `count`; `setCount(p=>p+1)` завжди бере актуальний. Стан **immutable** — не мутуй (`user.age = 30` не спрацює), створюй новий об'єкт (спред). Для складного стану → `useReducer` (питання 13).

## 10. `useEffect` (залежності, cleanup) ⭐

### L1 — Junior
`useEffect` — для **побічних ефектів** (запити, підписки, таймери, ручні маніпуляції DOM) — того, що не є чистим рендером.
```jsx
useEffect(() => {
  console.log("компонент відрендерився");
});
```

### L2 — Middle
```jsx
// Масив залежностей керує, КОЛИ виконувати:
useEffect(() => { /* кожен рендер */ });              // без масиву
useEffect(() => { /* лише при МОНТУВАННІ */ }, []);   // порожній масив
useEffect(() => { /* при зміні userId */ }, [userId]); // конкретні deps

// Cleanup — функція повернення (відписка, очищення)
useEffect(() => {
  const timer = setInterval(tick, 1000);
  return () => clearInterval(timer);    // cleanup при розмонтуванні/зміні deps
}, []);

// Типовий запит даних
useEffect(() => {
  fetch(`/api/user/${id}`).then(r => r.json()).then(setUser);
}, [id]);   // перезапит при зміні id
```

### L3 — Senior
**Життєвий цикл ефекту:** виконується **після** рендера (не блокує paint). Cleanup викликається **перед** наступним ефектом і при розмонтуванні → відписка/очищення (проти витоків, як listeners/timers у Node).

**Gotcha-и (найчастіші баги):**
- **Забуті залежності** — ефект використовує змінну, якої нема в deps → stale value. `exhaustive-deps` lint ловить.
- **Нескінченний цикл** — setState у ефекті без правильних deps → рендер → ефект → setState → ...
- **Cleanup обов'язковий** для підписок/таймерів/AbortController (інакше витік/гонки).

**React 18 StrictMode** навмисно **двічі** монтує в dev → ловить ефекти без cleanup. Сучасний тренд: **useEffect для синхронізації з зовнішнім** (не для похідного стану — те рахуй під час рендера); для запитів даних краще **React Query/SWR** (кеш, гонки, ретраї) замість ручного useEffect+fetch. Senior мінімізує useEffect (часто його не треба — обчислюй у рендері).

## 11. `useRef`

### L1 — Junior
`useRef` — «коробка», що **зберігає значення між рендерами** й **не викликає перерендер** при зміні. Часто — для доступу до DOM.
```jsx
const inputRef = useRef(null);
<input ref={inputRef} />
inputRef.current.focus();   // прямий доступ до DOM-елемента
```

### L2 — Middle
**Два застосування:**
```jsx
// 1. Доступ до DOM
const ref = useRef(null);
useEffect(() => ref.current.focus(), []);

// 2. Зберегти значення БЕЗ перерендеру (mutable, persists)
const renderCount = useRef(0);
renderCount.current++;   // не викликає перерендер (на відміну від useState)

// Зберегти попереднє значення, id таймера, тощо
const timerId = useRef();
```

### L3 — Senior
**useRef vs useState (ключова різниця):**
| | useState | useRef |
|---|---|---|
| Зміна → перерендер | ✅ | ❌ |
| Значення між рендерами | зберігає | зберігає |
| Для чого | дані, що впливають на UI | DOM, «позарендерні» значення |

`ref.current` — **мутабельна** «шухляда», що переживає рендери, але зміна **не тригерить** оновлення. Тому: DOM-доступ (focus, scroll, вимір), збереження id таймерів/попередніх значень, «інстанс-змінні» (як поля класу). **Не** для даних, що мають відображатись (там useState). Gotcha: не читай/пиши `ref.current` під час рендера (лише в ефектах/обробниках) — це побічний ефект.

## 12. `useMemo` та `useCallback` ⭐

### L1 — Junior
Обидва — **мемоізація** (кешування) для продуктивності:
- **`useMemo`** — кешує **обчислене значення**.
- **`useCallback`** — кешує **функцію**.
```jsx
const expensive = useMemo(() => heavyCompute(data), [data]);   // рахує лише коли data змінилось
const handler = useCallback(() => doSomething(id), [id]);      // та сама функція, поки id незмінний
```

### L2 — Middle
```jsx
// useMemo — не перераховувати важке щоразу
const sorted = useMemo(() => items.sort(compare), [items]);

// useCallback — стабільне посилання на функцію (щоб дитина не ререндерилась)
const handleClick = useCallback(() => setCount(c => c + 1), []);
<ExpensiveChild onClick={handleClick} />   // handleClick не міняється → memo дитини працює

// useCallback(fn, deps) ≡ useMemo(() => fn, deps)
```

**Gotcha:** мемоізація **не безкоштовна** (пам'ять + порівняння deps). **Не** оборачуй усе підряд — лише коли є реальна проблема (важке обчислення, ререндер дорогої дитини). Передчасна мемоізація — антипатерн.

### L3 — Senior
**Коли реально потрібні:**
- **`useMemo`** — дороге обчислення (сортування/фільтр великих даних, важкі трансформації), АБО стабільне посилання на об'єкт/масив (щоб не тригерити ефекти/memo дітей).
- **`useCallback`** — стабільна функція для `React.memo`-дитини (питання 16) або як залежність іншого хука (щоб не викликати його щоразу).

**Ключове:** обидва працюють лише в парі з **`React.memo`** (дитина) або залежностями хуків. `useCallback` без memo-дитини **марний** (функція стабільна, але дитина все одно ререндериться з інших причин). 

**Трейдоф:** економія обчислень/ререндерів ↔ overhead (кеш + порівняння deps + складність коду). React Compiler (React 19) обіцяє **авто-мемоізацію** → менше ручного useMemo/useCallback. Senior профілює (React DevTools Profiler) перед мемоізацією — не «про запас».

## 13. `useReducer`

### L1 — Junior
`useReducer` — альтернатива `useState` для **складного стану** (багато полів/переходів). Стан змінюється через **actions** (як Redux).
```jsx
const [state, dispatch] = useReducer(reducer, initialState);
dispatch({ type: "increment" });
```

### L2 — Middle
```jsx
function reducer(state, action) {
  switch (action.type) {
    case "increment": return { count: state.count + 1 };
    case "decrement": return { count: state.count - 1 };
    case "set": return { count: action.payload };
    default: return state;
  }
}

function Counter() {
  const [state, dispatch] = useReducer(reducer, { count: 0 });
  return (
    <>
      <span>{state.count}</span>
      <button onClick={() => dispatch({ type: "increment" })}>+</button>
    </>
  );
}
```

### L3 — Senior
**useReducer vs useState:**
- **useState** — простий стан (примітиви, незалежні поля).
- **useReducer** — **складна логіка переходів** (стан залежить від попереднього складно), багато пов'язаних полів, коли наступний стан = функція(попередній, action).

**Переваги:** логіка переходів **винесена** з компонента (тестований reducer — чиста функція), передбачувані переходи (як конечний автомат), легко логувати actions. **useReducer + Context** (питання 15) — «Redux без Redux» для середнього застосунку. Reducer — чиста функція (state, action) → newState, immutable. Senior обирає useReducer, коли useState стає плутаним (кілька setState, взаємозалежні поля) або для state-machine-логіки.

## 14. Custom hooks (власні хуки) ⭐

### L1 — Junior
**Власний хук** — функція, що починається з `use` і **переиспользовує** stateful-логіку між компонентами.
```jsx
function useToggle(initial = false) {
  const [value, setValue] = useState(initial);
  const toggle = () => setValue(v => !v);
  return [value, toggle];
}
// Використання:
const [isOpen, toggleOpen] = useToggle();
```

### L2 — Middle
```jsx
// Приклад — хук для запиту даних
function useFetch(url) {
  const [data, setData] = useState(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    setLoading(true);
    fetch(url)
      .then(r => r.json())
      .then(setData)
      .catch(setError)
      .finally(() => setLoading(false));
  }, [url]);

  return { data, loading, error };
}
// Використання в будь-якому компоненті:
const { data, loading, error } = useFetch("/api/users");
```

### L3 — Senior
Custom hooks — **головний механізм переиспользування логіки** в сучасному React (замінили HOC і render-props з класової епохи). Вони **композиційні** — хук може викликати інші хуки. Інкапсулюють stateful-логіку (стан + ефекти + обробники) під чистим API.

**Чому кращі за HOC/render-props:** без «wrapper hell» (вкладені обгортки), без колізій props, логіка групується за фічею. Приклади в екосистемі: `useQuery` (React Query), `useForm` (RHF), `useMediaQuery`, `useLocalStorage`, `useDebounce`. **Правило:** ділиш логіку між компонентами → custom hook. Це і абстракція, і тестованість (хук тестується `@testing-library/react-hooks`). Senior виносить повторювану логіку в хуки, тримає компоненти «тонкими» (лише розмітка + виклики хуків).

## 15. Context API (+ useContext) ⭐⭐

### L1 — Junior
**Context API** — спосіб передати дані **багатьом компонентам без «prop drilling»** (протягування props через багато рівнів). Для «глобального» стану: тема, мова, залогінений юзер.

```jsx
const ThemeContext = createContext("light");

function App() {
  return (
    <ThemeContext.Provider value="dark">
      <Toolbar />   {/* усі всередині мають доступ до теми */}
    </ThemeContext.Provider>
  );
}

function Button() {
  const theme = useContext(ThemeContext);   // дістати з контексту, БЕЗ props
  return <button className={theme}>Кнопка</button>;
}
```

### L2 — Middle
**Три частини:**
```jsx
// 1. Створити контекст
const AuthContext = createContext(null);

// 2. Обгорнути Provider'ом (передати value)
function App() {
  const [user, setUser] = useState(null);
  return (
    <AuthContext.Provider value={{ user, setUser }}>
      <Routes />
    </AuthContext.Provider>
  );
}

// 3. Спожити будь-де всередині
function Profile() {
  const { user } = useContext(AuthContext);
  return <div>{user?.name}</div>;
}
```

**Проблема, яку вирішує — prop drilling:**
```
App → Layout → Sidebar → Menu → UserBadge   ← передавати user через ВСІ 5 рівнів (біль)
   Context → UserBadge бере напряму ✅
```

### L3 — Senior
**Коли Context, а коли ні (важливо):**
- **Context добрий** для **низькочастотного** глобального стану: тема, локаль, авторизація, конфіг. Змінюється рідко.
- **Context поганий** для **високочастотних** оновлень: **усі** споживачі ререндеряться при зміні `value` (навіть якщо їм байдуже на змінену частину). Для частого/складного стану → **Zustand/Redux/Jotai** (селективні підписки).

**Gotcha перформансу:** `value={{ user, setUser }}` створює **новий об'єкт** щорендер → усі споживачі ререндеряться. Лікування: `useMemo` для value, розділення контекстів (окремо дані, окремо dispatch), або state-менеджер із селекторами.

| | Context API | Redux/Zustand |
|---|---|---|
| Для чого | низькочастотний глобал (тема/auth) | складний/частий стан |
| Ререндери | усі споживачі | селективні підписки |
| Boilerplate | мало | більше (Redux) / мало (Zustand) |
| Middleware/devtools | немає | є |

**Трейдоф:** Context — вбудований, простий, без залежностей ↔ ререндерить усіх споживачів, немає селекторів/middleware. Не «заміна Redux» — це інструмент для **передачі**, не для **керування** складним станом. Часто: Context + useReducer для середнього; Zustand/Redux — для великого/частого. Senior не пхає весь стан у Context (перформанс), розділяє за частотою оновлень.

## 16. Оптимізація ререндерів (React.memo) ⭐

### L1 — Junior
Компонент ререндериться, коли: **змінився його стан**, **ререндерився батько**, або **змінився контекст**. `React.memo` — пропустити ререндер, якщо **props не змінились**.
```jsx
const Child = React.memo(function Child({ value }) {
  return <div>{value}</div>;   // ререндер лише коли value змінюється
});
```

### L2 — Middle
**Коли компонент ререндериться:**
1. Змінився власний state (useState/useReducer).
2. **Ререндерився батько** (навіть якщо props дитини не змінились!).
3. Змінився спожитий Context.

```jsx
// Батько ререндериться → усі діти теж (за замовчуванням)
// React.memo зупиняє це, якщо props дитини незмінні:
const ExpensiveList = React.memo(({ items }) => { /* важкий рендер */ });

// АЛЕ: нестабільні props ламають memo
<ExpensiveList items={items} onClick={() => {}} />   // ❌ нова функція щорендер → memo марний
<ExpensiveList items={items} onClick={handleClick} /> // ✅ useCallback → стабільна
```

### L3 — Senior
**`React.memo`** — HOC, що робить shallow-порівняння props → пропускає ререндер, якщо не змінились. Працює лише в парі зі **стабільними props** (`useCallback` для функцій, `useMemo` для об'єктів/масивів — питання 12), інакше марний.

**Стратегія оптимізації (за пріоритетом):**
1. **Не оптимізуй передчасно** — профілюй (React DevTools Profiler).
2. **Структура** — підняти дорогі компоненти, щоб зайвий стан не тригерив їх; `children` як props (composition) уникає ререндерів.
3. **React.memo + useCallback/useMemo** — точково для дорогих піддерев.
4. **Віртуалізація** (react-window) — для довгих списків.

**Трейдоф:** мемоізація економить ререндери ↔ overhead (порівняння + пам'ять + складність). Більшість ререндерів **дешеві** (Virtual DOM diff) — оптимізуй лише реально дорогі. **React Compiler (React 19)** авто-мемоізує → зменшить ручну роботу. Senior профілює перед оптимізацією; часто краще рішення — реструктуризація, не memo.

## 17. Умовний рендеринг і списки

### L1 — Junior
```jsx
// Умовний рендер
{isLoggedIn ? <Dashboard /> : <Login />}      // тернарний
{isLoading && <Spinner />}                    // && (показати якщо true)
{error && <Error msg={error} />}

// Списки — map + key
{users.map(u => <UserCard key={u.id} user={u} />)}
```

### L2 — Middle
```jsx
// Кілька способів умовного рендеру
{count > 0 && <Badge count={count} />}         // && — але обережно з 0!
{items.length > 0 ? <List /> : <Empty />}      // тернарний для if/else

// Gotcha з &&:
{count && <Badge />}   // ❌ якщо count === 0 → відрендерить "0" (0 falsy, але рендериться!)
{count > 0 && <Badge />}   // ✅ явна булева умова
```

### L3 — Senior
**Gotcha `&&` з числами** (часта помилка): `{items.length && <List/>}` — якщо `length === 0`, `0` **рендериться** (0 — falsy, але React рендерить числа!) → на екрані «0». Використовуй явну умову (`length > 0 &&`) або тернарний.

**Списки:** `key` обов'язковий і стабільний (питання 6). Для великих списків (тисячі елементів) — **віртуалізація** (react-window/react-virtual): рендерити лише видимі елементи (проти лагів). Умовний рендер: тернарний для if/else, `&&` для «показати якщо», раннє повернення (`if (!data) return <Loading/>`) для читабельності складних умов.

## 18. Lifting state up (підняття стану)

### L1 — Junior
Коли **кілька компонентів ділять** стан — піднімаєш його до **спільного батька**, і передаєш вниз через props.
```jsx
function Parent() {
  const [value, setValue] = useState("");    // стан ТУТ (спільний предок)
  return (
    <>
      <Input value={value} onChange={setValue} />
      <Preview value={value} />               // обидва бачать той самий стан
    </>
  );
}
```

### L2 — Middle
**Проблема:** два сусідні компоненти мають бути синхронізовані (input + preview). Рішення — стан у спільного батька, вниз через props, зміни вгору через callbacks.

```
      Parent (state тут)
      /            \
   Input          Preview
 (onChange↑)      (value↓)
```

### L3 — Senior
**Lifting state up** — базовий патерн React для спільного стану (single source of truth у спільного предка). **Трейдоф:** просто й передбачувано ↔ якщо стан треба **дуже глибоко** або **багатьом розкиданим** компонентам → prop drilling → підняти занадто високо (весь застосунок ререндериться). Тоді → **Context** (питання 15) або state-менеджер.

**Правило балансу:** тримай стан **якнайближче** до тих, хто його використовує (локальний стан — краще); піднімай **лише** до спільного предка тих, хто ділить; для справді глобального — Context/store. «Colocation» (стан поруч із використанням) — сучасний принцип; передчасний глобальний стан — антипатерн (ускладнює, зайві ререндери).

## 19. Vite — що і навіщо? ⭐

### L1 — Junior
**Vite** — сучасний **інструмент збірки** (build tool) для фронтенду: миттєвий запуск dev-сервера й дуже швидке оновлення при збереженні (HMR). Замінює старий Create React App (CRA).
```bash
npm create vite@latest my-app -- --template react-ts
```

### L2 — Middle
| | Vite | CRA / Webpack (старе) |
|---|---|---|
| Dev-старт | миттєвий | повільний (бандлить усе) |
| HMR | миттєвий | повільніший |
| Двигун | esbuild (dev) + Rollup (build) | Webpack |
| Мова | Go (esbuild) — швидко | JS |

**Ключова ідея:** у dev Vite **не бандлить** усе наперед — використовує **нативні ES-модулі** браузера (браузер сам підвантажує імпорти), тому старт миттєвий незалежно від розміру проєкту.

### L3 — Senior
**Чому Vite швидкий (під капотом):**
- **Dev:** нативні ESM — браузер запитує модулі on-demand, Vite трансформує лише те, що потрібно (через **esbuild** — на Go, у ~10-100× швидший за JS-бандлери). Немає «зібрати весь бандл перед стартом».
- **Build (прод):** **Rollup** — оптимізований бандл із tree-shaking, code-splitting.
- **HMR** — точковий (оновлює лише змінений модуль, не перезбирає все).

**Чому витіснив CRA:** CRA (на Webpack) **офіційно застарілий** (React docs більше не рекомендують) — повільний старт/HMR на великих проєктах (бандлить усе наперед). Vite — сучасний стандарт для SPA. Для SSR/full-stack — Next.js (свій тулінг) або Remix. **Трейдоф:** Vite чудовий для SPA/бібліотек ↔ для складного SSR/RSC частіше Next.js. Це частина загального тренду (як ruff/uv у Python, esbuild/SWC замість Babel) — швидкі нативні інструменти.

## 20. Shadcn UI — що це? ⭐

### L1 — Junior
**Shadcn UI** — це набір гарних, готових React-компонентів (кнопки, форми, діалоги), але з **несподіваним підходом**: ти **не встановлюєш** його як бібліотеку — ти **копіюєш код компонентів у свій проєкт** через CLI.

```bash
npx shadcn@latest add button   # копіює код Button у твій проєкт
```

### L2 — Middle
**Головна відмінність від звичайних UI-бібліотек (MUI, Ant Design):**
| | Shadcn UI | MUI / Ant Design |
|---|---|---|
| Встановлення | **копіюєш код** у проєкт | `npm install` (залежність) |
| Володіння кодом | **ти** (код у тебе) | бібліотека (чорна скринька) |
| Кастомізація | повна (це твій код) | обмежена (API/теми) |
| Основа | **Radix UI + Tailwind CSS** | власні стилі |
| Оновлення | вручну | npm update |

**Основа Shadcn:**
- **Radix UI** — headless (без стилів) доступні примітиви (dialog, dropdown) — логіка + accessibility.
- **Tailwind CSS** — стилізація.
- Shadcn склеює їх у готові стилізовані компоненти, код яких **ти копіюєш і володієш**.

### L3 — Senior
**Shadcn — не бібліотека, а «колекція рецептів»** (copy-paste компоненти). Філософія: **«own your components»** — код компонента живе у **твоєму** репозиторії, тому:
- **Повна кастомізація** — це твій код, міняй як хочеш (не борешся з API бібліотеки).
- **Немає runtime-залежності** від UI-бібліотеки (лише Radix + Tailwind).
- **Немає «version lock»** — не залежиш від оновлень мейнтейнера.

**Трейдоф:**
- **+** Повний контроль, кастомізація, accessibility (Radix), сучасний вигляд, немає bundle-роздування від невикористаного.
- **−** Оновлення **вручну** (нема `npm update` для компонентів), код компонентів у твоєму репо (більше коду підтримувати), прив'язка до Tailwind + Radix.

**Коли обирати:** проєкти, де потрібен **контроль над дизайн-системою** й кастомний вигляд (не «типовий MUI-вигляд»), із Tailwind-стеком. **vs MUI/Ant** — ті швидші для «просто працює з коробки», централізоване оновлення, але менше контролю й «важчі». Shadcn — популярний сучасний вибір (особливо з Vite + TS + Tailwind), бо поєднує готовність із володінням кодом. Це зсув парадигми: від «встанови бібліотеку» до «згенеруй код, яким володієш».

## ✅ Чеклист React (інтерв'ю)

**Базово:** що таке React, JSX, функціональні компоненти, props vs state, Virtual DOM, keys.
**Hooks:** правила хуків, useState (functional update, stale closure), useEffect (deps, cleanup), useRef, useMemo/useCallback, useReducer, custom hooks.
**Стан/оптимізація:** Context API (коли/vs Redux, ререндери), React.memo, lifting state up, controlled vs uncontrolled.
**Тулінг:** Vite (чому швидкий, vs CRA), Shadcn UI (copy-paste, own your code, Radix+Tailwind).

## 🔑 Топ-питання
1. **Правила хуків + чому?** → верхній рівень; React відстежує за порядком виклику.
2. **useEffect cleanup + deps?** → відписка/очищення; deps керують коли; exhaustive-deps.
3. **useMemo vs useCallback?** → значення vs функція; лише з React.memo/deps.
4. **Коли Context, коли Redux?** → Context для рідкісного глобалу (тема/auth); Redux/Zustand для частого/складного (ререндери!).
5. **key чому не index?** → стабільна ідентичність; index ламає стан при сортуванні.
6. **Чому Vite > CRA?** → нативні ESM у dev (миттєвий старт), CRA застарілий.
7. **Shadcn — бібліотека?** → ні, copy-paste компоненти (own your code), Radix+Tailwind.

> ⚠️ React швидко розвивається: React 19 (React Compiler авто-мемоізація, Server Components, `use` hook, Actions). Звіряй з актуальними docs (react.dev). Багато «оптимізаційних» хуків (useMemo/useCallback) стануть менш потрібними з Compiler.

# Блок 1. Node.js — Junior

> Формат — **Принцип Сходів**: **L1 Junior** (аналогія + «коли») → **L2 Middle** (продакшн, best practices, gotchas) → **L3 Senior** (під капотом V8/libuv, трейдофи).
> Сучасний стек: Node.js 20+, ES2022+, TypeScript 5+.

---

## 1. Що таке Node.js?

### L1 — Junior
**Аналогія — двигун із машини, поставлений на човен.** JavaScript народився в браузері. **Node.js** дістав двигун JS (V8 від Google Chrome) із браузера й поставив його на сервер — тепер тим самим JS можна писати бекенд, CLI, скрипти.

**Точне визначення:** Node.js — це **середовище виконання (runtime)** для JavaScript **поза браузером**, побудоване на рушії **V8**, з **подієво-орієнтованою** (event-driven) архітектурою та **неблокуючим I/O**.

```javascript
// hello.js — запуск: node hello.js
console.log("Привіт із сервера, без браузера!");
```

### L2 — Middle
Node.js — **не мова й не фреймворк**, а runtime. Що він додає до V8:
- **Доступ до ОС** — файлова система (`fs`), мережа (`net`, `http`), процеси (`child_process`).
- **Модульну систему** — CommonJS (`require`) та ES Modules (`import`).
- **Глобали**, яких немає в браузері: `process`, `Buffer`, `global`, `__dirname`.
- **Подієвий цикл (event loop)** через бібліотеку **libuv** (питання 9, 13).

Типове застосування: REST/GraphQL API, real-time (WebSocket, чати), мікросервіси, CLI-інструменти, SSR (Next.js), build-тулінг.

**Gotcha:** Node.js ≠ JavaScript ≠ браузерний JS. У Node немає `window`, `document`, `DOM`; у браузері немає `fs`, `process`. Спільне ядро — мова ECMAScript + V8.

### L3 — Senior
Архітектурно Node.js — це **«клей» між трьома світами**:
1. **V8** (C++) — компілює й виконує JS (питання 9).
2. **libuv** (C) — event loop + асинхронний I/O + thread pool (питання 9, 13).
3. **Core modules** (JS + C++ bindings) — API над системними викликами.

**Чому це проривно:** традиційні сервери (Apache/PHP) виділяли **потік на кожен запит** (thread-per-request) → 10k з'єднань = 10k потоків = величезний overhead пам'яті й перемикань контексту (проблема C10k). Node.js обрав **один потік + event loop + неблокуючий I/O** → тисячі з'єднань на одному потоці майже без overhead (питання 5).

**Трейдоф:** це геніально для **I/O-bound** навантажень (більшість веб-серверів — чекають БД/мережу), але смертельно для **CPU-bound** (питання 3) — важке обчислення блокує єдиний потік і вішає всі з'єднання.

> ⚠️ Конкретні версії/API Node швидко еволюціонують (ESM за замовчуванням, вбудований fetch, test runner). Звіряй з docs цільової LTS-версії.

## 2. Переваги та недоліки Node.js

### L1 — Junior
**Аналогія — універсальний майстер.** Перевага: один інструмент (JS) і для фронту, і для бекенду — не треба перемикатися між мовами. Недолік: цей майстер чудовий у бігах між справами (I/O), але поганий у важкому фізичному навантаженні (обчислення).

### L2 — Middle
**Переваги:**
| Перевага | Чому |
|---|---|
| Швидкий I/O | неблокуюча модель, тисячі з'єднань (питання 5) |
| Одна мова full-stack | JS на фронті й бекенді — спільний код, типи, валідація |
| Величезна екосистема | **npm** — найбільший реєстр пакетів у світі |
| Real-time | природний для WebSocket, SSE, чатів, ігор |
| JSON-native | JS ↔ JSON без серіалізації-перекладача |
| Швидкий старт/прототипування | мінімум бойлерплейту |

**Недоліки:**
| Недолік | Лікування |
|---|---|
| Поганий для CPU-bound | worker_threads / винести в інший сервіс (питання 3, 6) |
| Callback hell (історично) | `async/await`, Promises |
| Динамічна типізація → баги | **TypeScript** |
| Один потік → один баг вішає все | кластеризація, обробка помилок |
| npm: залежності/безпека | аудит, lock-файли, мінімізація залежностей |

**Gotcha:** «Node швидкий» — точніше «Node швидкий для **I/O**». На бенчмарках CPU він програє Go/Rust/Java. Сила — у конкурентності I/O, не в сирій обчислювальній потужності.

### L3 — Senior
**Глибший трейдоф — single-threaded event loop:**
- **+** Немає блокувань/гонок даних на спільному стані (немає race conditions у класичному сенсі, бо JS-код виконується по одному), простіша ментальна модель, дешева конкурентність.
- **−** Один потік = одна точка відмови для CPU; необроблений `throw` у callback може покласти процес; масштабування **горизонтальне** (більше процесів), не вертикальне на ядра без `cluster`/`worker_threads`.

**npm — палиця з двома кінцями:** швидкість розробки ↔ **supply-chain ризик** (left-pad, event-stream інциденти), глибокі дерева залежностей, вага `node_modules`. Senior мінімізує залежності, використовує lock-файли, аудит (`npm audit`), і знає, що кожна залежність — потенційний вектор атаки.

**Зрілість vs хаос:** екосистема рухається швидко (нові фреймворки щомісяця) — це гнучкість ↔ нестабільність, fatigue, часті breaking changes. Senior обирає зрілі рішення з урахуванням bus factor, а не «наймодніше».

## 3. Для яких завдань Node.js не підходить?

### L1 — Junior
**Аналогія — спринтер на марафоні з гирями.** Node чудовий у швидких перемиканнях між легкими справами (обслужити запит, сходити в БД). Але якщо дати йому **одну важку обчислювальну справу**, він застрягне на ній і **перестане обслуговувати всіх інших** — бо потік **один**.

**Не підходить для CPU-важких задач.**

### L2 — Middle
**Погані кейси для Node:**
- **Важкі обчислення:** обробка відео/зображень, ML-тренування, складна криптографія, наукові розрахунки, парсинг гігантських даних у пам'яті.
- **Довгі синхронні операції** будь-якого роду — вони блокують event loop.

```javascript
// ❌ Блокує ВЕСЬ сервер на час обчислення — усі інші запити чекають
app.get("/fib", (req, res) => {
  res.json({ result: fibonacci(45) }); // важкий CPU → event loop стоїть
});
```

**Що робити натомість:**
- Винести CPU-роботу у **worker_threads** (питання 6).
- Окремий сервіс мовою під обчислення (Go, Rust, Python+NumPy).
- Черга задач (BullMQ) + воркери.

**Gotcha:** навіть «невинний» синхронний код шкодить: `JSON.parse` величезного об'єкта, `crypto.pbkdf2Sync`, `fs.readFileSync` великого файлу, регулярка з катастрофічним бектрекінгом (**ReDoS**) — усе це блокує цикл.

### L3 — Senior
Корінь проблеми — **єдиний event loop thread виконує весь твій JavaScript**. Поки твоя функція рахує, libuv не може запустити жоден інший callback → латентність усіх запитів зростає, з'єднання таймаутяться.

**Метрика для діагностики:** **event loop lag/delay** (`perf_hooks.monitorEventLoopDelay`). Якщо затримка циклу росте — десь блокуюча CPU-операція. APM-інструменти (Clinic.js, `--prof`) це ловлять.

**Архітектурний висновок Senior:** Node — для **I/O-orchestration** (координація БД, кешів, зовнішніх API, real-time). CPU-важке — **винось**: worker_threads (в межах процесу), окремий мікросервіс (мова під задачу), або offload у спеціалізовані системи (БД рахує агрегації, GPU-сервіс для ML). Правильна архітектура — «Node як диригент, важка робота — в інших виконавців».

## 4. Головні компоненти Node.js

### L1 — Junior
**Аналогія — автомобіль із кількох вузлів.** Node складається з кількох головних «вузлів», що працюють разом:
- **V8** — двигун (виконує JS).
- **libuv** — коробка передач + привід (асинхронний I/O, event loop).
- **Core-модулі** — приладова панель (готові API: `fs`, `http`, `crypto`).

### L2 — Middle
**Головні компоненти:**
| Компонент | Мова | Роль |
|---|---|---|
| **V8** | C++ | компіляція й виконання JS, керування пам'яттю/GC |
| **libuv** | C | event loop, async I/O, thread pool, крос-платформність |
| **Core modules** | JS + C++ | вбудовані API (`fs`, `http`, `stream`, `events`...) |
| **C++ bindings** | C++ | міст між JS і системними бібліотеками |
| **npm** | — | менеджер пакетів і реєстр |

Плюс ключові вбудовані абстракції: **EventEmitter** (подієва модель), **Streams** (потокові дані, питання 11), **Buffer** (бінарні дані), **process** (поточний процес).

**Gotcha:** `process`, `Buffer`, `global`, `__dirname`, `require` — це **глобали Node**, не частина мови JavaScript. У браузері їх немає.

### L3 — Senior
Як це збирається докупи (потік виконання):
1. Твій JS код → **V8** парсить і компілює в машинний код (питання 7).
2. Коли JS викликає async I/O (`fs.readFile`) → виклик іде через **C++ binding** до **libuv**.
3. **libuv** виконує операцію неблокуюче (через ОС-механізми: epoll/kqueue/IOCP) або через свій **thread pool** (для fs, dns, crypto).
4. Коли операція готова → libuv кладе callback у відповідну чергу **event loop** (питання 13).
5. Event loop повертає виконання у V8 → запускає твій callback.

**Трейдоф цієї архітектури:** розділення «виконання JS» (V8, один потік) і «I/O» (libuv, ОС + thread pool) дає неблокуючу конкурентність без складності багатопоточного коду. Ціна — потрібно мислити асинхронно (callbacks/promises) і не блокувати єдиний JS-потік.

> ⚠️ Список «головних компонентів» у різних джерелах трохи різниться (хтось додає llhttp, c-ares, OpenSSL). Суть — V8 + libuv + core API. Звіряй з офіційною архітектурною документацією Node.

## 5. Як один потік обробляє багато паралельних запитів?

### L1 — Junior
**Аналогія — один офіціант на 20 столиків.** Він не стоїть біля столу, чекаючи, поки гості оберуть страву. Прийняв замовлення → відніс на кухню → пішов до іншого столу → повернувся, коли страва готова. **Один** працівник обслуговує багатьох, бо **не простоює в очікуванні**.

Node так само: замість «чекати відповідь БД» він **передає запит і йде обслуговувати інших**, а коли БД відповіла — обробляє результат.

### L2 — Middle
Механізм: **event loop + неблокуючий асинхронний I/O**.
```javascript
// 1000 запитів приходять «одночасно»
app.get("/user/:id", async (req, res) => {
  const user = await db.query("SELECT * FROM users WHERE id = $1", [req.params.id]);
  res.json(user); // поки чекаємо БД — потік обслуговує ІНШІ запити
});
```
Ключове:
1. Запит на I/O (БД, файл, мережа) **не блокує** — Node делегує його libuv/ОС.
2. Потік **вільний** обробляти наступні запити.
3. Коли I/O завершився — callback стає в чергу, event loop його викликає.

**Gotcha:** це працює, **лише поки код неблокуючий**. Один синхронний/CPU-важкий виклик (питання 3) ламає всю модель — поки він рахує, **жоден** інший запит не обслуговується.

### L3 — Senior
**Конкурентність ≠ паралелізм.** Node дає **конкурентність** (багато задач прогресують, чергуючись на одному потоці), а **не паралелізм** (одночасне виконання на багатьох ядрах). Це **cooperative** модель: задачі «віддають» керування на точках `await`/завершення I/O, а не витісняються примусово.

**Як насправді виконується I/O:**
- **Мережа/сокети** — неблокуюче через ОС (epoll на Linux, kqueue на macOS, IOCP на Windows) — libuv питає ОС «що готове?» без потоків.
- **Файли, DNS, crypto** — через **thread pool libuv** (за замовчуванням 4 потоки, `UV_THREADPOOL_SIZE`), бо файловий I/O не має надійного неблокуючого ОС-API.

**Трейдоф vs thread-per-request (Apache/Java):** Node економить пам'ять і перемикання контексту (один стек замість тисяч), масштабується на десятки тисяч з'єднань. Ціна — **кооперативність**: один «жадібний» (блокуючий) callback вішає всіх; немає ізоляції збоїв між запитами в одному потоці. Для багатоядерності потрібен `cluster`/`worker_threads` (питання 6).

## 6. Чи можна використовувати кілька потоків? Якими модулями?

### L1 — Junior
**Так.** Хоч JS-код виконується в одному потоці, Node дає способи задіяти кілька потоків/процесів для важкої роботи.

Головний модуль — **`worker_threads`** (справжні потоки для CPU-задач).

### L2 — Middle
**Три механізми:**
| Модуль | Що дає | Коли |
|---|---|---|
| **`worker_threads`** | справжні потоки в одному процесі, спільна пам'ять (SharedArrayBuffer) | CPU-важкі задачі |
| **`child_process`** | окремі процеси (`fork`, `spawn`, `exec`) | запуск підпроцесів, ізоляція |
| **`cluster`** | кілька процесів Node на одному порту (по ядрах) | масштабування веб-сервера |

```javascript
import { Worker } from "node:worker_threads";

// Виносимо важке обчислення в окремий потік — event loop не блокується
const worker = new Worker("./heavy-task.js", { workerData: { n: 45 } });
worker.on("message", (result) => console.log("Готово:", result));
```

Плюс **libuv thread pool** (4 потоки за замовчуванням) — він **автоматично** обробляє async fs, dns, crypto. Ти ним не керуєш напряму, але він є.

**Gotcha:** `cluster` створює **процеси** (не потоки) — у кожного свій event loop і пам'ять, спілкуються через IPC. `worker_threads` — справжні **потоки** з можливістю ділити пам'ять через `SharedArrayBuffer`.

### L3 — Senior
**worker_threads vs child_process vs cluster — трейдофи:**
- **worker_threads:** легші за процеси, можуть ділити пам'ять (`SharedArrayBuffer` — без копіювання), передача даних через message passing (структуроване клонування) або transfer (zero-copy для ArrayBuffer). Для **CPU-паралелізму** в межах сервісу.
- **child_process:** повна ізоляція (окрема пам'ять, окремий V8), дорожчий старт, спілкування через IPC/serialization. Для запуску зовнішніх програм чи ізольованих задач.
- **cluster:** надбудова над child_process — форкає N воркерів (зазвичай = кількість ядер), майстер балансує з'єднання між ними (SO_REUSEPORT / round-robin). Для **горизонтального масштабування** одного сервера на всі ядра.

**libuv thread pool (важливий нюанс):** async-операції fs/crypto/dns виконуються в пулі з **4 потоків**. Якщо запустити 5 важких `crypto.pbkdf2` одночасно — п'ятий **чекає** звільнення потока (черга пулу). `UV_THREADPOOL_SIZE` (до 1024) це налаштовує. Це частий «прихований» bottleneck.

**Архітектурний висновок:** для CPU-bound — `worker_threads` (паралелізм у процесі) або окремий сервіс; для використання всіх ядер веб-сервером — `cluster` (або PM2/контейнери з N інстансами за балансувальником). Node **може** в багатопоточність, просто це не дефолт і вимагає явних дій.

> ⚠️ Деталі `SharedArrayBuffer`, transferable objects і поведінки thread pool — звіряй з docs `worker_threads`/libuv для своєї версії Node.

## 7. Node.js інтерпретує чи компілює код?

### L1 — Junior
**Аналогія — перекладач, що вчиться на льоту.** Спершу він перекладає речення «нашвидкуруч» (інтерпретація), але помічає фрази, які повторюються часто, і робить для них **готовий якісний переклад** (компіляція). Так працює V8: **і інтерпретує, і компілює**.

### L2 — Middle
Точна відповідь: **і те, і те** — це **JIT (Just-In-Time) компіляція**. V8 робить так:
1. JS → парсинг → **байт-код**.
2. Байт-код виконує **інтерпретатор (Ignition)** — швидкий старт.
3. V8 стежить, який код «гарячий» (виконується часто).
4. Гарячий код → **оптимізуючий компілятор (TurboFan)** перетворює в **оптимізований машинний код**.

```javascript
// Функцію в гарячому циклі V8 згодом скомпілює в нативний код
function sum(arr) {
  let total = 0;
  for (const x of arr) total += x; // "гаряча" → TurboFan оптимізує
  return total;
}
```

**Gotcha:** назвати Node «чисто інтерпретованим» — помилка на інтерв'ю. Коректно: «V8 використовує JIT — інтерпретатор Ignition + оптимізуючий компілятор TurboFan».

### L3 — Senior
**Конвеєр V8 (pipeline):**
```
JS-код → Parser → AST → Ignition (байт-код, інтерпретація)
                              ↓ (профілювання «гарячого» коду)
                          TurboFan → оптимізований машинний код
                              ↑ (deoptimization при зміні припущень)
```
- **Ignition** дає швидкий запуск (не треба чекати компіляції всього).
- **TurboFan** оптимізує на основі **припущень про типи** (speculative optimization): «ця функція завжди отримує числа» → генерує спеціалізований код.
- **Deoptimization:** якщо припущення порушується (раптом прийшов рядок замість числа) → V8 **відкочує** на байт-код. Тому **монолітні типи** в гарячих функціях критичні для перформансу.

**Практичний наслідок (senior-знання):** «hidden classes» та inline caches — V8 будує приховані класи об'єктів за їх формою. Об'єкти однакової форми (ті самі поля в тому ж порядку) → один hidden class → швидкий доступ. Зміна форми об'єкта «на льоту» (додавання полів динамічно, `delete`) ламає оптимізацію. Тому передбачувана структура об'єктів = швидкий код.

**Трейдоф JIT:** старт швидший за AOT-компіляцію (не чекаємо компіляції всього наперед) + адаптивна оптимізація під реальні дані ↔ overhead профілювання, прогрів (перші виклики повільніші), непередбачувані deopt-паузи.

> ⚠️ Назви фаз V8 (Ignition/TurboFan/Maglev/Sparkplug) і деталі pipeline змінюються між версіями V8. Звіряй з блогом V8 перед дослівним цитуванням.

## 8. Як читати великі файли в Node.js?

### L1 — Junior
**Аналогія — пити з пляшки, а не залпом відро.** Не намагайся завантажити весь гігантський файл у пам'ять одразу (захлинешся = out of memory). Читай **частинами (chunks)** через **стрими** — як ковтки.

```javascript
import { createReadStream } from "node:fs";

// ✅ Стрим — читаємо шматками, пам'ять O(1)
const stream = createReadStream("huge.log", { encoding: "utf8" });
stream.on("data", (chunk) => process(chunk)); // по шматку
stream.on("end", () => console.log("Готово"));
```

### L2 — Middle
**Неправильно vs правильно:**
```javascript
import { readFileSync, readFile } from "node:fs";

// ❌ Завантажує ВЕСЬ файл у пам'ять — OOM на великих файлах + блокує цикл
const data = readFileSync("huge.log", "utf8");

// ❌ Async, але все одно весь файл у пам'ять
readFile("huge.log", "utf8", (err, data) => { /* увесь файл */ });

// ✅ Стрим — постійна пам'ять незалежно від розміру файлу
import { createReadStream, createWriteStream } from "node:fs";
import { pipeline } from "node:stream/promises";

await pipeline(
  createReadStream("huge.log"),
  transformStream,            // обробка по шматках
  createWriteStream("out.log")
);
```

**Best practices:**
- `createReadStream` замість `readFileSync`/`readFile` для великих файлів.
- `pipeline()` (з `stream/promises`) замість ручного `.pipe()` — коректно обробляє помилки й cleanup.
- Для построкової обробки — `readline` поверх стриму.

**Gotcha:** `readFileSync` робить дві біди одночасно: (1) вантажить усе в RAM (OOM), (2) **блокує event loop** (синхронний) — вішає сервер.

### L3 — Senior
Ключова концепція — **backpressure (зворотний тиск)**. Якщо читаєш швидше, ніж записуєш/обробляєш — дані накопичуються в пам'яті буфера → OOM. `pipeline()`/`.pipe()` **автоматично** керують backpressure: коли споживач не встигає, стрим-джерело **призупиняється** (`pause()`), коли встигає — відновлюється (`resume()`).

```javascript
import { pipeline } from "node:stream/promises";
import { createReadStream, createWriteStream } from "node:fs";
import { createGzip } from "node:zlib";

// Стиснути гігантський файл, тримаючи пам'ять під контролем:
await pipeline(
  createReadStream("huge.log"),
  createGzip(),                 // backpressure поширюється по всьому ланцюгу
  createWriteStream("huge.log.gz")
);
```

**Трейдоф стримів:** постійна пам'ять (`O(1)`) і обробка даних, більших за RAM ↔ складніша модель (події, backpressure, обробка помилок на кожному етапі). Це той самий компроміс, що генератори vs списки в Python: лінива потокова обробка проти простоти «все в пам'яті».

`highWaterMark` — налаштування розміру буфера стриму (за замовчуванням 64KB для файлів, 16KB для object mode) — балансує пам'ять vs кількість операцій.

## 9. Що таке libuv і V8? Їхнє призначення?

### L1 — Junior
**Аналогія — двигун і коробка передач.**
- **V8** — це **двигун**: бере JS-код і виконує його (перетворює на машинні інструкції).
- **libuv** — це **коробка передач + привід**: керує асинхронним I/O і **event loop** — тим, як Node чекає на файли/мережу, не блокуючись.

V8 = «виконує JS», libuv = «робить I/O неблокуючим».

### L2 — Middle
| | **V8** | **libuv** |
|---|---|---|
| Хто створив | Google (для Chrome) | спільнота Node |
| Мова | C++ | C |
| Роль | компіляція/виконання JS, пам'ять, GC | event loop, async I/O, thread pool |
| Дає | сам JavaScript | конкурентність і крос-платформність |

**V8** робить: JIT-компіляцію (питання 7), керування пам'яттю (heap, garbage collection), реалізацію самої мови ECMAScript.

**libuv** робить: event loop (питання 13), асинхронні файлові/мережеві операції, thread pool (4 потоки), таймери, абстракцію над ОС (epoll/kqueue/IOCP — щоб працювало однаково на Linux/macOS/Windows).

**Gotcha:** V8 **не вміє** в I/O чи event loop — це чистий JS-рушій (той самий V8 у Chrome, але там event loop дає браузер, а в Node — libuv). Async-можливості Node — заслуга **libuv**, не V8.

### L3 — Senior
**Розподіл обов'язків (ключовий senior-інсайт):**
- **V8** — синхронний рушій: дай йому JS — він виконає й поверне результат. Сам по собі **нічого не знає про асинхронність, файли, мережу**. Він надає лише JS + механізм мікрозадач (Promise microtask queue — частина ECMAScript).
- **libuv** — постачає **асинхронність**: коли JS викликає `fs.readFile`, V8 не блокується — виклик іде через C++ binding у libuv, який виконує I/O неблокуюче й повертає callback у event loop.

**Як вони взаємодіють:**
```
V8 виконує JS → зустрічає async I/O → C++ binding → libuv
libuv (epoll/thread pool) робить I/O → готово → callback у чергу event loop
event loop → віддає callback назад у V8 → V8 виконує callback
```

**Тонкощі черг (зв'язок із питанням 10):**
- **Microtask queue** (Promises, `queueMicrotask`) — належить **V8** (це ECMAScript).
- **`process.nextTick` queue** — належить **Node** (libuv-рівень координації).
- **Macrotask phases** (timers, I/O, check) — належать **libuv** event loop.
Розуміння, **хто чим володіє**, пояснює порядок виконання (питання 10, 13).

> ⚠️ Точний поділ обов'язків (особливо хто володіє microtask queue) — деталь реалізації; концептуально V8 = JS-рушій + microtasks, libuv = event loop + I/O. Звіряй з docs.

## 10. Microtasks vs Macrotasks

### L1 — Junior
**Аналогія — термінові й звичайні справи.**
- **Microtasks** — «термінові дрібниці»: виконуються **негайно після поточної справи**, перш ніж узятися за щось велике. Приклад: `.then()` у Promise.
- **Macrotasks** — «великі заплановані справи»: одна за ітерацію циклу. Приклад: `setTimeout`.

**Правило:** після кожної macrotask Node **спорожнює всю чергу microtasks**, перш ніж брати наступну macrotask.

### L2 — Middle
| | Microtasks | Macrotasks |
|---|---|---|
| Приклади | `Promise.then/catch/finally`, `queueMicrotask`, `await` | `setTimeout`, `setInterval`, `setImmediate`, I/O callbacks |
| Коли | **після кожної** операції, до наступної macrotask | по одній за фазу event loop |
| Пріоритет | вищий | нижчий |

```javascript
console.log("1: синхронно");
setTimeout(() => console.log("4: setTimeout (macrotask)"), 0);
Promise.resolve().then(() => console.log("3: promise (microtask)"));
console.log("2: синхронно");

// Порядок: 1 → 2 → 3 → 4
// Спочатку весь синхронний код, потім ВСІ microtasks, потім macrotask
```

**Gotcha — `process.nextTick`:** у Node `process.nextTick` має **окрему чергу з ще вищим пріоритетом**, ніж Promise-microtasks. Порядок: синхронний код → `process.nextTick` queue → Promise microtasks → наступна macrotask.

### L3 — Senior
**Точний порядок спорожнення черг (Node):**
```
1. Виконати поточну синхронну операцію (один callback macrotask)
2. Спорожнити ВСЮ process.nextTick queue
3. Спорожнити ВСЮ Promise microtask queue
4. (повторити 2-3, якщо nextTick додав ще microtasks)
5. Перейти до наступної фази/macrotask event loop
```

```javascript
setTimeout(() => console.log("timeout"), 0);
setImmediate(() => console.log("immediate"));
Promise.resolve().then(() => console.log("promise"));
process.nextTick(() => console.log("nextTick"));
console.log("sync");
// sync → nextTick → promise → (timeout/immediate залежить, питання 13)
```

**Небезпека — голодування (starvation):** microtasks спорожнюються **повністю** перед наступною macrotask. Якщо microtask рекурсивно додає нові microtasks (нескінченний `.then` ланцюг або рекурсивний `process.nextTick`) → event loop **ніколи** не дійде до I/O/таймерів → сервер «зависає», хоча CPU зайнятий. Тому `process.nextTick` для важкої роботи — антипатерн.

**Чому це питають:** розуміння черг пояснює неочевидні баги порядку виконання, чому `await` «пропускає» наступний синхронний код, і чому `setTimeout(0)` не виконується «негайно» (стає в чергу macrotask **після** всіх microtasks). Це маркер, що кандидат розуміє event loop, а не зазубрив.

> ⚠️ Браузер і Node мають відмінності (`process.nextTick`/`setImmediate` — лише Node). Точний порядок між timers/immediate залежить від контексту (питання 13). Звіряй із docs event loop Node.

## 11. Що таке стрим (stream)?

### L1 — Junior
**Аналогія — конвеєр замість вантажівки.** Замість того, щоб привезти весь вантаж однією вантажівкою (завантажити файл у пам'ять цілком), стрим подає дані **по шматках на конвеєрі** — обробляєш кожен шматок і відпускаєш. Не треба тримати все одразу.

**Stream** — абстракція для **потокової обробки даних шматками (chunks)**, а не цілком.

```javascript
import { createReadStream } from "node:fs";
const stream = createReadStream("file.txt");
stream.on("data", (chunk) => console.log("шматок:", chunk.length));
```

### L2 — Middle
Стрими — **подієві** (EventEmitter): `data`, `end`, `error`, `finish`. Ключові переваги:
- **Пам'ять O(1)** — не залежить від розміру даних (питання 8).
- **Час до першого байта** — починаєш обробляти, не чекаючи всіх даних.
- **Композиція** — з'єднуються в конвеєр через `pipe`/`pipeline`.

```javascript
import { pipeline } from "node:stream/promises";
import { createReadStream, createWriteStream } from "node:fs";
import { createGzip } from "node:zlib";

// Конвеєр: читати → стиснути → записати, усе шматками
await pipeline(
  createReadStream("input.txt"),
  createGzip(),
  createWriteStream("output.txt.gz")
);
```

Де всюди стрими в Node: HTTP `req`/`res`, файли, `process.stdin/stdout`, сокети, crypto, zlib.

**Gotcha:** `request`/`response` в HTTP — це **стрими**. Тому `res.write()` шматками й `req.on("data")` — нативна потокова поведінка, а не «магія».

### L3 — Senior
Стрими розв'язують фундаментальну проблему — **обробка даних, більших за пам'ять**, і **backpressure** (питання 8). Без стримів сервер, що віддає 4GB-файл 100 клієнтам, тримав би 400GB у RAM; зі стримами — лише буфери по 64KB на з'єднання.

**Backpressure** — серце стримів: швидкий producer + повільний consumer → буфер росте → OOM. Стрими сигналізують про переповнення (`write()` повертає `false`, подія `drain`), і `pipe`/`pipeline` автоматично призупиняють джерело. Це той самий механізм, що в чергах розподілених систем (поглинання сплесків).

**Трейдоф:** стрими дають масштабованість по пам'яті й латентності ↔ складність (події замість лінійного коду, ручна обробка помилок на кожному етапі, легко «втратити» помилку в ланцюгу `.pipe()` — тому `pipeline()`). Концептуально це Node-аналог Python-генераторів: ліниве потокове проти eager «все в пам'яті».

## 12. Які види стримів?

### L1 — Junior
**Чотири види — за напрямком руху даних:**
- **Readable** — звідки **читаємо** (джерело): файл, HTTP-запит.
- **Writable** — куди **пишемо** (призначення): файл, HTTP-відповідь.
- **Duplex** — і читаємо, і пишемо (двосторонній): TCP-сокет.
- **Transform** — читаємо, **перетворюємо**, пишемо: gzip, шифрування.

### L2 — Middle
| Тип | Напрямок | Приклади |
|---|---|---|
| **Readable** | джерело даних | `fs.createReadStream`, `req`, `process.stdin` |
| **Writable** | приймач даних | `fs.createWriteStream`, `res`, `process.stdout` |
| **Duplex** | обидва (незалежно) | TCP-сокет (`net.Socket`) |
| **Transform** | вхід → перетворення → вихід | `zlib.createGzip`, `crypto`-cipher |

```javascript
import { Transform } from "node:stream";

// Власний Transform — переводить текст у верхній регістр шматками
const upper = new Transform({
  transform(chunk, encoding, callback) {
    callback(null, chunk.toString().toUpperCase());
  },
});
process.stdin.pipe(upper).pipe(process.stdout);
```

**Gotcha:** **Transform — це підвид Duplex.** Різниця: у Duplex читання й запис **незалежні** (як сокет: читаєш одне, пишеш інше); у Transform вихід є **функцією від входу** (перетворення того, що зайшло).

### L3 — Senior
**Ієрархія:** `Stream` (базовий EventEmitter) → `Readable`, `Writable` → `Duplex` (обидва) → `Transform` (Duplex, де output = f(input)) → `PassThrough` (Transform без змін, для дебагу/моніторингу).

**Два режими Readable (важливий нюанс):**
- **Flowing mode** — дані «течуть» автоматично через події `data` (push).
- **Paused mode** — споживач явно тягне через `.read()` (pull).
`.pipe()` і `for await...of` керують цим автоматично; ручне змішування режимів — джерело багів (втрачені chunks).

**Object mode:** за замовчуванням стрими працюють із `Buffer`/string, але `objectMode: true` дозволяє стримити **будь-які JS-об'єкти** — основа потокових пайплайнів обробки даних (ETL, побудова конвеєрів трансформацій).

**Трейдоф вибору:** Transform — найпотужніший для конвеєрів обробки (читай → трансформуй → пиши з backpressure через весь ланцюг), але вимагає коректної реалізації `_transform`/`_flush` і обробки помилок. Сучасний підхід — `stream/promises` + `for await...of` (async iterators) робить стрими лінійнішими й безпечнішими за класичні події.

## 13. Що таке event loop? З яких компонентів і як працює?

### L1 — Junior
**Аналогія — офіціант, що ходить по колу.** Event loop — це **цикл**, який постійно обходить «станції»: «є готові таймери? є готовий I/O? є відкладені справи?» — і виконує те, що готове. Він **ніколи не стоїть** і не блокується на одній справі (поки твій код неблокуючий).

Це **серце** неблокуючої моделі Node (питання 5).

### L2 — Middle
Event loop (від libuv) має **фази**, які проходяться по колу:
```
   ┌───────────────────────────┐
┌─>│         timers            │  setTimeout, setInterval
│  ├───────────────────────────┤
│  │     pending callbacks     │  відкладені системні callbacks
│  ├───────────────────────────┤
│  │       idle, prepare       │  внутрішнє
│  ├───────────────────────────┤
│  │          poll             │  ← новий I/O, чекає тут на події
│  ├───────────────────────────┤
│  │          check            │  setImmediate
│  ├───────────────────────────┤
│  │     close callbacks       │  socket.on('close')
│  └───────────────────────────┘
```
**Між фазами** (після кожного callback) спорожнюються `process.nextTick` + microtask черги (питання 10).

**Gotcha — `setTimeout(fn, 0)` vs `setImmediate(fn)`:** `setImmediate` спрацьовує у фазі **check** (після I/O), `setTimeout(0)` — у фазі **timers**. Усередині I/O-callback `setImmediate` **завжди** перший; у головному модулі їх порядок недетермінований (залежить від таймінгу старту).

### L3 — Senior
**Повний цикл однієї ітерації (tick):**
1. Виконати callbacks поточної фази (напр. усі готові timers).
2. Після **кожного** callback — спорожнити `process.nextTick` queue, потім microtask (Promise) queue.
3. Перейти до наступної фази.
4. У фазі **poll** — якщо немає роботи, event loop **блокується тут**, чекаючи нові I/O-події від ОС (epoll/kqueue) з таймаутом до найближчого таймера.

**Ключові тонкощі (senior-рівень):**
- **poll** — найважливіша фаза: тут Node «спить», чекаючи I/O, і прокидається на події. Це і є «неблокуюче очікування» — потік не зайнятий, ОС повідомить.
- **microtasks мають пріоритет над усіма фазами** — спорожнюються між кожним callback (питання 10). Тому Promise завжди «обганяє» setTimeout.
- **Голодування циклу:** важкий синхронний код у будь-якому callback блокує перехід до наступної фази → I/O не обробляється → латентність росте (питання 3). Метрика — event loop delay.

**Трейдоф архітектури:** event loop дає масову конкурентність I/O на одному потоці без замків/гонок ↔ вимагає, щоб **увесь** код був неблокуючим і кооперативним; будь-яка важка синхронна операція — глобальна проблема. Це фундаментальний компроміс Node: простота конкурентності ціною дисципліни «не блокуй цикл».

> ⚠️ Деталі фаз і поведінки timer vs immediate — специфіка libuv, мають нюанси між ОС і версіями. Звіряй з офіційним гайдом «The Node.js Event Loop».

## 14. Що таке логування і моніторинг?

### L1 — Junior
**Аналогія — щоденник і медогляд.**
- **Логування (logging)** — «щоденник» застосунку: записуємо, що сталося (запит прийшов, помилка виникла), щоб потім розслідувати.
- **Моніторинг (monitoring)** — «медогляд у реальному часі»: стежимо за «здоров'ям» системи (скільки запитів, латентність, помилки, пам'ять) і б'ємо на сполох при проблемах.

### L2 — Middle
**Логування в Node:**
```javascript
import pino from "pino"; // швидкий структурований логер
const log = pino();

log.info({ userId: 42, route: "/login" }, "користувач увійшов");
log.error({ err }, "помилка БД");
// Структуровані (JSON) логи → легко шукати/фільтрувати
```
- Бібліотеки: **pino** (швидкий), **winston** (гнучкий). Не `console.log` у проді.
- **Рівні:** trace/debug/info/warn/error/fatal.
- **Структуровані логи** (JSON), не рядки — щоб шукати/агрегувати (ELK, Loki).

**Моніторинг:**
- **Метрики:** RPS, латентність (p50/p95/**p99**), помилки, CPU/RAM, **event loop delay** (специфічно для Node!).
- Інструменти: Prometheus + Grafana, APM (Datadog, New Relic), Sentry (помилки).

**Gotcha:** **не логуй секрети** (паролі, токени, картки) — витік + порушення (GDPR). І `console.log` синхронний на TTY → у проді може блокувати; використовуй асинхронні логери (pino).

### L3 — Senior
*(Перетинається з підходом до observability — три стовпи: Logs, Metrics, Traces.)*

**Специфіка Node для моніторингу — event loop lag.** Унікальна й критична метрика: `perf_hooks.monitorEventLoopDelay()`. Якщо затримка циклу росте — десь блокуюча CPU-операція (питання 3), і **всі** запити деградують. У Python/Java цього показника немає — він прямий наслідок single-threaded моделі.

**Що моніторити в Node-сервісі:**
- **Event loop delay** — здоров'я єдиного потоку.
- **Heap usage / GC pauses** — V8 garbage collection (паузи = латентність).
- **Active handles/requests** — витоки з'єднань.
- RED-метрики (Rate/Errors/Duration), p99 латентності.

**Трейдоф логування:** детальні логи = безцінні для дебагу ↔ дорогі (обсяг, I/O, вартість зберігання), ризик витоку даних, синхронне логування шкодить event loop. Senior балансує: структуровані логи з рівнями + sampling для високочастотних подій, correlation ID для трасування запиту через сервіси, асинхронний transport. Logs (що сталося) + Metrics (скільки/як швидко) + Traces (де затик) разом дають повну картину «що й чому зламалось».

## 15. Чим відрізняється моноліт від мікросервісу?

### L1 — Junior
**Аналогія — універмаг vs вулиця бутіків.**
- **Моноліт** — один великий універмаг: усе під одним дахом, один вхід, одна каса. Просто, але якщо щось ламається — зачиняється весь.
- **Мікросервіси** — вулиця окремих бутіків: кожен сам по собі, свій вхід, свій персонал. Гнучко, але треба координувати між собою.

### L2 — Middle
| | Моноліт | Мікросервіси |
|---|---|---|
| Структура | один застосунок/деплой | багато незалежних сервісів |
| Зв'язок | виклики функцій у пам'яті | мережа (HTTP/gRPC/черги) |
| Деплой | усе разом | кожен окремо |
| Масштабування | весь застосунок | окремі сервіси |
| База даних | спільна | у кожного своя |
| Складність | проста на старті | розподілена |

**Best practice:** **починай з моноліту** (модульного), розбивай на мікросервіси, **коли є реальний біль** (команди заважають одна одній, частини масштабуються по-різному). «Microservices first» для стартапу — передчасна оптимізація.

**Gotcha:** мікросервіси не «спрощують» — вони **міняють** складність коду на складність розподіленої системи (мережеві збої, eventual consistency, distributed tracing, версіонування контрактів).

### L3 — Senior
**Node.js особливо популярний для мікросервісів** — легкий старт, швидкий I/O для міжсервісної комунікації, малий footprint контейнерів. Але це не робить мікросервіси «безкоштовними».

**Трейдофи (універсальні, не лише Node):**
- **Моноліт:** простий деплой/дебаг/тестування, транзакції в одній БД, низька латентність (виклики в пам'яті) ↔ tight coupling, важко масштабувати частинами, один краш = все, повільні великі релізи.
- **Мікросервіси:** незалежний деплой/масштабування/технології, ізоляція збоїв ↔ розподілена складність (мережа ненадійна, eventual consistency, saga замість транзакцій, distributed tracing, service discovery).

**Conway's Law:** архітектура відображає структуру команд. Мікросервіси виправдані, коли є **багато команд**, що мають розгортатися незалежно. Для однієї команди модульний моноліт зазвичай кращий.

**Senior-висновок:** обирай за організацією й навантаженням, не за модою. **Модульний моноліт** із чіткими межами часто оптимальний — він зберігає простоту, але дозволяє згодом виділити мікросервіс із готового модуля, коли з'явиться реальна потреба.

## 16. Різниця між `string` і `String`?

### L1 — Junior
**Аналогія — текст на папірці vs текст у папці з ярликами.**
- **`string`** (мала літера) — просте **значення**: сам текст `"привіт"`. Легке, швидке.
- **`String`** (велика літера) — **об'єкт-обгортка** навколо тексту, з методами. Важчий.

**Правило: завжди використовуй `string` (примітив), майже ніколи — `new String()`.**

```javascript
const a = "привіт";              // string — примітив
const b = new String("привіт");  // String — об'єкт (НЕ роби так)

typeof a; // "string"
typeof b; // "object" (!)
a === "привіт"; // true
b === "привіт"; // false (це об'єкт, не значення)
```

### L2 — Middle
**У JavaScript:**
- `string` — один із **примітивних типів** (як `number`, `boolean`).
- `String` — **конструктор/обгортка** (wrapper object).
- `String(x)` без `new` — корисна функція **конвертації** в рядок (це ок).
- `new String(x)` — створює об'єкт-обгортку (антипатерн).

**У TypeScript** (питання особливо актуальне):
```typescript
let a: string = "hi";   // ✅ примітивний тип — ЗАВЖДИ так
let b: String = "hi";   // ❌ тип об'єкта-обгортки — НЕ використовуй
```
TS-лінтери (`@typescript-eslint/ban-types`) **забороняють** `String`, `Number`, `Boolean` як типи — лише примітивні `string`, `number`, `boolean`.

**Gotcha:** `typeof new String("x") === "object"`, не `"string"`. Тому `new String("x") === "x"` → `false`. Порівняння об'єктів-обгорток ламає логіку.

### L3 — Senior
**Autoboxing (чому примітив має методи):** коли пишеш `"привіт".toUpperCase()`, JS **тимчасово** обгортає примітив у `String`-об'єкт, викликає метод, і **викидає** обгортку. Тому примітиви «мають» методи, не будучи об'єктами. Це прозоро й безкоштовно (V8 оптимізує), на відміну від явного `new String()`, що створює **постійний** об'єкт у heap.

**Чому `new String()` — антипатерн:**
- `typeof` повертає `"object"` → ламає перевірки типів.
- `===` порівнює за **ідентичністю об'єкта**, не за значенням → два `new String("x")` не рівні.
- Зайва алокація в heap (примітив живе на стеку/inline).
- У truthy-контексті `new Boolean(false)` — **truthy** (об'єкт завжди truthy!) — класична пастка з обгортками.

**Зв'язок із Python:** концептуально схоже на «усе є об'єкт» дискусію, але навпаки — JS має **і** примітиви (легкі значення), **і** об'єкти-обгортки, з autoboxing-містком між ними. Senior знає: працюй із примітивами, обгортки (`String`/`Number`/`Boolean` як конструктори) — майже ніколи не потрібні; як **функції конвертації** (`String(x)`, `Number(x)`) без `new` — корисні.

**TypeScript-нюанс:** `string` (примітивний тип) vs `String` (тип інстансів обгортки). `String`-тип приймає і примітиви (через структурну сумісність), і об'єкти-обгортки, тому ширший і небезпечніший — лінтери його банять. Завжди `string`.

> ⚠️ Деталі autoboxing і оптимізацій — специфіка рушія (V8). Поведінка `typeof`/`===` стандартизована ECMAScript і стабільна.

## 54. Відстань між елементами (повна задача)

```html
<div style="display: block">
  <div style="margin-bottom: 25px">AAA</div>
  <div style="margin-top: 21px">BBB</div>
</div>
```

### Відповідь: `display: block` → **25px**
Два блокові елементи в звичайному потоці → **вертикальні margin схлопуються** (п.53): беремо **більший**, не суму.
```
max(25px, 21px) = 25px   (а НЕ 25 + 21 = 46px)
```

### Чи зміниться при `display: flex`? — **Так, і двічі**
При `display: flex` стається **дві** зміни (обидві треба назвати):

**1. Схлопування зникає.** У flex/grid контейнерах margin **не схлопуються** ніколи.

**2. Напрямок стає горизонтальним.** `flex` за замовчуванням має `flex-direction: row` → AAA і BBB стають **в ряд** (поруч по горизонталі), а не один під одним.
```
display: flex (row) → AAA | BBB  (поруч)
```
Тепер `margin-bottom`/`margin-top` — це **вертикальні** margin, а елементи розташовані **горизонтально** → між ними по горизонталі **0px** (бо `margin-left`/`right` = 0). Вертикальні margin лише зсувають кожен елемент по cross-осі, але **проміжку між ними не створюють**.

**Якби було `flex-direction: column`** (вертикально, як у block):
```
max → НЕ діє (flex), тому 25 + 21 = 46px
```

### Senior-висновок (повна відповідь на інтерв'ю)
> «У `block` — 25px через схлопування вертикальних margin (береться більший). У `flex` схлопування зникає, **але** дефолтний `flex-direction: row` ставить елементи горизонтально, тож вертикальні margin між ними проміжку не дають — по горизонталі 0px. Якби flex-direction був `column`, було б 25+21=**46px** (бо у flex margin не схлопуються).»

Це показує розуміння **і** margin collapsing, **і** flex-моделі — більшість кандидатів забуває про зміну напрямку.

> ⚠️ Точна вертикальна позиція кожного елемента у flex-row залежить від `align-items` — але **проміжку між елементами** вертикальні margin не створюють.

## 55. Чому `<script>` перед `</body>`? Що буде в `<head>`?

### L1 — Junior
**Аналогія — не клич гостей до столу, поки страви не готові.** Браузер читає HTML **згори вниз**. Якщо скрипт зверху (`<head>`) звертається до елементів, яких **ще немає** (нижче в HTML) — він їх не знайде (`null`). Тому скрипт ставлять **перед `</body>`** — коли весь DOM уже існує.

```html
<body>
  <div id="app"></div>
  <script src="app.js"></script>  <!-- DOM уже готовий -->
</body>
```

### L2 — Middle
**Дві проблеми скрипта в `<head>` без атрибутів:**
1. **Блокування рендерингу:** парсер HTML **зупиняється**, завантажує й виконує скрипт, і лише потім продовжує → користувач бачить білий екран довше.
2. **DOM ще не існує:** `document.getElementById("app")` → `null`, бо `<body>` ще не розпарсений.

```html
<head>
  <script>
    document.getElementById("app"); // ❌ null — body ще не існує
  </script>
</head>
```

**Gotcha:** `<script>` без `defer`/`async` — **синхронний і блокуючий**: парсинг HTML стає на паузу на час завантаження + виконання скрипта. Багато таких скриптів зверху = повільне відображення сторінки.

### L3 — Senior
**Сучасний підхід — не «перед body», а `defer` у `<head>`** (питання 56):
```html
<head>
  <script src="app.js" defer></script>  <!-- сучасний best practice -->
</head>
```
`defer` дає краще за «скрипт перед body»: скрипт **завантажується паралельно** з парсингом (не блокує), а **виконується після** повного парсингу DOM, у порядку. Це поєднує неблокуючість і готовий DOM.

**Чому історично ставили перед `</body>`:** до `defer`/`async` це був єдиний спосіб гарантувати готовий DOM без блокування критичного рендеру. Зараз `defer` робить це чистіше (скрипти в `<head>`, але неблокуючі).

**Трейдоф:** «скрипт перед body» — проста класика, працює всюди ↔ скрипт починає завантажуватись пізно (після всього HTML). `defer` у head — завантаження стартує раніше (паралельно з парсингом), виконання після DOM → швидше. Senior: `defer` для застосункового коду, інлайн-критичне мінімально, важке — асинхронно/ліниво.

## 56. Атрибути `defer` і `async` у `<script>`

### L1 — Junior
Обидва роблять завантаження скрипта **неблокуючим** (паралельно з парсингом HTML), але **по-різному виконують**:
- **`async`** — виконати **щойно завантажився** (може перервати парсинг, порядок не гарантований).
- **`defer`** — виконати **після** повного парсингу HTML, **у порядку**.

```html
<script src="a.js" async></script>  <!-- незалежний скрипт -->
<script src="b.js" defer></script>  <!-- залежить від DOM/порядку -->
```

### L2 — Middle
```
Без атрибута:  [парсинг] ⏸ [завантаження+виконання] ▶ [парсинг далі]   ← блокує
async:         [парсинг───────] (паралельно: завантаження) ⏸виконання⏸ [парсинг]
defer:         [парсинг───────────────────] (паралельно: завантаження) → [виконання після DOM]
```

| | Завантаження | Виконання | Порядок | DOM готовий |
|---|---|---|---|---|
| (нічого) | блокує парсинг | одразу | так | ❌ |
| **async** | паралельно | щойно завантажено | **ні** | ❌ можливо |
| **defer** | паралельно | після парсингу | **так** | ✅ |

**Коли що:**
- **`async`** — незалежні скрипти, що нічого не чекають: аналітика, реклама, лічильники.
- **`defer`** — застосунковий код, що залежить від DOM або порядку завантаження.

**Gotcha:** `async`/`defer` працюють **лише для зовнішніх** скриптів (`src`). На інлайн-скриптах (`<script>код</script>`) **ігноруються**.

### L3 — Senior
**Тонкощі (senior):**
- **async — race condition порядку:** кілька `async`-скриптів виконуються в порядку **завантаження** (хто перший прийшов), не в порядку в HTML → не використовуй для взаємозалежних скриптів.
- **defer — гарантований порядок** + виконання **перед** `DOMContentLoaded`, після парсингу. Кілька `defer` виконуються в порядку оголошення.
- **module scripts** (`type="module"`) — **defer за замовчуванням** (тому ESM-скрипти неблокуючі автоматично).

**Трейдоф/вибір:**
- `async` — найшвидше виконання незалежного коду, але непередбачуваний порядок і момент (може стрельнути до готовності DOM).
- `defer` — передбачуваність (порядок + готовий DOM) ціною трохи пізнішого виконання (після парсингу).

Senior-правило: **defer** для більшості застосункового JS (передбачувано, неблокуюче, DOM готовий), **async** лише для справді незалежних сторонніх скриптів (аналітика). Критичний inline-CSS/JS — мінімізувати; важке — code splitting + lazy load.

## 57. Що таке `tabindex`? Де використовують?

### L1 — Junior
**`tabindex`** керує **навігацією клавішею Tab** — порядком і можливістю фокусу на елементі з клавіатури.

```html
<div tabindex="0">Тепер на мене можна перейти через Tab</div>
```

### L2 — Middle
**Три значення:**
| Значення | Поведінка |
|---|---|
| `tabindex="0"` | елемент фокусується в **природному** порядку (за позицією в DOM) |
| `tabindex="-1"` | фокусується **лише програмно** (`.focus()`), **не** через Tab |
| `tabindex="1"`+ | явний порядок Tab (**антипатерн!**) |

**Де використовують:**
- Зробити фокусованим елемент, що за замовчуванням не фокусується (`<div>`, `<span>`, кастомні віджети) — `tabindex="0"`.
- Програмний фокус без потрапляння в Tab-порядок (модалки, керування фокусом) — `tabindex="-1"`.

```html
<div role="button" tabindex="0">Кастомна кнопка</div>  <!-- доступна з клавіатури -->
<div id="modal" tabindex="-1">...</div>  <!-- фокусуємо через JS при відкритті -->
```

**Gotcha:** `tabindex="1"` і більше — **антипатерн доступності**: ламає природний порядок (елемент із `tabindex="1"` отримує фокус **раніше** за всі з `tabindex="0"`), створює хаотичну навігацію. Використовуй лише `0` і `-1`.

### L3 — Senior
`tabindex` — інструмент **доступності (a11y)** та **керування фокусом**. Нативні інтерактивні елементи (`<button>`, `<a href>`, `<input>`) **уже фокусовані** — їм tabindex не потрібен. Він потрібен, коли робиш інтерактивним **не-інтерактивний** елемент (антипатерн сам по собі, але інколи неминучий для кастомних віджетів).

**Senior-принципи:**
- **Семантика спершу:** використовуй нативні елементи (`<button>` замість `<div tabindex="0" role="button">`) — вони дають фокус, клавіатуру, ARIA «безкоштовно».
- **Focus management:** `tabindex="-1"` + `.focus()` — для пасток фокусу в модалках (focus trap), повернення фокусу після закриття діалогу, переходу до помилки форми.
- **Ніколи `tabindex > 0`** — ламає очікуваний порядок, кошмар для screen reader і клавіатурних користувачів.

**Трейдоф:** `tabindex` дає контроль над фокусом (потрібно для складних віджетів, модалок) ↔ легко зламати доступність (позитивні значення, фокус на неінтерактивному без ARIA). Правильна доступність — це семантичний HTML + мінімальний `tabindex` (0/-1) + ARIA там, де нативного елемента немає.

## 58. Чи число кратне 11?

```javascript
function isDivisibleBy11(myNumber) {
  return myNumber % 11 === 0;
}

isDivisibleBy11(121); // true
isDivisibleBy11(33);  // true
isDivisibleBy11(50);  // false
```

**Пояснення:** `%` — остача від ділення. Кратне 11 ⇒ остача 0.

**Senior-нюанси:**
- `0 % 11 === 0` → `true` (0 кратне всьому — коректно).
- Для **від'ємних**: `-22 % 11 === 0` → `true` (працює). Але `%` у JS зберігає знак діленого (`-23 % 11 === -1`), тож для перевірки кратності `=== 0` це не проблема.
- Не плутай із `Number.isInteger` — якщо вхід може бути дробовим/рядком, спершу валідуй: `Number.isInteger(myNumber) && myNumber % 11 === 0`.

## 59. Скопіювати масив без прив'язки за посиланням

```javascript
const original = [1, 2, 3];

// Способи (усі — SHALLOW копія):
const copy1 = [...original];        // spread (найпоширеніший)
const copy2 = original.slice();     // slice без аргументів
const copy3 = Array.from(original); // Array.from

copy1.push(4);
console.log(original); // [1, 2, 3] — оригінал не змінився ✅
```

**Чому потрібна копія:** `const b = a` копіює **посилання** — `b` і `a` вказують на **той самий** масив (зміна одного міняє інший). Spread/slice створюють **новий** масив.

**Gotcha — SHALLOW vs DEEP (критично):** ці способи копіюють лише **верхній рівень**. Вкладені об'єкти/масиви лишаються **спільними за посиланням**:
```javascript
const nested = [{ a: 1 }, [2, 3]];
const shallow = [...nested];
shallow[0].a = 99;
console.log(nested[0].a); // 99 — вкладений об'єкт СПІЛЬНИЙ! ❌
```

**Senior — для глибокої копії:**
```javascript
const deep = structuredClone(nested); // ✅ сучасно (Node 17+, браузери)
// старий хак: JSON.parse(JSON.stringify(nested)) — губить функції/Date/undefined
```

## 60. Що виведе консоль? (`delete trees[3]`)

```javascript
const trees = ["xyz", "xxxx", "test", "ryan", "apple"];
delete trees[3];
console.log(trees.length); // 5
console.log(trees[3]);     // undefined
```

**Відповідь: `5`, потім `undefined`.**

**Пояснення (це питання-пастка про `delete` на масиві):**
- `delete trees[3]` **видаляє значення**, але **не зсуває** елементи й **не міняє `length`** → лишається «дірка» (empty slot).
- `trees` стає `["xyz", "xxxx", "test", empty, "apple"]` — `length` усе ще **5**.
- `trees[3]` → `undefined` (доступ до порожнього слота).

**Senior-нюанс:** `delete` на масиві створює **sparse array** (розріджений масив із «дірками») — антипатерн (блок 3, п.29). Порожній слот (`empty`) — не те саме, що `undefined`-значення: `trees.hasOwnProperty(3)` → `false`, а методи типу `forEach`/`map` **пропускають** empty-слоти (але `for...of` дасть `undefined`). Для реального видалення з масиву — `splice(3, 1)` (зсуне, `length` стане 4) або `filter`.

## 61. Масив без дублікатів + масив лише дублікатів

```javascript
const source = [1, 2, 3, 5, 1, 5, 9, 1, 2, 8];

// 1. Без дублікатів (унікальні) — через Set
const unique = [...new Set(source)];
// [1, 2, 3, 5, 9, 8]

// 2. Лише ті, що дублювалися (унікальний список повторюваних)
const counts = {};
for (const x of source) counts[x] = (counts[x] || 0) + 1;
const duplicated = Object.keys(counts)
  .filter((k) => counts[k] > 1)
  .map(Number);
// [1, 2, 5]   (1 зустрічається 3 рази, 2 і 5 — по 2 рази)
```

**Пояснення:**
- **`unique`:** `Set` зберігає лише унікальні значення; spread перетворює назад у масив. O(n).
- **`duplicated`:** рахуємо частоту кожного, лишаємо ті, що зустрілись **> 1** разу. `[1, 2, 5]`.

**Альтернатива для duplicated (компактніше, з Set):**
```javascript
const seen = new Set();
const dups = new Set();
for (const x of source) {
  if (seen.has(x)) dups.add(x);  // бачили раніше → дублікат
  else seen.add(x);
}
const duplicated = [...dups];     // [1, 5, 2]
```

**Senior-нюанс:** уточни в інтерв'юера семантику «лише дублікати»: (а) **унікальний список** повторюваних значень `[1,2,5]` (найчастіше це мають на увазі), чи (б) **усі зайві входження** `[1, 5, 1, 2]` (кожне повторення понад перше). Перша інтерпретація через частоти/Set; друга — інша логіка. Уточнення = senior-сигнал.

## 62. Деструктуризація вкладеного об'єкта

```javascript
const obj = {
  name: "Vova", secondName: "Simba", age: 15,
  parents: {
    mother: { name: "Natali", secondName: "Simba", age: 38 },
    father: { name: "Oleg", secondName: "Simba", age: 45, job: "driver" },
  },
};

const {
  parents: {
    mother: {
      name: motherName,
      secondName: motherSecondName,
      age: motherAge,
      job: motherJob = "unemployed",   // дефолт, бо в mother НЕМАЄ job
    },
    father: { name: fatherName, secondName: fatherSecondName },
  },
} = obj;

const fatherInitials = { name: fatherName, secondName: fatherSecondName };

// Перевірка:
// motherName = "Natali", motherSecondName = "Simba", motherAge = 38
// motherJob = "unemployed"  ← дефолт спрацював
// fatherInitials = { name: "Oleg", secondName: "Simba" }
```

**Пояснення ключових прийомів:**
- **Вкладена деструктуризація:** `parents: { mother: { ... } }` — спускаємось у вкладені об'єкти.
- **Перейменування:** `name: motherName` — витягнути `name`, назвати `motherName`.
- **Дефолт:** `job: motherJob = "unemployed"` — якщо `job` відсутній (`undefined`), береться дефолт. У `mother` job немає → `"unemployed"`. ✅
- **fatherInitials** — окремо збираємо об'єкт лише з імені й прізвища (без age/job).

**Senior-нюанс:** дефолт у деструктуризації спрацьовує **лише на `undefined`** (не на `null`/`0`/`""`) — та сама семантика, що `??` (блок 3, п.39). `job: x = "unemployed"` для `job: null` дасть `null`, не дефолт. Тут це не проблема (job просто відсутній), але важливо пам'ятати.

## 63. Чи є відмінність між цими записами?

```javascript
function makeFunc() {}      // (1) Function Declaration
let myFunc = function();    // (2) ← це SYNTAX ERROR (немає тіла!)
```

### Відповідь: **так, є відмінність — і другий рядок узагалі некоректний**

**Спочатку — про синтаксис:** `let myFunc = function();` — **синтаксична помилка**. Function expression вимагає **тіло** `{}`: має бути `let myFunc = function () {};`. Як написано (`function()` з `;`) — невалідний JS.

**Якщо мали на увазі `let myFunc = function () {};`** — тоді це класичне **Declaration vs Expression** (блок 3, п.27):

| | `function makeFunc() {}` | `let myFunc = function () {}` |
|---|---|---|
| Тип | Function **Declaration** | Function **Expression** |
| Hoisting | піднята **повністю** (виклик до оголошення ✅) | лише змінна (TDZ, виклик до → помилка) |
| Ім'я функції | `makeFunc` (іменована) | анонімна (`myFunc.name` виведеться з присвоєння) |
| Прив'язка | у scope з компіляції | присвоюється в рантаймі |

```javascript
makeFunc();   // ✅ працює — declaration піднята
myFunc2();    // ❌ ReferenceError — expression у TDZ
let myFunc2 = function () {};
```

**Senior-нюанс:** головна практична різниця — **hoisting**. Declaration доступна в усій області з початку; expression — лише після рядка присвоєння. Тому declaration дозволяє «головна логіка зверху, хелпери знизу», а expression змушує оголошувати перед використанням (передбачуваніше). Багато styleguide радять expression через `const` за консистентність. І не забути: написане `function();` — це **помилка парсингу**, варто вказати інтерв'юеру.

## 64. Опціональний ланцюжок `?.`

```javascript
let array;                    // undefined
array?.slice(1, 3);           // undefined — НЕ кидає помилку!

if (array?.length) {          // array?.length → undefined → falsy
  array.slice(1, 3);          // не виконається (блок пропущено)
}
```

### Що таке `?.`
**Опціональний ланцюжок** — безпечний доступ до властивостей/методів: якщо ліва частина `null`/`undefined`, весь вираз **коротко замикається** на `undefined` замість того, щоб кинути `TypeError`.

```javascript
// Без ?. — помилка:
array.slice(1, 3);   // ❌ TypeError: Cannot read properties of undefined

// З ?. — безпечно:
array?.slice(1, 3);  // ✅ undefined (метод не викликається)
```

**Розбір коду в задачі:**
- `let array;` → `array === undefined`.
- `array?.slice(1,3)` → бачить, що `array` nullish → **зупиняється**, повертає `undefined`, `.slice` **не викликається** (немає помилки).
- `if (array?.length)` → `undefined` → falsy → тіло `if` **не виконується** → `array.slice` всередині безпечно не запускається.

**Senior-нюанси:**
- **Форми:** `obj?.prop` (властивість), `obj?.[key]` (динамічний ключ), `obj?.method()` (виклик), `arr?.[0]` (елемент).
- **Коротке замикання:** `a?.b.c.d` — якщо `a` nullish, **весь** ланцюг зупиняється (не лише `a?.b`).
- **Поєднання з `??`:** `user?.name ?? "Гість"` — безпечний доступ + дефолт (блок 3, п.39).
- **Не зловживати:** `?.` всюди ховає баги — якщо значення **завжди має бути**, краще явна перевірка/падіння (fail-fast), ніж тихе `undefined`, що тягнеться далі.

## 65. Асинхронний `.map()` об'єкта (повертає об'єкт, не Promise значень)

```javascript
async function asyncObjectMap(obj, callback) {
  const entries = Object.entries(obj);
  const mapped = await Promise.all(
    entries.map(async ([key, value]) => [key, await callback(value)])
  );
  return Object.fromEntries(mapped);
}

// Використання:
await asyncObjectMap({ qwe: 123, fds: 32 }, async (x) => x * 2);
// => { qwe: 246, fds: 64 }
```

**Пояснення (покроково):**
1. `Object.entries(obj)` → `[["qwe", 123], ["fds", 32]]` — масив пар.
2. `.map(async ([k, v]) => [k, await callback(v)])` → масив **промісів** пар `[ключ, результат]`.
3. `await Promise.all(...)` → чекає **всі** колбеки **паралельно**, отримує `[["qwe", 246], ["fds", 64]]`.
4. `Object.fromEntries(...)` → збирає назад в **об'єкт** `{ qwe: 246, fds: 64 }`.

**Чому `Promise.all`:** колбек асинхронний → кожен виклик повертає Promise. `Promise.all` чекає всі **паралельно** (блок 3, п.36) і повертає масив результатів. Без нього отримали б масив невирішених промісів.

**Senior-нюанси:**
- **Паралельно vs послідовно:** ця версія — **паралельна** (всі колбеки стартують одразу). Якщо потрібен **порядок/лімітування** (напр., не перевантажити API), — послідовний варіант через `for...of`:
```javascript
async function asyncObjectMapSeq(obj, callback) {
  const result = {};
  for (const [key, value] of Object.entries(obj)) {
    result[key] = await callback(value);   // по черзі
  }
  return result;
}
```
Трейдоф: паралельно — швидше (час найдовшого колбека), але всі одночасно (навантаження); послідовно — повільніше (сума), але контрольовано. Для багатьох ключів із мережевими колбеками — паралельно з **лімітом конкурентності** (`p-limit`).
- **Функція `async` повертає Promise** — тому `await asyncObjectMap(...)`. Вимога «повернути об'єкт, а не Promise» означає: **всередині** розв'язати всі проміси (через `await Promise.all`), щоб результат був готовим об'єктом значень, а не об'єктом промісів. ✅ виконано.

> Усі практичні приклади (60, 61, 62, 64, 65) перевірені запуском у Node — результати збігаються з очікуваними.

## 46. Що таке CORS? Де трапляється?

### L1 — Junior
**Аналогія — пропускний режим між будівлями.** Браузер за замовчуванням забороняє скрипту з одного сайту (origin) лізти по дані на **інший** сайт — це **Same-Origin Policy** (політика однакового джерела), захист від крадіжки даних. **CORS (Cross-Origin Resource Sharing)** — це механізм, яким сервер **дозволяє** певним чужим сайтам доступ, видаючи спеціальні заголовки.

**Origin** = протокол + домен + порт (`https://example.com:443`). Відрізняється хоч одне — це інший origin.

### L2 — Middle
**Де трапляється:** у **браузері**, коли JavaScript робить запит на **інший origin**:
```javascript
// Сторінка на https://myapp.com робить запит на https://api.other.com
fetch("https://api.other.com/data"); // ← cross-origin → діє CORS
```
Сервер `api.other.com` має відповісти заголовком:
```http
Access-Control-Allow-Origin: https://myapp.com
```
Інакше браузер **заблокує** доступ скрипта до відповіді.

**Що НЕ є CORS-обмеженим:** запити з того ж origin, `<img>`, `<script src>`, `<link>` (вони не читаються JS), запити з сервера на сервер (CORS — лише браузерне обмеження).

**Gotcha:** CORS перевіряє **браузер**, не сервер. `curl`/Postman/бекенд **не** обмежені CORS — тому «у Postman працює, у браузері CORS-помилка» — нормально.

### L3 — Senior
**Simple vs Preflighted requests:**
- **Simple** (GET/POST/HEAD, прості заголовки, певні Content-Type) — браузер шле запит **одразу**, але блокує **читання** відповіді, якщо немає `Access-Control-Allow-Origin`. Тобто запит **доходить** до сервера (побічні ефекти можливі!), браузер ховає лише відповідь.
- **Preflighted** — для «небезпечних» (PUT/DELETE/PATCH, кастомні заголовки, `application/json` у деяких випадках) браузер спершу шле **OPTIONS preflight**:
```http
OPTIONS /data
Origin: https://myapp.com
Access-Control-Request-Method: DELETE
```
Сервер відповідає, що дозволено (`Access-Control-Allow-Methods`, `-Headers`, `-Origin`), і лише тоді браузер шле справжній запит.

**Ключові заголовки сервера:** `Access-Control-Allow-Origin` (хто), `-Methods` (які методи), `-Headers` (які заголовки), `-Credentials` (чи можна cookies — тоді Origin **не може** бути `*`).

**Трейдоф/безпека:** CORS — це **не** захист сервера (його обходить будь-який не-браузерний клієнт), а захист **користувача браузера** від того, щоб зловмисний сайт читав його дані з інших сайтів (де він залогінений). Це доповнює, але не замінює серверну авторизацію (блок Python п.9/11). `Access-Control-Allow-Origin: *` на ендпоінті з приватними даними — діра.

> ⚠️ Точні правила «simple vs preflighted» (які Content-Type/заголовки) — звіряй зі специфікацією Fetch/MDN. Префлайт кешується (`Access-Control-Max-Age`).

## 47. Як отримати помилку CORS у консолі?

### L1 — Junior
Зробити з браузерної сторінки запит на **інший origin**, який **не дозволяє** твій origin:
```javascript
// На localhost:3000 виконуємо:
fetch("https://api.github.com/users") // або будь-який чужий домен без CORS-дозволу
  .then((r) => r.json());
// Консоль: "Access to fetch ... has been blocked by CORS policy:
//           No 'Access-Control-Allow-Origin' header is present"
```

### L2 — Middle
**Типові способи спровокувати CORS-помилку:**
1. Запит на чужий домен, що не віддає `Access-Control-Allow-Origin`.
2. Запит із `credentials: "include"`, а сервер віддає `Allow-Origin: *` (не можна разом із credentials).
3. Кастомний заголовок (`Authorization`, `X-Custom`), не дозволений у `Access-Control-Allow-Headers`.
4. Метод (PUT/DELETE), не дозволений у `Access-Control-Allow-Methods` (preflight провалюється).

**Як виглядає в консолі:**
```
Access to fetch at 'https://api.other.com' from origin 'http://localhost:3000'
has been blocked by CORS policy: No 'Access-Control-Allow-Origin' header...
```

**Gotcha:** мережева вкладка (Network) покаже, що запит **відбувся** (статус може бути 200!), але JS отримує помилку — браузер **заблокував читання відповіді** скриптом, а не сам запит (для simple requests).

### L3 — Senior
**Діагностика й виправлення (senior-підхід):**
- **Помилка на стороні сервера, не клієнта:** CORS лагодиться **на сервері** (додати заголовки), а не у фронт-коді. Фронт не може «обійти» CORS легально.
- **Перевір preflight:** якщо в Network є OPTIONS-запит, що падає — проблема в `Allow-Methods`/`Allow-Headers`.
- **`credentials` + `*`:** для cookies/auth `Access-Control-Allow-Origin` має бути **конкретним** origin (не `*`) + `Access-Control-Allow-Credentials: true`.

**Способи вирішення (за контекстом):**
- Налаштувати CORS на бекенді (`cors` middleware в Express — блок 2).
- **Proxy** у dev (Vite/CRA proxy) — запити йдуть через свій origin.
- Reverse proxy (Nginx) — фронт і API під одним origin у проді.

**Трейдоф:** `cors({ origin: "*" })` найпростіше, але небезпечно для приватних API (будь-який сайт може робити запити). Правильно — **allowlist** конкретних origin. Senior не «відключає CORS», а налаштовує **точний** дозвіл під довірені origin.

## 48. Основні HTTP-методи RESTful / CRUD

### L1 — Junior
**Чотири дії над даними (CRUD) ↔ HTTP-методи:**
| CRUD | HTTP | Дія |
|---|---|---|
| **C**reate | `POST` | створити |
| **R**ead | `GET` | прочитати |
| **U**pdate | `PUT` / `PATCH` | оновити |
| **D**elete | `DELETE` | видалити |

```
GET    /users      → список користувачів
GET    /users/1    → один користувач
POST   /users      → створити нового
PUT    /users/1    → замінити повністю
PATCH  /users/1    → оновити частково
DELETE /users/1    → видалити
```

### L2 — Middle
**Властивості методів (часте питання):**
| Метод | Safe (не змінює) | Idempotent | Тіло |
|---|---|---|---|
| GET | ✅ | ✅ | ні |
| POST | ❌ | ❌ | так |
| PUT | ❌ | ✅ | так |
| PATCH | ❌ | ❌* | так |
| DELETE | ❌ | ✅ | інколи |

- **Safe** — не змінює стан сервера.
- **Idempotent** — N однакових запитів = той самий ефект, що й 1.

**Gotcha — POST vs PUT:** POST **створює** і **не ідемпотентний** (два POST = два записи). PUT **замінює** і **ідемпотентний** (двічі покласти той самий ресурс = один результат). PATCH — часткове оновлення, формально **не** гарантовано ідемпотентний.

### L3 — Senior
*(Глибше — в основному HTTP-блоці.)* RESTful-дизайн (senior-принципи):
- **Ресурси як іменники** в URL (`/users`, не `/getUsers`) — дія виражається **методом**, не шляхом.
- **Idempotency** — основа надійності розподілених систем: при обриві мережі безпечно **повторити** лише ідемпотентний метод (GET/PUT/DELETE); для POST — **idempotency key**.
- **Статус-коди за призначенням:** 201 Created (POST), 204 No Content (DELETE), 200/200 (GET/PUT), 404, 409 Conflict.

**Трейдоф PUT vs PATCH:** PUT вимагає **повного** представлення ресурсу (заміна) → ідемпотентний, але передаєш усе. PATCH — лише зміни (дельта) → економніше, але складніше (формат patch, не завжди ідемпотентний, напр. «increment»). REST-чистота ↔ практичність: багато API використовують PATCH-семантику під виглядом PUT. Senior обирає за контрактом і потребами консистентності.

## 49. Що таке DNS?

### L1 — Junior
**Аналогія — телефонна книга інтернету.** Люди пам'ятають **імена** (`google.com`), а комп'ютери спілкуються за **IP-адресами** (`142.250.74.78`). **DNS (Domain Name System)** перекладає ім'я домену в IP-адресу.

```
google.com → DNS → 142.250.74.78
```

### L2 — Middle
**Процес резолюції (спрощено):**
1. Браузер перевіряє **кеш** (свій, ОС).
2. Питає **recursive resolver** (зазвичай провайдера або 8.8.8.8).
3. Resolver питає **root server** → скеровує до **TLD-сервера** (`.com`).
4. TLD → **authoritative server** домену → повертає IP.
5. IP кешується (за TTL), браузер з'єднується.

**Типи записів:**
| Запис | Призначення |
|---|---|
| **A** | домен → IPv4 |
| **AAAA** | домен → IPv6 |
| **CNAME** | псевдонім (домен → інший домен) |
| **MX** | поштові сервери |
| **TXT** | текст (SPF, верифікація) |
| **NS** | name servers домену |

**Gotcha:** DNS кешується за **TTL** — тому зміни DNS «розповсюджуються» не миттєво (від хвилин до годин). При міграції серверів TTL знижують заздалегідь.

### L3 — Senior
DNS — **перший крок** будь-якого мережевого запиту (блок Python п.14: DNS → TCP → TLS → HTTP) і часто **прихований внесок у latency**. Тому DNS кешується на багатьох рівнях (браузер, ОС, resolver) і використовується для оптимізації.

**DNS як інфраструктурний інструмент (senior):**
- **Load balancing / failover** — кілька A-записів, geo-DNS (повертає найближчий сервер).
- **CDN** — CNAME на edge-мережу (блок Python п.66).
- **Service discovery** — у Kubernetes/мікросервісах внутрішній DNS резолвить імена сервісів.
- **Безпека** — DNSSEC (підпис записів), DNS-over-HTTPS/TLS (приватність), але й вектор атак (DNS spoofing, cache poisoning).

**Трейдоф кешування:** TTL балансує **свіжість** (низький TTL — швидкі зміни, але більше запитів) vs **продуктивність/навантаження** (високий TTL — менше запитів, але повільне поширення змін). Senior керує TTL свідомо: низький перед плановою міграцією, високий для стабільних записів.

## 50. Способи центрування елемента (горизонтально + вертикально)

### L1 — Junior
**Сучасний дефолт — Flexbox (3 рядки):**
```css
.parent {
  display: flex;
  justify-content: center; /* по горизонталі */
  align-items: center;     /* по вертикалі */
}
```
Або ще коротше — **Grid:**
```css
.parent {
  display: grid;
  place-items: center;     /* і по горизонталі, і по вертикалі */
}
```

### L2 — Middle
**Усі способи (знати кілька):**
```css
/* 1. Flexbox — найуніверсальніший */
.parent { display: flex; justify-content: center; align-items: center; }

/* 2. Grid — найкоротший */
.parent { display: grid; place-items: center; }

/* 3. Absolute + transform — коли не можна чіпати батька-flex */
.child {
  position: absolute;
  top: 50%; left: 50%;
  transform: translate(-50%, -50%);
}

/* 4. margin: auto (лише горизонтально, потрібна ширина) */
.child { width: 200px; margin: 0 auto; }

/* 5. Absolute + margin auto (всі сторони 0) */
.child { position: absolute; inset: 0; margin: auto; width: 200px; height: 100px; }
```

**Gotcha:** `text-align: center` центрує лише **inline-контент** (текст, inline-block), не блокові елементи. `margin: 0 auto` працює горизонтально **лише** для блоків із заданою шириною.

### L3 — Senior
**Чому `transform: translate(-50%, -50%)` працює:** `top/left: 50%` зміщують **верхній лівий кут** елемента в центр → елемент зміщений вправо-вниз. `translate(-50%, -50%)` відсуває назад на **половину розміру самого елемента** (% у transform рахується від розміру елемента, не батька) → точний центр. Перевага: працює без знання розмірів елемента.

**Трейдоф вибору:**
- **Flexbox/Grid** — сучасний дефолт: чисто, адаптивно, без хаків, керує і дочірніми. Grid `place-items: center` — найлаконічніший. ✅ для нового коду.
- **Absolute + transform** — коли елемент має бути **поза потоком** (оверлеї, модалки, тултіпи) або не можна зробити батька flex/grid. Виймає з потоку (не впливає на сусідів).
- **margin auto** — лише горизонтально для блоків (legacy, але робоче).

Senior обирає Flexbox/Grid за замовчуванням; absolute+transform — для оверлеїв/позиціонування поза потоком. Знання кількох способів показує розуміння **CSS layout моделей** (потік, flex, grid, absolute).

## 51. Звернутися до всіх посилань із `.com` у кінці href + приховати

### L1 — Junior
**Селектор атрибута `[attr$="value"]`** — `$=` означає «закінчується на».
```css
a[href$=".com"] {
  display: none;  /* приховати */
}
```

### L2 — Middle
**Селектори атрибутів (повний набір):**
```css
a[href="https://example.com"] {}   /* точний збіг */
a[href^="https://"] {}             /* ^= починається з (secure links) */
a[href$=".com"] {}                 /* $= закінчується на */
a[href*="google"] {}              /* *= містить підрядок */
a[target="_blank"] {}              /* зовнішні (нова вкладка) */
```

**Приховати — два способи (різні!):**
```css
a[href$=".com"] { display: none; }      /* прибрати з потоку (місця не займає) */
a[href$=".com"] { visibility: hidden; } /* сховати, але МІСЦЕ лишається */
```

**Gotcha:** `$=".com"` спіймає й `evil.com.ua` якщо href = `...com` — буквально «закінчується на .com». Для href типу `https://site.com/page` (з шляхом після) `$=".com"` **не** спрацює (закінчується на `/page`). Точніше — `[href*=".com"]` (містить) або складніше.

### L3 — Senior
**`display: none` vs `visibility: hidden` vs `opacity: 0` (важлива різниця):**
| | Видимий | Займає місце | Клікабельний | У DOM |
|---|---|---|---|---|
| `display: none` | ❌ | ❌ | ❌ | так (але «нема» в layout) |
| `visibility: hidden` | ❌ | ✅ | ❌ | так |
| `opacity: 0` | ❌ | ✅ | **✅** | так |

`opacity: 0` — елемент **невидимий, але клікабельний** (часта пастка/вектор clickjacking). `display: none` — повністю виключений із рендеру (не читається screen-reader'ами теж — нюанс доступності).

**Трейдоф/безпека:** CSS-приховування — лише **візуальне**, дані лишаються в HTML/DOM (доступні через DevTools, скрейпери, JS). Приховати посилання через CSS ≠ видалити їх. Для справжнього видалення — серверний рендеринг без них або видалення з DOM через JS. Селектори атрибутів потужні для стилізації за патернами (зовнішні посилання, типи файлів `[href$=".pdf"]`), але не для безпеки.

## 52. Чи можна зробити фіксацію кліку через HTML+CSS? Приклад

### L1 — Junior
**Так — через «checkbox hack».** CSS не має «подій», але є псевдоклас **`:checked`** для чекбоксів/радіо — він **запам'ятовує** стан кліку. Поєднуємо прихований чекбокс + label + селектори.

```html
<input type="checkbox" id="toggle" hidden>
<label for="toggle">Натисни мене</label>
<div class="content">Я з'являюсь після кліку!</div>
```
```css
.content { display: none; }
#toggle:checked ~ .content { display: block; } /* після кліку — показати */
```

### L2 — Middle
**Кілька CSS-only способів «зафіксувати» взаємодію:**
```css
/* 1. Checkbox hack — стан перемикається й ТРИМАЄТЬСЯ */
#toggle:checked ~ .menu { display: block; }

/* 2. :target — реагує на клік по якорю (#id в URL) */
.modal:target { display: block; }   /* <a href="#modal"> відкриває */

/* 3. :focus / :focus-within — поки елемент у фокусі */
.dropdown:focus-within .menu { display: block; }

/* 4. :active — ЛИШЕ поки кнопка натиснута (не фіксує!) */
button:active { background: red; }
```
Плюс нативний `<details>/<summary>` — розкривний блок без JS:
```html
<details><summary>Клік</summary>Прихований вміст</details>
```

**Gotcha:** `:active` — **не** фіксація (працює лише поки тримаєш кнопку). Для **збереженого** стану кліку потрібен `:checked` (checkbox hack) або `:target`.

### L3 — Senior
**Чому checkbox hack працює:** `:checked` — єдиний CSS-механізм, що **зберігає булевий стан** без JS. Комбінатори (`~` загальний сусід, `+` прямий сусід) дозволяють із цього стану стилізувати **інші** елементи. Так роблять CSS-only меню, акордеони, таби, модалки.

**Трейдофи CSS-only інтерактиву:**
- **+** Працює без JS (прогресивне покращення, швидко, надійно), менше коду.
- **−** **Доступність (a11y):** checkbox hack часто ламає семантику й клавіатурну навігацію/screen readers, якщо не додати ARIA. `<details>` і `:focus-within` — доступніші.
- **−** Обмеженість: складна логіка/стан неможливі; це «хак», не архітектура.

**Senior-позиція:** для простого toggle (меню, акордеон) — нативний `<details>` або `:focus-within` (доступні, семантичні). Checkbox hack — знати як прийом, але для серйозного UI з вимогами доступності краще JS із правильним ARIA. CSS-only добрий для прогресивного покращення (працює, поки JS вантажиться), не для повноцінної інтерактивності.

## 53. Що таке «схлопування» (margin collapsing)?

### L1 — Junior
**Схлопування полів (margin collapsing)** — коли **вертикальні** відступи (`margin-top`/`margin-bottom`) сусідніх блоків **не додаються**, а «зливаються» в один — **більший** із двох.

```html
<div style="margin-bottom: 30px">Блок 1</div>
<div style="margin-top: 20px">Блок 2</div>
<!-- Відстань між ними = 30px (більший), а НЕ 50px! -->
```

### L2 — Middle
**Три ситуації схлопування:**
1. **Сусідні блоки** — `margin-bottom` першого + `margin-top` другого → беруть **більший**.
2. **Батько-дитина** — якщо немає `border`/`padding`/контенту між ними, `margin-top` дитини «вилазить» назовні батька.
3. **Порожній блок** — власні `margin-top` і `margin-bottom` схлопуються між собою.

```css
/* Беруть більший: max(30, 20) = 30px, не 50px */
.first { margin-bottom: 30px; }
.second { margin-top: 20px; }
```

**Як ЗАПОБІГТИ схлопуванню:**
- Додати `padding` або `border` між елементами.
- Зробити батька **flex/grid контейнером** (у них margin **не** схлопується!).
- `overflow: hidden`/`auto` на батьку (створює BFC).

**Gotcha:** схлопуються **лише вертикальні** margin (top/bottom), і **лише** у звичайному потоці (block flow). **Горизонтальні** (left/right) **ніколи** не схлопуються. У **flex/grid** схлопування **немає**.

### L3 — Senior
Margin collapsing — наслідок **block formatting context (BFC)** і історичної моделі CSS (типографська традиція: відступи між абзацами не повинні подвоюватись). Це **навмисна** поведінка для текстового потоку, але часто несподівана для розробників.

**Що створює новий BFC (зупиняє схлопування):** `overflow` ≠ visible, `display: flow-root` (чистий спосіб!), `flex`/`grid` контейнери, `position: absolute/fixed`, float. Всередині нового BFC margin не схлопуються з зовнішнім.

```css
.container { display: flow-root; } /* сучасний чистий спосіб ізолювати BFC */
```

**Трейдоф/сучасна практика:** схлопування — джерело «чому між блоками не той відступ» багів. Сучасний підхід **уникає** покладання на margin-collapsing: використовують **flex/grid із `gap`** (передбачуваний відступ, без схлопування) або **single-direction margin** (лише `margin-bottom`, методологія «owl selector» `* + *`). Senior розуміє механізм (щоб дебажити legacy), але в новому коді віддає перевагу `gap` у flex/grid — там відступи однозначні.

## 54. Якою буде відстань між двома елементами?

> ⚠️ **Питання обрізане** — немає коду/значень двох елементів. Відповім на **принцип** (це продовження п.53 про схлопування) і покажу, як рахувати. **Скинь конкретний HTML/CSS — дам точну цифру.**

### L1 — Junior
**Якщо це два вертикально розташовані блоки з `margin`** — відповідь майже завжди про **схлопування** (п.53): відстань = **більший** із двох margin, **не** їх сума.

```html
<div style="margin-bottom: 20px">A</div>
<div style="margin-top: 30px">B</div>
<!-- Відстань = max(20, 30) = 30px (НЕ 50px!) -->
```

### L2 — Middle
**Алгоритм підрахунку відстані (типові кейси):**
| Ситуація | Відстань |
|---|---|
| Два блоки, margin 20 + 30, звичайний потік | **30px** (схлопування, max) |
| Те саме, але батько `display: flex/grid` | **50px** (схлопування немає, сума) |
| Між ними `padding`/`border`/контент | **50px** (схлопування розірване) |
| Один має `float`/`position: absolute` | margin **не** схлопуються |
| Flex/grid із `gap: 40px` | **40px** (gap, незалежно від margin) |

**Питання-пастка зазвичай перевіряє:** чи знаєш ти, що `margin-bottom: 20px` + `margin-top: 30px` = **30px**, а не 50px (схлопування).

### L3 — Senior
Точна відповідь залежить від **контексту форматування**:
- **Block flow (звичайний):** вертикальні margin **схлопуються** → `max(margins)`.
- **Flex/Grid контейнер:** margin **не** схлопуються → `margin1 + margin2`; а `gap` додає свій відступ.
- **Розірване схлопування:** будь-який `padding`/`border`/інлайн-контент/новий BFC між елементами → margin додаються.

**Щоб дати точну цифру, потрібно знати:**
1. Значення `margin` (top/bottom) обох елементів.
2. Тип батьківського контейнера (block / flex / grid).
3. Чи є `padding`/`border`/`gap`/інші елементи між ними.
4. `position`/`float` елементів.

**Скинь HTML+CSS — порахую крок за кроком і поясню, чому саме така відстань.** Найімовірніше, очікувана відповідь — демонстрація розуміння **margin collapsing** (п.53).

# ЧАСТИНА A — LLM (основи)

## 1. Що таке LLM і як він працює?

### L1 — Junior
**Аналогія — надпотужне автодоповнення.** LLM (Large Language Model) — це модель, натренована **передбачати наступне слово (токен)** за попереднім текстом. З мільярдів текстів вона вивчила закономірності мови й «знань». Ти даєш початок → вона продовжує найімовірнішим продовженням.

### L2 — Middle
- **Архітектура** — переважно **Transformer** (механізм **attention** — «на що звертати увагу в контексті»).
- **Тренування:** (1) **pre-training** — передбачення токенів на величезному корпусі; (2) **fine-tuning / RLHF** — донавчання бути корисним і безпечним.
- **Інференс:** авторегресійна генерація — токен за токеном, кожен залежить від попередніх.
- Приклади: Claude (Anthropic), GPT (OpenAI), Gemini, Llama.

### L3 — Senior
**Ключове розуміння:** LLM — це **ймовірнісна модель розподілу наступного токена** `P(token | контекст)`. Вона не «розуміє» й не «шукає в базі» — вона **генерує статистично правдоподібний** текст. Звідси і сила (гнучкість, узагальнення), і слабкості (галюцинації, п.7 — впевнено вигадує, бо оптимізована на правдоподібність, не на істинність).

**Трейдоф масштабу:** більше параметрів/даних → кращі можливості (scaling laws), але дорожче, повільніше, більше галюцинацій без grounding. Тому в проді LLM **доповнюють** інструментами (tool use, п.9) і знаннями (RAG, п.11) — модель як «розум», зовнішні системи як «пам'ять і руки».

## 2. Токени і токенізація

### L1 — Junior
**Аналогія — LEGO-цеглинки тексту.** LLM бачить не букви й не слова, а **токени** — шматочки (слово, частина слова, символ). «tokenization» ≈ `token` + `ization` (2 токени). Приблизно **1 токен ≈ 4 символи** англійською / ~¾ слова.

### L2 — Middle
- **Токенізатор** розбиває текст на токени (BPE — Byte Pair Encoding — популярний метод).
- **Чому важливо:** ліміти (context window), **вартість** (плата за токени — вхід + вихід), швидкість — усе рахується в токенах.
- Різні мови/код токенізуються **по-різному**: не-англійська й код часто дають **більше** токенів на той самий зміст.

### L3 — Senior
**Практичні наслідки для інженера:**
- **Не рахуй токени «на око»** й **не використовуй чужий токенізатор** (напр. `tiktoken` від OpenAI недорахує для Claude) — у кожної моделі свій. Використовуй **офіційний count-tokens** ендпоінт/бібліотеку моделі.
- **Вартість = (input + output) токени.** Довгий контекст = дорого й повільно → звідси **prompt caching** (кешування стабільного префікса) і **RAG** (тягнути лише релевантне, п.11).
- Різні версії моделі можуть мати **різний токенізатор** → перерахуй бюджети при апгрейді.

> ⚠️ Співвідношення «1 токен ≈ 4 символи» — груба оцінка; реальність залежить від мови/тексту. Точно — лише через count-tokens конкретної моделі.

## 3. Контекстне вікно (context window)

### L1 — Junior
**Аналогія — робочий стіл обмеженого розміру.** **Context window** — скільки токенів модель може «тримати перед очима» одночасно (вхід + вихід). Не влізло — «падає зі столу» (модель цього не бачить).

### L2 — Middle
- Вимірюється в токенах (напр. 200K, 1M).
- Включає **все**: system prompt + історію діалогу + документи + відповідь.
- Перевищив → помилка або втрата початку (у чатах — обрізають старі повідомлення).

### L3 — Senior
**«Більше вікно» ≠ «клади все підряд»:**
- **Вартість/латентність** ростуть із розміром контексту (кожен токен обробляється).
- **«Lost in the middle»** — моделі гірше «пам'ятають» інформацію в **середині** довгого контексту (краще на початку/кінці). Тому важливе — на краях.
- **RAG (п.11)** часто кращий за «запхати весь корпус у вікно»: тягнеш **лише релевантне** → дешевше, точніше, масштабованіше.

**Техніки для довгих діалогів:** обрізання, **сумаризація/компакція** історії, RAG по історії. Трейдоф: велике вікно — простота (усе поміщається) ↔ вартість + деградація уваги. Senior балансує: контекст — дорогий ресурс, не смітник.

## 4. Temperature та параметри семплінгу

### L1 — Junior
**Аналогія — регулятор «творчості».** **Temperature** керує випадковістю вибору наступного токена:
- **Низька (0–0.3)** — детерміновано, передбачувано (факти, код, класифікація).
- **Висока (0.7–1+)** — різноманітно, креативно (тексти, ідеї).

### L2 — Middle
| Параметр | Що робить |
|---|---|
| **temperature** | випадковість вибору (0 = майже завжди найімовірніший токен) |
| **top_p** (nucleus) | вибір з найімовірніших токенів, що сумарно дають p ймовірності |
| **top_k** | вибір лише з k найімовірніших |
| **max_tokens** | максимум токенів у відповіді |
| **stop sequences** | де зупинити генерацію |

**Best practice:** керуй **або** temperature, **або** top_p (не обома). Для факт-задач/JSON — низька temperature; для креативу — вища.

### L3 — Senior
**temperature ≠ гарантія детермінізму.** Навіть `temperature=0` **не гарантує** ідентичний вихід між запусками (числові нюанси, батчинг на сервері). Для «однаковості» покладайся на **промпт і структуру**, не на temperature.

**Нюанс сучасних моделей:** деякі reasoning-моделі (з «мисленням») **не приймають** temperature/top_p взагалі — керування йде через рівень «зусиль» (effort) і промпт. Тому: перевіряй, які параметри підтримує **конкретна** модель — API може відхилити застарілі. Для структурованого виводу краще **structured outputs / JSON schema**, ніж «temperature=0 + сподівання».

> ⚠️ Підтримка семплінг-параметрів різниться між моделями/версіями (деякі reasoning-моделі їх відхиляють). Звіряй з docs конкретної моделі.

## 5. Що таке embeddings (вектори)?

### L1 — Junior
**Аналогія — GPS-координати для змісту.** **Embedding** — це перетворення тексту у **вектор чисел** так, що **близькі за змістом** тексти мають **близькі** вектори. «кіт» і «кошеня» — поруч; «кіт» і «трактор» — далеко.

```
"собака" → [0.21, -0.45, 0.88, ...]   (напр. 1536 чисел)
"пес"    → [0.19, -0.43, 0.85, ...]   ← близько (схожий зміст)
```

### L2 — Middle
- Роблять їх **embedding-моделі** (окремі від LLM): Voyage, OpenAI, Cohere, open-source (sentence-transformers).
- **Схожість** міряють **косинусною подібністю** (cosine similarity) — кут між векторами.
- Застосування: **семантичний пошук**, RAG (п.11–13), рекомендації, кластеризація, дедуплікація, класифікація.

```python
# косинусна подібність (спрощено)
import numpy as np
def cosine(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))
# ~1 = дуже схожі, ~0 = не пов'язані, ~-1 = протилежні
```

### L3 — Senior
**Чому це працює:** embedding-модель навчена так, що **семантика → геометрія**. Пошук «за змістом» стає пошуком **найближчих векторів** (nearest neighbors) — на відміну від keyword-пошуку (збіг слів). Тому «як скинути пароль» знайде «відновлення доступу», хоч слова різні.

**Трейдофи/нюанси:**
- **Розмірність** (напр. 384 vs 1536 vs 3072): більше — точніше, але дорожче по пам'яті/швидкості. Деякі моделі підтримують **Matryoshka** (обрізати вектор без перетренування).
- **Domain match:** загальна embedding-модель гірша на вузькому домені (юридика, медицина) — інколи потрібні спеціалізовані/донавчені.
- **Embeddings ≠ LLM:** це окрема модель; Anthropic, наприклад, **не має власної** embeddings-моделі й рекомендує сторонні (Voyage AI). Вибір embedding-моделі критичний для якості RAG (п.16).

> ⚠️ Конкретні розмірності/можливості (Matryoshka тощо) залежать від embedding-моделі — звіряй з її docs.

## 6. Prompt engineering — техніки

### L1 — Junior
**Аналогія — чітке ТЗ виконавцю.** Як сформулюєш — таке й отримаєш. **Prompt engineering** — мистецтво писати інструкції для LLM так, щоб отримати потрібний результат.

### L2 — Middle
**Ключові техніки:**
| Техніка | Суть |
|---|---|
| **System prompt** | задати роль/контекст/правила («ти — досвідчений редактор») |
| **Few-shot** | дати приклади вхід→вихід (модель наслідує формат) |
| **Chain-of-Thought (CoT)** | «думай крок за кроком» → краще на логіці/математиці |
| **Структура** | чіткі секції, теги, формат виводу (JSON) |
| **Constraints** | явні обмеження («відповідай ≤ 3 речення») |

```
System: Ти — асистент підтримки. Відповідай коротко, українською.
User: [приклад питання] → [приклад відповіді]   ← few-shot
User: [реальне питання]
```

### L3 — Senior
**Порядок і структура промпту важать:**
- **Стабільне — на початку** (system, інструкції, приклади), **змінне — в кінці** (питання) → дружньо до **prompt caching** (незмінний префікс кешується → дешевше/швидше).
- **CoT** покращує reasoning ціною токенів/латентності; сучасні reasoning-моделі роблять це вбудовано («thinking»).
- **Structured outputs** (JSON schema) надійніше за «попроси JSON у промпті».
- **Prompt injection** — вразливість: недовірений текст (з документа/користувача) може «перехопити» інструкції. Захист: розділяти інструкції й дані, не давати недовіреному тексту авторитету system-рівня, санітизація.

**Трейдоф:** промпт-інжиніринг — дешево й швидко (не треба тренувати), але крихко (модель/версія змінилась — промпт «поплив»). Для стабільності — тести промптів (eval-и), версіонування, структуровані виходи.

## 7. Галюцинації — що це і як боротись?

### L1 — Junior
**Аналогія — впевнений студент, що не вчив.** **Галюцинація** — коли LLM **впевнено вигадує** неправду: неіснуючі факти, фейкові посилання, вигадані API. Звучить переконливо, але хибно.

### L2 — Middle
**Чому виникають:** модель оптимізована на **правдоподібність**, не на істинність (п.1). Вона «заповнює пропуск» найімовірнішим, навіть якщо не знає.

**Як зменшити:**
- **RAG (п.11)** — дати справжні документи як контекст, просити відповідати **лише з них**.
- **Grounding + цитати** — вимагати посилання на джерело.
- **«Не знаю» дозволено** — явно дозволити зізнатись у незнанні.
- **Нижча temperature** для факт-задач.
- **Tool use (п.9)** — рахувати/шукати реальними інструментами, не «в голові».
- **Верифікація** — окрема перевірка виводу (self-check, друга модель).

### L3 — Senior
**Галюцинації не «баг», а наслідок природи** генеративної моделі — повністю не усунути, лише **керувати ризиком**. Ключ — **не покладатись на «пам'ять» моделі** для фактів/приватних/свіжих даних:
- **RAG** переносить джерело істини **зовні** (модель переказує знайдене, не вигадує).
- **Constrained generation / structured outputs** обмежують форму → менше простору для вигадок.
- **Human-in-the-loop / verification** для критичних доменів.

**Трейдоф:** grounding (RAG, цитати, tools) різко знижує галюцинації ↔ додає складність (пайплайн retrieval, латентність). Senior проєктує систему так, щоб LLM **не був єдиним джерелом правди** там, де ціна помилки висока.

## 8. Fine-tuning vs Prompting vs RAG

### L1 — Junior
Три способи «навчити» модель під задачу:
- **Prompting** — просто добре попросити (інструкції/приклади в промпті). Найпростіше.
- **RAG** — дати знання **в контекст** під час запиту (зовнішня пам'ять).
- **Fine-tuning** — **донавчити** модель на своїх даних (змінити ваги).

### L2 — Middle
| | Prompting | RAG | Fine-tuning |
|---|---|---|---|
| Змінює модель? | ні | ні | **так (ваги)** |
| Свіжі/приватні дані | ні | **так** (легко оновити) | застигають у вагах |
| Вартість запуску | низька | середня (retrieval) | навчання дороге |
| Найкраще для | поведінка, формат | **знання, факти** | стиль, формат, вузький домен |

### L3 — Senior
**Правило вибору (за задачею):**
- Потрібні **знання/факти** (документація, приватні дані, свіже) → **RAG** (оновлюєш базу, не модель).
- Потрібна **поведінка/стиль/формат** → **prompting** спершу, **fine-tuning** якщо промпт не витягує стабільно.
- **Не тюнь для знань** — це антипатерн: факти застигають у вагах (не оновиш без перетренування), дорого, ризик забування (catastrophic forgetting).

**Порядок ескалації (senior):** prompt engineering → few-shot → RAG → (лише якщо треба) fine-tuning. Часто **RAG + гарний промпт** закриває 90% потреб дешевше й гнучкіше за тюнінг. Комбінують: fine-tune для формату/тону + RAG для знань.

## 9. Tool use / Function calling

### L1 — Junior
**Аналогія — дати помічнику телефон і калькулятор.** LLM сам не вміє точно рахувати, ходити в БД чи в інтернет. **Tool use (function calling)** дає моделі «інструменти»: вона каже «виклич функцію `get_weather('Київ')`», **ти виконуєш**, віддаєш результат — модель продовжує з реальними даними.

### L2 — Middle
**Як працює (цикл):**
```
1. Ти описуєш інструменти: name, description, input_schema (JSON schema).
2. Модель вирішує викликати → повертає tool_use (яку функцію, з якими аргументами).
3. ТИ виконуєш функцію у своєму коді.
4. Віддаєш результат назад (tool_result).
5. Модель формує фінальну відповідь (або кличе ще інструмент).
```
```json
{
  "name": "get_weather",
  "description": "Get current weather for a city",
  "input_schema": {
    "type": "object",
    "properties": {"city": {"type": "string"}},
    "required": ["city"]
  }
}
```
**Ключове:** **модель не виконує** інструмент — вона лише **просить**; виконання (і безпека) — на тобі.

### L3 — Senior
Function calling — міст між **ймовірнісним** LLM і **детермінованим** світом (API, БД, обчислення). Це основа **агентів**: модель у циклі планує → кличе інструменти → спостерігає → діє далі.

**Best practices/трейдофи:**
- **Описи інструментів критичні** — модель обирає інструмент за description (пиши, **коли** його викликати).
- **Безпека:** інструменти з побічними ефектами (запис у БД, надсилання, платежі) — гейти/підтвердження, валідація аргументів (модель може передати неправильне). Ніколи не виконуй сліпо.
- **Паралельні виклики** — модель може попросити кілька інструментів одразу.
- **Ідемпотентність/помилки** — повертай помилку інструмента моделі (вона адаптується), не «ковтай».

**Зв'язок з MCP (п.17):** function calling — це **можливість моделі** (вивести виклик); **MCP** — це **стандарт**, як інструменти/дані **під'єднуються** до застосунку. Інструменти з MCP-сервера подаються моделі саме через function calling.

## 10. Streaming

### L1 — Junior
**Аналогія — друкує на очах, а не мовчить 30 секунд.** **Streaming** — модель віддає відповідь **токен за токеном** у реальному часі (як у ChatGPT «друкується»), а не чекає повної генерації.

### L2 — Middle
- Технічно — **SSE (Server-Sent Events)**: сервер шле потік подій (початок, дельти тексту, кінець).
- **Навіщо:** краще UX (миттєвий фідбек), і **уникнення таймаутів** на довгих відповідях (не чекаєш хвилину одним запитом).
- Обробляєш події: `content_block_delta` (шматок тексту) → додаєш у UI.

### L3 — Senior
**Коли обов'язково:** довгий вихід, великий `max_tokens`, reasoning-моделі (можуть «думати» хвилини) → без стримінгу ризик HTTP-таймауту. Стрімінг тримає з'єднання живим і показує прогрес.

**Трейдофи/нюанси:**
- Стрімінг ускладнює обробку (події замість одного об'єкта), парсинг **структурованого** виводу на льоту, підрахунок токенів наприкінці.
- Для tool use у стрімі — аргументи інструмента приходять дельтами (`input_json_delta`), треба акумулювати.
- **Backpressure/скасування** — вміти обірвати генерацію (користувач передумав) → економія токенів.

**Зв'язок з Node/Python:** стрімінг LLM лягає на async I/O (asyncio / Node streams) — це той самий патерн потокової роздачі, що в блоках про стрими.

# ЧАСТИНА B — RAG

## 11. Що таке RAG і навіщо?

### L1 — Junior
**Аналогія — іспит із відкритою книгою.** Замість покладатись на «пам'ять» LLM (яка застаріла й вигадує), **RAG (Retrieval-Augmented Generation)** спершу **знаходить** релевантні документи у твоїй базі, кладе їх у контекст, і модель відповідає **на основі знайденого**.

### L2 — Middle
**Навіщо (вирішує 3 проблеми LLM):**
1. **Knowledge cutoff** — модель не знає свіжого/після навчання.
2. **Приватні дані** — модель не знає твоїх внутрішніх документів.
3. **Галюцинації** (п.7) — grounding у реальні джерела.

**Плюс:** оновлюєш **базу**, не модель (дешево, миттєво); даєш **цитати** (звідки відповідь).

### L3 — Senior
RAG переносить **джерело істини** з ваг моделі у **зовнішнє сховище** → модель стає «reasoning-движком над наданим контекстом», а не «базою знань». Це фундаментальний архітектурний вибір: **retrieval + generation** замість «все в моделі».

**Трейдоф RAG vs довгий контекст vs fine-tuning:**
- **RAG** — точковий, дешевий, оновлюваний, з цитатами ↔ складність пайплайну (retrieval якість вирішує все, п.16).
- **Запхати все у вікно** — просто, але дорого/повільно й «lost in the middle» (п.3).
- **Fine-tuning** — не для знань (п.8).

Senior: RAG — дефолт для «питання-відповідь по документах/знаннях». Головний ризик — **якість retrieval**: погано знайшов → погана відповідь (garbage in, garbage out).

## 12. Як працює RAG-пайплайн?

### L1 — Junior
Дві фази:
```
ІНДЕКСАЦІЯ (заздалегідь):
  документи → розбити на шматки (chunks) → embeddings → зберегти у vector DB

ЗАПИТ (у рантаймі):
  питання → embedding → знайти схожі chunks → додати в промпт → LLM відповідає
```

### L2 — Middle
```
1. LOAD      — завантажити документи (PDF, HTML, БД...)
2. CHUNK     — розбити на шматки (п.14)
3. EMBED     — кожен chunk → вектор (п.5)
4. STORE     — у vector DB (п.13)
──────────────── (індекс готовий) ────────────────
5. QUERY     — питання користувача
6. EMBED Q   — питання → вектор
7. SEARCH    — знайти top-k найсхожіших chunks (similarity)
8. AUGMENT   — вставити знайдене в промпт як контекст
9. GENERATE  — LLM відповідає на основі контексту (+ цитати)
```

### L3 — Senior
Кожен крок — точка якості й оптимізації:
- **Chunking (п.14)** — надто дрібно = втрата контексту; надто крупно = шум/дилюція.
- **Embedding-модель (п.5)** — визначає якість пошуку; domain-match критичний.
- **Retrieval (п.16)** — top-k, поріг подібності, **reranking**, **hybrid search**, метадані-фільтри.
- **Augment** — як вставити (порядок, кількість, ліміт контексту), інструкція «відповідай лише з наданого».
- **Generate** — grounding, цитати, «не знаю якщо немає в контексті».

**Оцінка (senior обов'язково):** RAG треба **вимірювати** — retrieval metrics (recall@k, precision), generation metrics (faithfulness — чи відповідь ґрунтована; answer relevance). Без eval-ів RAG «наче працює», але тихо галюцинує на крайових запитах.

## 13. Vector database та векторний пошук

### L1 — Junior
**Аналогія — бібліотека, впорядкована за змістом, не за алфавітом.** **Vector DB** зберігає embeddings і **швидко знаходить найближчі** вектори до запиту (семантично схожі документи).

### L2 — Middle
- **Приклади:** Pinecone, Weaviate, Qdrant, Milvus, Chroma, **pgvector** (розширення PostgreSQL).
- **Операція:** `similarity_search(query_vector, k=5)` → топ-5 найсхожіших chunks.
- Зберігають: вектор + оригінальний текст + **метадані** (джерело, дата, автор — для фільтрації).
```python
results = vector_db.search(query_embedding, top_k=5,
                           filter={"source": "docs", "lang": "uk"})
```

### L3 — Senior
**Під капотом — ANN (Approximate Nearest Neighbor):** точний пошук найближчих серед мільйонів векторів — `O(n)` (повільно). Тому використовують **наближені** індекси:
- **HNSW** (граф) — швидкий, точний, популярний.
- **IVF** (кластери), **PQ** (стиснення) — компроміси швидкість/пам'ять/точність.

**Трейдоф:** ANN дає sub-linear пошук ціною **невеликої втрати точності** (recall < 100%). Параметри індексу (ef, nprobe) балансують швидкість↔точність.

**Вибір рішення:**
- **pgvector** — якщо вже є Postgres (одна БД, транзакції, метадані-JOIN) — часто достатньо.
- **Спеціалізовані** (Pinecone/Qdrant) — для величезних масштабів, гібридного пошуку, managed.

**Метадані-фільтрація** — критична: «шукай схоже, але лише в документах цього юзера/за останній місяць» → поєднання векторного + структурного фільтра.

## 14. Стратегії чанкінгу (chunking)

### L1 — Junior
**Аналогія — різати книгу на шматки для картотеки.** Документи великі — їх **ріжуть на chunks** (шматки), бо embedding робиться на шматок, і в контекст кладуть шматки, не цілі книги. Питання — **як різати**.

### L2 — Middle
**Стратегії:**
| Стратегія | Суть |
|---|---|
| **Fixed-size** | по N токенів/символів (просто) |
| **Overlap** | шматки з перекриттям (щоб не розірвати думку на межі) |
| **Recursive** | за структурою (абзаци → речення), зберігає межі |
| **Semantic** | різати там, де змінюється тема |
| **Document-based** | за секціями/заголовками (Markdown, розділи) |

**Типово:** ~200–800 токенів на chunk + **overlap** ~10–20%.

**Gotcha:** **надто дрібно** → chunk без контексту («він сказав» — хто?); **надто крупно** → шум дилютує релевантність + дорогий контекст.

### L3 — Senior
**Чанкінг — найнедооцінніший важіль якості RAG.** Погане розбиття губить відповіді, хоч embedding і БД ідеальні:
- **Розрив контексту** — визначення й приклад у різних chunks → retrieval знайде один без іншого.
- **Гранулярність ↔ точність:** дрібні chunks — точний матч, але фрагментарні; крупні — повний контекст, але шумні й «розмиті» за embedding.

**Просунуті техніки:**
- **Overlap** — компроміс проти розривів.
- **Contextual retrieval** — додати до кожного chunk короткий контекст документа (щоб «він» став зрозумілим) перед embedding — суттєво піднімає recall.
- **Parent-child / small-to-big** — шукати по дрібних, але подавати в контекст батьківський (більший) chunk.
- **Metadata** — заголовок секції, джерело в кожен chunk.

Senior: chunking **під тип контенту** (код ≠ проза ≠ таблиці) і **міряти** вплив на retrieval. Немає універсального розміру — тестуй на своїх даних.

## 15. Embeddings і similarity (косинус) у RAG

### L1 — Junior
*(Розширення п.5 у контексті RAG.)* Пошук у RAG = **знайти chunks, чий вектор найближчий до вектора питання**. «Близькість» міряють **косинусною подібністю** (кут між векторами): ~1 = дуже схоже.

### L2 — Middle
```python
# Індексація: chunk → вектор (зберігаємо)
# Запит: питання → вектор → знайти top-k за cosine similarity
import numpy as np
def cosine(a, b):
    return np.dot(a, b) / (np.linalg.norm(a) * np.linalg.norm(b))

# vector DB робить це ефективно через ANN (п.13), не перебором
```
**Метрики подібності:** cosine (найпоширеніша, ігнорує довжину), dot product, Euclidean. Для нормалізованих векторів cosine ≈ dot.

### L3 — Senior
**Ключове:** якість RAG-пошуку = якість **embedding-моделі** × **метрики** × **чанкінгу**. Однакова метрика (cosine), різні embedding-моделі → різний recall.

**Нюанси:**
- **Симетрія query/document:** деякі embedding-моделі мають окремі режими для «запиту» й «документа» (asymmetric) — використовуй правильний, інакше гірший матч.
- **Нормалізація** векторів — щоб cosine коректний; більшість моделей віддають нормалізовані.
- **Обмеження чистого semantic search:** погано ловить **точні терміни/коди/абревіатури/рідкісні слова** (embedding «розмиває») → тому **hybrid search** (п.16): semantic + keyword (BM25).

Senior не покладається лише на cosine-топ-k — додає reranking і hybrid (п.16) там, де точність критична.

## 16. Проблеми RAG і як покращити

### L1 — Junior
RAG «просто працює» на демо, але в проді має проблеми: **знаходить не те**, **пропускає релевантне**, **плутає точні терміни**. Головне — **якість retrieval** (що знайшли, те й відповіли).

### L2 — Middle
**Типові проблеми й рішення:**
| Проблема | Рішення |
|---|---|
| Пропускає релевантне (low recall) | більший top-k, кращий embedding, кращий чанкінг |
| Не ловить точні терміни/коди | **hybrid search** (semantic + BM25 keyword) |
| Топ-k не найрелевантніші | **reranking** (cross-encoder переранжує) |
| Питання неоднозначне | **query rewriting / expansion** |
| Шум у контексті | пороги подібності, менше але точніше |
| Мало контексту в chunk | contextual retrieval, parent-child (п.14) |

### L3 — Senior
**Просунутий RAG (за важливістю):**
- **Reranking** — retrieval дає top-50 (швидко, груба cosine), потім **cross-encoder** переранжує до top-5 (точно, дорого). Найбільший приріст точності на одиницю зусиль.
- **Hybrid search** — поєднати **dense** (embeddings, семантика) + **sparse** (BM25/keyword, точні слова) → ловить і зміст, і терміни. RRF (reciprocal rank fusion) для злиття.
- **Query transformation** — rewrite (уточнити), **HyDE** (згенерувати гіпотетичну відповідь → шукати за нею), decomposition (розбити складне питання).
- **Metadata filtering** — звузити пошук (юзер, дата, тип).
- **Contextual retrieval / reranking** разом дають найбільший приріст.

**Головне — evaluation:** без метрик (retrieval recall/precision, faithfulness, answer relevance — інструменти типу RAGAS) неможливо покращувати системно. Senior будує **eval-набір** і оптимізує за ним, а не «на око». RAG — це **retrieval-задача** передусім; generation вторинна.

# ЧАСТИНА C — MCP (Model Context Protocol)

## 17. Що таке MCP?

### L1 — Junior
**Аналогія — USB-C для AI-застосунків.** Раніше кожен інструмент/джерело даних під'єднували до кожного AI-застосунку **по-своєму** (M×N інтеграцій — хаос). **MCP (Model Context Protocol)** — **відкритий стандарт** (від Anthropic), єдиний «роз'єм»: будь-який MCP-клієнт може під'єднатись до будь-якого MCP-сервера.

### L2 — Middle
- **MCP** — відкритий протокол для з'єднання AI-застосунків із **зовнішніми інструментами, даними й контекстом** стандартним способом.
- Замість «M застосунків × N інтеграцій» → «M клієнтів + N серверів», що говорять **однією мовою**.
- Приклади MCP-серверів: GitHub, Google Drive, Slack, бази даних, файлова система — кожен експонує свої можливості через MCP.

### L3 — Senior
MCP розв'язує **проблему інтеграційного вибуху** (M×N): без стандарту кожна пара «застосунок ↔ джерело» — окрема кастомна інтеграція. MCP робить це **N+M** через спільний протокол — як HTTP стандартизував веб, USB — периферію.

**Що дає архітектурно:**
- **Розв'язка (decoupling):** автор AI-застосунку не знає деталей GitHub API; автор GitHub MCP-сервера не знає про застосунок — вони говорять MCP.
- **Переиспользування:** один MCP-сервер працює з Claude Desktop, IDE, кастомним агентом — будь-яким MCP-клієнтом.
- **Екосистема:** спільнота пише сервери, усі клієнти їх отримують.

**Трейдоф:** стандартизація (переиспользування, екосистема) ↔ ще один шар абстракції/протоколу; для однієї простої інтеграції прямий виклик API простіший. MCP окупається при **багатьох** інтеграціях/клієнтах.

> ⚠️ MCP активно розвивається (специфікація, транспорти). Звіряй деталі з офіційною специфікацією на modelcontextprotocol.io.

## 18. Архітектура MCP (host, client, server, transport)

### L1 — Junior
Три ролі:
```
[MCP HOST] ── містить ── [MCP CLIENT] ──протокол──> [MCP SERVER] ── дані/інструменти
 AI-застосунок            конектор                    експонує можливості
 (Claude Desktop, IDE)    (1 на сервер)                (GitHub, БД, файли)
```

### L2 — Middle
| Компонент | Роль |
|---|---|
| **Host** | AI-застосунок, що використовує LLM (Claude Desktop, IDE, твій агент) |
| **Client** | живе всередині host'а; тримає **1:1** з'єднання з одним сервером |
| **Server** | експонує можливості (tools, resources, prompts, п.19) — обгортка над GitHub/БД/файлами |
| **Transport** | як клієнт і сервер спілкуються |

**Транспорти:**
- **stdio** — сервер як **локальний процес** (запуск на твоїй машині; файли, локальні інструменти).
- **Streamable HTTP / SSE** — **віддалений** сервер по мережі.

Спілкування — на **JSON-RPC** (стандартні повідомлення request/response).

### L3 — Senior
**Розподіл ролей (ключове):**
- **Host** керує LLM і оркеструє; **client** — тонкий конектор (один на сервер, ізоляція); **server** — незалежний, нічого не знає про конкретний host.
- **1:1 client↔server** дає ізоляцію: збій/компрометація одного сервера не тече на інші.

**Безпека (senior-акцент):** MCP-сервер отримує доступ до даних/дій (БД, файли, API) → **питання довіри й дозволів**:
- Автентифікація до серверів (OAuth-токени тощо) — зберігати безпечно (не в промпті!).
- **Дозволи/підтвердження** на небезпечні дії (як tool use, п.9) — сервер може писати/видаляти.
- Недовірений MCP-сервер = вектор атаки (prompt injection через дані, ексфільтрація).

**Трейдоф stdio vs HTTP:** stdio — просто, локально, без мережі (dev, локальні інструменти) ↔ лише на одній машині; HTTP — віддалено, масштабовано, багатокористувацько ↔ auth, мережа, безпека. Вибір за сценарієм.

> ⚠️ Деталі транспортів і спеки MCP еволюціонують — звіряй з modelcontextprotocol.io.

## 19. Примітиви MCP: tools, resources, prompts

### L1 — Junior
MCP-сервер може експонувати **три види можливостей**:
- **Tools** — **дії/функції**, які модель може викликати (пошук, запис у БД, надіслати).
- **Resources** — **дані/контекст** для читання (файли, записи БД, документи).
- **Prompts** — **готові шаблони** промптів/робочих процесів.

### L2 — Middle
| Примітив | Що це | Хто «керує» | Аналог |
|---|---|---|---|
| **Tools** | функції з побічними діями | **модель** вирішує викликати (model-controlled) | function calling (п.9) |
| **Resources** | контекст лише для читання | **застосунок/користувач** обирає (app-controlled) | файли/дані в контекст |
| **Prompts** | параметризовані шаблони | **користувач** обирає (user-controlled) | слеш-команди/сніпети |

```
Tool:     "run_query(sql)" — виконати запит
Resource: "file:///docs/readme.md" — прочитати документ у контекст
Prompt:   "code_review" — готовий шаблон рев'ю з параметрами
```

### L3 — Senior
**Три примітиви = три «осі керування»** (хто ініціює):
- **Tools — model-controlled:** модель сама вирішує викликати (через function calling, п.9) → потужно, але потребує гейтів безпеки (побічні ефекти).
- **Resources — application-controlled:** застосунок/юзер вибирає, що подати в контекст (не модель) → безпечніше (read-only, керований доступ).
- **Prompts — user-controlled:** явно викликані користувачем (напр. слеш-команда) → передбачувано.

Цей поділ — про **хто має владу ініціювати**: розділення знижує ризик (модель не читає що завгодно сама; небезпечні дії — через tools із дозволами). У API інструменти з MCP-сервера подаються моделі як звичайні tools (function calling), а resources/prompts — механізми подачі контексту.

**Трейдоф:** багатство примітивів (гнучкість) ↔ складність (треба розуміти, що є що, і правильно налаштувати дозволи). Senior проєктує MCP-інтеграцію за принципом найменших привілеїв: resources (read-only) за замовчуванням, tools (дії) — свідомо, з гейтами.

> ⚠️ Точна семантика примітивів і control-моделей — звіряй зі специфікацією MCP (вона розвивається).

## 20. MCP vs function calling — у чому різниця й навіщо MCP?

### L1 — Junior
Їх **плутають**, але це **різні рівні**:
- **Function calling (п.9)** — **можливість моделі** вивести «виклич функцію X» (модельний рівень).
- **MCP** — **стандарт**, як інструменти/дані **під'єднуються** до застосунку (інтеграційний рівень).

MCP-сервер **експонує** інструменти → застосунок **подає їх моделі** через function calling. Вони **доповнюють**, не конкурують.

### L2 — Middle
```
Function calling: модель ↔ застосунок
   "модель, ось інструменти; скажи, який викликати"  (одна модель, твій код)

MCP: застосунок ↔ зовнішні джерела
   "будь-який AI-застосунок ↔ будь-який сервер інструментів/даних"  (стандарт)
```
**Без MCP:** для кожного джерела (GitHub, Slack, БД) пишеш **власну** інтеграцію під **свій** застосунок → M×N.
**З MCP:** пишеш/береш **MCP-сервер раз** → працює з **усіма** MCP-клієнтами.

### L3 — Senior
**Це два шари, що складаються:**
1. **Function calling** — механізм: як модель **виражає** намір викликати інструмент (і як ти вертаєш результат). Прив'язаний до конкретного застосунку/коду.
2. **MCP** — протокол: як інструменти/дані **стандартно постачаються** застосункам. Прив'язаний до екосистеми.

**Ланцюг:** MCP-сервер оголошує tools → MCP-клієнт у host'і отримує їх → host подає моделі як function-calling tools → модель кличе → клієнт маршрутизує виклик у сервер → результат назад.

**Навіщо MCP поверх «просто function calling»:**
- **Переиспользування** — один сервер, багато клієнтів (не переписуєш інтеграцію під кожен застосунок).
- **Розв'язка** — автор інтеграції й автор застосунку незалежні.
- **Екосистема** — спільні сервери, стандартна безпека/дозволи.

**Трейдоф:** для **однієї** моделі + **кількох** власних інструментів — чистого function calling досить (MCP — зайвий шар). MCP окупається при **багатьох джерелах × багатьох клієнтах** або коли хочеш готові сторонні сервери. Senior обирає: прямий function calling для простого/замкненого; MCP — для інтеграційного масштабу й переиспользування.

> ⚠️ MCP — молодий стандарт, що активно розвивається (нові транспорти, можливості, безпека). Для актуальних деталей — офіційна специфікація modelcontextprotocol.io.

## ✅ Чеклист (інтерв'ю)

**LLM:** передбачення токенів (не «пошук»), context window + вартість, temperature/семплінг, embeddings + cosine, prompt engineering, галюцинації + grounding, fine-tuning vs prompting vs RAG, tool use, streaming.
**RAG:** навіщо (знання/свіже/приватне/анти-галюцинації), пайплайн (chunk→embed→store→retrieve→augment→generate), vector DB + ANN, чанкінг (недооцінений важіль), hybrid search + reranking, **evaluation**.
**MCP:** відкритий стандарт Anthropic («USB-C for AI»), host/client/server/transport, примітиви (tools/resources/prompts + хто керує), MCP ≠ function calling (стандарт інтеграції vs можливість моделі — доповнюють).

## 🔑 Топ-питання
1. **LLM «знає» факти?** → ні, генерує ймовірне; для фактів — RAG/tools (інакше галюцинації).
2. **RAG vs fine-tuning?** → RAG для знань (оновлюване), fine-tuning для стилю/формату; не тюнь для знань.
3. **Що вирішує якість RAG?** → **retrieval** (чанкінг, embedding, reranking, hybrid) + evaluation.
4. **MCP vs function calling?** → стандарт інтеграції (MCP) vs можливість моделі викликати (FC); доповнюють.
5. **Три примітиви MCP?** → tools (model-controlled дії), resources (read-only контекст), prompts (шаблони).
6. **Навіщо MCP?** → прибирає M×N інтеграцій; один сервер — усі клієнти.

> ⚠️ LLM/RAG/MCP-екосистема швидко рухається. MCP-специфіку звіряй з modelcontextprotocol.io; embedding/vector/reranking-інструменти й моделі змінюються — перевіряй актуальні docs. Для роботи саме з Claude API (tool use, MCP-конектор) — офіційна документація Anthropic.
